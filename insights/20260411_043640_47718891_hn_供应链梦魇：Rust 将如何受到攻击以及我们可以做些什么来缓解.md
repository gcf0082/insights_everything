## 摘要

本文讨论了 Rust 生态系统中供应链安全的严重问题。根据 Adam Harvey 的分析，约 17% 最受欢迎的 Rust crates 包含与代码仓库不匹配的代码。社区提出了多种缓解措施，包括依赖 Rust Foundation 管理的"祝福"库、供应商依赖、硬件安全令牌、可重现构建等。

## 核心问题

### 1. 代码不匹配问题
- 约 17% 最受欢迎的 crates.io 包包含与源代码仓库不匹配的代码
- 发布不经过 GitHub，直接从本地机器发布，crates 可以包含生成的代码
- 某些差异只是时间戳、哈希等微小差异

### 2. 供应链攻击向量
- 维护者 quit 或成为恶意行为者
- 账户被 compromise
- 恶意 PR 引入
- DNS 接管（.rs 域名为塞尔维亚）
- rustup 使用 curl | sh 方式安装，无签名

### 3. 构建链问题
- 官方 Rust 发布不是完全可重现的
- 不是完全源代码引导（需要旧版本 Rust 编译新版本）
- 存在"信任信任"攻击风险

## 缓解措施

### 1. 官方层面
- Rust Foundation 接管"祝福"库并提供资金和审计
- 由.foundation 协调 CVE 报告标准化
- Chainguard 即将推出 Rust 库支持

### 2. 技术方案
- 使用硬件安全令牌存储签名密钥
- TPM/安全启动保护开发者密钥
- 供应商依赖（vendor）
- 可重现构建
- 沙箱化依赖
- 极详细的效果系统在编译时强制依赖权限

### 3. 社区实践
- 延迟 6 个月更新依赖版本
- 公司内部 crates.io 镜像
- cargo audit 和 RustSec 咨询数据库

## 争议与讨论

### 关于"大标准库"
- 有观点认为 Go 的综合性标准库是最佳防御
- 反驳：Go 和 Python 程序仍有大量依赖
- 标准库也存在安全问题（"电池包含"策略导致的安全漏洞）
- 分离标准库与语言 spec 的做法更灵活

### 关于 AI 的作用
- Claude Mythos 等 LLM 可能改变风险格局
- 自动化代码审查、漏洞检测、性能分析
- 可以询问 LLM 代理工作流审查仓库与包的差异
