## 事件概述

2026年5月11日，Mini Shai-Hulud 蠕虫通过劫持 CI/CD 流水线，窃取 CI/CD 密钥，入侵了多个 @tanstack npm 包。这些包每周有数百万次下载，是最广泛使用的 React 路由库之一。攻击在广泛使用的 React 路由库中注入了 2.3 MB 的混淆窃取凭证有效载荷，并通过被劫持的 OIDC 令牌通过项目自己的 GitHub Actions 发布流水线发布恶意版本，使受污染的包携带有效的 SLSA  provenance 证明，这是首个记录在案的自我传播 npm 蠕虫。

## 攻击机制

### 攻击三阶段

#### 第一阶段：在 Fork 中准备有效载荷

攻击者在 TanStack/router 的 fork 中创建了一个恶意 commit (`79ac49ee`)，包含两个文件：

1. **`package.json`**：定义了虚假的 `@tanstack/setup` 包，其 `prepare` 生命周期钩子运行恶意 JS 文件
2. **`tanstack_runner.js`**：2,339,346 字节的混淆 JavaScript 文件（主要恶意有效载荷）

#### 第二阶段：向已发布的包注入有效载荷

对每个受感染包的修改：

- 在 `package.json` 中添加 `optionalDependencies` 字段指向攻击者的 fork commit
- 在包根目录放置 `router_init.js` 文件（2,341,681 字节）

#### 第三阶段：通过合法 CI/CD 流水线发布

受污染的包携带有效的 SLSA provenance 证明，尽管测试步骤失败，但恶意代码利用工作流的环境 OIDC 令牌直接发布到 npm，绕过了工作流自身的发布步骤。

### 有效载荷分析

恶意载荷使用多层混淆：

- **十六进制变量名**：所有变量和函数名使用 `_0x` 前缀模式
- **字符串表轮换**：大数组存储编码字符串，通过校验和匹配进行轮换
- **AES-256-GCM 加密**：敏感字符串（域名、路径、令牌模式）使用 AES-256-GCM 加密

### 凭证收集

恶意软件使用 10 个专用收集器类从多个来源收集密钥，包括：

- GitHub JWT 令牌：`ghs_\d+_[A-Za-z0-9_-]+\.[A-Za-z0-9_-]+\.[A-Za-z0-9_-]+`
- 旧版 GitHub 服务器令牌：`ghs_[A-Za-z0-9]{36,}`

### 自我传播机制

蠕虫使用窃取的令牌进行身份验证后，会识别令牌有写访问权的其他包，注入相同的恶意有效载荷，并发布新的受污染版本。这形成了一个自我传播循环。

## 安全建议

### 个人开发者

1. **降级到安全版本**：将每个受影响的包降级到最后一个干净版本
2. **删除 node_modules 并重新安装**：`rm -rf node_modules && npm install`
3. **轮换凭证**：如果使用受污染版本运行了 `npm install`，轮换该机器上可访问的任何 npm 令牌、GitHub PAT 和云 API 密钥
4. **检查 ~/.npmrc 外泄**：恶意软件读取 homedir 和文件系统密钥，检查 `~/.npmrc` 中的存储令牌并轮换

### CI/CD 环境（关键）

1. **立即轮换所有 CI 密钥**：GitHub 令牌、npm 令牌、`NX_CLOUD_ACCESS_TOKEN`、云提供商凭证以及工作流环境中可用的任何其他密钥
2. **审计 GitHub Actions 运行**：检查受污染版本发布后发生的运行，查找意外的 npm 发布事件
3. **检查下游传播**：如果在安装了受污染版本的 CI 运行期间发布了任何包，这些发布的版本也可能受到污染
4. **检查 npm 访问令牌**：运行 `npm token list` 并撤销任何不认识的令牌

### 检查命令

```bash
# npm 检查
grep "@tanstack/" package-lock.json | grep -v node_modules

# pnpm 检查
grep "@tanstack/" pnpm-lock.yaml

# yarn 检查
grep "@tanstack/" yarn.lock

# 搜索受污染版本
grep -E "(draftlab|draftauth|taskflow-corp|tolka)" package-lock.json pnpm-lock.yaml yarn.lock 2>/dev/null

# 搜索恶意文件
find node_modules -name "router_init.js" -type f 2>/dev/null

# 搜索恶意可选依赖
grep -r "@tanstack/setup" node_modules/*/package.json 2>/dev/null
```

## 关键洞察

1. **SLSA Provenance 局限性**：SLSA provenance 仅确认哪个流水线产生了工件，而不确认流水线是否按预期运行。受污染的构建步骤可以产生有效证明但实际上是恶意软件。

2. **OIDC 令牌滥用**：攻击者利用工作流的环境 OIDC 令牌直接发布到 npm，绕过了工作流自身的发布步骤。

3. **双有效载荷机制**：攻击使用双重有效载荷传递机制：安装时执行的恶意 `optionalDependencies` 条目，以及可被直接调用的嵌入式混淆 JavaScript 文件。

4. **多重混淆层**：恶意代码使用十六进制变量名、字符串表轮换和 AES-256-GCM 加密来抵抗静态分析。

5. **自我传播能力**：蠕虫不仅窃取凭证，还使用窃取的令牌来感染其他包并自主传播，这是供应链攻击的重大升级。
