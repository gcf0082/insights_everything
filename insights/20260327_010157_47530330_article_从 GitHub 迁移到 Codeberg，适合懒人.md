# 洞察报告：从 GitHub 迁移到 Codeberg

**洞察链接**：https://unterwaditzer.net/2025/codeberg.html  
**发布时间**：2025-09-06  
**作者**：Markus Unterwaditzer  
**报告生成日期**：2026-03-27

---

## 摘要

本文作者分享了将代码仓库从 GitHub 迁移到 Codeberg 的实践经验。主要涵盖仓库迁移、静态网站托管、CI/CD 配置等方面的解决方案，适合有意迁移但担心工作量的开发者参考。

---

## 主要内容

### 1. 仓库迁移

仓库迁移（包括 issues、pull requests、releases 及相关附件）是整个过程中最简单的部分。Codeberg 提供了从 GitHub 直接导入的功能，操作简单且效果出色。导入过程会保留 issue 编号、标签和作者信息，用户体验远超那些将问题跟踪器数据导入 GitHub 的繁琐方案。

### 2. 静态网站托管

对于 GitHub Pages 用户，可以使用 [codeberg.page](https://codeberg.page/) 进行替代。虽然官方警告该服务不提供可用性 SLO，但作者表示目前使用中从未遇到宕机。此外，2025年9月的更新还推荐了其他替代方案，如 [grebedoc.dev](https://grebedoc.dev) 和 [https://www.statichost.eu/](https://www.statichost.eu/)。

### 3. CI/CD 配置

CI 配置是迁移过程中最复杂的部分。GitHub 提供了免费的 macOS 运行器和公开仓库的无限容量，这些优势在 Codeberg 上将不再拥有。建议的解决方案包括：
- **跨平台编译**：针对编程语言进行跨编译
- **自托管运行器**：为 Forgejo Actions 配置自托管运行器

作者推荐使用 **Forgejo Actions** 而非 Woodpecker CI。虽然 Woodpecker 在 Codeberg 上更稳定，但 Forgejo Actions 对从 GitHub Actions 迁移的用户更加友好——UI 和 YAML 语法几乎完全相同，现有的 actions 生态系统基本可以直接使用。例如，原有的 GitHub Actions 配置 `uses: dtolnay/rust-toolchain` 在 Forgejo Actions 中只需改为 `uses: https://github.com/dtolnay/rust-toolchain` 即可。

### 4. macOS 运行器

如果必须使用 macOS 运行器，作者建议继续使用 GitHub Actions。具体方案是：将代码推送到 Codeberg，利用 GitHub Actions 进行 CI，然后通过 Forgejo Actions 轮询 GitHub API 将 CI 状态同步回 Codeberg。

### 5. 旧仓库处理

完成迁移后，只需更新原 GitHub 仓库的 README 并将其归档即可。也可以配置 Codeberg 向 GitHub 推送新提交，但这会导致用户仍可在 GitHub 上提交 PR 和评论。某些项目通过禁用 GitHub 仓库的 issues 功能来解决此问题，但这会带来破坏性影响，因为所有 issues 会返回 404。另一个方案是编写 GitHub Action 自动关闭所有 pull requests。

---

## 结论

作者认为 Codeberg 的迁移过程比预想的更为简便，特别是仓库导入功能已经相当成熟。主要挑战在于 CI/CD 配置，但通过使用 Forgejo Actions 可以大幅降低迁移成本。
