# 洞察报告：GitHub 之前的开源世界

## 基本信息

- **洞察链接**: https://lucumr.pocoo.org/2026/4/28/before-github/
- **作者**: Armin Ronacher
- **发布日期**: 2026年4月28日
- **标签**: github, open-source, thoughts
- **语言**: English

## 文章摘要

本文是 Armin Ronacher 对开源软件历史的回顾，以及对 GitHub 当前状态的深思。作者回顾了自己在 GitHub 出现之前的开源经历，探讨了 GitHub 对开源社区的深远影响，并表达了对 GitHub 现状的失望与担忧。

## 核心洞察

### 1. 作者的开源历程

在 GitHub 出现之前，作者的开源项目经历了几次迁移：
- 最初托管在 SourceForge
- 然后使用自己架设的 Trac 和 Subversion 仓库
- 之后迁移到 Bitbucket（当时仍是 Git 的可行替代方案）
- 最终才迁移到 GitHub

这种经历在当时很常见——开发者不仅要编写代码，还要扮演系统管理员的角色。

### 2. GitHub 对开源社区的意义

GitHub 不仅仅是一个代码托管平台，它：
- 塑造了作者的"开源身份"
- 促成了许多专业关系和友谊的开始
- 让项目发现变得容易
- 让贡献代码变得对普通人可理解
- 标准化了"开源协作在公开环境下进行"的理念
- 意外地成为了开源软件的"档案馆"

### 3. GitHub 之前的开源世界

在 GitHub 出现之前：
- 开源世界要小得多，人们依赖的项目数量有限
- 依赖关系不是轻率的决定，需要了解维护者、发布流程和社区
- 声誉至关重要，开发者通过长期贡献建立信任
- 项目通常自己托管基础设施（服务器、Trac、邮件列表等）
- 许多依赖会被直接 vendoring 到项目中

### 4. 微依赖问题的根源

GitHub 和 npm 的结合产生了"微依赖"现象：
- 发布代码几乎无摩擦
- 消费代码几乎无摩擦
- 包图的增长速度超过任何人的理解能力
- 间接产生了供应链安全问题

### 5. GitHub 的现状与衰退

作者对 GitHub 当前状态的批评：
- 产品不稳定，功能频繁变化
- Copilot AI 噪音过多
- 领导层不明确
- 平台不再是为主创社区设计

一些重要项目开始迁移到其他平台（如 Ghostty 迁移到自托管，Strudel 和 Tenacity 迁移到 Codeberg）。

### 6. 分散化的代价

回归"多个铸造厂、多个服务器"的模式：
- **好处**：恢复自主权，减少对单一公司决策的依赖
- **风险**：Issues、Reviews、设计讨论、安全公告等社交上下文可能消失
- **历史教训**：Google Code 和 Bitbucket 的关闭证明了许多平台无法长期维持

### 7. 需要公共档案馆

作者呼吁：
- 建立一个公开、有趣、资金充足的的开源软件档案
- 类似捐赠基金或公共资助的机构
- 任务不是赢得市场份额，而是确保最重要的成果不消失
- 保存源代码、发布产物、元数据和足够的项目背景

## 总结

作者既不想回到过去（充满失效的 tarball 链接和废弃的 Trac 实例），也不想假装过去二十年是正常或永久的。GitHub 书写了开源的精彩篇章，如果这一章正在结束，下一章应该从中学习，同时也要从之前的历史中汲取教训：更容易迁移项目、更容易镜像社交上下文、更容易保存发布版本，让一家公司的动摇不再成为所有人的文化危机。
