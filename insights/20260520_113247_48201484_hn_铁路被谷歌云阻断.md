## 事件概述

Railway（一家云托管平台）的 Google Cloud 账户被突然封禁，导致其服务全面下线。这是 GCP 对创业公司采取的激进做法的最新案例。

## 主要讨论观点

### 1. GCP 的封禁历史

- 这不是 GCP 第一次封禁创业公司账户
- 每年都会发生类似事件，而 AWS 和 Azure 很少这样做
- 有人提到 GCP 封禁了 hn.algolia.com 等服务

### 2. GCP 与 AWS/Azure 的对比

- **支持者观点**: GCP 是三大云服务商中体验最好的，但因这种做法而毁掉口碑
- **反驳观点**: GCP 实际上也有严重宕机事件，包括约一年前的全球性宕机和巴黎数据中心洪水导致的数据永久丢失
- **AWS 优势**: 即使 AWS 出现宕机，通常能在几分钟到几小时内恢复，且影响范围通常限于单个区域

### 3. 自动化问题

- 推测 GCP 封禁是自动化系统驱动的，缺乏人工介入
- 账户代表（Account Executive）轮换频繁，通常无法提供有效帮助
- 有人建议使用 GCP 经销商来获得更好的支持渠道

### 4. 业务影响

- Railway 客户受到影响，有人转向 Render 等竞争对手
- Railway 创始人表示：Railway 网络是 AWS、GCP 和 Metal 之间的环形mesh，将增加 AWS 和 Metal 的分片来提高可用性

### 5. 多云策略的重要性

- "不要把所有鸡蛋放在一个篮子里"
- 备份策略需要跨账户、跨云服务商
- 域名注册商也需要考虑隔离风险

### 6. 对 Railway 本身的批评

- Railway 的数据库删除 API 也会同时删除备份，这是设计缺陷
- 有人因此对 Railway 失去信任，选择迁移到其他平台

## 社区情绪

- 普遍对 GCP 的做法表示不满
- 对 Railway 的支持与批评并存
- 很多人开始反思云服务商锁定风险
