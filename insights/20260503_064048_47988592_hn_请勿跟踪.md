## 洞察摘要

本帖介绍了一个名为 "Do_not_track" 的项目，旨在帮助用户禁用各种 CLI 工具、SDK 和框架的遥测数据收集功能。该项目收集了多种工具的 "DO_NOT_TRACK" 环境变量，用户可以通过设置这些变量来阻止工具向开发者发送使用数据。

## 详细内容

### 主要功能

- 提供一个统一的页面，列出各种工具的禁用遥测环境变量
- 帮助用户集中管理隐私设置
- 涵盖多种常见的开发工具和框架

### 社区反应

社区对此展开了热烈讨论，主要观点包括：

1. **隐私权问题**：许多人认为隐私应该被视为一项基本权利，而非可以被金钱利益滥用的东西。

2. **现有方案**：有人指出 toptout.me 已经处理了许多类似问题，还有 do-not-track-cli 工具可以统一处理这些环境变量。

3. **DNT 失败历史**：评论者指出浏览器中的 "Do Not Track" (DNT) 功能几乎被所有广告商忽视，最终被浏览器厂商移除。

4. **更深层的解决方案**：
   - 使用 DNS 黑名单阻止遥测域名
   - 使用网络命名空间限制进程访问互联网
   - 运行自己的 DNS 服务器并屏蔽追踪域名

5. **对默认值的不满**：许多人批评软件默认启用遥测的做法，认为应该默认禁用，用户主动选择开启。

### 实际案例

评论中提到了多个具体工具的禁用方法，包括：

- Hugging Face Transformers: `HF_HUB_DISABLE_TELEMETRY=1`, `HF_HUB_OFFLINE=1`
- Python 转换库: `TRANSFORMERS_OFFLINE`, `HF_DATASETS_OFFLINE`
- Deno: `DENO_NO_UPDATE_CHECK=1`
- Semgrep: `SEMGREP_SEND_METRICS=off`
- Storybook: `STORYBOOK_DISABLE_TELEMETRY=1`
- Next.js: `NEXT_TELEMETRY_DISABLED=1`
- Serverless: `SLS_TELEMETRY_DISABLED=1`

## 结论

该帖子反映了开发者社区对软件遥测收集的广泛关注。虽然技术上可以通过环境变量禁用部分遥测，但许多人认为这应该是软件的基本原则而非可选功能。部分用户呼吁通过法律手段强制保护用户隐私。
