## 事件概述

发帖者表示其数据库遇到了一个 UUID v4 碰撞：两个不同时间生成的 UUID v4 产生了完全相同的值：

```
b6133fd6-70fe-4fe3-bed6-8ca8fc9386cd
```

- 第一个 UUID 来自 2025 年约一年前生成的一条记录
- 第二个 UUID 是今天新插入的文档生成的
- 数据库仅有约 15,000 条记录
- 使用的是 npm 的 `uuid` 包：`import { v4 as uuidv4 } from "uuid";`

发帖者认为这在统计上"不可能"，但确实发生了。

## 关键洞察

### 1. UUID v4 碰撞并非不可能

虽然从统计学上 UUID v4 碰撞概率极低（约 1/2^122），但并非不可能。关键在于随机数生成器的质量，而非单纯的概率问题。

### 2. 熵源问题是主要原因

用户 **jandrewrogers**（38分钟前，最高赞评论）指出：

> UUIDv4 的安全性依赖于高质量的熵源假设。这一假设可能因硬件缺陷、软件 bug 或开发者不理解"高质量熵"的含义而被破坏。检测熵源是否损坏相对昂贵，因此几乎没人会做。他们通常在发生碰撞时才会发现问题。

> UUIDv4 在许多高可靠性系统中是明确禁止使用的。

### 3. uuid npm 包存在代码变更问题

用户 **jbverschoor** 深入调查发现 `uuid` 包在 3 周前发生了一个关键变更：

- **旧代码**：`rng.ts` 中使用 `slice()` 创建新副本
- **新代码**：`const rnds8 = new Uint8Array(16)` 被声明为 const，每次调用共享同一数组，导致生成的 UUID 被覆盖

这个 bug 导致：
- 同步/串行调用：`a` 和 `b` 引用相同数组，内容完全相同
- 异步调用：在 await 后 UUID 被覆盖，最终返回相同值

### 4. 熵源不足的可能原因

- 低质量设备上的随机数生成器未正确初始化
- 虚拟机环境可能"虚拟化掉所有熵"
- `crypto.getRandomValues()` 在某些环境（如 Google Bot）中可能是确定性的

### 5. 替代方案

社区建议的替代方案包括：
- **序列号**：高可靠性系统通常使用序列而非随机 UUID
- **UUID v7**：包含时间戳，基于时间的排序 UUID
- **Snowflake 风格**：Twitter 使用的 128 位分布式 ID 生成方案
- **CUID2**：一种抗碰撞的 ID 生成方案
- **使用时间戳+随机组合**：将日期时间信息纳入 ID 生成

### 6. 有趣的故事

有用户分享了一个"UUID 生成微服务"的笑话：一个初创公司有一个专门的 3 人团队来维护一个 UUID 生成服务，该服务会检查生成的 UUID 是否已被使用过后再返回。这被其他用户调侃为过度工程的典型案例。

## 技术建议

1. 检查 `uuid` 包版本，回滚到使用 `slice()` 的旧版本
2. 考虑迁移到 UUID v7 以利用时间戳保证唯一性
3. 在数据库中实现唯一性约束和碰撞重试机制作为备用方案
4. 确保随机数生成器正确初始化，特别是在虚拟机环境中
