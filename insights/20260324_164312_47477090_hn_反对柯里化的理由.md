## 内容摘要

这是一篇关于函数式编程中**柯里化（Currying）**问题的文章。作者认为柯里化虽然是一个优雅的数学概念，但在实际编程中存在以下问题：

### 1. 隐式部分应用导致歧义

在柯里化语言中，以下代码会产生歧义：

```haskell
let f = foobinade a b
```

读者无法立即判断：
- `f` 是 `foobinade` 完全应用 `a` 和 `b` 的结果
- `f` 是一个等待更多参数的偏函数

### 2. 编译器错误信息不清晰

当函数被错误地应用了过少的参数时，编译器无法立即检测到错误。例如，如果 `foobinate` 需要3个参数：

```haskell
let f = foobinade a b  -- 不会产生错误！
```

### 3. 作者提出的解决方案

使用显式的占位符语法（如 `$`）来区分部分应用和完全应用：

```haskell
let f = foobinade(a, b)        -- 完全应用
let f = foobinade(a, b, $)     -- 部分应用
```

## 社区讨论要点

### 支持方观点

1. **paldepind2**: 认为显式部分应用语法更清晰，有助于编译器和人类理解
2. **twic**: 认为柯里化是"可爱的特性"，在大型代码库中对健康不利
3. **munchler**: 反对此观点，认为柯里化的链式调用能力是其主要优势

### 反对方观点

1. **riwsky**: 认为"部分应用的结果"和"柯里化函数"本质上是相同的
2. **raincole**: 指出柯里化提供的额外自由度应该可选（opt-in），而非默认
3. **tikhonj**: 认为函数式编程中函数作为值是核心概念，柯里化是自然的

### 补充观点

- **jhhh**: 指出柯里化允许在中间步骤进行处理和缓存，适合日志框架等场景
- **vq**: 提到柯里化可以让编译器更好地优化高阶函数
- **Coalton语言**: 最近移除了柯里化，采用固定参数数量函数

## 语言层面的相关讨论

| 语言 | 做法 |
|------|------|
| **Standard ML** | 使用元组参数而非柯里化 |
| **OCaml** | 支持柯里化但推荐使用命名参数 |
| **Roc** | 采用与本文相似的结论 |
| **Scala/Virgil** | 支持 `_` 占位符语法进行部分应用 |
| **Coalton** | 已移除柯里化，采用固定参数 |

---

## 总结

这场讨论反映了函数式编程社区对**显式vs隐式**的持续争论。柯里化在理论上优雅，但可能在实际项目中造成可读性和维护性问题。作者建议采用显式的部分应用语法，让程序员明确表达意图，而不是依赖语言的隐式行为。
