# 洞察报告：Claude Code 在复杂工程任务中可用性问题

## 基本信息

- **洞察链接**: https://github.com/anthropics/claude-code/issues/42796
- **Issue编号**: #42796
- **标题**: [MODEL] Claude Code is unusable for complex engineering tasks with the Feb updates
- **发起人**: stellaraccident
- **创建日期**: 2026年4月2日
- **标签**: area:model, bug, model

## 问题概述

2026年2月的更新导致Claude Code在复杂工程工作流中出现严重质量回退。通过对17,871个思考区块和234,760次工具调用的定量分析，发现思考内容精简（thinking redaction）的 rollout 与测量到的质量下降精确相关。

## 核心发现

### 1. 思考精简时间线与质量回退吻合

| 时间段 | 可见思考 | 已精简 |
|--------|----------|--------|
| 1月30日 - 3月4日 | 100% | 0% |
| 3月5日 | 98.5% | 1.5% |
| 3月7日 | 75.3% | 24.7% |
| **3月8日** | **41.6%** | **58.4%** |
| 3月10-11日 | <1% | >99% |
| 3月12日后 | 0% | 100% |

质量回退在3月8日被独立报告——恰好是精简思考超过50%的日期。

### 2. 思考深度在精简前就已下降

| 时间段 | 估计思考长度(字符) | 相对基线 |
|--------|-------------------|----------|
| 1月30日 - 2月8日 (基线) | ~2,200 | — |
| 2月下旬 | ~720 | -67% |
| 3月1-5日 | ~560 | -75% |
| 3月12日后 (完全精简) | ~600 | -73% |

思考深度在2月下旬已下降67%，远在精简开始之前。

### 3. 行为影响：测量到的质量指标

| 每个会话的提示数 | 35.9 | 27.9 | -22% |

### 5. 为什么扩展思考对这些工作流至关重要

受影响的工作流涉及：
- 50+并发代理会话进行系统编程（C、MLIR、GPU驱动）
- 30+分钟的自主运行，包含复杂的多文件更改
- 广泛的项目特定约定（5,000+词的CLAUDE.md）
- 代码审查、工单管理和迭代调试
- 在良好期间一个周末合并了191,000行代码

扩展思考是模型执行以下操作的核心机制：
- 在行动前规划多步骤方法
- 从CLAUDE.md回忆和应用项目特定约定
- 在输出前Catch自己的错误
- 决定是否继续工作或停止
- 在数百次工具调用中保持连贯推理

当思考变浅时，模型默认为可用的最便宜操作：不阅读就编辑、不完成就停止、规避失败责任、选择最简单的修复而不是正确的修复。

### 6. 建议的改进方案

- **思考分配的透明度**：如果思考token被减少或限制，依赖深度推理的用户需要知道
- **"最大思考"层级**：运行复杂工程工作流的用户愿意为保证深度思考支付更多费用
- **API响应中的思考token指标**：即使思考内容被精简，在使用响应中暴露`thinking_tokens`也能让用户监控推理深度
- **-power用户的金丝雀指标**：Stop hook违规率（0 → 10/天）可以作为质量回退的前瞻性指标

### A.7 约定偏移

在良好时期，模型可靠地遵循约定——阅读CLAUDE.md是会话初始化的一部分，深度思考允许模型回忆并将约定应用到每次编辑。

思考减少后，约定遵守情况明显下降：
- 缩写变量名（buf、len、cnt）再次出现
- 清理模式（if-chain instead of goto）被违反
- 关于删除代码的注释被保留
- 时间引用（"第二阶段"、"稍后完成"）出现在代码中

## 附录B：降级成本

### Token使用：2026年1月至3月

| 指标 | 1月 | 2月 | 3月 |
|------|-----|-----|-----|
| 活跃天数 | 31 | 28 | 28 |
| 用户提示 | 7,373 | 5,608 | 5,701 |
| API请求（去重） | 97* | 1,498 | 119,341 |
| 总输入（含缓存） | 4.6M* | 120.4M | 20,508.8M |
| 总输出token | 0.08M* | 0.97M | 62.60M |
| 估计Bedrock成本 | $26* | $345 | $42,121 |
| 估计每日成本 | — | $12 | $1,504 |

3月API请求增加80倍，输出token增加64倍，估计成本增加122倍——而用户提示数量基本持平（5,608 vs 5,701）。

### 成本解释

深度思考时：
- 编辑前充分阅读代码（每次编辑6.6次读取）
- 第一次尝试就做对更改
- 会话自主运行30+分钟无需干预
- 一次API请求完成有意义的工作

不思考时：
- 不阅读就编辑（每次编辑2.0次读取）
- 更改错误，需要纠正循环
- 会话每1-2分钟就停滞，需要人工干预
- 每次干预生成多个额外API请求
- 失败的工具调用浪费输出token

## 结论

数据表明，扩展思考token不是"可有可无"，而是模型执行多步骤研究、遵守约定和仔细修改代码的结构性需求。当思考深度减少时，模型的工具使用模式从研究优先转变为编辑优先，产生了用户报告的质量问题。

对于运行大规模复杂工程工作流的企业用户，保证深度思考可能反而会减少Anthropic的服务成本——即使单个请求的花费更多，因为深度思考的模型在一次请求中就能做对，而浅思考的模型需要10次请求才能跌跌撞撞达到同样结果。
