## 摘要

本文探讨了 JavaScript 生态系统中依赖树膨胀的三大根源，以及我们可以采取的应对措施。

---

## 三大膨胀根源

### 1. 过时的运行时支持（安全性和跨域）

**问题描述**：许多小型工具包（如 `is-string`、`hasown`）提供了本应由平台原生提供的功能。

**产生原因**：
- **支持极旧的引擎**：部分开发者需要支持 ES3（如 IE6/7），这些环境缺少 ES5 特性如 `Array.prototype.forEach`、`Object.keys` 等
- **防止全局命名空间被修改**：Node.js 使用 "primordials" 模式保护自身不被全局污染破坏，部分开发者将此理念应用于包开发
- **跨域值传递**：从 iframe 传递到父窗口的值（如 `new RegExp()`）与原窗口的构造函数不同，`instanceof` 检查会失败

**问题所在**：绝大多数开发者使用的是现代 Node.js 或常青浏览器，完全不需要这些兼容性代码。极小部分人的需求却让所有人承担了成本。

---

### 2. 原子化架构

**问题描述**：将代码拆分成极细粒度的包，导致依赖图极其复杂。

**示例**：
- `shebang-regex`：仅包含一个正则表达式 `^#!(.*)/`
- `arrify`：将值转换为数组 `Array.isArray(val) ? val : [val]`
- `path-key`：获取当前平台的 PATH 环境变量键名

**问题所在**：
- **单用途包**：许多原子包仅被一个包使用，本质上等同于内联代码，但增加了获取成本
- **重复**：同一包可能有多个版本共存，如 nuxt 依赖树中出现 `is-docker`（2版本）、`is-wsl`（2版本）等
- **供应链风险**：维护者被入侵会导致大量小包被攻陷，带来安全风险

---

### 3. 过时的 "Ponyfill"

**问题描述**：为已原生支持的功能使用 ponyfill，且从未移除。

**常见示例**：
- `globalthis` - 为 `globalThis` 提供的 ponyfill（2019年已广泛支持，每周下载4900万次）
- `indexof` - 为 `Array.prototype.indexOf` 提供的 ponyfill（2010年已广泛支持，每周下载230万次）
- `object.entries` - 为 `Object.entries` 提供的 ponyfill（2017年已广泛支持，每周下载3500万次）

**问题所在**：这些 ponyfill 在当时是有意义的，但当功能已被所有主流引擎长期支持后，它们应该被移除。实际情况是它们被永久保留，无人问津。

---

## 解决方案

### 1. 使用 knip 移除未使用的依赖

knip 是一个工具，可以帮助发现和移除未使用的依赖、死代码等。

### 2. 使用 e18e CLI 检测可替换的依赖

```bash
npx @e18e/cli analyze
```

该命令可以识别哪些依赖可以用原生功能替代，或有更好的替代方案。

### 3. 使用 npmgraph 可视化依赖树

npmgraph 可以帮助可视化依赖树，识别哪些分支是孤立且不必要的。

### 4. 模块替换项目

[module-replacements](https://github.com/es-tooling/module-replacements) 项目记录了哪些包可以用原生功能或更轻量的替代品替换。

---

## 结论

整个 JavaScript 社区正在为极少数人的特殊需求付出代价。这些膨胀源于一个"更黑暗的时代"，当时许多现代 API 和跨兼容性功能还不存在。问题在于我们从未真正向前迈进。

解决方案是扭转局面：让那小部分有特殊需求的人使用他们专门的栈，其他人则使用现代、轻量且广泛支持的代码。
