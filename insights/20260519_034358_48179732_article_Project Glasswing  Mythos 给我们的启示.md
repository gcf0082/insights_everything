## 摘要

Cloudflare 在过去几个月对多种安全专用的大语言模型（LLM）进行了测试，将其应用于自身关键基础设施的实时代码中。本文分享了测试观察结果、模型的优势与不足，以及在扩大使用规模前需要进行的架构和流程改进。

---

## 核心发现

### Mythos Preview 的突破性进展

Mythos Preview 是 Anthropic 推出的安全专用 LLM，在测试中展现出显著进步：

**1. 漏洞利用链构建能力**

真正的攻击很少使用单一漏洞，而是将多个小攻击原语组合成可工作的漏洞利用。Mythos Preview 能够将多个漏洞原语组合成完整可行的漏洞利用链，推理过程看起来像高级研究人员的分析工作。

**2. 概念验证生成能力**

发现漏洞与证明其可利用性是两回事。Mythos Preview 都能完成——它编写触发漏洞的代码，在临时环境中编译运行，根据结果验证假设。如果失败，模型会分析失败原因并调整假设重试。

### 模型拒绝问题

Mythos Preview 虽然没有公开发布模型的那些额外安全防护，但模型本身会对某些请求产生自发的拒绝。这种自发的拒绝并不一致——相同的任务以不同方式表述或在不同上下文中呈现，可能产生完全不同的结果。

这意味着未来任何可广泛使用的安全 LLM 都必须在此基础上增加额外的安全防护措施。

## 信号与噪声问题

安全漏洞分类中最困难的部分是判断哪些是真实的、哪些可利用、哪些需要立即修复。

### 噪声率的两大主导因素

**1. 编程语言**

C 和 C++ 提供直接内存控制，导致缓冲区溢出、越界读写等漏洞。内存安全语言（如 Rust）在编译时消除了这些漏洞。使用不安全的语言编写的项目产生的误报明显更多。

**2. 模型偏见**

人类研究人员会告诉你发现了什么以及置信度。模型不会。询问模型查找漏洞，它就会找到漏洞，无论代码中是否真的存在。结果充满了"可能""理论上可以""或许"等模糊表述。在分类队列中，每个投机性发现都会消耗人工注意力和 token 成本。

Mythos Preview 在这方面有明显改进，特别是将多个漏洞组合成可工作的概念验证的能力。带有 PoC 的发现是可以采取行动的发现。

## 为什么通用编码代理无法胜任

当 Cloudflare 去年开始 AI 辅助漏洞研究时，最直接的方法是让通用编码代理指向任意代码库并查找漏洞。这种方法确实会产出发现，但无法有效覆盖真实代码库并识别有价值的发现。

### 两个主要原因

**1. 上下文问题**

编码代理针对单一工作流进行了优化：构建功能、修复 bug、重构。它们摄入大量源代码，保持单一假设并迭代。这正好与漏洞研究的性质相反——漏洞研究是狭窄且并行的。人类研究人员选择特定目标进行深入研究，然后针对不同功能、安全边界或漏洞类别重复这个过程数千次。

**2. 吞吐量问题**

单流代理一次只做一件事，但真实代码库需要同时针对多个组件测试多个假设。使用通用编码代理进行手动调查效果不错，但对于实现高覆盖率来说是错误的工具。

## 漏洞发现框架

Cloudflare 构建了一个漏洞发现框架，包含以下阶段：

| 阶段 | 功能 | 重要性 |
|------|------|--------|
| **侦察** | 自上而下读取代码库，分配给各子系统代理，产生架构文档和初始任务队列 | 为下游代理提供共享上下文，减少漫游问题 |
| **狩猎** | 每个任务对应一个攻击类别加范围提示，并行运行约50个代理，每个代理可扩展多个探索子代理 | 大部分工作发生的地方，多个狭窄任务并行而非一个 exhaustive 代理 |
| **验证** | 独立代理重新读取代码并尝试推翻原始发现，使用不同 prompt | 捕获第一代理会遗漏的噪声 |
| **Gapfill** | 标记未充分覆盖的区域并重新排队 | 对抗模型偏向已成功攻击类别的倾向 |
| **去重** | 共享同一根因的发现合并为单条记录 | 变体分析是一项功能，不是重复膨胀 |
| **追踪** | 对于共享库中的确认发现，追踪代理展开检查攻击者输入是否真正从系统外部到达该漏洞 | 将"存在缺陷"转化为"存在可利用的漏洞" |
| **反馈** | 可达追踪成为消费者代码库中新狩猎任务 | 闭环，管道随运行改进 |
| **报告** | 代理根据预定义模式编写结构化报告 | 输出是可查询的数据而非自由格式文本 |

## 对安全团队的意义

对 Mythos Preview 最强烈的反应是关于速度——扫描更快、打补丁更快、压缩响应周期。但仅仅更快是不够的。

打补丁更快不会改变产生补丁的流程形态。如果回归测试需要一天，无法在不跳过它的情况下实现两小时 SLA，而跳过回归测试时引入的 bug 往往比试图修复的 bug 更严重。

更困难的问题是漏洞周围应该是什么架构。原则是即使存在 bug 也要让攻击者更难利用。这意味着：位于应用前方的防御层阻止漏洞被触达；应用设计使得代码某一部分的缺陷无法给攻击者提供访问其他部分的途径；能够同时向代码运行的每个位置推出修复，而不是等待各个团队单独部署。

## 结论

Cloudflare 在受控环境中使用 Mythos Preview 对自身代码进行了研究。通过这项工作发现的每个漏洞都经过分类、验证，并在 Cloudflare 正式漏洞管理流程中进行了修复。

同一能力既能帮助我们发现自身代码中的漏洞，在错误手中也会加速对互联网每个应用的攻击。Cloudflare 站在数百万应用前方，上述架构原则正是其产品为客户构建的基础。
