## 基本信息

- **洞察链接**: https://read.thecoder.cafe/p/linux-broke-postgresql
- **原文标题**: How Linux 7.0 Broke PostgreSQL: The Preemption Regression Explained
- **作者**: Teiva Harsanyi
- **发布日期**: 2026年4月29日
- **主题**: Linux 内核调度策略变更导致 PostgreSQL 性能下降的技术分析

---

## 摘要

2026年4月3日，AWS 工程师 Salvatore Dipietro 在 Linux 内核邮件列表发布了一个补丁，揭示了在 96 vCPU Graviton4 机器上运行 Linux 7.0 时，PostgreSQL 吞吐量下降至 Linux 6.x 的一半左右。本报告详细分析了这一问题的根本原因、影响机制以及解决方案。

---

## 问题描述

### 测试环境与结果

- **硬件**: 96 vCPU Graviton4 处理器
- **工作负载**: pgbench 基准测试，简单更新操作，比例因子 8,470（约 8.47 亿行表），1,024 个客户端，96 个线程
- **结果对比**:
  - Linux 6.x: 98,565 事务/秒
  - Linux 7.0: 50,751 事务/秒（下降约 49%）

### 性能分析

通过 `perf` 工具分析发现，55% 的 CPU 时间消耗在单个函数 `s_lock` 上，该函数位于 `StrategyGetBuffer` 中，负责在共享缓冲区池中查找可用缓冲页。

---

## 问题根源

### 1. Linux 7.0 调度策略变更

Linux 7.0 在现代 CPU 架构上移除了 `PREEMPT_NONE` 选项，只保留：
- `PREEMPT_FULL`: 完全抢占
- `PREEMPT_LAZY`: 延迟抢占（Linux 6.12 引入）

`PREEMPT_LAZY` 设计作为吞吐量工作负载的替代方案，对大多数服务器软件影响不大，但 PostgreSQL 触发了特定边缘情况。

### 2. PostgreSQL 内存管理机制

PostgreSQL 使用固定大小（8 KB）的数据页存储数据，并维护共享缓冲区池来缓存常用数据页。在测试中，`shared_buffers` 配置为 120 GB。

关键组件：
- **后端进程**: 每个客户端连接对应一个独立的后端进程
- **StrategyGetBuffer**: 查找可用缓冲区的关键函数，使用单一全局自旋锁保护临界区

### 3. 自旋锁与页故障的交互

**自旋锁原理**: 线程在锁可用前持续轮询检查，而非休眠。这适用于极短的临界区，假设锁持有者会极快释放。

**问题触发过程**:
1. PostgreSQL 共享缓冲区池默认使用 4 KB Linux 内存页
2. 120 GB 缓冲区池意味着约 3,100 万个内存页
3. 首次访问未映射的虚拟地址会触发 **次要页故障**（minor page fault）
4. 在基准测试过程中，新区域不断进入工作集，故障持续发生

**关键问题**: 当后端持有自旋锁时触发页故障，锁持有者被困在内核故障处理程序中，所有等待该锁的其他后端持续自旋浪费 CPU 周期。

### 4. 抢占模型的影响

| 场景 | 行为 | 影响 |
|------|------|------|
| PREEMPT_NONE (Linux 7.0 之前) | 故障处理完成后快速恢复，锁快速释放 | 自旋线程等待时间可接受 |
| PREEMPT_LAZY (Linux 7.0 及之后) | 调度器可能在故障处理中期抢占该进程 | 锁持有时间 = 故障处理时间 + 调度等待时间，等待的自旋线程数量成倍放大性能损失 |

在 96 vCPU 机器上，数百个后端同时自旋，额外等待时间被所有等待线程累积，导致灾难性性能下降。

## 解决方案

### 大页（Huge Pages）

Linux 支持大于默认 4 KB 的内存页：
- **2 MB 大页**: 120 GB 需约 61,440 个页（减少 99.8%）
- **1 GB 大页**: 120 GB 只需约 120 个页

**优势**:
1. 大幅减少潜在页故障数量
2. 降低 TLB（Translation Lookaside Buffer）压力
3. 减少热路径上的 TLB 未命中和页表遍历

**配置方法**:
PostgreSQL 通过 `huge_pages` 参数控制：
- `off`: 禁用
- `on`: 强制启用
- `try`: 默认值，尝试使用大页，不可用时回退到 4 KB 页

**注意事项**:
- 大页需预分配和预留，即使 PostgreSQL 未完全使用也会占用内存
- 需合理评估生产环境权衡

### 其他方案

内核工程师 Peter Zijlstra 建议 PostgreSQL 采用 Restartable Sequences（rseq）机制检测并恢复抢占，但 PostgreSQL 社区对此兴趣不大，因为这需要依赖内核特性来恢复 Linux 7.0 之前免费获得的功能。

---

## 总结

1. **根本原因**: Linux 7.0 移除 `PREEMPT_NONE` 后，`PREEMPT_LAZY` 允许在页故障处理期间抢占进程，延长了自旋锁持有时间

2. **影响规模**: 120 GB `shared_buffers` 使用 4 KB 页意味着约 3,100 万次潜在首次访问故障

3. **性能损失**: 55% CPU 时间消耗在自旋锁等待上

4. **解决方案**: 启用大页（2 MB 或 1 GB）可将潜在故障减少数个数量级，同时降低 TLB 压力，使性能回归消失
