## 问题概述

macOS 26.3.1 悄悄破坏了 `/etc/resolver/` 机制，这是一个长期稳定的 Apple 文档化功能，用于配置自定义顶级域名的 DNS 解析。受影响的包括 `.internal`、`.test`、`.home.arpa`、`.lan` 等非 IANA 注册的 TLD。

## 技术细节

### 受影响的 TLD

| TLD | 状态 | 备注 |
|-----|------|------|
| `.internal` | 损坏 | IETF 草稿特殊用途 TLD |
| `.test` | 损坏 | RFC 6761 §6.2 — 明确保留用于本地测试 |
| `.home.arpa` | 损坏 | RFC 8375 — IANA 保留用于家庭私有网络 |
| `.lan` | 损坏 | 广泛使用的约定 |

### 根本原因

在 macOS 26 中，`mDNSResponder` 拦截了这些自定义/私有 TLD 的查询，并将其作为 mDNS（多播 DNS）处理，从不咨询 `/etc/resolver/` 中指定的单播 DNS 服务器。这意味着使用 dnsmasq、bind、unbound 等本地 DNS 服务器的开发者将无法解析 `*.internal`、`*.test` 等私有域名。

### 三大 DNS 问题

macOS 26.1 引入了三个独立的 DNS 问题：

1. **加密 DNS 配置文件安装失败**: 通过 .mobileconfig 文件安装加密 DNS（如 Mullvad）时失败，错误信息为 "The 'VPN Service' payload could not be installed"
2. **/etc/resolver/ 机制损坏**: 自定义 TLD（如 .internal、.local、.test）的解析被破坏
3. **VPN DNS 集成失败**: 使用自定义 DNS 的 VPN 服务在启用加密时无法解析内部域名

## 社区讨论要点

- 许多开发者依赖此功能进行本地开发（如 Docker 容器访问）
- `.test` 特别严重，因为 RFC 6761 明确保留用于本地/私有 DNS 测试
- 有用户指出使用 AI 撰写 bug 报告存在争议，原始报告错误地提到了"macOS 25"
- 有人将此与 Linux 进行比较，认为系统更新破坏 DNS 设置并非 macOS 独有
- 有消息人士称 iOS 27 可能是类似 Snow Leopard 的更新，主要聚焦 bug 修复和性能改进

## 临时解决方案

1. **手动 /etc/hosts 条目**: 可靠但需要 sudo，不支持通配符
2. **使用 *.localhost**: 仅限浏览器
3. **公共通配符 DNS 服务**: 如 nip.io 或 lvh.me，可系统范围使用但需要互联网

## 关键建议

- 如果你的工作流程依赖自定义 DNS、加密 DNS 配置、VPN 或 .internal/.local 域名用于开发，请推迟 macOS 26 升级
- 如果已安装加密 DNS 配置文件，请保留它们——在 macOS 26.1 中删除后无法重新安装
- Apple 尚未对此问题发表任何声明
