# 洞察报告：离开GitHub，转向Forgejo

## 基本信息

- **洞察链接**: https://jorijn.com/en/blog/leaving-github-for-forgejo/
- **文章标题**: Why I'm leaving GitHub for Forgejo
- **作者**: Jorijn Schrijvershof
- **发布日期**: 2026年4月
- **语言**: 英文

## 核心摘要

作者将代码从GitHub迁移到自托管的Forgejo实例，原因是担心平台所有权归属而非服务中断。作者提到荷兰政府也做出了类似选择，于2026年4月27日推出了code.overheid.nl，这是一个自托管的Forgejo平台供荷兰政府使用。

## 关键发现

### 1. 停机事件并非主要原因

2026年4月的停机事件确实严重：658个仓库的合并提交被回退，PR和Issues服务也经历了超过6小时的宕机。但作者指出，真正的问题在于GitHub正在全力投入AI功能，CTO在道歉中表示需要将容量扩大30倍来应对AI驱动的负载增长。

### 2. GitHub不再拥有独立的CEO

2025年8月11日，GitHub前CEO Thomas Dohmke离职，微软没有任命继任者。GitHub被并入微软的CoreAI部门——一个负责构建Copilot和AI堆栈的团队。这意味着GitHub的品牌虽在，但独立领导权已不存在。

### 3. 训练数据默认选项的改变

2026年4月24日起，GitHub将Copilot Free、Pro和Pro+用户的交互数据（包括输入、输出、代码片段和相关上下文）用于AI模型训练，默认为Opt-in而非Opt-out。更关键的是，没有仓库级别的开关，每个贡献者都必须自行选择退出。

### 4. 司法管辖风险

GitHub和微软都是美国公司，受FISA Section 702和CLOUD Act管辖。微软自己的律师在法国参议院作证时表示，他无法保证法国数据免受美国政府的秘密访问。

### 5. 荷兰政府的选择

荷兰政府选择Forgejo而非GitLab，理由是Forgejo完全开源，没有开放核心分裂，提供数字自治所需的全部自由。欧洲其他国家如德国使用GitLab，法国则只是聚合器而非自托管。

### 6. 为何选择Forgejo而非GitLab

两个关键因素：
- **许可证**：GitLab是开放核心，许多功能位于企业版付费许可后；Forgejo在2024年8月从MIT重新许可为GPLv3+，明确保持Copyleft立场
- **治理**：Forgejo由Codeberg e.V.（柏林注册的非营利组织）运营，有成员投票的公开预算制度

### 7. 技术架构

作者在单台Intel NUC（64GB RAM）上运行：
- Forgejo v15 LTS（PostgreSQL 17 + Traefik via Docker）
- Incus管理的KVM虚拟机运行Forgejo Actions runner
- 五层安全防护：KVM隔离 → gVisor运行时 → 每周销毁重建 → nftables出口过滤 → 作用域绑定的runner令牌

## 代价与权衡

- **发现与社交图谱**：贡献者通过GitHub找到作者，迁移后需要将公开仓库存档并指向新地址
- **GitHub Actions生态脆弱性**：Forgejo Actions追求熟悉度而非兼容性，部分功能需要使用Fork版本
- **无Dependabot**：作者使用Renovate替代
- **无全天候Vendor支持**

## 何时不值得迁移

- 团队没有基础设施运维能力或意愿
- 严重依赖GitHub特有功能（GitHub Apps、Codespaces、Copilot Workspace、Advanced Security）
- 贡献者群体依赖GitHub社交图谱
- 没有为Runner实施适当安全措施的能力

## 结论

这不是"GitHub不可靠"的故事，而是关于平台所有权和控制权的根本转变。荷兰政府的类似决定表明这一趋势不再是边缘化的。真正的挑战不在于Forgejo部署本身，而在于Runner的安全性——它需要执行来自外部的未信任代码。
