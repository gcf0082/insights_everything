## 内容摘要

本文是关于一个名为 **Lubeno** 的代码协作平台的广告博文，该平台专注于 **堆叠式拉取请求（Stacked PRs）** 工作流。作者认为传统的 PR 模式存在效率问题，提出了重新设计代码审查流程的理念。

### 核心观点

1. **堆叠式 PR 的优势**：允许开发者创建多个小型、自包含的变更，可以逐个审查而无需等待整个 PR 合并
2. **变更传播**：子分支自动继承父分支的修改，减少重复工作
3. **改进代码审查体验**：让审查者可以专注于单个提交而非一次性审查整个 PR

---

## 讨论热点

### 1. 堆叠式 PR 的风险与价值

**反对意见**：
- 在未审查的变更基础上继续开发可能导致浪费时间，因为基础变更可能被拒绝
- Git 和大多数开发工作流是线性的，这是有原因的

**支持意见**：
- 大多数情况下基础变更基本会通过，最多有小调整而非重大修改
- 从之前的 PR 分支而不是从开发分支继续，可以减少合并冲突
- 允许后端和前端审查者分别审查各自负责的提交，而非必须同时审查整个 PR

### 2. 小团队 vs 大团队的工作流

有评论指出：
- GitHub 的 PR 模式非常适合开源项目（需要整体评估）
- 但对于小团队，PR 审查成为瓶颈，开发者花费大量时间等待审查
- 建议采用更小、更频繁的提交，直接合并到主分支
- 这是原始持续集成理念的精神

### 3. 工具推荐：Jujutsu (jj)

多个评论者推荐使用 **Jujutsu**（jj）：
- 一个 Git 兼容的版本控制系统
- 使堆叠式 PR 变得非常简单
- 启用 RERERE 和 autostash 可以消除 80% 的繁琐操作
- 对于不熟悉 jj 的命令，仍可使用原生 git 命令

### 4. GitHub 的现状与未来

- GitHub 官方宣布正在开发堆叠 PR 功能，目前处于 private beta 阶段
- 追踪 issue: https://github.com/github/roadmap/issues/1218
- 目前需要使用第三方工具（如 Graphite）来实现堆叠 PR

### 5. 其他观点

- **性能问题**：博客加载速度较慢（4.85秒到8秒），作者已致歉并表示会优化
- **AI 辅助**：有人提出可以使用 AI 将大型 PR 分解为可消化的部分
- **本地优先工具**：有人希望有本地优先的代码审查工具，可与 AI agents 配合使用
- **Gerrit vs GitHub**：Gerrit 在处理代码变更方面比 GitHub 做得更好

## 总结

这篇讨论反映了开发者对当前 PR 工作流的普遍不满，以及对更高效代码协作工具的渴望。堆叠式 PR 作为一种改进方案，得到了不少开发者的认可，但也有人对其风险表示担忧。Jujutsu 等工具的出现表明这一需求正在被满足，同时 GitHub 官方也在开发相关功能。
