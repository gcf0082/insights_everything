## 项目概述

Re_Gent 是一个开源版本控制系统，专为 AI Agent 工作流设计。目前仅支持 Claude Code。该工具旨在解决 AI Agent 在执行任务过程中的版本控制问题，让用户能够追踪 Agent 的操作历史、回答"为什么这样做"、"何时删除这个文件夹"等问题，并支持类似 git 的 `/rewind`、`/compact` 和 `bisect` 等功能。

---

## 核心观点

### 1. 项目定位与价值

- **解决的问题**：AI Agent 执行任务时缺乏版本控制能力，用户无法追踪 Agent 的决策历史
- **核心功能**：追踪 Agent 操作、支持历史回滚、支持 bisect 定位问题
- **与 Git 的关系**：不是替代 Git，而是建立在 Git 之上，在最终保存代码时才与 Git 交互

### 2. 用户反馈与建议

#### 2.1 传播方面的建议
- **宣传语需要改进**：不少人误解项目是让 Agent 使用 Git，实际是解决不同的问题
- 建议将宣传语改为 "Version Control for AI agent activity"
- 建议加速 GitHub 页面上的演示视频，减少用户流失

#### 2.2 技术功能的建议
- **对话历史检索**：用户希望能搜索特定对话，如"找到昨天关于某话题的讨论"
- **RTK 集成**：是否与 RTK（token 节省工具）兼容
- **特性分支追踪**：是否支持分支管理
- **垃圾回收机制**：历史重写（如 rebase）时的垃圾回收，以及数据简化机制

#### 2.3 扩展性方面的建议
- 目前仅支持 Claude Code，计划支持其他主流 Agent
- 考虑发布到 aqua-registry 等 mise 兼容的注册表

### 3. 社区的主要观点

#### 3.1 质疑声音
- **"直接用 Git"派**：认为用 pre-commit hook 让 Agent 每次操作自动 commit 即可解决
- 作者回应：这会污染 Git 历史，且开销过大
- **"Agent 已经会用 Git"派**：Agent 已经能够通过命令行使用 Git
- **"Flux Capacitor for AI agents"派**：调侃这类工具的本质就是原有工具的重新包装

#### 3.2 正面反馈
- **Jujutsu (jj) 工作流**：用户 Zambyte 分享了使用 jj 的工作流程，与 Re_Gent 概念相似
  - 使用 `jj new` 创建新 change
  - 每次 prompt 后自然检查 diff
  - 使用 `jj evolog` 查看工作 change 的 diff log
  - 使用 `jj restore` 撤销部分或全部操作
  - 满意后再用 jj commit 提交
- **隔离人类与 Agent 的 Git 历史**：将 Agent 操作与人类操作分离，保持人类 Git 历史清晰
- **追踪意图的价值**：在 git commit 中记录意图（why）和决策依据

#### 3.3 相关项目对比
- **Cloudflare Artifacts**：类似的 Agent 版本控制方案
- **entire.io**：基于 Git 构建，需要先 commit 才能追踪
- **jujutsu (jj)**：已有用户将 jj 应用于 Agent 工作流
- **triblespace-rs**：类似的 DAG + 知识图谱方案，支持 P2P 同步

### 4. 有趣评论

- 有用户分享 Agent 曾经执行 `git rebase` 和 `git force push` 导致冲突
- 有用户认为 Git 应该作为人类防御 AI 垃圾代码的工具，而非自动化
- 有用户提到 Cursor 的 ESC-ESC 快捷键（相当于回滚功能），Claude Code 目前缺失

---

## 总结

Re_Gent 是一个针对 AI Agent 工作流的版本控制系统，解决了 Agent 操作透明度和可追溯性问题。虽然面临"直接用 Git"的质疑，但其设计思路（追踪意图、支持 bisect、保持人类 Git 历史清洁）获得了部分用户认可。项目目前处于早期阶段，功能相对简单，测试用例较少，但概念独特，值得关注。社区讨论中也提到了 jujutsu (jj) 作为替代方案的实践。
