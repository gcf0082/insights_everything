## 文章摘要

本文作者结合自己多年的编程经验，对比了Lisp/Scheme与Haskell的优缺点，解释了为什么他更倾向于使用Lisp/Scheme而不是Haskell。

### 主要观点

#### 1. Haskell的优点

- 拥有最复杂、最强大的类型系统
- 引入了许多数学概念：代数数据类型、模式匹配、函子、单子、Monoid等
- 在学术界和研究领域影响力大
- 由PhD、计算机科学研究者、范畴论研究者等组成的高质量社区

#### 2. Haskell的痛点

- **原型开发困难**：即使有多年经验，写一个简单的数据转换（如Haskell转XML）需要花费大量时间处理依赖和Monadic API
- **Monads税**：Monads虽然优雅，但感觉像是沉重的抽象税，让原型开发不够直观快速
- **DSL割裂**：Hackage上有数百个DSL，每个都完全不同，学习曲线高。例如解析XML、处理JSON、写PDF需要三个不同的DSL

#### 3. Scheme/Lisp的优势

- **简洁灵活**：允许用更简单的术语表达复杂系统
- **实用主义**：可以随时使用`(write ...)`而不是像Haskell那样需要处理IO Monad
- **强大的宏系统**：可以轻松重塑语言，构建自己的DSL
- **优秀的REPL**：支持增量开发、强大的调试能力，可以暂停、检查对象、修改值、甚至在生产环境中即时重定义函数

#### 4. REPL交互式工作流

- **增量开发**：逐函数、逐行编写和测试，获得即时反馈
- **强大调试**：无需添加print语句重启，可以随时暂停、检查、修改值
- **快速原型**：即时尝试新库或API，比只读文档快得多

## 讨论焦点（Top Comments）

1. **Racket的宏系统**：有人指出虽然 Lisp 宏很强大，但实际上很多 Racket 开发者不需要自己写宏，因为标准库已经包含

2. **S-Expressions**：关于 Lisp 开发者如何处理 JSON 和 XML，有评论指出转换为 S-Expressions 是常见的做法

3. **热修复能力**：讨论哪些 Lisp 方言真的支持连接到运行程序、调试和热修复。Common Lisp、Clojure、Racket 都支持，但可能需要一些工具

4. **类型系统**：有人提到从 Scheme 转向 Common Lisp 后，很怀念类型系统。Coalton 被推荐为 Common Lisp 的类型系统方案

5. **Haskell 语法批评**：有评论批评 Haskell 语法是"word salad"，每行都要读多次才能理解结构

6. **Clojure 推荐**：很多人推荐 Clojure 作为结合 Lisp 优势和现代生态的方案

## 结论

| 维度 | Haskell | Scheme/Lisp |
|------|---------|-------------|
| **类型系统** | 极强 | 较弱/可选 |
| **原型速度** | 慢（需要处理Monads） | 快（直接写） |
| **宏能力** | Template Haskell | 原生强大宏系统 |
| **DSL构建** | 困难 | 简单 |
| **REPL** | GHCi | 强大交互式 |
| **适用场景** | 学术研究、企业级 | 原型、快速开发、个人项目 |

**核心观点**：Haskell 是编程语言的"柏拉图理想"，而 Scheme/Lisp 才是作者认为的"最佳平衡点"——既有函数式编程的美感，又不失实用性。
