## 问题概述

Apple M4和M5芯片在macOS上引入了外部4K显示器HiDPI模式的回归问题，导致用户无法在4K显示器上使用完整分辨率的HiDPI模式。

**核心问题**：
- 4K显示器（3840x2160）现在最高只能使用3360x1890的HiDPI模式
- 这意味着渲染缓冲区仅为6720x3780，而非所需的7680x4320（2倍缩放）
- M2/M3芯片没有此限制

**用户面临的选择**：
1. 使用全屏工作区（3840x2160），但文本模糊（HiDPI被禁用）
2. 使用缩小的工作区（3360x1890），文本清晰但UI元素过大

## 技术分析

### 问题根源

根据原文作者的分析：

1. **非硬件限制**：DCP（显示协处理器）在M2 Max和M5 Max上报告了相同的显示参数
   - MaxActivePixelRate: 497,664,000
   - MaxW: 3840
   - MaxH: 2160

2. **软件层面限制**：差异出现在WindowServer的模式列表中
   - M2 Max: 包含3840x2160@2x HiDPI模式
   - M5 Max: 该模式不存在，最高仅到3360x1890@2x

3. **动态资源分配**：BetterDisplay开发者waydabber解释称，M4系列采用了新的动态资源分配策略，系统会尽可能低地分配帧缓冲区大小，为可能连接的其他显示器预留空间

### 已尝试的解决方案（均未成功）

| 尝试方法 | 结果 |
|----------|------|
| Display Override Plist | 无效，M5 Max拒绝枚举该模式 |
| EDID补丁（软件覆盖） | 无效，无法欺骗DCP生成有效模式 |
| 刷写显示器EEPROM | 失败，LG显示器EEPROM可能为只读 |
| IOKit注册表修改 | 失败，内核驱动拒绝用户空间属性写入 |
| WindowServer缓存清除 | 无效 |
| 减少连接显示器数量 | 无效 |
| SkyLight私有API | 返回错误1000，模式验证失败 |

## 社区洞察

### 有趣的发现

1. **给Tim Cook发邮件有效**：一位用户分享经验称，给Tim Cook发邮件解决了他的DisplayPort DSC bug问题

2. **BetterDisplay的局限性**：即使使用BetterDisplay，也无法在M4/M5机器上设置4K HiDPI模式

3. **这不是"正常"的Retina配置**：部分评论认为这是非常规的高倍超采样配置，但作者指出使用标准LG UltraFine 4K显示器（通过标准Thunderbolt连接）不应被视为非标准配置

4. **Windows处理得更好**：多位用户指出，Windows 11可以轻松处理27英寸4K显示器的150%缩放，macOS在这方面的体验较差

5. **Subpixel抗锯齿的缺失**：由于Apple在2018年左右移除了子像素抗锯齿，用户不得不通过超采样来获得可接受的文本渲染效果

### 建议的显示器配置

| 显示器尺寸 | 推荐分辨率 |
|------------|------------|
| 24寸 | 4K (2x缩放) |
| 27寸 | 5K (2x缩放) |
| 32寸 | 6K (2x缩放) |

## 可能的解决方

1. **Apple修复**：需要Apple修改缩放比例限制或添加用户-facing覆盖选项
2. **使用5K或8K面板**：更高分辨率面板可能不会遇到相同限制
3. **EDID欺骗**：通过软件EDID覆盖谎报显示器原生分辨率为8K（但实际显示器无法处理该信号）

---

## 结论

这是Apple M4/M5芯片上的一个软件回归问题，影响所有使用第三方4K显示器的用户。该问题并非硬件限制，而是macOS在帧缓冲区分配策略上的改变。用户目前没有有效的临时解决方案，需要等待Apple发布官方修复。
