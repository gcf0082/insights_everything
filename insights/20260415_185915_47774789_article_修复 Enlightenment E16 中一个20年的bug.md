- **作者**: Kamila Szewczyk

## 摘要

本文描述了作者在 Enlightenment E16（1997年发布的窗口管理器）中修复一个可追溯到2006年的严重Bug的过程。该Bug的根源是一个使用了Newton算法的文本截断函数存在缺陷，导致在处理特定长度的PDF窗口标题时造成无限循环卡死。

## 背景

Enlightenment E16 是由 Carsten Haitzler 于1997年创建的窗口管理器，至今仍有活跃的社区用户使用。作者就是其中之一。作者日常使用E16作为窗口管理器，在准备课程幻灯片时，打开一个PDF文件导致整个桌面冻结。

## 问题分析

### 症状表现
- 打开特定PDF文件时整个X11会话冻结
- 使用gdb调试发现程序并未死锁，而是在 `__imlib_font_cache_glyph_get` 中持续调用不同的字形索引
- Frame 8 (`TextstateTextFitMB` at `text.c:350`) 是恒定不变的

### 根本原因
问题出在 `TextstateTextFitMB` 函数中的Newton风格搜索算法。该算法用于在文本过长时从中间截断文字（middle-ellipsis truncation）。

关键代码是一个无限循环，用于估算需要删除多少个字符以使文本适应宽度限制：
```c
for (;;)
{
    nc2 = (wc_len - nuke_count) / 2;
    // 构建新文本 = 前nc2个字符 + "..." + 尾部字符
    ts->ops->TextSize(ts, new_line, 0, pw, &hh, &ascent);
    width = *pw;
    nc2 = textwidth_limit - width;
    cw = width / len_n;
    if (nc2 >= 0 && nc2 < 3 * cw)
        break;  // 适应宽度，在3个字符内
    // Newton步长更新
    if (nc2 > 0)
        nuke_count -= (nc2 <= 2 * cw) ? 1 : (nc2 + cw / 2) / cw;
    else
        nuke_count += (-nc2 <= 2 * cw) ? 1 : (-nc2 + cw / 2) / cw;
}
```

### 问题所在
1. **缺少迭代限制**: Newton方法可能不收敛或发散，这里没有最大迭代次数限制
2. **震荡**: 在特定条件下（特定文本长度291像素的标题槽），算法在两个状态间无限震荡
3. **退出条件过严**: 只接受 `nc2` 在 `[0, 3*cw)` 范围内

## 修复方案

作者进行了三个防御性修改：

1. **限制迭代次数**: 最多32次迭代，超过限制后如果当前试验适合就接受，否则将 `nuke_count` 加1重试
2. **限制最小nuke_count**: 在循环内部将 `nuke_count` 地板值设为1，防止负修正导致头部尾部重叠的退化字符串
3. **限制最小cw**: 将 `cw` 地板值设为1，防止除零错误和零宽度测量导致的病态行为

## 复现条件

任何WM_NAME足够长以使middle-ellipsis搜索进入overshoot机制的窗口都可以复现：
```
Kickoff.pdf — Introduction to Information Theory Session 1: kickoff & first topic
```
（81个宽字符，包含em-dash，约291像素的边框标题槽，平均字体宽度约3像素/字符）
