## 洞察摘要

2026年4月，Canonical披露了uutils（GNU coreutils的Rust重写版）在Ubuntu 26.04 LTS默认启用前发现了44个CVE漏洞。这次外部审计揭示了一个重要事实：即使使用Rust编写、由经验丰富的开发者完成，这些漏洞也没有被Rust的借用检查器、clippy lint或cargo audit捕获。

本文深入分析了这些漏洞，展示了Rust安全保证的边界，对于编写系统级Rust代码的开发者来说，这是目前最集中的学习资源。

## 主要漏洞类型分析

### 1. 跨系统调用的TOCTOU竞态条件（最大类漏洞）

**问题描述**：对路径执行一次检查，然后执行另一次操作，两次调用之间攻击者可以替换路径组件。

**典型案例 CVE-2026-35355**：
```rust
// 1. 删除目标
fs::remove_file(to)?;
// 2. 创建目标 - 路径在这里被重新解析！
let mut dest = File::create(to)?;
```

攻击者在步骤1和2之间可以将`to`替换为指向`/etc/shadow`的符号链接，从而覆盖特权文件。

**解决规则**：使用文件描述符而非路径进行操作，或使用`OpenOptions::create_new(true)`确保文件不存在。

### 2. 权限设置时机错误

**问题描述**：先创建文件/目录，然后再设置权限，会导致短暂的无权限窗口。

```rust
fs::create_dir(&path)?;
fs::set_permissions(&path, Permissions::from_mode(0o700))?;
// 在此窗口，其他用户可以打开该目录
```

**解决规则**：使用`OpenOptions::mode()`和`DirBuilderExt::mode()`在创建时直接设置正确权限。

### 3. 路径字符串比较不等于文件系统身份

```rust
// 不安全：只比较字符串
if file == Path::new("/") {}
// 可被 /..、/./、符号链接绕过
```

**解决规则**：使用`fs::canonicalize()`解析路径后再比较，或比较`(dev, inode)`对。

### 4. UTF-8边界处理不当

Rust的`String`和`&str`始终是UTF-8，但Unix路径和工具处理的是原始字节。

**案例 CVE-2026-35346（comm命令）**：
```rust
print!("{}", String::from_utf8_lossy(ra)); // 静默损坏非UTF-8数据
```

**解决规则**：对Unix系统代码使用`OsStr`、`Path`、`PathBuf`、`Vec<u8>`或`&[u8]`，避免UTF-8转换。

### 5. Panic即拒绝服务

每个`unwrap`、`expect`、索引操作在处理不受信任的输入时都可能导致进程崩溃。

**案例 CVE-2026-35348（sort --files0-from）**：
```rust
let path = std::str::from_utf8(bytes).expect("无法解析");
```

非UTF-8文件名会导致整个进程崩溃，使cron作业失效。

**解决规则**：将错误转换为错误码而非panic，使用`?`、`get`、`checked_*`等安全方法。

### 6. 错误信息丢失

`chmod -R`和`chown -R`返回最后一个文件的退出码而非最严重的，导致脚本误以为操作成功。

**解决规则**：跟踪所有错误，返回最严重的退出码。

### 7. 与原工具行为差异

**案例 CVE-2026-35369（kill -1）**：
- GNU：`-1`表示"signal 1"
- uutils：将`-1`解释为向PID -1发送信号，Linux上这意味着杀死所有可见进程

**解决规则**：bug-for-bug兼容性是安全功能，行为偏离会导致shell脚本做出错误决策。

### 8. 信任边界跨越前的解析

**CVE-2026-35368（chroot）**：最严重的漏洞，本地root代码执行。

```rust
chroot(new_root)?;
// 仍在攻击者控制的文件系统内
let user = get_user_by_name(name)?; // 加载新root下的共享库
```

**解决规则**：在跨越信任边界前解析所有输入。

## Rust成功防止的漏洞类型

值得注意的是，以下类型的漏洞在Rust版本中完全未出现：
- 无缓冲区溢出
- 无use-after-free
- 无double-free
- 无共享可变状态的数据竞争
- 无空指针解引用
- 无未初始化内存读取

相比之下，GNU coreutils在近年来仍有此类漏洞：
- `pwd`缓冲区溢出（9.11, 2026）
- `numfmt`越界读取（9.9, 2025）
- `unexpand --tabs`堆缓冲区溢出（9.9, 2025）
- `split --line-bytes`堆覆盖（CVE-2024-0684）

## 讨论热点

### 核心观点

1. **缺乏Unix经验**：Rust开发者虽然懂Rust，但对Unix API和陷阱经验不足，导致许多"新手"错误。

2. **重写风险**：rewriting perfectly good code was a colossal mistake，成熟代码包含数十年积累的隐性知识。

3. **Rust定位**：Rust不是银弹，它只能消除某些类型的bug，不能防止所有bug。

4. **API设计**：Rust标准库倾向于使用基于路径的API而非基于句柄的API，这与Unix最佳实践相悖。

### 建议

1. 创建基于文件句柄的封装库
2. 在CI中运行GNU coreutils测试套件
3. 对untrusted input禁用`unwrap`、`expect`
4. 优先使用`openat`系列系统调用
