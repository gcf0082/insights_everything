# 洞察报告：Windows 95 防御机制——阻止安装程序用旧版本覆盖文件

**洞察链接**: https://devblogs.microsoft.com/oldnewthing/20260324-00/?p=112159

**发布日期**: 2026年3月24日

**作者**: Raymond Chen

**分类**: Old New Thing

---

## 摘要

本文讲述了 Windows 95 如何应对安装程序错误地用旧版本文件覆盖系统组件的问题。在 16 位 Windows 时代，许多系统组件是可分发的，程序安装时可以携带这些系统组件的副本并安装到系统中。官方指南要求安装程序在发现系统中已存在同名文件时，应比较版本号，只在待安装文件版本更高时才覆盖。然而，许多安装程序并未遵循这一指南，它们会直接覆盖任何阻挡其安装的文件，将 Windows 95 的组件替换为 Windows 3.1 版本，造成严重问题。

---

## 背景

在 16 位 Windows 时代，系统组件是可分发的。这意味着使用这些组件的程序可以将组件副本随程序一起分发，并在安装时将组件安装到系统中。

**官方安装指南**：
- 如果系统中已存在该系统组件，应比较现有文件的版本号与待安装文件的版本号
- 仅当待安装文件版本号更高时才覆盖现有文件
- 如果现有文件版本号更高，则应保留现有文件

这是因为 Windows 保持向后兼容性，新版本的文件仍可被旧程序正常使用。

---

## 问题

实际上，许多安装程序并不遵循这一指南。它们会直接覆盖任何阻挡其去路的文件，无论现有文件的版本号有多高。当这些安装程序在 Windows 95 上运行时，它们会用 Windows 3.1 版本的组件替换 Windows 95 版本的组件，这给系统带来了巨大的灾难。

---

## Windows 95 的解决方案

### 备份目录策略

Windows 95 采用了一种独特的方法来解决这个问题：

1. **备份目录**：Windows 95 在 `C:\Windows\SYSBCKUP` 隐藏目录中保留了常用被覆盖文件的备份副本

2. **安装后检查**：每当安装程序完成后，Windows 95 会检查这些常用被覆盖的文件是否确实被覆盖

3. **版本比较与恢复**：
   - 如果覆盖文件的版本号高于 SYSBCKUP 目录中的版本，则将新版本复制到 SYSBCKUP 目录保存
   - 如果覆盖文件的版本号**低于** SYSBCKUP 目录中的版本，则将 SYSBCKUP 中的备份复制回原位置，覆盖那个错误的替换

4. **本质**：Windows 95 在每个安装程序完成后"回头检查工作"，修正安装程序造成的错误

### 其他方案及其失败原因

在采用上述方案之前，Windows 95 曾尝试过其他方法，但都失败了：

1. **直接阻止覆盖**：一些早期设计试图直接阻止安装程序覆盖文件，但这带来了更多问题：
   - 某些安装程序会将此视为安装失败并放弃安装
   - 有些会显示错误消息并询问用户如何处理（用户当然不知道该怎么处理）
   - 更极端的安装程序会："好吧，我无法覆盖文件，那就重启系统然后通过批处理文件来覆盖，看你们能不能阻止我"

2. **重定向写入**：将写入重定向到虚拟文件也不行，因为某些安装程序有验证步骤，会检查磁盘上的文件校验和，发现覆盖失败后会报错退出

3. **最终方案**：最有效的方案是允许安装程序覆盖任何它想覆盖的文件，然后由系统"事后清理"造成的混乱

## 额外的行业应对措施

一些组件厂商采取了更激进的策略：他们提供自己的安装程序，并告诉其他安装程序："你们不允许直接安装这些组件文件，必须运行我们的自定义安装程序。是的，这会破坏你们安装程序的 UI，但既然你们安装文件的方式已经证明不可信任，这是你们自己的问题。"

## 评论精选

1. **Alexey Shiryaev**：Windows 95 如何判断这是安装程序而不是普通应用程序复制文件？当时是否已有安装 API？

2. **Joshua Hudson**（回复）：它并没有判断。我曾有一次手动强制降级一个组件（早已忘记原因），不得不与 SYSBACKUP 目录对抗。

3. **Almighty Toomre**：这比"程序可以使用系统组件"还要糟糕！以 WinSock（网络堆栈）为例，外部开发者需要创建自己的网络堆栈版本，这让 Windows 能够支持在 TCP/IP 和以太网最终胜出之前存在的各种网络协议。

4. **Koro Unhallowed**：我记得你不应该自己动手做这件事，而应该调用 VerInstallFile 来完成。我清楚地记得对话框会显示被替换的系统文件的两侧版本号，并询问用户想怎么做。

---

## 结论

Windows 95 的 SYSBCKUP 机制是一个巧妙的后发制人策略，它不阻止安装程序的错误行为，而是在安装完成后自动修复这些问题。这种"事后补救"的方法最终被证明比试图事先阻止更加有效。
