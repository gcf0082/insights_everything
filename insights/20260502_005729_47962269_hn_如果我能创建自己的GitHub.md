## 洞察摘要

本文作者阐述了对理想代码托管平台的愿景，引发了关于现有GitHub工作流程改进方向的广泛讨论。

## 核心主题

### 1. PR审批流程的争论

讨论焦点在于PR审批应该是布尔值（批准/不批准）还是多层级系统：

- **支持布尔值观点**：代码要么可以合并，要么不能，这是本质上的二元决策
- **Gerrit模式**：采用-2到+2的评分系统
  - -2：强烈不建议合并
  - -1：建议改进
  - +1：审核者缺乏足够知识或权限
  - +2：批准合并
- **现实考量**：部分审核者只理解代码的某些部分，但会点击"批准"，导致批准变得毫无意义；多人分工审核可解决此问题

### 2. GitHub替代方案

参与者讨论了多种替代平台：

- **Tangled**：基于jj的版本控制系统，支持stacked PRs，可在树莓派上运行
- **Radicle**：点对点、去中心化的代码托管，支持私有仓库
- **SourceHut**：简洁的git服务，专注于核心功能
- **Gitea**：功能丰富的自托管方案
- **Grasp**：基于Nostr构建

### 3. 离线工作需求

讨论强调PR和issues应该能够离线使用：

- PR审批和评论应该可以在本地进行
- 与代码仓库类似，issues数据也可以存储在git中
- 离线操作可减少等待CI反馈的往返周期

### 4. 私有仓库问题

- 私人项目是许多开发者的核心需求
- 替代方案需支持私有仓库才能真正具备竞争力
- Radicle支持选择性公开仓库
- 托管自建节点是私有仓库的解决方案

### 5. CI/CD可本地运行

- 希望能本地运行Actions获得与CI相同的结果
- 这主要由构建系统的可重现性决定
- 构建系统应该做到： hermetic、跨平台、随处运行

### 6. GitHub的护城河

- GitHub最大的优势是用户基数和网络效应
- 替代方案不仅需要技术上改进，还需足够吸引用户迁移
- 微软对AI的投入可能导致GitHub关注点偏离开发者体验

### 7. 代码审查工具的历史

- Gerrit的评论与代码版本绑定，版本更新后评论自动消失
- Git-appraise使用git notes存储审查数据
- git-bug用git存储bugs
- 这些元数据可以完全存储在git仓库中，增加便携性

---

## 主要结论

1. **PR审批多元化**：多层级审批系统（如Gerrit）比简单布尔值更能反映现实审查场景
2. **去中心化趋势**：离线优先、本地运行的工具正在获得更多关注
3. **集成是关键**：GitHub成功在于将代码托管、代码审查、问题追踪、CI四大功能无缝集成
4. **替代方案仍有机会**：但需在私有仓库支持和生态系统建设上做出突破
