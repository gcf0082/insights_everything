## 原文摘要

Sebastian Raschka 的文章探讨了编码代理（coding agents）的核心组成部分，包括：

1. **实时仓库上下文（Live Repo Context）** - 代理在执行任务前收集项目信息，如Git仓库状态、分支、项目文档等
2. **Prompt塑造与缓存复用** - 构建稳定的prompt前缀，避免每次交互都重新处理工作区摘要
3. **工具访问与使用** - 提供预定义的工具列表，支持文件读取、搜索、执行命令等
4. **上下文精简** - 通过裁剪、摘要和去重管理上下文膨胀
5. **结构化会话内存** - 分离工作内存和完整 transcripts，支持会话恢复
6. **子代理委托** - 支持并行化子任务，但需控制上下文继承和边界

## HN热门评论摘要

### 核心主题：Spec-Driven（规范驱动）vs Chat-Driven（聊天驱动）

**beshrkayali** 提出了一个重要观点：长上下文昂贵且可能引入噪音。他认为规范驱动的代码生成是聊天式编程的对立面。通过Ossature工具，用户编写规范描述行为，系统在写代码前审核规范的一致性和完整性，然后生成构建计划。每个任务声明需要哪些规范部分和上游文件，LLM只看到所需信息。

**hansonkd** 补充道：目前大多数编码代理缺失的是"真实来源的中心"。之前人们对公司正在构建的东西有上下文，现在编码代理每次都重新开始。他认为未来是"chat -> spec -> code"模式，而不是现有的"chat -> code"。

**comboy** 进一步阐述：用户只与修改规范的代理A对话，代理B只根据规范生成代码。他强调"意图是神圣的"，应该始终被存储。

### 关于Harness复杂性的讨论

**armcat** 评论：对LLM周围加上简单状态机并赋予bash访问权限所释放的力量感到难以置信。

**emp17344** 回应：如果你看了Claude Code的泄露，你会发现这个harness绝不简单。它是一个庞大的、迷宫般的系统，但这是让LLM变得确定性和有用所必需的。

**xstas1** 提出一个有趣的假设：它之所以如此庞大，可能是因为它是用Claude Code自己构建的。

**senko** 指出：Claude Code有57个左右的工具，而Pi只有4个约束工具。它确实可以工作。

### 关于模型能力的讨论

**IceWreck** 质疑：作者推测GLM-5可以与GPT-5.4或Claude Opus 4.6媲美，但人们已经这样做了一年多。

**MrScruff** 回复：他在实践中发现，Anthropic的模型确实更好。

**vidarh** 评论：如果说有什么区别的话，harness对于其他模型更重要，因为它们更笨。你可以通过harness来补偿一些愚蠢，但无法完全弥补。

### 关于OpenCode的讨论

**barnabee** 分享：我发现OpenCode比Claude Code好得多，他宁愿购买OpenRouter API积分而不是Claude Max，因为Claude Code不够好。他对OpenCode可以用少量自定义命令完成大部分工作感到惊讶。

### 关于Waterfall（瀑布式开发）的讨论

**dboreham** 简单评论：Waterfall！

**AnimalMupp et** 回应：瀑布式开发有两个问题。首先，如果实现时间过长，世界已经变了而规范没有跟上。其次，规范往往有缺口，只有在尝试实现时才会发现。但对于AI，如果一天或一周就能生成代码，世界没有那么多时间变化。Ossature（或其他工具）会警告缺口而不是编造东西，这不是真正的瀑布式开发，而是规范的迭代开发。

## 关键洞察

1. **规范驱动开发复兴**：越来越多的开发者认识到规范（spec）作为"源代码"的价值，spec -> code的分离可能是未来的方向
2. **Harness的重要性**：文章的推测是，良好的harness设计可能比模型本身更重要——将GLM-5放入类似的harness可能达到GPT-5.4的水平
3. **上下文管理是核心竞争力**：Claude Code有512K LOC，庞大的harness包含工具、技能、上下文管理和子代理，这些都是有原因的
4. **意图存储的重要性**：讨论中反复强调"意图是神圣的"，需要被存储和追踪
5. **开源Harness的可能性**：简单的harness + 好的模型可能就能实现很好的效果
