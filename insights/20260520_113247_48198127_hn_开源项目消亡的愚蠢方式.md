## 核心内容摘要

这篇文章系统性地分类总结了开源项目走向"死亡"（即无人维护、无法使用）的各种方式。作者引用了之前"Weekend at Bernie's"分析，指出最依赖的开源包中有很大一部分已经死亡，并详细阐述了导致这一结果的多条路径。

## 主要分类总结

### 一、维护者离开

1. **幽灵维护者**: 最常见的情况——维护者只是停止回应，项目在沉默中消亡
2. **企业孤儿**: 公司开源后因重组或裁员放弃维护
3. **论文孤儿**: 学生为学术项目创建的代码，毕业后无人接手
4. **资金断崖**: 资助到期后项目无法持续维护
5. **被挖走**: 维护者被公司雇佣后无暇顾及开源项目
6. **继承僵局**: 原维护者失联，接管者无法获得发布权限

### 二、维护者仍存在但项目停滞

1. **倦怠高原**: 偶尔回应但核心问题长期搁置
2. **善意僵尸**: 只有机器人自动更新，没有真正维护
3. **监护权争夺**: 多个共同维护者意见不合导致项目冻结
4. **部落知识消失**: 了解核心逻辑的人已离开
5. **毒性把关**: 维护者对贡献者态度恶劣，驱赶潜在接手人

### 三、破坏和接管

1. **被劫持**: 通过社会工程获取维护权限后植入恶意代码（如xz后门事件）
2. **抗议软件**: 维护者故意破坏自己的包（如colors和faker事件）

### 四、发布流程断裂

1. **维护但不发布**: 代码有更新但无人能发布新版本
2. **无法发布的主分支**: 与上次发布版本差异过大成为破坏性更新
3. **构建考古**: 无法重现构建环境
4. **影子维护**: 实际开发在内部进行，公开发布只是偶尔同步
5. **版本断裂**: 旧版本仍在使用但已无人维护

### 五、不可抗力

1. **制裁滞留**: 维护者因地区制裁无法推送
2. **下架**: 因DMCA或商标纠纷被从注册表中移除

### 六、世界变了

1. **平台淘汰**: 依赖已废弃的运行时
2. **传递性死亡**: 依赖的依赖死了导致项目无法使用
3. **API突变**: 依赖的外部服务关闭或收费
4. **被替代**: 原生语言功能取代了库的需求

### 七、项目分裂

1. **分叉 limbo**: 分歧导致多个分叉，无一明确胜出
2. **许可证陷阱**: 项目改用非开源许可证，社区分叉存在但未统一
3. **开源核心空洞化**: 有趣的开发移到商业版本

## HN 讨论亮点

1. **维护成本问题**: 运行开源项目的实际工作量（问题分类、安全披露、CI、发布周期、依赖维护）远超解决原始问题的工作量

2. **依赖地狱**: 依赖膨胀和依赖腐化使解决方案的持久性降低，零依赖的老项目仍然屹立不倒

3. **GitHub 作为泥潭**: GitHub 已成为开源工作的"陷阱"，尤其对于被"培训"得像企业一样行事的开发者

4. **品牌建设 vs 问题解决**: 过去"开源项目"意味着"我有问题，这是我的解决方案"；现在更多是为了建立个人品牌、展示技能

5. **bot 和 SEO 攻击**: 安全扫描器通过 PR 在 README 中植入徽章链接的现象愈演愈烈

6. **范围蔓延**: 最健谈的用户推动的功蔓延能可能导致项目失去焦点
