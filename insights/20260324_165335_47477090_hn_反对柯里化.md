# 洞察报告：反对柯里化的理由

## 基本信息

- **链接**: https://news.ycombinator.com/item?id=47477090
- **原文标题**: A case against currying
- **原文链接**: https://emi-h.com/articles/a-case-against-currying.html
- **HN发布时间**: 1天前
- **得分**: 111 points
- **评论数**: 129 comments

## 核心观点

### 1. 柯里化是隐式的偏函数应用

在柯里化语言中，代码 `let f = foobinade a b` 无法立即判断 `f` 是 `foobinade` 应用于 `a` 和 `b` 的结果，还是 `foobinade` 部分应用于部分参数后返回的函数。使用显式语法可以立即明确意图：

```haskell
-- 显式语法
let f = foobinade(a, b)  -- 完整应用
let f = foobinade(a, b, $)  -- 部分应用
```

### 2. 更好的错误信息

在柯里化语言中，如果函数被错误地应用于参数不足的情况，编译器无法立即检测到错误。例如，如果 `foobinate` 接受3个参数，`let f = foobinade a b` 不会产生任何错误。而在显式语法中，编译器可以立即检测到缺失的参数。

### 3. 代码可读性问题

除非读者已经记住了函数签名，否则无法区分 `f` 是一个柯里化函数还是一个实际结果。这对代码阅读者不友好，而代码阅读的频率通常高于编写频率。

## 赞成观点

### 1. 函数链式调用的便利性

柯里化的主要好处之一是能够将函数调用链接在一起。使用管道操作符 `|>` 可以使代码更具可读性：

```fsharp
let result =
    input
        |> foobinade a b
        |> barbalyze c d
```

### 2. 函数式编程的理论基础

函数式编程的观点认为，"对 a 和 b 进行 foobinade 操作"本质上就是"将 foobinade 应用于部分参数"。这种区分在理论上是没有意义的。

### 3. 优化可能性

柯里化允许编译器在部分应用函数时进行优化，例如避免每次调用时重复创建闭包。

## 讨论焦点

### 1. Coalton语言移除柯里化

Coalton函数式语言最近移除了柯里化，改用固定元数函数。主要原因：
- 更清晰的类型错误
- 更简单的语义

### 2. 命名参数的讨论

在业务软件中，命名参数通常比位置参数更可取。柯里化和元组参数列表都需要记住参数名称，这增加了认知负担。

### 3. 显式vs隐式

许多开发者认为偏函数应用应该是可选的（opt-in），而不是语言默认行为。

## 总结

这场讨论反映了函数式编程中一个持久的争论：柯里化作为一种理论上的优雅特性，在实践中可能带来可读性和调试方面的挑战。关键在于找到适当的平衡点——既保留偏函数应用的便利性，又通过显式语法提高代码的可理解性。
