## 摘要

这是一篇关于从 Supabase 迁移到 Clerk 再到 Better Auth 的博客文章引发的热烈讨论。讨论的核心问题是：开发者是否应该将用户认证委托给第三方 SaaS 服务，还是选择自托管的开源库方案。

## 主要讨论观点

### 1. 是否应该使用第三方认证服务

**反对方的观点：**
- 认证实际上没有那么复杂，只是比较繁琐
- 使用第三方服务会增加系统的复杂性和故障点
- 第三方服务的可靠性往往不如预期，Clerk 多次出现宕机
- 使用 SaaS 意味着将数据和控制权交给第三方
- B2C 应用通常只需要 Apple 和 Google OAuth，不需要复杂的 SSO

**支持方的观点：**
- SSO、SAML、SCIM、OIDC、OAuth、2FA、密码less等需求会迅速变得复杂
- 企业级应用需要这些功能，维护成本很高
- 专业的认证服务有独立的保险，如果数据泄露可以追责
- 认证虽然不难，但涉及大量重复性工作，可能偏离核心业务

### 2. Better Auth 的优势

Better Auth 作者（Bereket）亲自参与讨论，解释了 Better Auth 的设计理念：
- 用户数据存储在开发者自己的数据库中
- 提供完整的认证功能，包括 OAuth 提供商支持
- 开源且免费，开发者也可以选择付费托管服务
- 高度可定制，可以添加自定义认证方式（如 iframe postMessage）

### 3. Clerk 的问题

多位用户抱怨 Clerk 的问题：
- **可靠性问题**: 多次出现宕机，导致整个应用无法使用
- **功能缺失**: 缺少审计日志和版本控制，无法追踪配置变更
- **RBAC 限制**: 角色绑定到组织而非用户，无法实现全局管理员概念
- **库体积问题**: JavaScript 包体积大，加载缓慢
- **调试困难**: 库代码经过混淆，TypeScript 代码充满难以理解的抽象
- **移动端缺陷**: 仍然无法实现离线认证

### 4. 其他认证方案比较

**自托管方案：**
- Lucia Auth: 已停止维护，转为学习资源
- Django、Laravel、Ruby on Rails 等框架自带认证功能
- authentication-zero (Rails)
- authentication-zero (Elixir/Phoenix)
- Keycloak、Authentik 等开源方案

**SaaS 方案：**
- Auth0: 可靠性较高，但也有人反映问题
- WorkOS AuthKit: 支持企业功能，但 API 不完整
- Logto: 轻量级开源替代方案
- Stack Auth: 存在严格的速率限制和性能问题

### 5. 系统可靠性考量

一位用户引用了一个重要的观点：
> 构建复杂系统的硬性教训是，可靠性是关键组件可靠性的**最小值**。

更准确地说，联合可用性是关键路径中所有组件可用性的**乘积**。如果软件、认证层和云提供商各有 99% 的可用性，最终可用性仅为 97%。

有用户分享了实际经历：CloudFlare 故障导致依赖它的 Auth0 服务也无法使用，整个认证链断裂。

## 总结

这场讨论反映了开发者社区对认证方案的不同看法：

1. **对于简单应用**：使用框架自带的认证功能或开源库即可，无需引入外部依赖

2. **对于企业级应用**：第三方服务可能更合适，但要注意供应商锁定和数据控制权问题

3. **Better Auth 的定位**：介于完全自托管和 SaaS 之间，提供开源库的灵活性同时拥有完整的认证功能

4. **核心考量因素**：可靠性、数据控制权、成本、维护复杂度、功能需求

无论选择哪种方案，关键在于减少系统中的组件数量，选择可靠的解决方案，避免不必要的复杂性。
