## 一、核心问题概述

Async Rust 承诺提供零成本抽象（zero cost abstractions），但在嵌入式系统中这一问题尤为明显——每个字节的二进制大小都很重要，而 async 引入了大量代码膨胀（bloat）。虽然这个问题在桌面和服务器上不那么明显（因为有更多内存和计算资源），但在嵌入式环境中却非常突出。

作者在本文中深入分析了 async 生成的 MIR（中间表示），探讨了编译器层面的优化机会，并提交了一个 Rust Project Goal 来推动这些改进。

## 二、生成的_future_结构解析

### 2.1 状态机构成

以一个简单的异步函数为例：

```rust
fn bar() -> impl Future<Output = i32> {
    async {
        foo().await + foo().await
    }
}
```

该函数生成了 360 行 MIR（中间表示）代码，而同步版本仅需 23 行。生成的状态机包含以下状态：

- **Unresumed (0)**：初始状态
- **Returned (1)**：完成后返回 Ready 的状态
- **Panicked (2)**：处理 panic 后的状态
- **Suspend0 (3)**：第一个 await 点，包含第一个 foo future
- **Suspend1 (4)**：第二个 await 点，包含第一个 foo 的结果和第二个 foo future

### 2.2 Returned 和 Panicked 状态的问题

**关键发现**：Future 在 `Returned` 状态被再次轮询时会 panic，但这其实是不必要的。

- `Future::poll` 是一个安全函数，调用它不应导致任何未定义行为（UB）
- 即使 future 已完成，再次轮询也应该是安全的
- 当前的实现选择在完成后 panic，而非简单返回 `Pending`
- 这种设计引入了额外的代码路径，阻碍了 LLVM 的优化

### 2.3 无 await 时的状态机膨胀

对于一个没有任何 await 的简单 async 函数：

```rust
fn foo() -> impl Future<Output = i32> {
    async { 5 }
}
```

编译器仍然生成了完整的状态机（包含 Unresumed、Returned、Panicked 三个状态），这完全是多余的。一个理想的实现应该无需任何状态，每次轮询直接返回 `Poll::Ready(5)`。

## 三、编译器优化机会

### 3.1 用 Poll::Pending 替代 panic

**优化方案**：在 release 构建中，将 Returned 状态的 panic 行为替换为返回 `Poll::Pending`。

**测试结果**：2%-5% 的二进制体积缩减（在嵌入式固件上）

这类似于 `overflow-checks = false` 的设计——debug 构建保留 panic 以便立即发现错误，release 构建则获得更小的体积。

当使用 `panic=abort` 时，甚至可以完全移除 Panicked 状态。

### 3.2 无 await 时无需状态机

**优化方案**：如果 async 块没有任何 await，则不生成状态机，直接返回 ready。

**测试结果**：0.2% 的二进制体积缩减

虽然优化幅度不大，但这是一个相对简单的优化，值得实现。

### 3.3 Future 内联

**关键问题**：生成的 Rust futures 从不被内联，每次 future 实现后，LLVM 和链接器才有内联机会，但为时已晚。

**优化机会**：对于以下常见模式：

```rust
async fn foo(blah: SomeType) -> OtherType { ... }
async fn bar(blah: SomeType) -> OtherType {
    foo(blah).await
}
```

当前的编译器会生成 `bar` 的状态机来调用 `foo` 的状态机，这是极大的浪费。其实 `bar` 可以直接返回 `foo` future。

### 3.4 状态折叠

对于包含相同代码路径的 async 函数，可以将重复状态合并为单一状态。

**示例**：优化前的代码生成 456 行 MIR，优化后仅需 302 行。

```rust
// 优化前（生成重复状态）
pub async fn process_command() {
    match get_command() {
        CommandId::A => send_response(123).await,
        CommandId::B => send_response(456).await,
    }
}

// 优化后（无重复状态）
pub async fn process_command() {
    let response = match get_command() {
        CommandId::A => 123,
        CommandId::B => 456,
    };
    send_response(response).await;
}
```

## 四、测试结果汇总

| 优化项 | 二进制体积节省 |
|--------|-------------|
| Returned 状态不 panic | 2-5% |
| 无 await 则无状态机 | 0.2% |
| 两项结合 | 约 3% 性能提升（x86 synthetic benchmark）|

Future 内联应有更大效果，但需在实际系统中 benchmark 才能确认。

## 五、作者提议的编译器改进

作者希望在 Rust 编译器中实现以下改进：

1. **Release 模式下 Returned 状态不再 panic**：改为返回 Poll::Pending
2. **无 await 的 async 块不生成状态机**：直接返回 ready
3. **Future 内联**：支持单 await 点的 futures 内联
4. **状态折叠**：合并相同状态

## 七、总结

Async Rust 虽然是一个非常强大的特性，但距离其承诺的零成本抽象还有很大的改进空间。本文揭示了编译器在生成状态机时存在多个优化机会：

1. 不必要 panic 行为导致的代码膨胀
2. 无 await 时的状态机过度生成
3. Future 无法内联的架构限制
4. 重复状态的缺失优化

这些问题在嵌入式系统中尤为突出，但也会影响桌面和服务器应用。作者正在寻求资金支持来完成这些编译器改进，预计 €30k 可以完成大部分工作。
