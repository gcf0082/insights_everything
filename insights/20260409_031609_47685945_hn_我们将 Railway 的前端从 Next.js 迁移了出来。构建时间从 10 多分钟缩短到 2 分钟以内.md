# 洞察报告：Railway 将前端从 Next.js 迁移出来

## 基本信息

- **洞察链接**: https://news.ycombinator.com/item?id=47685945
- **原文标题**: We moved Railway's frontend off Next.js. Builds went from 10+ mins to under 2
- **原文链接**: https://blog.railway.com/p/moving-railways-frontend-off-nextjs
- **发布平台**: Hacker News
- **发布时间**: 13小时前
- **得分**: 123 points
- **评论数**: 107 comments

## 核心内容

Railway 团队将其前端从 Next.js 迁移出来，构建时间从超过10分钟降至不到2分钟。

## 主要讨论观点

### 1. 性能问题的直接证据
- Railway 的 domains 页面加载数据量达10.8MB
- DOM 准备就绪需要2秒
- 完整渲染需要17秒
- 页面存在大量未优化的图片（约占数据量的一半）
- 背景PNG图片达3.57MB，可转换为AVIF格式节省3MB

### 2. 框架选择讨论
- **推荐尝试Rails**：有评论建议Railway考虑使用Rails
- **HTMX + Django方案**：用户回归Django + Wagtail + HTMX，从1700个依赖降至65个
- **TanStack Start**：多个用户提到从Next.js迁移到Vite + TanStack Router
- **Astro**：用户从Next.js迁移到Astro，主要因为Vercel托管静态页面费用较高

### 3. 构建工具优化
- **Turbopack**：部分用户指出使用Turbopack可将构建时间从6分钟降至2分钟
- 有人认为Railway可能跳过了Turbopack直接迁移到Vite
- Webpack → Turbopack 被认为是更明智的初步迁移

### 4. 竞争关系提示
- Next.js由Vercel开发，是Railway的竞争对手
- 有用户认为这是基本的免责声明
- 虽然Next.js是开源项目，但其设计目的是引导开发者使用Vercel

### 5. 前端过度工程化反思
- **web dev is a sewer**：用简单的服务器端渲染（Jinja/minijinja + Bootstrap + jQuery + HTMX）无需构建步骤
- **回归传统**：用户怀念HTML/jQuery/handlebars的时代
- **Blazor方案**：C#的Blazor templating可以实现无需JS逻辑的纯服务端渲染
- 现代Web开发"像飞蛾扑火一样追求复杂性"

### 6. 托管选择
- 使用$5/月的Ubuntu/Debian VPS + nginx
- 维护成本包括：TLS证书、SSH密钥、安全更新、监控等
- 使用Hetzner + Claude AI进行管理

### 7. 编译时间对比
- 有人指出构建Web前端比编译Linux内核花费的时间还长
- 7分钟的Next.js构建时间被认为是"荒谬的"

## 总结

本次讨论反映了开发者社区对现代前端框架（特别是Next.js）构建时间过长和膨胀问题的广泛关注。多位开发者分享了从Next.js迁移到更轻量级方案（Vite、TanStack、Astro、HTMX+Django等）的经验，整体构建时间显著改善。讨论也涉及了Vercel与Railway之间的竞争关系，以及前端开发过度工程化的趋势。
