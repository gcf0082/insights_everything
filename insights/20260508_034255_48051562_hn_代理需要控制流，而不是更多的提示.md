## 核心观点总结

本文的核心论点是：**可靠的智能体（agents）需要确定性的控制流编码在软件中，而不是越来越复杂的提示链**。

### 核心问题

作者指出，如果你已经不得不使用"MANDATORY"或"DO NOT SKIP"这样的措辞，说明你已经触碰到了提示工程的瓶颈。想象一种编程语言，其中语句只是"建议"，函数返回"成功"的同时却在"幻觉"——随着复杂性增加，推理变得不可能，可靠性也会崩溃。

### 关键论据

1. **提示链缺乏递归可组合性**：软件通过库、模块和函数实现递归可组合性，而提示链不具备这种特性。代码暴露可预测的行为，支持局部推理；提示则是非确定性的、弱规范的、难以验证的。

2. **可靠性需要确定性脚手架**：将逻辑从 prose 文本转移到运行时需要确定性脚手架——明确的状态转换和验证检查点，将 LLM 视为组件而非系统本身。

3. **错误检测的必要性**：在容易发生静默失败的系统，没有激进错误检测的智能体只是在快速得出错误结论。

## 社区讨论要点

### 1. 核心解决方案：LLM 生成确定性代码

社区高度认同的核心思路是：**使用 LLM 生成确定性代码，而非让 LLM 直接执行任务**。

- 用户 **rnxrx** 指出，应该让 LLM 生成可重复、可验证的确定性脚本，而非让 LLM 本身处理任务。
- 用户 **bwestergard** 强调，当提示达到极限时，需要从运行时使用 LLM 完成任务转向使用 LLM 编写软件来完成任务。

### 2. 架构模式：确定性流程 → 非确定性决策 → 确定性工具

用户 **vishvananda** 提出了一个强大的模式：

```
确定性智能体流 → 非确定性决策 → 确定性工具
```

这种模式将非确定性决策 sandwich 在确定性层之间。

### 3. 验证层的重要性

多个用户强调了验证层的关键作用：

- 用户 **tim-projects** 表示："实现质量控制门后，一切都运作得更好。"
- 用户 **arian_** 简洁地总结："控制流告诉智能体允许做什么，但不能告诉你智能体实际做了什么。两者都很重要。每个人都在构建权限层，几乎没人构建验证层。"

### 4. 具体实践建议

- **chrismarlow9** 建议在提示中添加："遇到任何模糊的边缘情况时，请咨询我"
- **QuercusMax** 让 Claude 为自己编写 shell 脚本来处理常见案例
- **dnautics** 建议 TDD 执行器模式：第一阶段只允许修改测试文件，第二阶段只允许修改代码文件

### 5. 工具推荐

社区提到了几个有助于构建确定性控制流的工具和框架：

- DSPy 和 PydanticAI：允许在程序的控制流中结构化 LLM 调用
- Langgraph、BAML
- AgentScript（Salesforce）

## 结论

这篇讨论揭示了一个重要的共识：**与其试图让 LLM 变得确定性，不如围绕 LLM 构建确定性系统**。核心思路是：

1. 使用 LLM 生成确定性代码（脚本、工具）
2. 将 LLM 作为组件嵌入到确定性框架中
3. 构建强有力的验证层来检测错误
4. 在关键决策点保持人类在环

这实际上是在用"老式编程"的思维方式来驾驭新的人工智能技术。
