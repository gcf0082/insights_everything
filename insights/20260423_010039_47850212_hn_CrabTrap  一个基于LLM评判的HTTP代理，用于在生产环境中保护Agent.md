## 内容摘要

CrabTrap是Brex团队开源的一个HTTP/HTTPS代理，专门用于在生产环境中保护AI代理(agents)的安全。它通过拦截AI代理的所有出站请求，并使用LLM-as-a-judge(LLM作为裁判)的方式来判断请求是否符合安全策略。这是针对AI代理安全问题的创新解决方案，旨在防止数据泄露和未授权的外部API调用。

## 技术架构

### 1. 核心工作原理
- 设置HTTP_PROXY和HTTPS_PROXY环境变量，所有出站请求通过CrabTrap路由
- 无需SDK或包装器，透明代理，对代理无侵入
- 对于HTTPS，CrabTrap执行TLS拦截：生成自签名证书，解密后再建立上游连接

### 2. 两层安全检查
**静态规则层**：URL模式匹配（前缀、精确或glob）
- 规则匹配时立即决策，无需LLM调用
- 拒绝规则优先于允许规则

**LLM裁判层**：静态规则不匹配时使用
- 接收完整请求上下文和自然语言安全策略
- 返回结构化JSON决策（ALLOW或DENY）及原因

### 3. 安全特性
- SSRF保护：阻止私有网络请求（RFC 1918、环回链路等）
- DNS重绑定防护
- 提示注入防御：请求载荷JSON编码，策略内容JSON转义
- 每IP速率限制：令牌桶算法（默认50 req/s，突发100）
- 完整的审计日志（PostgreSQL）

## 讨论要点

### 1. LLM-as-judge的安全性
- 讨论认为"LLM作为裁判是错误的安全原语"
- 提示注入链终止于出站POST，凭证在裁判看到请求时已被读取
- 在传输层放置裁判更适合作为审计层而非执行层
- 内核级控制告诉你代理做了什么，代理告诉你代理试图泄露什么

### 2. 检测与预防的区别
- 结构化JSON转义和头部限制适合检测
- 但不适合预防——不同层不同问题
- 裁判只看到HTTP body，可控制请求的攻击者也可以塑造裁判的上下文窗口

### 3. 延迟vs安全权衡
- 裁判在每次调用运行还是仅在触发确定性策略后运行？
- 在支付/广告领域，误批准电话的影响范围足够大
- "另一个LLM说OK"感觉像是用一个黑箱换两个黑箱

### 4. 混合方案
- 静态规则层 + LLM裁判层的组合
- 先用静态规则分析，疑似问题再交给LLM裁判
- 避免每个请求都经过LLM（成本高）
- 适合任务关键领域（医疗、国防）

### 5. LLM的局限性
- LLM本质是非静态的：无确定性输出，无法像传统函数那样验证
- 添加另一层LLM只是增加另一层"瑞士奶酪模型"，祈祷孔不对齐
- 无法事前预测LLM是否能通过

### 6. 深度防御
- 分层不会固有地使事物更不安全，通常使其更安全
- 但危险在于可能给人虚假的完全安全感
- 安全层是概率性而非确定性的

## 关键观点

1. **代理安全很重要**：AI代理可以调用外部服务（Slack、Gmail、GitHub等），需要护栏

2. **透明代理优势**：无需修改代理代码，通过环境变量即可使用

3. **写入操作安全**：没有代理可以通过LLM策略裁判修改外部系统

4. **数据泄露防护**：所有出站流量可见且可控

5. **审计追踪**：每个请求和决策都记录到PostgreSQL

## 技术细节

### 配置示例
```yaml
approval:
  mode: llm

llm_judge:
  enabled: true
  provider: bedrock-anthropic
  eval_model: us.anthropic.claude-3-5-haiku-20241022-v1:0
  aws_region: us-west-2

database:
  url: postgres://crabtrap:secret@postgres:5432/crabtrap
```

### 部署方式
- Docker快速启动
- 自动生成CA证书
- 代理默认端口8080，管理UI端口8081

### 安全头部处理
- 安全相关头部优先处理
- 总头部内容限制4KB，防止提示膨胀攻击
- 请求body截断16KB并明确警告模型
- 多部分请求替换为结构化摘要

## 总结

CrabTrap代表了AI代理安全领域的一个重要创新，通过HTTP代理层 + LLM裁判的组合来保护生产环境中的AI代理。虽然讨论中对其作为唯一安全原语的有效性存在质疑，但作为深度防御的一层，结合静态规则和速率限制等传统安全措施，可以显著提高AI代理系统的安全性。
