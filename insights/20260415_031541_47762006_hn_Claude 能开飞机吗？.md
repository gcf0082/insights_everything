## 摘要

本讨论探讨了是否可以借助 Claude（大型语言模型）来驾驶飞机。作者通过一个实验，让 Claude 控制飞行模拟器，观察其能否完成飞行任务。讨论的焦点不仅是技术可行性，更涉及自动化与人机协作的根本性问题。

---

## 主要观点

### 1. LLM 的技术局限

- **延迟问题**: LLM 的 token 生成速度远低于实时控制所需的响应时间，正常飞行需要毫秒级响应，但 LLM 只能做到每秒两位数的 token 输出
- **不适合作为驾驶模型**: 输出 token 并期望在正确时间采取行动的延迟令人生畏，理想的飞行控制系统应该是输入神经元直接获取飞机所有数据，输出神经元直接控制所有动作
- **控制回路问题**: 实验中 Claude 使用纯 P（比例）控制回路而非 PID 回路，这是一个糟糕的选择

### 2. 自动化 vs 人类飞行员

- **人类仍然是必要的**: 飞行员需要保持在决策循环中，处理自动驾驶系统无法应对的意外情况，如风切变、跑道被占、系统故障等
- **Sully Sullenberger 的观点**: "无论你是手动驾驶还是使用技术辅助，你最终都是通过脑海中开发和维护一个准确的实时模型来驾驶飞机——这包括飞机本身、环境和态势"
- **问责问题**: 机器人无法承担责任，而人类可以，这是航空业坚持人类驾驶员的核心原因

### 3. 自动驾驶系统的现状

- **现有技术已足够成熟**: 自动驾驶系统早已能够驾驶商业客机，甚至可以完成自动着陆（Autoland），自1970年代就已存在
- **LLM 在航空领域的角色定位**: 大多数飞行员认为，应该让传统的自动驾驶系统做它擅长的工作，而 LLM 可以用于识别意外情况作为警告系统
- **安全冗余**: 人的价值在于作为隐式的"安全担保"，如果飞行不安全，飞行员不会每天都冒险

### 4. 类比其他自动化领域

- **与自动驾驶汽车的比较**: 同理，人们对 Waymo 撞猫事件的反应表明，一次事故就可能让公众失去对自动驾驶的信任
- **与 vibe coding 的比较**: 有评论指出，希望"vibe coding"爱好者也在听这个讨论——不要过度依赖不了解的自动化
- **道德困境**: 有人提到"某处某人正在用它来决定谁生死"，那为什么不应该把它连接到飞行模拟器上呢？

---

## 结论

实验表明 Claude 无法在实时环境中成功驾驶飞机。主要原因不是智力问题，而是 LLM 的输入输出延迟无法满足实时控制的需求。更重要的是，这引发了一个根本性问题：无论如何训练，LLM 都不应该是关键安全决策系统的核心。航空业的经验表明，人类飞行员作为最终决策者和安全冗余的角色是不可或缺的。
