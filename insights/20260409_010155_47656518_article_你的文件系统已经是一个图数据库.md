## 核心观点

作者认为，文件系统配合 Markdown 和维基链接（wikilinks）本身就是一套图数据库系统：

- **文件即节点**：每个 Markdown 文件是一个节点
- **维基链接即边**：`[[target]]` 语法创建语义关联
- **文件夹即分类**：采用 PARA 分类体系组织文件
- **LLM 即查询引擎**：大语言模型可以自然语言方式遍历和理解这个"图数据库"

## 核心价值

### 解决什么问题

- **痛点**：工作中需要从 Slack、文档、邮件、会议记录中重新组装上下文，耗时且容易遗漏
- **方案**：将所有知识和上下文存储为互联的 Markdown 文件，形成可检索的知识库
- **效果**：编写设计文档、性能分析、入职指南等产出物时，直接从知识库获取完整上下文

### 架构设计

采用 Tiago Forte 的 **PARA** 分类体系：

```
/projects/{name}     # 项目文件夹
/areas/{topics}      # 领域/主题
/people/{slack_handle}  # 人员
/daily/{year}/{month}/{day}/   # 日记
/meetings/{year}/{month}/{day}/ # 会议记录
```

每个文件通过维基链接相互关联，形成图结构。LLM 可以自然语言查询这个结构。

### 实际工作流程

1. **会后**：AI Agent 自动创建会议笔记，链接到相关人员和项目
2. **积累**：每个人的笔记成为时间线，每个项目积累所有相关材料
3. **创作**：让 Agent 根据项目文件夹中的完整上下文起草文档，产出质量远高于冷启动提示

### 与单纯使用 LLM 的区别

| 维度 | 单纯使用 LLM | 使用知识库 + LLM |
|------|-------------|------------------|
| 上下文 | 无 / 需要手动提供 | 自动包含完整历史 |
| 产出质量 | 一般 | 根据真实项目历史定制 |
| 效率 | 每次重复描述背景 | 直接继承历史上下文 |

## 当前挑战

**自动化收件箱处理** 尚未完全解决：

- 各种来源的內容（网页剪藏、会议笔记、Slack 保存）需要进入收件箱
- LLM 擅长总结和分类，但定义"已处理"的统一标准很困难
- 太 rigid 则导致知识库 drift into chaos，太 loose 则无法保持一致性

## 起步建议

1. **创建文件夹结构**：projects、areas、people、daily，即使空着也能提供分类框架
2. **会后链接**：每次会议后让 Agent 创建笔记并链接到相关人员和项目
3. **基于上下文创作**：下次需要写文档时，让 Agent 基于相关文件夹中的内容起草

## 核心洞见

> 知识库不是为建 wiki 而建 wiki，而是构建一个输入层，让未来每一次 LLM 交互都变得更好。每次会议笔记、每个关联决策、每个归档材料都会提升后续查询的质量。
>
> 工作会不断累积，这才是真正令人兴奋的新事物。
