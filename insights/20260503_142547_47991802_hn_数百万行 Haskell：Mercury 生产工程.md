## 摘要

本文介绍了 Mercury（一家金融科技公司）如何在生产环境中使用 Haskell 构建了数百万行代码的系统。Mercury 是一家处理巨额资金的银行，传统观点认为使用 Haskell 这样的小众语言应该是一场灾难，但事实上系统运行良好，经历了超增长、SVB 危机（5天内收到 20 亿美元新存款）、监管审查等考验。文章强调了"天生可观测"（observable by construction）的设计原则，以及通过类型系统编码业务规则的重要性。

## 主要讨论观点

### 1. Haskell vs Rust 生产力对比

一位用户分享了个人体会：在 Haskell 工作几年后，在 Rust 中的生产力仍是 Haskell 的两倍。Haskell 存在一些容易踩坑的地方，语言有时近乎"只能写"的特性，工具链与 Nix 紧密耦合，语言扩展到处都是，编译错误也需要适应。

### 2. 类型驱动的安全模式

许多评论者高度赞扬了**User -> LoggedInUser -> AccessControlledLoggedInUser** 这种类型化流程，认为这种模式可以有效防止 Web 应用中的认证授权 bug。评论者指出这种模式在 Rust 和 TypeScript 中也能很好地实现，但在行业中严重未被充分利用。

### 3. Mercury 产品体验

多位用户分享了对 Mercury 产品的积极评价：
- 可以创建数百张有限额的虚拟借记卡
- 多账户管理，自动规则拉取资金
- 出色的财务可视化，了解每笔支出去向
- API 支持完整，包括 MCP
- 免费商业账户

### 4. Haskell 招聘策略

评论者讨论了招聘策略：
- 雇佣没有 Haskell 经验的学习者有助于从零建立团队文化
- 使用 Haskell 作为招聘过滤器，吸引高质量候选人
- 需要过滤那些只喜欢"摆弄"Haskell 而不愿意构建实际产品的人

### 5. 关于是否因为 Haskell 成功

存在质疑：Mercury 的成功是否真的是因为 Haskell？评论者认为：
- 无法在没有 Haskell 的情况下运行公司两次来对比
- 使用 Haskell 的人本身可能就更具技术热情和动力
- 良好的工程文化可能是更重要的因素

### 6. Option Types vs Untagged Unions

讨论了类型设计的不同方式：
- Haskell 的 Maybe 类型要求在每个调用点修改代码
- TypeScript/Scala 的 untagged unions 可以保持调用点不变
- 但有人认为为了这个便利付出 null 的代价太高

### 7. 构建挑战

关于 Haskell 构建的挑战：
- 跨编译困难
- 静态链接二进制文件是挑战
- 构建缓存依赖 Docker
- 编译速度慢，生成的二进制文件大
