## 核心观点

作者认为在接手新代码库时，首先不是打开代码，而是运行一系列Git命令来诊断项目状态。提交历史可以反映出：谁构建了项目、问题集中在哪里、团队是在信心满满地发布还是在小心翼翼地避坑。

---

## 洞察一：什么文件变化最频繁

```bash
git log --format=format: --name-only --since="1 year ago" | sort | uniq -c | sort -nr | head -20
```

**用途**：列出过去一年中变更最多的20个文件。

**解读**：
- 文件变更频繁不一定代表它有问题，可能只是活跃开发
- 但如果一个文件频繁变更却没人愿意维护，这就是最清晰的代码库拖累信号
- 这种文件每次修改都是补丁摞补丁，小改动的爆炸半径不可预测
- 2005年微软研究显示，基于变更量的指标比复杂度的指标更能预测缺陷

**建议**：将前5个高变更文件与bug热点命令交叉参考，既高变更又高bug的文件是最大的风险点。

---

## 洞察二：谁构建了这个项目

```bash
git shortlog -sn --no-merges
```

**用途**：列出所有贡献者，按提交数量排序。

**解读**：
- 如果一个人贡献了60%以上的代码，这就是"巴士因子"问题
- 如果这个人已经离职六个月，那就是危机信号
- 如果总体贡献榜的第一名在近6个月没有出现（`git shortlog -sn --no-merges --since="6 months ago"`），需要立即标记给客户

**注意**：如果团队使用squash-merge工作流，这个输出反映的是谁合并了PR，而不是谁写了代码。

## 洞察三：Bug集中在哪些文件

```bash
git log -i -E --grep="fix|bug|broken" --name-only --format='' | sort | uniq -c | sort -nr | head -20
```

**用途**：找出包含bug相关关键词的提交，列出这些提交中变更最多的文件。

**解读**：
- 将这个列表与变更热点列表对比
- 同时出现在两个列表中的文件是最高风险代码：它们不断出问题，不断被打补丁，但从未被真正修复
- 这依赖于团队的提交信息规范程度

---

## 洞察四：项目是在加速还是衰败

```bash
git log --format='%ad' --date=format:'%Y-%m' | sort | uniq -c
```

**用途**：按月统计提交数量，显示整个仓库的历史。

**解读**：
- 稳定的节奏是健康的
- 某月提交数突然减半，通常有人离职
- 6到12个月内持续下降的曲线说明团队正在失去动力
- 周期性高峰然后平静说明团队是批量发布而不是持续交付

**案例**：作者曾向一位CTO展示提交速度图，CTO说"那是我们失去第二位高级工程师的时候"。这是团队数据，不是代码数据。

---

## 洞察五：团队有多频繁在救火

```bash
git log --oneline --since="1 year ago" | grep -iE 'revert|hotfix|emergency|rollback'
```

**用途**：查找revert和hotfix的频率。

**解读**：
- 一年中少数几次是正常的
- 每几周都有revert说明团队不信任部署流程
- 这反映出更深层的问题：不可靠的测试、缺失的staging环境、或使回滚比预期困难的部署管道
- 零结果也是信号：要么团队稳定，要么没人写描述性的提交信息

---

## 总结

这五个命令只需几分钟运行。它们不能告诉你一切，但你会知道应该先读什么代码，以及读的时候要找什么。这就是花第一天有条理地阅读代码库和花第一天到处闲逛的区别。

这是作者进行代码库审计时的第一个小时的做法。
