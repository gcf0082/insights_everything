## 概述

OpenSSL 4.0.0 版本发布，成为本周Hacker News最热门的话题之一。该版本带来了多项重要更新，其中最受关注的是对 Encrypted Client Hello (ECH) 的支持。

## 主要功能更新

### 1. Encrypted Client Hello (ECH) 支持

这是本次更新最受关注的功能。ECH 是 RFC 9849 定义的标准，可以加密 TLS 握手中的 SNI（Server Name Indication）信息，保护用户访问的域名隐私。

**当前支持状态**:
- **Cloudflare**: 自2023年起支持
- **Chrome**: 支持
- **Firefox**: 119版本起默认启用
- **Safari**: 尚未在生产环境默认启用，仅在 iOS/macOS 有实验性开关
- **Nginx**: 1.29.x 和 1.30.x 版本支持

### 2. QUIC 支持改进

OpenSSL 3.4.1 终于提供了独立的 QUIC API，允许开发者使用第三方 QUIC 实现而无需依赖 OpenSSL 内置的 QUIC 堆栈。这对 HTTP/3 实现来说是重要进步。

## 社区讨论焦点

### ECH 的实用性争议

关于 ECH 是否真正保护隐私，社区存在激烈讨论：

**支持方观点**:
- ECH 可以防止网络观察者看到用户访问的具体域名
- 结合 DOH (DNS over HTTPS) 可以提供更好的隐私保护
- 对于使用 CDN 的网站（如 Cloudflare）特别有价值

**质疑方观点**:
- ECH 无法隐藏服务器 IP 地址，对于独立服务器意义有限
- 只有当服务器与其他大量用户共享 IP 时（即大型 CDN）才有意义
- 俄罗斯已经封锁了 Cloudflare 的 ECH 连接

### OpenSSL 质量担忧

多位评论者表达了对 OpenSSL 现状的担忧：

**性能问题**:
- OpenSSL 3.x 相比 1.1.1 版本性能显著下降（10%-99%）
- OSSL_PARAM 架构导致性能损失和代码可读性下降

**测试问题**:
- OpenSSL CI 测试极其不稳定
- 3.0.4 版本的关键 RSA 溢出漏洞实际上被 CI 捕获，但因为只在特定 CPU 上触发被误判为测试不稳定而忽略

**开发者体验**:
- 设置 Ed25519 签名上下文等操作变得极其复杂
- 需要构造 OSSL_PARAM 数组，查找 provider，编译时无法验证

**项目动态**:
- Python cryptography 库正在考虑弃用 OpenSSL
- Node.js 工作组也在讨论迁移到 BoringSSL

## 升级注意事项

- 从 3.x 升级到 4.0 预计比当年从 1.x 升级到 3.x 更为平滑
- Engines 支持被移除，但大多数依赖已在 Fedora 等发行版中迁移
- 主要 Linux 发行版的包维护者需要时间更新到新版本

## 总结

OpenSSL 4.0.0 带来了重要的安全功能（ECH 支持），但在软件质量和性能方面仍面临社区质疑。ECH 的广泛采用仍需要时间，主要浏览器和服务器的支持正在逐步推进，但企业网络和部分国家的封锁可能成为障碍。
