## 摘要

Terminal User Interfaces (TUI) 正在卷土重来。本文探讨了 TUI 复兴的原因，包括跨平台开发的困难、原生 GUI 框架的碎片化、以及 AI 编码助手的兴起。

---

## 核心洞察

### 1. 原生 GUI 开发陷入困境

**Windows**: 微软从未有过连贯的 GUI 策略。从 MFC 到 WinForms、WPF、Silverlight、WinUI 到 MAUI，每一个新框架都在失败后被新的取代。

**Linux**: GTK 和 Qt 是两大主导框架，但它们主要在 Linux 上使用。不同的发行版、桌面环境和硬件组合使得测试变得几乎不可能，大多数公司选择 Electron 或放弃 Linux 原生应用。

**macOS**: 苹果曾是人机界面指南的代名词，如今却在破坏自己曾经坚持的设计一致性——忽略费茨定律、让窗口调整变得几乎不可能、给每个菜单添加图标。

### 2. Electron 并非完美解决方案

- 内存消耗虽然在过去十年有所下降
- 视觉一致性缺失
- 缺乏键盘驱动的工作流
- 沙盒内的 HTML 无法访问系统菜单
- 开发者逐渐忘记添加菜单来执行应用中可用的操作

### 3. TUI 的优势

- **速度快**: 启动快、运行快、使用快
- **易于自动化**: 可远程运行，无需 X forwarding
- **跨平台一致性好**: 在不同操作系统上表现一致
- **信息密度高**: 没有现代 GUI 设计中常见的过度留白
- **适合键盘操作**: 无需鼠标切换
- **AI 友好**: AI 可以快速生成 TUI 代码

### 4. AI 驱动的 TUI 热潮

- Claude Code 和 Codex 的成功推动了 TUI 的流行
- AI 可以快速生成可用的 TUI，而生成 GUI 需要更多时间和精力
- Go + Lipgloss + Bubbletea、Rust 的 ratatui 等库使得构建 TUI 变得容易
- 现代语言（Rust、Go）使得构建 TUI 变得非常简单

### 5. 设计师/产品经理的困境

TUI 强制开发者仔细设计界面：
- 80 列文本空间没有地方添加大量留白来实现"时尚"和"现代"外观
- 产品经理几乎没有可以"精简"的东西

## 社区讨论精选

### 赞成 TUI 的观点

1. **Cyberpunk 美学**: 使用 TUI 让自己看起来像"黑客"，即使只是在"vibe coding"

2. **效率优先**: 使用 tmux 窗格可以同时运行多个 TUI 和终端，而 VSCode 无法同时打开数十个实例

3. **开发速度**: 用 Go + lipgloss + bubble tea 可以通过单个 prompt 在几分钟内生成所需的 TUI

4. **远程工作**: TUI 非常适合低摩擦的远程工作，通过 SSH 部署应用无需用户安装

5. **资源占用**: TUI 内存占用极小，16GB 内存的用户避免 Electron 应用

### 反对/质疑的观点

1. **TUI 并非总是更快**: 一些新的 TUI 需要四秒才能启动，性能并不总是优于 GUI

2. **并非总是更简单**: GUI 工具包默认配置下可以非常快速地启动小型应用

3. **Electron 本身不是问题**: "内存消耗过大"是糟糕软件的错，而非 Electron 的错

4. **这只是时尚**: 一位评论者认为 TUI 的兴起主要是由 Claude Code 驱动，但这可能只是短期趋势

5. **未来预测**: 2026 年下半年形势可能会转变，SwiftUI 等现代框架使原生 UI 开发变得更加简单

### 有趣的观点

1. **JIS 键盘**: 日本键盘布局的小空格杆更舒适，Henkan 和 Muhenkan 键可以绑定为 Control 键

2. **Vim 的学习曲线**: Vim 唯一的难点是 reaching for Escape，将 CapsLock 重新映射为 Escape 可以解决这个问题

3. **Claude Code 近乎 Electron + TUI**: 有评论指出 Claude Code 几乎是 Electron 应用实现了 TUI 层

## 结论

TUI 的回归反映了现代应用开发的几个现实问题：
1. 原生 GUI 框架的持续碎片化
2. Electron 作为跨平台解决方案的局限性
3. AI 辅助编程降低了 TUI 开发门槛
4. 用户对高效键盘驱动工作流的需求

尽管有人认为这只是短期趋势，但 TUI 在可预见的未来仍将保持其相关性和吸引力。
