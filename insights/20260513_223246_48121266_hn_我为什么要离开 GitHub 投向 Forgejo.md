## 摘要

本文作者分享了从 GitHub 迁移到自托管 Forgejo 的原因和实施过程。2026年4月27日，荷兰政府推出了 code.overheid.nl，一个自托管的 Forgejo 实例用于托管政府源代码。作者也在同一周将自己的代码迁移到了自托管的 Forgejo 实例。迁移的主要原因包括：GitHub 频繁的服务中断、Copilot 训练数据的默认 opt-in 政策、以及 GitHub 不再有独立的 CEO（现已被整合到微软的 CoreAI 部门）。荷兰政府的选择表明，这种从中心化平台转向开源自托管方案的决策已不再是"边缘"行为。

---

## 关键要点

### 1. 停机问题并非主因
- 2025年5月至2026年4月期间，GitHub 共记录257起事件，其中48起为重大故障
- 2026年4月23日的合并队列故障导致658个仓库和2092个拉取请求被静默回滚
- GitHub CTO 公开道歉并表示需要30倍的容量扩展来应对AI驱动的负载增长

### 2. GitHub 不再有独立 CEO
- 2025年8月11日，Thomas Dohmke 辞任 GitHub CEO
- 微软未进行替换，而是将 GitHub 整合进 CoreAI 部门
- GitHub 的品牌虽然保留，但独立领导权已不存在

### 3. 训练数据默认 opt-in
- 2026年4月24日起，Copilot Free/Pro/Pro+ 用户的交互数据默认用于AI模型训练
- 没有仓库级别的 opt-out 选项
- 私人仓库的豁免范围比预期更窄

### 4. 法律管辖权风险
- GitHub 和微软都是美国公司，受 FISA Section 702 和 CLOUD Act 管辖
- 微软自己的律师在法国参议院听证会上宣誓，无法保证欧盟数据免受美国政府秘密访问

### 5. 荷兰政府的选择
- 荷兰政府选择 Forgejo 而非 GitLab
- 原因是 Forgejo 完全开源，没有 open-core 分裂，提供数字自主所需的全部自由
- 这是机构层面的认可，证明该决策不再是"边缘"行为

## 社区讨论热点

### 去中心化与集中化
- Git 生来就是去中心化的，但围绕 Git 的工具集中到了 GitHub
- 许多用户认为应该将代码推送到多个远程服务器，而不是依赖单一平台

### AI 训练数据争议
- 用户对 GitHub 用所有开源仓库训练 Copilot 表示不满
- 部分用户明确表示不会将个人代码提交到 GitHub

### 自托管的挑战
- 自托管的优势：完全控制、无AI爬虫、可避免微软的监控
- 自托管的难点：维护基础设施、发现性降低、社交图谱丢失

### 替代方案
- **Fossil**: 将整个仓库状态、wiki、问题跟踪、论坛打包成单个文件
- **GitSocial**: 允许带走社交图谱和协作历史，支持跨仓库的拉取请求
- **Tangled**: 基于 AT 协议的去中心化方案，正在开发 PR 堆叠功能
- **SourceHut**: 提供基础 git 托管服务

### Forgejo 的优势
- 完全开源，采用 GPLv3+ 许可证
- 由 Codeberg e.V. 非营利组织管理
- 支持联合（federation），实例之间可以互联

### 关于许可证的争论
- 有用户认为"纯开源"已被大公司滥用为"企业福利"
- 建议采用更强有力的许可证，如 CC BY-NC-SA
- 其他人认为非商业许可证过于极端

---

## 结论

这场讨论反映了开发者社区对中心化平台日益增长的不信任。荷兰政府的选择为这一趋势增添了机构层面的背书。迁移到自托管方案虽然需要付出基础设施维护的代价，但换取的是数据主权和平台独立性。随着 Forgejo 等开源方案的功能和稳定性不断提升，这一趋势可能会继续加速。
