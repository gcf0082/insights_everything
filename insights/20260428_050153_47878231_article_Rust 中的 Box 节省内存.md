- **发布日期**: 2026-04-23
- **主题**: 通过Box优化Rust结构体内存占用
- **语言**: 英文（原文）/ 中文（报告）

## 摘要

本文作者通过修改Rust结构体的内存布局和JSON反序列化方式，将一个真实程序的内存占用从895 MB降低到420 MB，成功节省了475 MB（约53%）的内存。

## 背景与问题

### 实际用例

作者的プログラム需要将AWS SDK的JSON模型文件反序列化为"Smithy Shape"结构体。这些文件包含数千个结构定义，其中大部分包含可选字段，如：

```rust
#[derive(Clone, Deserialize, Serialize)]
pub struct SmithyShape {
    pub shape_type: SmithyShapeType,
    pub operations: Vec<SmithyReference>,
    pub members: FxHashMap<String, SmithyReference>,
    pub key: Option<SmithyReference>,
    pub value: Option<SmithyReference>,
    pub member: Option<SmithyReference>,
    pub input: Option<SmithyReference>,
    pub output: Option<SmithyReference>,
    pub traits: SmithyTraits,
}
```

### 问题分析

原始实现占用895 MB内存。通过分析发现，大多数可选字符串字段为`None`，这是优化的关键点。

## 关键技术洞察

### 1. Rust结构体内存布局

在64位平台上，一个字（word）占用8字节：

- `String` 需要3个词（指针、大小、容量）共24字节，加上堆上的实际内容
- `Option<String>` 由于编译器优化，仍占24字节（不额外占用一个字节来标记None）

### 2. 嵌套结构体的内存问题

当一个结构体包含另一个结构体时，即使内部字段全为`None`，也会占用完整空间：

```rust
// 包含5个Option<String>的结构体，即使全为None也要占120字节
pub struct SmithyServiceTrait {
    pub sdk_id: Option<String>,
    pub arn_namespace: Option<String>,
    pub cloud_formation_name: Option<String>,
    pub cloud_trail_event_source: Option<String>,
    pub endpoint_prefix: Option<String>,
}
```

### 3. 与其他语言的对比

在Java、Python、JavaScript等语言中，`null`只占用一个指针大小的空间。但在Rust中，`Option<BigStruct>`即使为`None`也要占用`BigStruct`的全部空间。

## 解决方案

### 核心策略

1. **检测空结构体**：识别所有字段为`None`的结构体
2. **使用Box包装**：将大结构体放到堆上
3. **使用Option<Box<...>>**：进一步减少占位空间

### 具体实现

将结构体从：

```rust
pub struct SmithyReference {
    pub target: ShortShapeId,
    pub traits: SmithyTraits,
}
```

修改为：

```rust
pub struct SmithyReference {
    pub target: ShortShapeId,
    pub traits: Option<Box<SmithyTraits>>,
}
```

实现自定义反序列化器：

```rust
fn deserialize_boxed_traits<'de, D: Deserializer<'de>>(
    deserializer: D
) -> Result<Option<Box<SmithyTraits>>, D::Error> {
    let traits = SmithyTraits::deserialize(deserializer)?;
    if traits.is_empty() {
        Ok(None)
    } else {
        Ok(Some(Box::new(traits)))
    }
}
```

### 内存节省效果

- `Option<BigStruct>` 在 `None` 时：占用完整的 `BigStruct` 空间
- `Option<Box<BigStruct>>` 在 `None` 时：只占用一个词（8字节）

通过此优化，内存从895 MB降到420 MB，节省475 MB。

### 性能说明

- 反序列化CPU开销增加（因为先反序列化再丢弃）
- 但整体任务反而更快（因为减少了内存压力）
- 使用大量Box会导致堆碎片化，但在本案例中不是问题

## 验证方法

使用带有统计功能的内存分配器（如tikv-jemallocator）来测量：

```rust
#[cfg(feature = "profile")]
#[global_allocator]
static ALLOC: tikv_jemallocator::Jemalloc = tikv_jemallocator::Jemalloc;
```

通过比较反序列化前后的内存使用量来验证效果。

## 总结

### 关键要点

1. **复合结构体可能消耗大量内存** - 即使字段为空也会占用空间
2. **`Option<BigStruct>` 即使为`None`也占用`BigStruct`的全部空间**
3. **使用`Option<Box<BigStruct>>`可以打破这个链条** - `None`只占用一个词
4. **使用Serde反序列化时仍可进行这些优化**

### 适用场景

- 处理大量JSON数据反序列化的程序
- 包含大量可选字段的数据结构
- 内存敏感的应用

### 注意事项

- 这个优化会增加一定的CPU开销
- 大量Box可能导致堆碎片化
- 需要权衡内存节省与代码复杂度
