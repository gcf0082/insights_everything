## 一、项目概述

**sllm** 是一个让多个开发者共同分担 GPU 节点费用的服务。该服务的核心理念是通过"组队"（cohort）的方式，让多个用户共享一个专用 GPU 节点，从而降低运行大型语言模型的成本。

运行 DeepSeek V3 (685B) 模型需要 8×H100 GPUs，成本约为 $14,000/月。大多数开发者只需要 15-25 tok/s 的性能，sllm 允许开发者组成小组共享一个专用节点。

## 二、服务特点

### 2.1 定价模式

- **起步价**: $5/月（小模型）
- **DeepSeek R1**: $40/月
- **DeepSeek V3**: 价格未明确显示（已知问题）

### 2.2 技术特性

- **API兼容性**: OpenAI兼容（使用 vLLM）
- **隐私保护**: 完全私密，不记录任何流量
- **计费方式**: 使用 Stripe 保存卡片信息，组队完成后才扣款
- **性能指标**:
  - TTFT 平均 < 2秒
  - TTFT 最坏情况 10-30秒
  - 模型权重永久驻留在 VRAM 中

### 2.3 可用模型

根据网站显示，目前可用的模型包括：

- llama-4-scout-109b
- qwen-3.5-122b
- glm-5-754b
- kimi-k2.5-1t
- deepseek-v3.2-685b
- deepseek-r1-0528-685b

## 三、运作机制

### 3.1 组队模式

sllm 通过" cohorts "（小组）的方式运营：

1. 用户加入一个小组，贡献一部分 GPU 节点费用
2. 当小组满员后，用户获得 API 密钥
3. 每个小组有固定数量的插槽和承诺期
4. 小组满员后，新小组开放

### 3.2 等待机制

- **7天窗口计划**: 如果7天内组队未完成，将自动取消并释放卡片
- **目前状态**: 服务仍处于早期阶段，尚未有组完成

## 四、热门评论分析

### 4.1 关于"嘈杂邻居"问题

开发者担心如果有人运行大量24/7的重型任务，会导致 TTFT（首 token 时间）下降。创始人回应使用 vLLM 的连续批处理来缓解，但承认物理 VRAM/计算资源有限。正在考虑加入基于信用的按任务计费模式来避免资源竞争。

### 4.2 关于公平性和资源争用

- 大型长查询可能占用整个系统
- 解决方案：实施速率限制和排队机制确保公平性
- 建议与不同时区或不同工作/睡眠时间的人组队
- 有人提到"激增定价"（surge pricing）作为潜在方案

### 4.3 关于定价比较

- $40/月 for DeepSeek R1 vs OpenRouter $40 可获得 105M tokens
- 创始人回应：20 tok/s 是平均值，低峰期可以获得更高的吞吐量
- 有人指出这个服务不是为了价格优势，而是为了控制依赖

### 4.4 关于商业模式

有评论认为商业模式不可行，因为 AI 行业目前都在亏损运营。创始人回应人们不会24小时使用，只有在输出生成时才计数，并且考虑时区差异，大部分时间可能是低利用率。

## 六、与其他服务比较

| 特性 | sllm | OpenRouter | vast.ai |
|------|------|------------|---------|
| 计费方式 | 组队月费 | 按量计费 | 按量计费 |
| 模型选择 | 固定 | 灵活 | 通用 |
| 隐私 | 不记录流量 | 可能记录 | 可能记录 |

- **vs OpenRouter**: OpenRouter 是按量计费，可以自由选择模型
- **vs vast.ai**: 可用于通用 GPU 访问（非 LLM 用途）
- **vs 本地运行**: 有开发者提到在 Framework Desktop 上运行本地 122B 模型约 20 tok/s

## 八、结论

1. **创新性**: 组队共享 GPU 节点的概念被认为是创新和有趣的
2. **主要担忧**: 资源争用、组队时间不确定性、商业模式可行性
3. **目标用户**: 需要 24/7 运行 LLM 的开发者，不追求极致速度但需要稳定访问
4. **发展阶段**: 仍处于早期，尚未有组完成，正在收集反馈
