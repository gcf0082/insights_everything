## 洞察摘要

本文介绍了 Archestra AI 公司如何利用 Git 的 `--author` 标志来阻止其 GitHub 仓库中的 AI 机器人垃圾内容（AI slop）。

### 解决方案

1. **启用 GitHub 限制**：开启"要求先前贡献"（Require prior contribution）标志，只有在仓库中有过提交的贡献者才能创建 PR 和评论
2. **CI 脚本验证**：创建了一个 CI 脚本，用户需要访问他们的网站并通过验证码验证身份
3. **共同提交机制**：通过验证的用户会通过 GitHub REST API 在仓库中创建一个小小的共同提交（co-authored commit），使其成为"已验证"的贡献者

### 成效

- 在第一周就成功阻止了至少 500 个机器人
- 大大减少了垃圾 PR 和问题报告的数量
- 真实用户的贡献质量得到保障

## 社区讨论热点

### 1. 安全隐患

有评论者指出这种方法存在安全隐患：当用户有一个 PR 被合并后，就不再需要审批。恶意用户可以通过提交一个简单的修正（如拼写错误）来满足此要求，然后获取更高的权限。建议启用"要求所有外部贡献者审批"设置。

### 2. GitHub 的责任

许多评论者认为 GitHub 才是解决这个问题的关键，而不是由开源项目自己想出变通办法。有人指出 GitHub 没有动力去阻止 AI 机器人，因为这符合其商业模式（Copilot 订阅）。

### 3. 信任系统建议

- **积分系统**：类似 Stack Exchange 的声誉系统，根据用户的历史贡献质量来评估
- **信任圈**：由项目维护者担保的用户组
- **签名提交**：使用 GPG 签名来验证提交者身份
- **PR 令牌系统**：初始只给用户一个 PR 令牌，优质 PR 可获得更多令牌

### 4. ELO 评分系统争议

有人提议使用类似 ELO 的评分系统来评估贡献者质量，但遭到反对：这类系统容易被 AI 操纵，而且新用户会被排斥在外。

### 5. 方法的局限性

- 这些 onboarding 提交仍然是 git 历史的一部分，占据了约 10% 的提交量
- 只能解决 PR 和问题评论的噪音，无法阻止邮箱通知
- 依赖 GitHub 不改变这些功能的风险

## 结论

这是一个巧妙但治标不治本的解决方案。社区普遍认为要从根本上解决 AI 机器人垃圾内容问题，需要 GitHub 平台层面采取更有效的措施，而不是由各个项目自己想出各种变通办法。
