## 内容摘要

这是一篇关于JavaScript中**无效代理对（Invalid Surrogate Pairs）**的技术博客文章，作者分享了他最喜欢的Bug故事。

### 背景

作者在迁移一个传统编辑器到协作编辑系统时（使用TipTap + ProseMirror + Yjs），发现编辑器有时会静默停止保存内容。用户可以继续编辑，但更改不会同步到Yjs文档，下次打开页面时会丢失数据。这个问题非常罕见且难以复现。

### 根本原因

问题的根源在于**UTF-16编码的代理对分割**：

1. **问题场景**：当用户在同一位置插入两个多字节emoji（如🟢和🔴）时
2. **技术细节**：JavaScript使用UTF-16编码，emoji等超出U+FFFF的字符需要用两个16位代码单元表示，称为**代理对（Surrogate Pair）**
3. **触发条件**：当CRDT操作恰好在代理对的中间位置分割时，会产生**孤立的代理（Orphaned Surrogate）**
4. **致命后果**：当这些无效字符串传递给`encodeURIComponent`时，会抛出`URIError`，导致同步完全停止

### 关键技术概念

| 概念 | 说明 |
|------|------|
| **Code Units（代码单元）** | JavaScript内部使用的16位原始值，`.length`计数的就是这个 |
| **Code Points（代码点）** | Unicode定义的单字符，如U+1F920（🤠）需要两个代码单元 |
| **Grapheme Clusters（字素簇）** | 人类感知为"一个字符"的单位，如👩‍🚀实际包含3个代码点 |

### 解决方案

1. **临时方案**：添加离线支持和全局错误监听器捕获URIError
2. **真正修复**：
   - lib0库被修复，检测到孤立代理时替换为Unicode替换字符
   - 将emoji定义为ProseMirror的原子节点类型，防止被分割
3. **现代方案**：使用`Intl.Segmenter`按字素簇分割字符串

### 后续影响

这个Bug成为了公司内部的"经典案例"，同事们会故意发送🟢🔴这个组合来调侃作者。一旦了解这个问题，会发现类似的bug在很多地方存在——比如用`name[0]`来获取用户名字首字母的工具。

## 社区讨论要点

1. **现代语言处理Unicode的方式**：大多数现代语言使用UTF-8，不再会遇到代理对问题
2. **Intl.Segmenter**：推荐使用此API来进行安全的字符串分割
3. **Rust的字符串处理**：Rust确保无效的Unicode字符串根本无法存在
4. **JavaScript的问题**：JS没有区分代码单元和标量值类型的机制

---

## 结论

这个Bug揭示了JavaScript使用UTF-16编码的历史遗留问题，以及在处理多字节字符时面临的挑战。作者强调："能够正常工作本身就很了不起。"
