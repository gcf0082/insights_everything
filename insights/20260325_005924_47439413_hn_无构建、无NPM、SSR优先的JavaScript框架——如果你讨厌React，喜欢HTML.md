## 主题概述

这是一个关于新兴 JavaScript 框架 Qite 的讨论帖，该框架主打"无构建、无 NPM、SSR 优先"，面向讨厌 React、喜欢 HTML 的开发者。讨论的核心话题是：**响应式（reactivity）是否是现代 UI 复杂性的根源**。

## 核心讨论观点

### 1. 响应式是"原罪"？

- **观点**: 响应式（自动响应数据变化）本意是让开发更简单，但实际上引入了"状态"概念，使其独立于 UI 和数据源
- **结果**: 从此开始了状态管理的复杂之路，包括：双向数据绑定 → Flux → Redux → state vs props → sagas → prop drilling → hooks → context API → 有状态/无状态组件 → 不可变性 → 浅拷贝/深拷贝等
- **游戏开发类比**: 游戏开发者认为 UI 很简单，因为游戏有状态，主渲染函数根据当前状态绘制所有图形，按帧率调用，且渲染函数不能改变程序状态

### 2. 对 React 的批评

- Facebook 2010 年时交互性恰到好处，因为关注点分离在语言层面：HTML 结构、CSS 展示、JS 行为
- "组件化"把所有东西整合在一起，导致简单的下拉菜单也要成为组件并发出 10 个请求
- React 是"一团糟"，它不固执己见，任何人都可以用任何方式做任何事
- 多团队协作时，每个团队使用不同的代码组织、状态管理库和编程范式
- "我从未见过一家产品优秀的公司使用 React。代码总是在重写，每 3 周迁移一次，或者干脆有 bug"
- Vue 和 Svelte 做得更好

### 3. 反驳观点

- HTML+CSS+JS 三元组如果真的适合创建 UI，就不会出现"寒武纪大爆发"式的不同方案
- 响应式是对旧 GUI 工具包中 MVC 痛苦的回应
- 无响应式的心理模型更简单（Mithril、Imba 性能也很好）
- 游戏可以每帧重绘一切，但 DOM 不行
- 浏览器有固有约束：CSS 级联样式、非 DOM 渲染层、HTML 元素原生状态、网络异步调用
- Web 从未被设计为富应用分发+执行层，后来演变成这样是有真正复杂性的
- React 可以完全无状态、函数式，状态存在其他地方

### 4. 关于 Remix 3

- Remix 已更名为 React Router
- Remix 3 将不使用 React，也可能不使用响应式
- 相关演讲: https://www.youtube.com/watch?v=iZl0IKj0HHc

### 5. 技术讨论

- **即时模式 vs 保留模式 UI**: 游戏是即时模式，每帧从状态重绘；UI 通常用保留模式
- **虚拟 DOM diff**: 这是另一个复杂性来源
- **受控组件 vs 非受控组件**: 受控组件每次按键触发状态变化和重渲染，性能差；非受控组件在 DOM 中存储状态，焦点管理复杂
- **Web Components**: Lit 框架等可实现互操作组件，但标签名需要破折号、属性绑定麻烦

### 6. 其他观点

- 用 Preact 无响应式 + MVC 模式：View 触发事件 → Controller 更新 Model → Controller 调用 render 更新 View
- 好的 UI 框架（Vue）在响应式和简单之间取得平衡
- 响应式让开发者能构建更复杂的 UI，收益被复杂性抵消
- 状态管理 vs 事件总线：状态管理完全可测试，可形式化为状态机
- "最终会有人推出无 JS、无 NPM、无 SSR 的框架，然后我们只是重新发明了 HTML"
- HTMZ 概念被提及
- 应使用 `<form>` 元素而不是 `<div data-part="form">`

## 总结

这场讨论反映了开发者对现代前端生态复杂性的普遍不满。核心争论在于：响应式编程范式是否导致了不必要的复杂性，还是解决了一个真实的问题。支持者认为它简化了状态同步，批评者认为它引入了一层额外的复杂性，且 Web 浏览器环境的限制使得问题更加复杂。游戏开发者的简单模型在 Web 环境中难以直接应用，因为 DOM 不是游戏画布。许多开发者呼吁回归更简单的方法，如使用标准 HTML 元素和更少的构建工具。
