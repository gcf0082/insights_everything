## 研究摘要

该论文报告了一项针对IoT传感器平台的固件开发对比研究。两个团队并发开发相同功能，一个使用C语言，另一个使用Rust语言，经过数月的分析和迭代开发，得出了以下结论：

1. **内存占用与执行速度**: 没有强有力的理由偏好C超过Rust
2. **Ariel OS**: Rust实现的运行时 footprint 小于先进的裸机C栈
3. **结论**: Rust是当前固件开发的可行选择

---

## 主要讨论观点

### 1. Rust在嵌入式领域的优势

**支持方观点**:
- 语言设计和工具链整体更优（枚举、命名空间、无头文件、`cargo run --release`直接可用）
- 更好的错误信息
- 易于构建可靠程序的结构
- cortex-m、defmt、probe-rs等工具成熟
- PAC项目表现出色

**质疑方观点**:
- 需要自己编写HAL和库
- 许多有用的Rust库似乎需要nightly版本
- 存在长期维护担忧

### 2. 长期稳定性争议

**核心争论**: Rust演进太快，不适合需要运行数年甚至数十年的代码

**反驳观点**:
- 代码不会因为语言演进而停止运行
- 可以使用旧的工具链
- Rust使用Editions（2015、2018、2021、2024）来引入破坏性变更而不分裂生态系统
- 每个版本都被新编译器版本无限期支持
- 保持旧工具链是嵌入式开发的标准做法

**实际问题**:
- 使用`rust-toolchain.toml`和Cargo.lock可以锁定版本
- 第三方依赖会过时，这是所有语言的共同问题
- 嵌入式C项目很少有外部依赖，而Rust生态激励使用依赖

### 3. 与Ada/SPARK的对比

部分评论者认为在高完整性任务关键领域，Ada/SPARK是更安全的选择：
- Ada有正式规范和经过验证的工具链
- 在航空航天、铁路、汽车等领域有 decades 的 legacy
- 2026年1月生效的内存安全软件强制令推动了对Rust的兴趣
- 但Rust缺乏稳定的语言规范

### 4. 批评意见

- 实验只有一个case，证据不够充分
- Rust团队有Ariel OS开发者支持，不够公平
- 便携性声称缺乏实际测试
- C项目设置和代码复制per target的描述不准确

## 技术细节

### 性能优化
- STM内部人员通过比较C和Rust固件的精确MCU和外设初始化、收紧I2C时序、启用MCU指令缓存，实现了超过三分之一的性能提升

### 内存安全
- Rust使用unsafe关键字时并非真正内存安全
- 与外部C库交互时内存安全保证失效

### 版本问题
- 即使crate被撤下，如果lockfile中有版本，仍可下载和构建
- 依赖项lockfile可以跨平台构建

---

## 结论

该HN讨论反映了嵌入式开发社区对Rust vs C的传统争论。根据论文和评论：

1. **Rust对于嵌入式是可行的选择**，尤其对于新项目
2. **长期维护**是合理担忧，但可以通过版本锁定缓解
3. **工具链成熟度**在特定平台（Espressif RiscV、Cortex-M ARM、STM32）已经很完善
4. **生态系统库**正在改善，但尚需时间
