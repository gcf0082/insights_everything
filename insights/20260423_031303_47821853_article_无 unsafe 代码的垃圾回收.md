# 洞察报告：无 unsafe 代码的垃圾回收库

**链接**: https://fitzgen.com/2024/02/06/safe-gc.html
**作者**: Nick Fitzgerald
**发布日期**: 2024-02-06
**标签**: Rust, 垃圾回收, unsafe

## 核心观点

作者 Nick Fitzgerald 实现了一个名为 `safe-gc` 的 Rust 垃圾回收库，其核心理念是：**整个库的实现不包含任何 unsafe 代码**。这在 Rust GC 库领域是独特的，因为通常 GC 库即便提供安全 API，其内部实现也会大量使用 unsafe 代码。

## 主要内容

### 1. 为什么需要 unsafe

传统 GC 库面临的核心问题是：如何让用户定义 GC 管理的类型并向收集器报告"GC 边"（指向其他 GC 对象的引用）。如果遗漏任何边，收集器可能错误地认为对象不可达并回收它，导致 use-after-free 错误。这个 `Trace` trait 通常被设计为 unsafe，让用户承担安全不変性。

### 2. safe-gc 的设计

- **Gc<T>**：GC 管理对象的引用类型
- **Trace trait**：安全 trait，用户实现它来报告 GC 边
- **Heap**：分配 GC 对象的堆，可以触发垃圾回收
- **Root<T>**："根化"的引用，在 GC 期间保持对象存活
- **访问方式**：不通过解引用 Gc<T> 访问对象，而是通过 `heap[&gc]` 索引访问，这是避免 unsafe 的关键

### 3. 实现细节

- **存储结构**：使用 HashMap 按 TypeId 映射到 Arena<T>，Arena 内部使用 Vec 存储
- **空闲列表**：管理空闲槽位，仅管理索引而非原始内存块
- **GC 算法**：采用简单的 mark-and-sweep（标记-清除）算法
- **标记阶段**：从根集开始递归标记所有可达对象
- **清除阶段**：回收未标记的对象，将槽位加入空闲列表
- **根集管理**：每个 Arena 有自己的 RootSet，可包含重复项

### 4. 安全特性

即使用户未正确实现 Trace 或未对 Gc<T> 进行根化，访问也已回收的对象时：

1. 对象恰好被其他引用 keeping alive → 访问成功，bug 隐藏
2. 槽位为空（FreeListEntry::Free）→ panic
3. 槽位被新对象占用 → 访问到错误对象（ABA 问题）

无论哪种情况，都是内存安全的，不会导致内存损坏或 use-after-free。

### 5. 复制收集器的尝试

作者曾尝试实现 copying collector（复制收集器），但因所有权和借用问题未能成功。mark-and-sweep 只用了半小时就原型就成功了。

## 总结

`safe-gc` 是 Rust GC 库设计空间中的一个有趣尝试，证明了可以实现完全不含 unsafe 代码的 GC 库。虽然性能可能不是最优的，但它是安全的——只要 Rust 语言本身安全，safe-gc 就是安全的。
