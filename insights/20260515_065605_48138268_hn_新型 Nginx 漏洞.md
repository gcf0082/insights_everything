## 基本信息

- **洞察链接**: https://news.ycombinator.com/item?id=48138268
- **标题**: New Nginx Exploit
- **来源**: Hacker News
- **发布时间**: 5小时前
- **得分**: 246 points
- **评论数**: 57 comments

## 漏洞概述

这是一个关于 **CVE-2026-42945**（被称为 "NGINX Rift"）的安全漏洞发现。该漏洞是 Nginx 的 `ngx_http_rewrite_module` 中的一个堆缓冲区溢出漏洞，已经存在了 **18年**（自2008年的0.6.27版本）。

### 漏洞技术细节

- **漏洞类型**: 堆缓冲区溢出（heap buffer overflow）
- **根本原因**: 经典的两遍脚本引擎bug（长度计算与实际拷贝传递之间的不一致，涉及 `ngx_escape_uri`）
- **触发条件**: 需要非常特定的配置模式：
  1. 使用 `rewrite` 指令（带未命名捕获如 `$1`/`$2`，且在替换字符串中包含 `?`）
  2. 紧跟着使用 `set`、`if` 或另一个 `rewrite` 指令

### 漏洞利用

- 公开的 PoC（概念验证代码）可以将此漏洞转化为**未经身份验证的远程代码执行（RCE）**
- 利用技术：跨请求堆布局（heap feng shui）+ 池清理指针损坏（pool cleanup pointer corruption）
- PoC 假设 ASLR（地址空间布局随机化）已被禁用

- **F5 补丁版本**: 1.31.0 和 1.30.1

## 社区讨论焦点

### 1. ASLR 保护的有效性

社区对此存在激烈争论：

- **一方观点**: POC 因为禁用了 ASLR 才能工作，所以大多数用户是安全的
- **另一方观点**: 撰写漏洞报告的人声称有办法可靠地绕过 ASLR，应该默认相信这一点；ASLR 只是一个纵深防御技术，最终被绕过只是时间和技能问题

### 2. 实际风险评估

- 使用 nginx 10年的用户表示从未同时使用 `rewrite` 和 `set`
- 某些情况下会在顶层设置变量，然后在 location 块中用 rewrite 覆盖，这些变量可用于日志或其他"全局"上下文
- 默认的 Nginx PHP 集成使用了 `fastcgi_split_path_info` 和 `set $path_info $fastcgi_path_info`，可能存在风险

### 3. 补丁状态

- **Ubuntu**: 已于今晨修复（1.24.0-2ubuntu7.8）
- **Debian**: 尚未为 trixie 发布补丁
- **Debian 安全追踪**: https://security-tracker.debian.org/tracker/CVE-2026-42945

### 4. 缓解措施

F5 建议的临时缓解措施：将未命名捕获（如 `$1`、`$2`）替换为命名捕获（如 `$user_id`、`$section`）

### 5. 替代方案讨论

社区讨论了使用内存安全语言编写的 Web 服务器替代方案，如：
- **Caddy** (Go语言): 使用体验不错，但有数千个二进制文件依赖各种插件组合
- **Jetty** (Java): 有 shell 注入等漏洞历史
- **Traefik**: 配置更详细，但对于负载均衡、路径重写等复杂场景很好用

同时讨论了使用 SELinux 和 AppArmor 等主动防御措施。

## 总结

这是一个影响深远的漏洞，因为任何在生产环境中使用 Nginx 进行 URL 重写的用户都可能受到影响。然而，漏洞利用需要非常特定且相对罕见的配置模式。社区普遍建议：

1. **尽快打补丁**: 不要因为可能有缓解措施就掉以轻心
2. **检查配置**: 如果使用 rewrite 模块，检查是否使用了未命名捕获
3. **不要依赖 ASLR**: ASLR 是一个有用的缓解措施，但不是万能的
4. **使用命名捕获**: 作为临时缓解措施，将 `$1`、`$2` 改为命名捕获
