# 洞察报告：Zig 项目的反 AI 贡献政策

**洞察链接**: https://simonwillison.net/2026/Apr/30/zig-anti-ai/

**来源**: Simon Willison's Weblog

**发布日期**: 2026年4月30日

**标签**: JavaScript, Open Source, AI, Zig, Generative AI, LLMs, AI-assisted programming, Anthropic, AI ethics, Bun

---

## 内容摘要

Zig 项目拥有目前最严格的反 LLM 政策之一，其官方行为准则明确规定：

- 禁止使用 LLM 处理 issue
- 禁止使用 LLM 提交 pull request
- 禁止在 bug tracker 上使用 LLM 撰写评论（包括翻译），鼓励使用英语但不强制要求

最引人注目的 Zig 项目是 Bun JavaScript 运行时，该项目于 2025 年 12 月被 Anthropic 收购，毫无疑问大量使用 AI 辅助。Bun 运营着 Zig 的自有分支，并在近期通过添加"并行语义分析和多个 codegen 单元到 llvm 后端"实现了 4 倍的编译性能提升。然而，Bun 团队表示目前没有计划将这些改进上游回归，因为 Zig 严格禁止 LLM 生成的贡献。

## 核心观点：贡献者扑克

Zig 软件基金会社区副总裁 Loris Cro 在 "Contributor Poker and Zig's AI Ban" 文章中解释了这一严格禁令的理由：

在成功的开源项目中，项目维护者最终会面临收到的 PR 数量超过处理能力的情况。按照常理，为了最大化投资回报，应该停止接受不完美的 PR，但这并非 Zig 项目的做法。相反，Zig 尽最大努力帮助新贡献者完成他们的工作，即使他们需要一些帮助。

Zig 重视贡献者本身而非他们的贡献。每个贡献者都代表了 Zig 核心团队的投资——审核和接受 PR 的主要目的不是要落地新代码，而是帮助培养新的贡献者，让他们最终成为值得信赖且多产的社区成员。

LLM 辅助完全打破了这个模式。即使 LLM 帮助提交了一个完美的 PR，Zig 团队在审核工作中花费的时间并不能帮助他们增加新的、可靠的、可信的贡献者。

这就是"贡献者扑克"名称的由来——就像玩纸牌游戏一样，"玩的是人，而不是牌"。在贡献者扑克中，你赌的是贡献者本人，而不是他们第一个 PR 的内容。
