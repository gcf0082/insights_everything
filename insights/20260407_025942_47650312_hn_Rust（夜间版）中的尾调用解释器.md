## 内容摘要

本文讨论了博主 Matt Keeter 使用 Rust nightly 新增的 `become` 关键字实现的一个尾调用解释器。文章重点介绍了 Rust 语言对尾调用优化的支持现状以及相关技术讨论。

### 核心主题

1. **Rust 的 `become` 关键字**
   - 近期添加到 nightly Rust 的新关键字，用于确保尾调用优化（TCO）
   - 与普通 `return` 的区别：`become` 会在返回前先销毁局部变量
   - 使用方式：`become foo(arg1, arg2)` 替代 `return foo(arg1, arg2)`
   - 目前仅支持同步函数，不支持 async 函数

2. **尾调用优化的重要性**
   - 对于需要深度递归的解释器实现至关重要
   - 没有 TCO 的情况下，栈会不断增长直到溢出
   - 即使在 release build 中，编译器也未自动优化掉栈的增长

3. **为什么 VM/解释器有时比直接代码更快**
   - 指令缓存效率（instruction cache efficiency）
   - 分支预测（branch prediction）
   - 案例：Go 的 hyperpb 解析器比默认生成的 protobuf 代码快 3 倍

## 技术要点

1. **尾调用优化（TCO）**：让尾调用重用调用者的栈帧而非新建
2. **become 关键字**：Rust 新的保证尾调用的语法
3. **drop 问题**：Rust 中所有值在作用域结束时都会被隐式销毁，`become` 需要在调用前处理这些销毁
4. **异步兼容性**：当前 RFC 不允许在 async 函数中使用 become
