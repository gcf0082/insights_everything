## 核心观点总结

### 1. 当前问题：云端AI依赖的隐患

当前软件开发中，开发者倾向于直接调用 OpenAI 或 Anthropic 等云端 AI 服务来实现功能。这种做法带来了三大核心问题：

- **脆弱性**：应用在服务器宕机或账户到期时立即停止工作
- **隐私侵犯**：用户数据被传输到第三方AI提供商，带来数据保留、合规、审计等复杂问题
- **复杂度提升**：功能从简单的UX特性变成了依赖网络状况、供应商稳定性、速率限制、计费系统的分布式系统

### 2. 核心论点：本地AI应该是常态

- 现代设备的算力已经远超十年前，且配备专用神经引擎，却让它闲置而等待远程服务器响应
- "AI无处不在"不是目标，**有用的软件才是目标**
- 如果功能可以在本地完成，引入云端依赖就是自找麻烦

### 3. 实践案例：Brutalist Report 的设备端摘要

作者展示了一个成功案例——Brutalist Report 的原生 iOS 应用：

- 新闻摘要**完全在设备上生成**，使用 Apple 本地模型 API
- 无需服务器中转、无用户日志、无供应商账户
- 用户数据始终保留在本地

### 4. Apple 生态的本地AI工具链

文章介绍了 Apple 平台提供的本地AI开发方式：

**基础用法**：
```swift
import FoundationModels
let model = SystemLanguageModel.default
let session = LanguageModelSession { /* 提示词 */ }
let response = try await session.respond(options: .init(maximumResponseTokens: 1_000)) {
  articleText
}
```

**结构化输出**（推荐）：
```swift
@Generable
struct ArticleIntel {
  @Guide(description: "One sentence. No hype.") var tldr: String
  @Guide(description: "3–7 bullets. Facts only.") var bullets: [String]
  @Guide(description: "Comma-separated keywords.") var keywords: [String]
}
```

Apple 推动将 AI 输出从非结构化文本转向**类型化数据**，让模型直接生成结构化对象，而非解析 JSON。

### 5. 本地AI的适用场景

本地AI最适合的场景：**转换用户自有数据**

- 摘要、分类、提取、改写、规范化
- 输入数据已在设备上（用户刚加载的内容）
- 输出轻量、快速且私密

本地AI不是用来替代整个互联网搜索的，而是作为**数据转换器**嵌入应用中。

### 6. 对"本地模型不够智能"的回应

作者认为：

- 大多数应用功能不需要能写莎士比亚或通过律师考试的模型
- 它们只需要可靠地完成：摘要、分类、提取、改写或规范化
- 对于这些任务，本地模型可以做得非常出色

---

## 关键行动建议

1. **优先考虑本地**：如果功能可以在本地完成，就不要引入云端依赖
2. **使用结构化输出**：定义 Swift struct 让模型生成类型化数据，而非解析自由文本
3. **将AI作为子系统**：不要把AI简单粘合成聊天框，要作为有可靠行为的真实子系统
4. **只在必要时使用云端**：云端模型应该只用于真正需要其能力的地方

## 核心金句

> "You don't build trust with your users by writing a 2,000 word privacy policy. You build trust by not needing one to begin with."

（你不需要写2000字的隐私政策来建立用户信任。不需要隐私政策才是建立信任的方式。）
