## 摘要

本讨论聚焦于一个名为 "Days without GitHub incidents" (https://www.dayswithoutgithubincident.com/) 的网站，该网站显示 GitHub 已连续 293 天未发生重大事故。讨论的焦点在于这个"事故计数器"是一种幽默的 meme 网站，还是对 GitHub 真实服务可靠性的合理批评。

## 核心讨论话题

### 1. GitHub 稳定性问题
- 许多人认为 GitHub 近期频繁出现服务中断问题，特别是 GitHub Actions 和 CI/CD 流程
- 有用户提到"vibe coding"环境下，平台不可靠会导致团队被困数日无法部署
- 有人指出 GitHub 提交量同比增长 14 倍，这可能是导致问题的原因之一

### 2. 状态页面的争议
- 评论者批评 GitHub 状态页面将各项服务分开统计的做法是"SLA-fudging"
- 有人认为 git push/pull、issues、PRs、actions 是核心功能，任何一部分出问题都应被视为平台故障
- 也有人认为分开统计对不同用户有参考价值

### 3. 对 Microsoft 的批评
- 一些评论者认为 Microsoft 作为 trillion dollar company，应该能更好地处理扩展问题
- 有人指出 GitHub 出售给 Microsoft 后，服务质量在下降
- 部分用户开始考虑迁移到 self-hosted 替代方案如 Forgejo、CodeBerg 等

### 4. 为 GitHub 辩护的观点
- 有人认为扩展到 GitHub 的规模非常复杂，不是简单的问题
- 维护服务的工程师们同样辛苦，应该得到理解和支持 (#hugops)
- 有人认为批评者过度简化了分布式系统的复杂性

### 5. 自托管解决方案
- 有用户提到使用 self-hosted forgejo instance
- 有人讨论 Phabricator 作为自托管选择
- AtProto/Bluesky 的"Knot"系统被提及作为数据自托管方案

## 主要结论

讨论反映了开发者社区对 GitHub 可靠性日益增长的不满情绪，同时也展现了关于平台责任、技术复杂性和开源生态的不同观点。核心问题是：作为事实上的行业标准，GitHub 是否应该为用户提供更稳定的服务。
