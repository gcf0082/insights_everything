# 洞察报告：MAD Bugs - 即使"cat readme.txt"也不安全

**洞察链接**：https://blog.calif.io/p/mad-bugs-even-cat-readmetxt-is-not

**发布日期**：2026年4月17日

---

## 摘要

本文揭示了iTerm2中的一个严重安全漏洞，攻击者可以通过简单的`cat readme.txt`命令实现任意代码执行。该漏洞利用了iTerm2的SSH集成功能和PTY（伪终端）协议的信任问题。

## 背景

### iTerm2的SSH集成功能

iTerm2提供了SSH集成功能，通过一个名为"conductor"的远程辅助脚本来实现更丰富的远程会话管理。该功能的工作流程如下：

1. iTerm2通过`it2ssh`启动SSH集成
2. iTerm2向远程会话发送一个引导脚本（conductor）
3. 远程脚本成为iTerm2的协议对等方
4. 双方通过终端转义序列进行通信

### PTY（伪终端）

iTerm2作为终端模拟器，通过PTY与shell进行通信。在SSH会话中：
- iTerm2写入字节到PTY
- SSH进程接收并转发到远程机器
- 远程的conductor从stdin读取数据

## 核心漏洞

### 漏洞本质

**信任失败**：iTerm2接受来自非信任源的SSH conductor协议。具体来说，iTerm2会接受终端输出中的伪装的远程conductor消息，而不是仅接受来自真实conductor会话的消息。

这意味着恶意文件、服务器响应、banner或MOTD可以打印：
- 伪造的`DCS 2000p` hook
- 伪造的`OSC 135`回复

从而让iTerm2进入一个虚假的SSH集成交换状态。

### 利用机制

1. **构造恶意文件**：readme.txt文件包含伪造的conductor协议序列
2. **触发漏洞**：受害者运行`cat readme.txt`
3. **状态机推进**：iTerm2自动发送`getshell()`和`pythonversion()`请求
4. **利用sshargs**：利用精心构造的`sshargs`参数
5. **代码执行**：通过base64编码的`run(...)`命令执行任意代码

### PTY混淆

在合法SSH集成中，iTerm2写入base64编码的命令到PTY，SSH转发到远程conductor。在漏洞利用中，iTerm2仍然写入到PTY，但没有真实的SSH conductor，本地shell接收这些作为普通输入。最终的base64块可以被设计成有效的可执行路径。

## 复现步骤

1. 生成漏洞利用文件：
   ```bash
   python3 genpoc.py
   unzip poc.zip
   ```

2. 运行漏洞：
   ```bash
   cat readme.txt
   ```

需要从包含`ace/c+aliFIo`可执行脚本的目录运行`cat readme.txt`。

## 总结

这是一个非常严重的安全漏洞，因为它展示了看似无害的日常操作（如读取文本文件）如何导致任意代码执行。该漏洞利用了iTerm2对终端输出的过度信任，通过精心构造的终端转义序列来伪装成SSH集成协议的一部分。
