## 文章概述

本文作者从函数式编程的角度，对柯里化（currying）风格提出了质疑。作者认为，虽然柯里化在纯函数式语言中是一种优雅的惯例，但实际上存在一些被忽视的缺点。文章探讨了三种多参数函数的定义风格：参数列表风格、柯里化风格和元组风格，并分析了它们的优缺点。

---

## 主要内容

### 1. 三种函数定义风格

**参数列表风格**（Imperative Style）
```rust
fn f(p1: P1, p2: P2, p3: P3) -> R { ... }
```

**柯里化风格**（Curried Style）
```haskell
f p1 p2 p3 = ...
-- 类型: f :: P1 -> P2 -> P3 -> R
```

**元组风格**（Tuple Style）
```haskell
f(p1, p2, p3) = ...
-- 类型: f :: (P1, P2, P3) -> R
```

理论上，这三种风格在类型上是同构的（isomorphic），可以相互转换。

### 2. 部分应用（Partial Application）

柯里化风格被认为的主要优势是部分应用变得简单自然：

```haskell
add' = add 1  -- 返回一个函数 \y -> (\z -> 1 + y + z)
add'' = add' 2  -- 返回一个函数 \z -> 1 + 2 + z
add'' 3  -- 返回 6
```

然而，作者指出部分应用并非柯里化独有的特性。通过简单的语法糖，参数列表风格和元组风格同样可以实现部分应用：

```haskell
add' = add(1, $, $)
add'' = add'(2, $)
add''(3)  -- 返回 6
```

作者认为这种写法实际上**更具可读性**，因为它展示了数据流的方向。

更重要的是，元组风格支持任意参数的部分应用，而柯里化风格默认只支持第一个参数的部分应用。

### 3. 柯里化的缺点

#### 性能开销
每次调用多参数柯里化函数时，都会创建中间函数对象。例如 `add 2 3` 会先创建 `\y -> add 2 y`，然后再应用到 `3`。

#### 类型形状不自然
柯里化函数的类型 `P1 -> P2 -> P3 -> R` 会导致类型形状不对称：
- 输入多个值 → 需要分散在多个参数中
- 输出多个值 → 需要返回元组

这导致柯里化函数与高阶函数的组合变得困难：

```haskell
sayHi name age = "Hi I'm " ++ name ++ " and I'm " ++ show age
people = [("Alice", 70), ("Bob", 30), ("Charlotte", 40)]

-- 错误：sayHi 是 String -> Int -> String，但 people 是 (String, Int) 元组
conversation = intercalate "\n" (map sayHi people)
```

需要使用 `uncurry sayHi` 来转换类型。

#### 与证明助手的不兼容
作者在使用 Rocq 证明助手时，遇到过柯里化函数与谓词类型不匹配的问题。类型 `P1 -> ... -> Pn -> State R` 无法与 `In -> State Out` 统一。

### 4. 柯里化的优势场景

作者也承认在**依赖类型语言**中，柯里化风格确实是更好的选择：

```coq
Definition plus1 (n : nat) (i : fin n) : fin (n + 1) := ...
-- 类型: forall n : nat, fin n -> fin (n + 1)
```

当返回类型依赖于输入值时，柯里化风格的类型签名更加清晰。

---

## 结论

作者并不主张完全放弃柯里化，而是建议：
1. 在设计新的函数式语言或标准库时，可以考虑默认使用元组风格
2. 为部分应用提供替代语法支持
3. 根据具体场景选择合适的风格

对于某些高阶函数（如 `map`、`fix`），柯里化定义确实更优雅；但对于大多数场景，元组风格更加合理。
