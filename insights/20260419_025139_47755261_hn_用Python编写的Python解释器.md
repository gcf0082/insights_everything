## 洞察摘要

本讨论围绕一篇介绍用Python编写Python解释器的文章展开，该项目名为Byterun，来自《500 Lines or Less》一书。

## 核心讨论内容

### 1. 自举问题争论

最受关注的评论讨论了用Python写Python解释器的实际效用问题：

- 解释器仍然需要CPython来运行，使新解释器变得冗余
- 如果向自定义解释器添加新语言特性，需要在运行时执行整个链条：CPython → 旧解释器 → 新解释器 → 目标程序，链条会越来越长，性能指数级下降
- 相比之下，自托管编译器的每次迭代都会被缓存为编译后的二进制文件

### 2. PyPy与RPython

- PyPy通过使用名为RPython的受限最小Python子集来实现自举
- RPython实际上是编译型语言，PyPy并非用解释型语言编写的解释器
- RPython可以看作Python的一个子集，可以在CPython下运行

### 3. 解释器与编译器的区别

- 解释器不将字节码翻译成原生机器指令，那是JIT编译器的工作
- weval项目可以将解释器转换为编译器，但这是部分求值器，不是解释器

### 4. 其他观点

- 文章关于Perl的解释方式是不正确的，Perl实际上会被转换为AST和操作码树
- 使用`getattr`来分发指令是一个优雅的模式，在许多真实解释器中出现
- 这是所谓的"元循环解释器"(Metacircular Interpreter)
