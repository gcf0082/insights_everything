## 内容摘要

这篇文章介绍了现代SQLite中许多不为人所熟知的功能特性，引起了社区的广泛讨论。

## 核心洞察

### 1. STRICT 模式（严格模式）

STRICT表是SQLite 3.37（2021年11月）引入的重要特性，能够强制执行类型检查，防止类型不匹配的数据被写入数据库。

**用户反馈：**
- 用户`crazygringo`分享了一个真实案例：在使用Python和NumPy时，NumPy整数和Python整数会被不同地存储到SQLite中（前者存为二进制blob，后者存为整数），导致无法比较。启用STRICT模式后可以及时发现这类类型问题。
- 用户`momo_dev`表示STRICT模式帮助他捕获了一个悄然工作的类型错误。

**讨论焦点：**
- 部分用户认为STRICT应该是默认值
- SQLite作者Dr. Hipp亲自回应：如果是默认值，无数现有应用将崩溃，SQLite安装在数十亿设备上，即使0.001%的回归率也会影响大量用户

### 2. JSON 功能

SQLite的JSON功能（json1扩展）从3.9版本就存在（2015年），但在3.38版本（2022年）才成为标准发行版的一部分。

**关键特性：**
- `->>` 操作符可以返回JSON中的任意类型（文本、整数、浮点、NULL），而不像PostgreSQL总是返回文本
- 支持表值函数 `json_each`、`json_tree` 用于遍历JSON数组和对象

### 3. 灵活类型系统 vs 严格类型

这是讨论最热烈的话题。

**支持灵活类型的观点（Dr. Hipp本人）：**
- SQLite的灵活类型与JSON配合良好
- 脚本语言中所有值都是单一超类的子类，强制严格类型并不能提高可靠性
- 现实世界的数据集几乎都有无效条目，严格类型规则无法解决这个问题
- 类型检查不等于数据验证

**支持严格类型的观点：**
- 当多个应用读写同一数据库时，需要共享的schema契约
- 通用工具（如报表系统）依赖稳定的数据类型

### 4. 全文搜索（FTS5）

**用户反馈：**
- FTS5对于简单的FTS需求很方便，但对于更复杂的模糊搜索或子词搜索不够实用
- 例如搜索"Daemon"无法匹配包含"DaemonSet"的文档
- 建议使用trigram tokenizer来改进
- 讨论还涉及了Turso的多写入器支持（MVCC）等生产环境特性

### 5. 窗口函数和生成列

- 窗口函数早已存在，被用户用于复杂的数据分析
- 生成列从SQLite 3.31（2020年1月）开始支持

### 6. 生产环境使用

- 有用户正在开发SQLite的生产监控和安全层（Rust agent）
- Turso提供了高并发的多写入器支持
- Windows系统本身也大量使用SQLite

---

## 关键结论

1. **SQLite比大多数人想象的更强大**：JSON支持、窗口函数、生成列、FTS5等功能已存在多年
2. **严格模式是可选的**：虽然STRICT模式提供了更强的类型检查，但由于兼容性问题，不会成为默认选项
3. **类型检查 ≠ 数据验证**：即使使用STRICT模式，仍需在应用层进行数据验证
4. **Dr. Hipp亲自参与讨论**：SQLite作者在HN上直接回应了关于类型系统的讨论
