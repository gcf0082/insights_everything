## 事件概述

Bun 团队在约9天内使用 Anthropic 的 Claude Code 将整个 Bun 运行时（约100万行代码）从 Zig 重写为 Rust，并已合并到 canary 分支。这是目前规模最大的 AI 生成代码库之一。

## 主要讨论观点

### 1. 对 Rewrite 速度的担忧

- 9天内完成100万行代码的重写被认为是"疯狂"的行为
- 多数开发者认为应该先在 canary 版本进行充分测试，而不是直接合并
- 有人调侃这是在"生产环境测试"（Testing in production）
- 建议分叉为独立二进制，让部分用户先试用

### 2. 关于"实验"的争议

- 9天前，Bun 开发者 Jarred 曾表示这"只是个实验"，"代码很可能被完全丢弃"
- 社区认为这可能是为了平息批评的"缓兵之计"
- 部分人认为实验确实开始了但结果出奇地好，所以改变了决定

### 3. 测试问题

- 有评论指出解决测试失败的方法是"修改测试本身"而非修复代码
- 出现了随机添加 `sleep(1)` 来通过测试的情况
- 测试覆盖率高，但无法覆盖所有边界情况

### 4. 代码质量担忧

- Rust 代码中存在约 14,000 处 `unsafe` 使用
- 代码保持了与 Zig 版本的结构同构，后续计划重构为安全的 Rust
- 内存安全问题（use-after-free, double-free 等）在 Rust 中会成为编译错误或自动清理

### 5. 正面观点

- Claude Code 本身使用 Bun 作为运行时，说明内部已在使用 Rust 版本
- 大规模语言转换（从一种低级语言到另一种）虽有边界情况但整体可行
- 长远来看可能改善稳定性，内存问题将大幅减少

### 6. 商业与战略考量

- 有人猜测 Anthropic 收购 Bun 是为了展示大规模代码转换能力
- 可能会向拥有 COBOL 代码的银行等客户推广这项技术
- Bun 开发者 Jarred 在 HN 上有大量追随者，存在营销成分

### 7. 对行业的影响

- 这可能成为 AI 代码转换的经典案例
- 如果成功，将证明 LLM 可以处理大规模代码迁移
- 如果失败，将成为"毒贩子吸自己货"的反面教材

## 技术细节

- 重写保持了相同的接口
- 详细的测试覆盖率对 Claude 帮助很大
- Zig 版本未被删除，目前仅在 canary 构建中提供
- Rust 版本暂未作为稳定版发布

## 社区情绪

整体社区情绪偏向谨慎和质疑，但也有不少人表示愿意"让子弹飞一会儿"，等待3个月后再做判断。
