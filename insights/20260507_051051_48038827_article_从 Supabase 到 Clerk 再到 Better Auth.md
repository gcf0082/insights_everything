## 文章摘要

本文记录了 Val Town 从 Clerk 认证服务迁移到 Better Auth 的全过程。作者分享了三年多使用 Clerk 的经验教训，以及最终选择 Better Auth 的原因。

## 核心问题

### 1. Clerk 试图同时充当用户表和会话表

作者认为不应该将用户表完全托管给第三方服务。Clerk 曾经发表博客文章和视频，建议用户"删除用户表"，但作者强烈建议不要这样做。

### 2. Clerk 作为用户表表现不佳

- **严重速率限制**: Clerk API 速率限制为每秒 5 次请求（针对整个账户），这对社交网站来说是致命的
- **社交功能不兼容**: Val Town 作为社交平台，需要显示其他用户的头像和用户名，但 Clerk 的设计假设用户只查看自己的头像
- **数据同步复杂性**: 需要通过 webhook 将 Clerk 数据同步到本地数据库，导致两个用户数据源，增加了复杂性
- **用户状态不一致**: 注册过程复杂，用户可能在某一时刻只有 Clerk 账户但没有本地数据库记录

### 3. Clerk 成为单点故障

- Cookie 方式的用户会话需要频繁刷新，每次刷新都需要调用 Clerk API
- Clerk 宕机时，整个网站不可用，而不仅仅是登录/登出功能
- 自 2025 年 5 月以来，Clerk 的可用性一直在两个九到三个九之间徘徊

## 为什么不立即迁移

1. **迁移成本高**: Val Town 不希望频繁重写系统，这会影响开发速度和团队稳定性
2. **Clerk 有优点**: Clerk 提供了完整的 SDK 支持（Remix、Fastify、Express），并且在管理后台和反滥用方面做得不错
3. **选择有限**: 大多数开源认证解决方案都很古老且缺乏维护
4. **适合场景**: Clerk 非常适合简单的、主要使用前端的应用程序

## 为什么选择 Better Auth

1. **高质量代码**: 代码质量高，与不同框架的集成良好
2. **真正的开源项目**: 核心是开源的，可以独立运行
3. **无供应商锁定**: 不再依赖第三方服务来管理会话和用户认证
4. **巧妙的商业模式**: 付费服务基本是无状态的，不参与会话管理
5. **数据自主**: 所有数据由 Val Town 自己管理

## 迁移过程

在 LLM 的帮助下，Val Town 花了两个星期同时支持 Clerk 和 Better Auth 两种认证方式。所有处理认证的端点都接受两种 cookie，用户逐渐迁移到 Better Auth。最终手动完成了纯 Better Auth 的认证系统。

## 经验教训

1. **上游依赖风险**: 系统的可靠性取决于最薄弱的关键组件
2. **产品适用性**: 产品可能对很多用例都很成功，但不一定适合你的特定问题
3. **技术演进**: 软件世界变化很快，正确的解决方案可能在你需要时还不存在，但可能一年后会出现
