## 摘要

作者 Nick Kossolapov 用 F# 编写了一个 Game Boy 模拟器 "Fame Boy"。他拥有8年软件工程师经验，但从未理解计算机实际是如何工作的，因此决定通过模拟一台游戏机来学习。

## 关键内容

### 项目背景
- 首先完成了 "From NAND to Tetris" 课程来理解计算机基础知识
- 然后用 F# 写了 CHIP-8 模拟器作为练习
- 最终完成了完整的 Game Boy 模拟器，支持声音，可在桌面和网页运行

### 技术亮点

**F# 领域建模的优势**
- 使用 F# 的类型系统对 CPU 指令进行建模，将512个操作码简化为58条指令
- 通过 `From` 和 `To` 类型确保无法表示非法状态
- 标志位处理使用内联函数，性能提升约10%

**性能优化**
- 最初使用区分联合 (DU) 进行内存地址映射，导致每秒分配数百万个对象
- 移除 DU 直接访问数组后，FPS 翻倍
- Debug 模式 vs Release 模式性能差异巨大 (约1000 FPS vs 45 FPS)

**跨平台**
- 使用 Fable 将 F# 编译为 JavaScript
- 发现 Fable 将 uint8 转换为 JS Number 时不会截断，需要手动处理

### AI 使用
- 初期故意不使用 AI 来学习，但用 AI 生成测试用例
- "Timer Winter" 期间调试20小时无果，最终 Claude Opus 几分钟内解决问题
- 后期用 AI 辅助优化性能，成功将部分基准测试性能翻倍

## 热门评论观点

### 1. F# 与性能
- F# 性能可以很好，但需要谨慎选择特性
- 在热循环中使用可变数据和数组比完全不可变数据结构更快
- 持久化数据结构是另一个工具，在某些场景下适用

### 2. Fable 的问题
- Fable 将 .NET uint8 转换为 JavaScript Number 时不截断，造成 bug
- 这类隐藏的行为变化很难检测

### 3. 关于 AI 的态度
- 有人赞赏作者不使用 AI 从零编写代码的精神
- 也有人认为即使想自己写代码，2026年没必要为一个问题折腾20小时
- AI 作为学习工具生成测试用例是合理的用法

### 4. F# 的现状
- F# 长期活在 C# 阴影下
- 与 C# 互操作性会削弱 F# 的函数式特性保证
- 有人认为 C# 正在借鉴 F# 的特性但实现得很差
