## 核心内容

### 问题描述

文章探讨了SSH协议缺少类似HTTP的"Host header"功能所带来的问题。在HTTP协议中，客户端发送请求时会包含Host header，服务器可以据此将多个域名指向同一个IP地址的流量路由到不同的后端服务。这种机制使得基于SNI（Server Name Indication）的反向代理成为可能。

然而，SSH协议没有这样的机制。当多个虚拟机（VM）共享同一个IP地址时，无法像HTTP那样通过SNI来区分不同的SSH连接目标。

### 解决方案

文章提出了一种创新方案：为每个虚拟机分配一个独立的公网IPv4地址。通过在SSH连接时使用特定的IP地址（而非域名），可以唯一标识目标虚拟机。

exe.dev的联合创始人crawshaw解释说，他们购买了一个/22的IPv4地址块（约1012个IP地址），专门用于让客户能够直接SSH到他们的虚拟机。

## 讨论热点

### IPv4与IPv6的争论

这是讨论中最热门的话题。

- **支持IPv6的观点**：建议提供IPv6-only选项，IPv4作为额外付费功能。IPv6地址资源丰富，成本更低。
- **反对IPv6的观点**：
  - exe.dev联合创始人crawshaw指出，他所在地区的唯一一家提供千兆网络的ISP不支持IPv6
  - 即使IPv6已经部署，数据中心之间的连接在双栈环境下也完全使用IPv4
  - 他们发现LAX数据中心的20台机器IPv6配置损坏，几个月都未被发现

- **全球IPv6部署现状**：
  - 瑞士：没有一家蜂窝网络提供IPv6
  - 新西兰：除一家外所有固定线路提供商都支持IPv6，但移动网络全部仅支持IPv4
  - 加拿大：Bell Canada不向家庭用户提供IPv6
  - 法国：许多4G运营商不支持IPv6
  - 许多地区数百万用户完全没有IPv6支持

### SSH密钥安全问题

另一个热门讨论点是SSH公钥的隐私问题：

- SSH默认会向服务器发送所有公钥
- 服务器可以拒绝每个密钥直到获取用户的所有公钥，从而识别用户身份
- 可通过`ssh whoami.filippo.io`测试此问题
- 建议使用SSH证书格式或为每个主机使用不同的密钥

### 技术替代方案

讨论中提到的其他解决方案：

1. **SRV记录**：支持主机名+端口的DNS记录格式，但支持有限
2. **ProxyJump**：使用跳板机作为中介
3. **端口分配**：不同虚拟机使用不同端口
4. **SSH反向代理**：使用gliderlabs/ssh库实现自定义SSH代理

## 总结

这篇洞察揭示了SSH协议在现代云服务环境中面临的独特挑战。核心问题是没有类似HTTP Host header的机制来区分共享IP上的不同服务。

社区讨论表明：
1. IPv6的全面部署仍面临重大障碍
2. IPv4地址仍然必要，但成本正在下降
3. SSH协议设计确实存在一些长期被忽视的安全和隐私问题

解决方案需要在用户体验（简单易用的域名访问）和技术实现（共享IP、IPv4/IPv6兼容性）之间取得平衡。
