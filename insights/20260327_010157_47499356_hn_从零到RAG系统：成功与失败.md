## 文章概述

本文作者分享了为公司构建企业内部RAG（检索增强生成）系统的完整经历。系统的核心需求是：基于公司近十年来的所有项目文档（原始数据约1TB），构建一个可以用自然语言提问并返回答案及原始文档引用的聊天工具，特别需要处理OrcaFlex文件（海上浮体动力学仿真软件）。

最终系统成功索引了**451GB**的文档，生成了**738,470个向量**，索引文件大小为**54GB**。

## 遇到的六大问题及解决方案

### 问题一：技术选型

**挑战**：之前从未接触过RAG系统，需要从零开始。

**解决方案**：选择成熟度高且易用的开源方案——Ollama运行本地LLaMA模型，LlamaIndex作为RAG框架，Python作为开发语言。

> 作者提示：原以为这是几周的项目，结果大错特错。

### 问题二：文档混乱

**挑战**：数据源是Azure上的一个文件夹，包含数百GB、数千个文件，格式多样，毫无结构可言。

**解决措施**：
1. 添加文件扩展名过滤，排除无价值文件类型
2. 排除规则示例：

| 类别 | 排除的扩展名 |
|------|-------------|
| 视频 | mp4, avi, mov, mkv, webm... |
| 图片 | jpg, png, gif, bmp, tiff... |
| 可执行文件 | exe, dll, bat, sh... |
| 压缩文件 | zip, rar, 7z, tar, gz... |
| 仿真文件 | sim, dat |
| 临时文件 | tmp, temp, cache, log... |
| 备份文件 | bak, old, bkp... |
| 邮件 | msg, pst, eml... |

3. 排除CSV、JSON等低价值格式

**结果**：文件数量减少**54%**，RAM不再溢出。

### 问题三：索引451GB文档

**挑战**：使用LlamaIndex默认的JSON格式存储，面临以下问题：
- 服务重启需从头开始处理所有文档（耗时数天）
- JSON格式对大规模搜索不优
- 内存溢出

**解决方案**：
- 切换到专用向量数据库ChromaDB（基于SQLite）
- 实现批处理pipeline：每批处理150个文件
- 添加检查点（checkpoint）系统，保存进度

**成果**：支持多会话索引、可恢复、便于备份。

### 问题四：GPU性能不足

**挑战**：笔记本电脑集成显卡无法满足需求，CPU处理500MB文档需4-5小时。

**解决方案**：租用配备NVIDIA RTX 4000 SFF Ada（20GB VRAM）的虚拟机。

**结果**：经过2-3周，索引完成，虚拟机关闭。成本：**184欧元**（Hetzner）。

### 问题五：用户体验

**解决方案**：
- Flask构建API
- Streamlit构建前端
- 定制LLM响应模板，显示信息来源和文档下载链接

### 问题六：磁盘空间限制

**挑战**：生产环境VM只有100GB磁盘，无法存储全部原始文档。

**解决方案**：原始文档保留在Azure Blob Storage，响应中附带带有SAS令牌的下载链接，用户可直接从云端下载。

## 学到的经验教训

1. **内存管理**：批量处理（每批150个文件），批次间显式调用垃圾回收器，释放内存后再处理下一批。

2. **错误容忍**：解析失败的文件（如损坏的PDF）记录日志后跳过，单个问题文件不应阻止整批处理。

3. **检查点系统**：保存最后完成的批次、处理的节点总数、时间戳。重启后可从中断处继续。

4. **监控脚本**：添加各种状态的监控脚本——`index-progress`、`index-watch`、`index-speed`、`index-checkpoint`、`index-failed`。

## HN评论热点讨论

### 1. RAG是否已死？

- **观点A**：有人认为随着LLM上下文窗口越来越大（如Gemini 2M），RAG变得不必要
- **观点B**：另一方认为这需要看数据规模——整个法律图书馆、Wikipedia、451GB数据仍然远超上下文容量
- **共识**：多数开发者认为RAG仍然重要，但需要更智能的实现方式

### 2. 数据质量是关键

- 评论者普遍认同：**数据 ingestion（摄取）是核心**
- 简单使用docling将PDF转markdown然后存入向量数据库的做法被批评为"荒谬"
- 高精度RAG需要专门的ETL设计，可能需要将信息分离到SQL和图数据库

### 3. 上下文质量vs数量

- 研究表明：过度填充上下文反而降低LLM性能
- 小型本地LLM在精简聚焦的上下文中表现更好
- AGENTS.md和CLAUDE.md等配置文件如果过时，会产生"自信的谎言"

### 4. 重新排序（Reranking）的重要性

- 建议使用小型便宜的embedding模型 + 单独的reranker
- 这比使用高端embedding模型更经济有效

## 结论

这是一个成功的RAG系统构建案例，尽管过程中遇到诸多挑战。作者的核心建议是：**投入时间构建高质量的数据。如果源数据不够相关，LLM无法生成好的答案。**

系统最终运行快速、可靠且对同事真正有用。
