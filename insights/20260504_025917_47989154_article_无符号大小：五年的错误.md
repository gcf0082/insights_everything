- **原文标题**: Unsigned sizes: a five year mistake
- **语言**: 英文
- **来源**: C3 官方博客

## 文章摘要

本文是 C3 语言作者对过去五年决策的反思，探讨了为什么 C3 从默认使用无符号类型（unsigned）转向默认使用有符号类型（signed）。作者承认这是一个"五年错误"，并详细解释了背后的技术原因和经验教训。

## 核心内容

### 无符号类型的已知陷阱

1. **无限循环问题**：
   ```c
   for (uint x = 10; x >= 0; x--) { ... }  // 无限循环！
   ```

2. **比较问题**：
   ```c
   uint a = 0;
   int b = -1;
   if (a > b) { ... }  // 在C中两者都提升为无符号，导致比较失败
   ```

C3 为了解决这些问题实现了安全的无符号/有符号比较。

### 关键转折点：模运算问题

C3 原本选择让 `int + uint` 提升为 `int`，以减少强制转换。但当计算 `(foo + a) % 2` 且 `foo` 超过 `INT_MAX` 时，结果变得不可理解。正确的做法是使用 `(foo + a) % 2U`。

这个问题之所以严重，是因为在其他地方"忽略符号转换"都能正常工作，导致开发者难以察觉潜在的符号问题。

### 环形缓冲区的陷阱

使用无符号类型计算偏移时：
```c
// 错误的写法（对于无符号）
index = ((start - offset_back) % length + length) % length;
// 正确的写法
index = (start + length - (offset_back % length)) % length;
```

编译器无法检测出第一种写法的错误。

### 历史背景

- **C 语言**：最初围绕有符号整数设计，`sizeof` 的类型被标准化为无符号的 `size_t`
- **Java**：90年代决定完全放弃无符号类型，以消除相关 bug
- **Go**：作为对 C++ 问题的反应，由深谙无符号类型代价的人创建，却选择了有符号类型

### 有符号类型的优势

1. **边界更安全**：对于 32 位有符号整数，边界在 ±20 亿；对于无符号是 0 到约 40 亿，无符号的"不安全边界"更接近
2. **溢出更明显**：有符号溢出产生无效的负数；无符号溢出产生看似合理但错误的值
3. **实际需求**：在现代 64 位机器上，内存耗尽前无法用完完整的有符号 64 位整数

### C3 的改变

C3 0.8.0 版本将默认的尺寸类型从 `usz`（无符号）改为 `sz`（有符号）：
- `sz` 成为默认的尺寸和长度类型
- `usz` 成为需要显式使用的类型
- 移除了有符号与无符号之间的隐式转换

作者承认，这个改变花费了比预期更长的时间，证明了"无符号类型是默认"这个习惯根深蒂固——即使 Go 和 Java 已经展示了有符号类型的正确做法。

## 结论

C3 的经验表明，无符号类型虽然看起来更"正确"（因为尺寸不可能为负），但实际上带来了更多隐蔽的 bug 和复杂性。有符号类型虽然范围较小，但边界更安全，溢出更容易被发现。这一改变被社区称为"szmageddon"（尺寸末日），但作者认为这是正确的方向。
