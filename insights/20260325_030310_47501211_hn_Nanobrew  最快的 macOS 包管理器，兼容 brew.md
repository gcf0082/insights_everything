## 项目概述

Nanobrew 是一个用 Zig 编写的 macOS 包管理器，声称是兼容 Homebrew 的最快选择。其宣传的安装时间为 3.5ms，声称比 Homebrew 快 7,000 倍。

## 主要讨论观点

### 1. 兼容性质疑

Homebrew 项目负责人 (mikemcquaid) 指出：如果 Nanobrew 不执行 Ruby，就无法真正兼容 Homebrew。因为 Homebrew 的配方 (formulae) 是用 Ruby 编写的，安装脚本可以执行任意计算。要实现完全兼容，需要运行 Ruby。他同时提到 Homebrew 官方正在开发 Rust 前端，这将实现真正的完整兼容。

### 2. 速度重要性辩论

- **支持提速的观点**: 用户反映 Homebrew 更新/升级过程非常缓慢，尤其是在安装小工具时可能需要先等待 20 分钟的自动更新。类比 Linux 世界中 APT (C++) 比 yum (Python) 快，最终 yum 被 dnf (C++) 取代。
- **质疑提速的观点**: 有用户认为安装包并不频繁，每次节省几秒钟的体验影响不大。更有人比喻：人们每天花在洗澡、倒水等事情上的时间远比运行包管理器长。

### 3. 与其他工具的对比

- **Zerobrew**: 另一个实验性的 Homebrew 替代方案，建议与 Homebrew 并行使用而非完全替代。支持 Linux。
- **Nix**: 有用户从 Homebrew 迁移到 Nix 以获得更高效的包管理体验。
- **Bun/pnpm**: 作为 Node.js 包管理器的更快替代品，最终 npm 仍然是默认选项，但用户会根据需求选择。

### 4. 技术细节

- Nanobrew 使用 Homebrew 的 JSON API 来获取包元数据
- 使用自己的依赖解析器和链接器来拉取 Homebrew 的预编译包 (bottles)
- 当前版本不支持 casks 命令，无法安装某些应用

### 5. 其他问题

- 有用户指出 Nanobrew 将包安装到与 Homebrew 不同的目录，移除 Nanobrew 后无法用 Homebrew 管理这些包
- 讨论了 Homebrew 放弃对旧版 macOS 支持的问题 (只支持最新 3 个版本)
- 对 Gatekeeper 相关的变更表示担忧

## 总结

Nanobrew 代表了 macOS 包管理器领域的一个新尝试，专注于性能提升。然而，社区对其"完全兼容" Homebrew 的声明持怀疑态度，因为完全兼容需要处理 Ruby 配方。Homebrew 官方的 Rust 前端正在开发中，可能会提供更权威的解决方案。用户对速度与兼容性之间的权衡也有不同看法，但普遍认为竞争有助于推动主流工具改进性能。
