## 内容摘要

Python 3.14 引入了一个全新的增量垃圾回收器（Incremental GC），但在生产环境中收到了大量关于显著内存压力的报告。Python 团队决定在 Python 3.14 和 3.15 两个版本中都回滚此项更改，恢复使用 Python 3.13 的分代垃圾回收器（Generational GC）。

### 关键问题

1. **内存泄漏问题**：有用户在实际生产环境中发现服务内存持续增长，经调查发现是 HTTP 客户端（aiohttp）为每个请求重新创建，导致 SSL 库的内存增长速度超过释放速度。

2. **增量GC的局限性**：增量GC在某些边缘情况下表现不佳，特别是涉及引用循环时。有用户指出 httpx 库创建的引用循环是增量GC的最坏情况。

3. **缺乏PEP流程**：该更改没有经过Python增强提案（PEP）流程，引发了社区关于是否应该对这类核心运行时更改进行更严格审核的讨论。

### 社区反应

- 大多数开发者认为回滚是正确的决定
- 有开发者担心Python会变得像Java一样需要大量调优
- 有人建议如果需要JIT性能可以选择PyPy，但PyPy支持滞后
- 部分开发者认为应该考虑使用Go等语言

### 技术细节

- 分代GC自2000年由Guido van Rossum合并，经过 decades 的实际使用和优化
- 增量GC虽然对大多数用户有效，但在少数情况下会使用更多内存
- Python 3.15 将继续提供性能分析器功能

---

## 总结

Python团队在3.14版本中引入的增量垃圾回收器由于在生产环境中引发显著的内存压力问题，已决定在3.14和3.15版本中回滚至传统的分代垃圾回收器。这一决定得到了社区的广泛支持，同时也引发了对Python核心运行时更改应否经过更严格审核流程的讨论。
