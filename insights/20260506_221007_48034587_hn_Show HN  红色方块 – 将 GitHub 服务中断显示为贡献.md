# 洞察报告：Red Squares – GitHub outages as contributions

**链接**: https://news.ycombinator.com/item?id=48034587

**项目地址**: https://red-squares.cian.lol/

**发布时间**: 2026年5月6日（3小时前）

**评分**: 465 points | 94 comments

## 项目概述

Red Squares 是一个创意可视化网站，将 GitHub 的宕机时间以类似 GitHub 贡献图的形式呈现。不同于传统的状态页面，该项目使用红色方块来表示 GitHub 不可用的时间，用颜色深浅表示宕机严重程度。

## 关键发现

### 1. 工作日 vs 周末宕机率差异显著

- 工作日正常运行时间: 89.3%
- 周末正常运行时间: 96.5%
- 62% 的工作日发生过_incident，而周末仅 11%
- 周二至周四 是"危险区域"，周日几乎是另一个服务

### 2. 官方与非官方状态页差异巨大

- 官方 GitHub 状态页显示近乎 100% 可用
- 第三方独立状态页（如 mrshu.github.io/github-statuses）显示真实可用性约85%
- 官方 SLA条款与实际用户体验差异引发争议

### 3. 可能原因分析

**技术层面**:
- Azure 基础设施问题频繁
- GitHub 部署频率高（每60-90分钟一次）
- AI/Agent 代码激增导致负载无法应对

**管理层**:
- 优先推送新功能而非核心服务稳定性
- 使用微软内部技术栈而非原有架构
- 将资源投入 AI 工具开发而非基础架构

### 4. 用户反应与建议

- 部分用户转向自托管方案（Forgejo、codeberg）
- 建议 GitHub 实施"周末部署"策略
- 呼吁引入真正的容错和回滚机制

---

## 总结

Red Squares 用极具创意的方式揭示了 GitHub 在工作日的高频宕机问题。该项目引发的讨论反映了用户对 GitHub 稳定性下降的不满，尤其是在微软收购后。有趣的是，宕机时间分布与工作日高度相关，暗示部署频率可能是主要因素之一。
