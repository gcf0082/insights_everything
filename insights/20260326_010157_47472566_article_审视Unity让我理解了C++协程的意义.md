## 核心观点

作者通过研究 Unity 如何在 C# 中使用协程，终于理解了 C++ 协程的真正价值所在。

## 主要内容

### 1. 协程的实际应用场景

- C++ 协程出现 6 年了，但生产代码中很少见到
- 原因：协程本身是底层特性，需要复杂的底层代码集成
- 更重要原因是缺乏具体示例（不像 async IO 有明确需求）

### 2. Unity 的启发

Unity 在 C# 中使用协程处理"短暂行为"（如特效生成）：

```csharp
IEnumerator Fade()
{
    for (float alpha = 1f; alpha >= 0; alpha -= 0.1f)
    {
        c.a = alpha;
        renderer.material.color = c;
        yield return null;  // 等待下一帧
    }
}
```

### 3. 为什么需要协程？

简单特效可以用 lambda + 可变变量实现，但复杂序列操作（如 TimeWarp 示例）会变成**状态机**，代码丑陋难读：

- 跳跃到左边
- 向右走4步
- 举手
- 再跳一次

协程将此状态机转换为**易读的函数**，这是协程的核心价值。

### 4. C++23 实现方案

使用 `std::generator<std::monostate>`：

```cpp
std::generator<std::monostate> TimeWarp(GameObject& obj)
{
    obj.transform.position.x -= 1.f;
    co_yield {};  // 等待下一帧
    
    for (int i = 0; i < 4; ++i)
    {
        obj.transform.position.x += 0.2f;
        co_yield {};
    }
    // ...
}
```

### 5. 简单的协程执行器

作者不到一小时实现了一个 Unity 风格的协程运行器（约 100 行代码），核心思路：
- 维护 effects 列表和迭代器
- 每帧推进所有迭代器
- 移除已完成的协程

### 6. 额外收益

可以改为返回可绘制对象而非副作用，从而实现**并行化**（使用 TBB）。

---

## 总结

协程最适合场景：**将复杂的状态机转换为易读的顺序代码**。Unity 的做法证明，即使使用"hack"（用 yield 表示等待下一帧），也比手动管理状态机更优雅。C++26 的 `execution` 库将提供更完善的协程支持，但现有的 generator 方案已足够解决实际问题。
