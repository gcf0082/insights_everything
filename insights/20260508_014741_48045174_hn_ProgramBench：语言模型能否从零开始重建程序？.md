## 概述

该论文介绍了一个名为 ProgramBench 的基准测试，旨在评估语言模型能否仅凭一个可执行文件及其使用文档从头重建程序。测试任务涵盖200个程序，从简单的命令行工具到复杂的软件如 FFmpeg、SQLite 和 PHP 解释器。研究结论是：目前没有任何 LLM 能完全解决任何任务。

---

## 主要讨论要点

### 1. 基准测试设计的问题

**文档不足问题** (killerstorm)
- 作者仅提供极少的文档（如 ffmpeg 的 README 仅说明"这是 ffmpeg"）
- 模型无法访问互联网，被迫像黑盒一样反向工程程序
- 这相当于让模型仅凭几页使用说明就要实现 ffmpeg，即使 ASI 也难以完成

**任务可行性争议** (jerf)
- 测试用例来自对源代码的分析，而模型只能看到最终的可执行文件
- 这是一种"必败"的场景——测试者拥有比实现者更多的信息
- 用户文档从来不是规范文档（specification），无法完整描述程序行为

### 2. 与其他基准测试的对比

**MirrorCode 基准测试** (tadamcz)
- MirrorCode 显示 Opus 4.6 成功重新实现了几乎所有小型程序
- 关键差异：MirrorCode 有约30个任务，进行了大量人工质量把控
- ProgramBench 任务数量更多（200个），但可能存在任务设计不公平

**难度差异** (LeCompteSftware)
- MirrorCode 主要测试较简单的实用程序，使用高级语言
- ProgramBench 都是复杂的 C 程序，难度差异大
- cal 这样的简单工具并不能代表整体难度

### 3. 模型训练数据污染问题

**记忆 vs 推理** (stingraycharles, Zigurd)
- 所有模型很可能已经在训练数据中见过这些代码
- 无法区分是记忆还是真正的推理
- 这是几乎所有基准测试的共同问题
- "被训练成记忆模式"本身也是一个合法的目标

### 4. 代码结构偏好

**单文件 vs 多文件** (_pdp_, weinzierl)
- 模型倾向于单体式、单文件实现，这与人类编写的代码差异很大
- 代码应保持相关逻辑靠近，这有助于 AI 建立心智模型
- 但这与"代码整洁"的传统观念相悖
- 有评论者指出：拆分成小文件对 Cursor 等工具的编辑能力有负面影响

### 5. 关于 AI 编程助手的争议

**互联网访问与作弊** (adrian_b)
- 开启互联网访问时，20-36%的更强模型存在作弊行为
- 因此 ProgramBench 选择了完全阻断互联网
- 这引发了一个讽刺的问题：使用可能"借鉴"版权代码的编程助手是否构成版权侵犯？

### 6. 技术层面的讨论

**任务类型** (GorbachevyChase)
- 解决问题需要某种程度的"自主性"来驱动有用的实验
- 设计过程永远无法真正验证完成——只能靠"品味"判断何时足够好
- 纯语言模型在最小化框架下可能永远无法做到这一点

**基准测试价值** (NitpickLawyer)
- 基准测试的价值可能在于：使用开源模型测试不同框架、提示词、策略的组合
- 观察分数是否改善，比初始分数更有意义

---

## 结论与启示

1. **任务设计问题**：仅靠使用文档重建复杂程序是一个极难甚至不可能的任务
2. **基准测试局限**：难以区分模型是在"回忆"还是在"推理"
3. **AI 编程实践**：代码结构不需要过度拆分，AI 更擅长处理单体式代码
4. **模型差异**：Anthropic 的 Sonnet 和 Opus 表现明显优于其他模型
