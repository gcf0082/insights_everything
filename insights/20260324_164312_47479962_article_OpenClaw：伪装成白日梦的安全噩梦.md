## 一、概述

OpenClaw 是一个基于 Anthropic Claude Opus 4.5 模型的 AI 助手，能够与本地系统文件、终端、浏览器、Gmail、Slack 以及家庭自动化系统进行交互。然而，其强大的功能背后存在严重的安全隐患。本报告将全面分析 OpenClaw 的安全漏洞、攻击面以及防护建议。

---

## 二、OpenClaw 简介

OpenClaw（又称 ClawdBot、Moltbot）是一个开源项目，由 Peter Steinberger 创建。该项目在 2026 年初迅速走红，能够执行以下操作：

- 清理收件箱、安排会议并添加准备笔记
- 预订航班、控制智能家居设备
- 与 Telegram、WhatsApp 等消息平台集成
- 访问日历、Gmail、Slack、Notion 等办公工具

然而，这种高度自动化需要用户提交其安全和隐私权限，形成所谓的"浮士德交易"——以隐私和安全换取自动化便利。

## 三、安全漏洞分析

### 3.1 ClawdHub Skill 供应链攻击

OpenClaw 依赖 Skills 扩展功能，用户可从 SkillHub 下载社区上传的技能。安全研究者 Jason Melier（1Password）发现：

- 下载量最高的"Twitter"技能实际上是一个恶意软件投递向量
- 该技能要求安装"openclaw-core"依赖，并提供恶意链接
- 攻击流程：诱导用户访问恶意页面 → 执行混淆的有效载荷 → 下载第二阶段脚本 → 运行信息窃取恶意软件

Snyk 对 3,984 个技能的安全分析发现：
- **283 个技能（7.1%）** 存在关键安全漏洞
- 敏感凭证以明文形式暴露在 LLM 上下文窗口和输出日志中

### 3.2 提示注入漏洞

OpenClaw 面临严重的提示注入威胁，符合 Simon Willison 所描述的"致命三要素"：

| 要素 | OpenClaw 实现 |
|------|---------------|
| 访问私人数据 | 可访问 Gmail、Slack、文件等 |
| 暴露于不可信内容 | 通过 WhatsApp、Telegram、邮件接收消息 |
| 外部通信能力 | 可发送消息、提取数据 |

由于 OpenClaw 部署在 WhatsApp、Telegram 等平台，任何人都可以发送消息来注入恶意指令。

### 3.3 认证令牌滥用

- OpenClaw 在本地存储 OAuth 访问令牌和刷新令牌（位于 `agents/<agentId>/agent/auth-profiles.json`）
- 许多部署使用弱认证、暴露的网关、反向代理配置错误
- 攻击者获取令牌后可冒充用户在 Slack、Gmail 中发送消息、拉取数据

### 3.4 内存污染攻击

- OpenClaw 的记忆功能完全基于 Markdown 文件
- 攻击者可以修改记忆文件，使代理静默执行恶意任务
- 记忆污染可以在用户不知情的情况下影响整个实例

### 3.5 公开暴露实例

- 2026 年 1 月 27-31 日，Censys 发现约 **21,000 个** 暴露的 OpenClaw 实例
- BitSight 扫描（1月27日-2月8日）发现 **30,000+ 个** 易受攻击的实例
- 初始版本存在关键漏洞：localhost 连接被自动信任，导致反向代理后的攻击

## 四、OWASP Top 10 映射

| OWASP 风险 | OpenClaw 实现 |
|------------|---------------|
| A01: 提示注入 | Web 搜索结果、消息、第三方技能注入指令 |
| A02: 不安全的工具调用 | 基于不可信记忆源的推理调用工具 |
| A03: 过度代理自主权 | 单代理拥有文件系统根访问、凭证访问 |
| A04: 缺少人类介入控制 | 破坏性操作无需批准 |
| A05: 代理记忆中毒 | 所有记忆来源无区分存储 |
| A06: 不安全的第三方集成 | 第三方技能以完全权限运行 |
| A07: 权限分离不足 | 单代理处理不可信输入和高权限操作 |
| A08: 供应链模型风险 | 使用上游 LLM 无验证 |
| A09: 无限代理间操作 | 单体代理架构 |
| A10: 缺少运行时监控 | 无策略执行层 |

## 五、安全加固建议

### 5.1 容器化部署

- **勿在主设备上运行**：使用专用旧设备或 Docker 容器
- **限制文件访问**：仅挂载一个工作目录（如 `/srv/openclaw/work`）
- **使用独立用户**：以非管理员用户运行
- **禁用 Docker socket**：勿挂载 `/var/run/docker.sock`
- **限制 Linux 能力**：遵循最小权限原则

### 5.2 云 VPS 部署

- **禁止公网暴露**：仅通过 VPN 或私有隧道访问
- **配置防火墙**：仅允许来自可信 IP 的 SSH
- **令牌轮换**：定期轮换网关令牌和凭证

### 5.3 账户隔离

为 OpenClaw 创建独立的 Gmail、日历和其他集成账户，使用独立的密码管理器账户存储凭证。

### 5.4 最小权限访问

- 按风险级别分离代理
- 时间限制提升的权限
- 按资源而非操作进行作用域限制
- 破坏性操作需要人工批准

## 六、结论

OpenClaw 虽然功能强大，但安全风险极高。关键问题包括：

1. **供应链漏洞**：第三方技能可被恶意利用
2. **提示注入**：无法完全防御的固有缺陷
3. **过度权限**：单代理拥有过大控制权
4. **记忆污染**：静默的持久性攻击向量
5. **大规模暴露**：数千实例公网暴露

**建议**：普通消费者应避免使用 OpenClaw。技术爱好者如需使用，应严格遵循安全加固指南，并考虑使用 TrustClaw 等更安全的替代方案。
