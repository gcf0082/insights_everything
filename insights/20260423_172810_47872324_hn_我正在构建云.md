## 核心洞察摘要

David Crawshaw（Tailscale 联合创始人）宣布正在构建一家新的云服务公司 exe.dev，并获得了 A 轮融资。他在博客中深入剖析了当前云计算的问题，以及为什么他认为现在是解决这些问题的最佳时机。

---

## 关键观点

### 1. 对现有云的不满

Crawshaw 直言不讳地表达了对现有云服务的失望：

> "I do not like the cloud today."

他认为现有云服务的核心抽象模型是错误的，包括：
- **VM 资源模型错误**：用户被迫购买固定的 CPU/内存组合，而非购买资源后运行任意数量的 VM
- **存储性能问题**：SSD 的seek时间从10ms降至20μs，但云厂商仍在使用远程块设备，网络开销导致性能下降超过10倍
- **网络成本过高**：云厂商的流量出口费用是普通数据中心的10倍

### 2. Kubernetes 无法解决根本问题

文章中最犀利的观点之一：

> "Making Kubernetes good is inherently impossible, a project in putting (admittedly high quality) lipstick on a pig."

他认为 Kubernetes 是一个试图解决"不可能问题"的产品——让云既可移植又可用，但这在本质上无法实现。

### 3. AI Agent 改变游戏规则

Crawshaw 认为 AI Agent 的出现使得更好的云计算基础设施成为必需：

- Agent 使编写代码变得更容易，将产生更多软件（Jevons 悖论）
- 现有云抽象迫使 Agent 在思考如何适配云上浪费大量上下文窗口
- 需要更简单的云基础设施，让 Agent 和人类都能更高效

### 4. exe.dev 的解决方案

exe.dev 试图解决的核心问题：
- **VM 隔离**：用户购买 CPU 和内存，运行自己想要的 VM
- **本地 NVMe 存储**：块设备异步复制到其他机器
- **全球区域部署**：让机器靠近用户
- **Anycast 网络**：为用户提供低延迟入口点
- **TLS 代理和认证代理**：保护新 VM 不直接暴露在公网

### 5. 社区讨论焦点

HN 社区讨论的核心议题：

| 主题 | 观点 |
|------|------|
| **Kubernetes 过度使用** | 许多开发者用 K8s 处理小型应用，用 "简历驱动开发" 来解释这一现象 |
| **简单架构的价值** | 单 VM + Debian + Kamal 的组合比复杂 K8s 集群更稳定、更便宜 |
| **Hetzner 的优势** | 与 AWS/DO 相比，Hetzner 提供更好的性价比（测试显示 IOPS 差距可达100倍） |
| **云成本结构** | 云计算定价并非基于成本，而是基于市场愿意支付的价格 |

## 延伸讨论

### LLMs 是否会导致更多低质量软件？

部分评论者担忧 AI Agent 会产生大量无人使用的软件（"shovelware"），但也有人认为这是解放——人们可以构建满足个人需求的定制化软件，而不是为亿万人构建通用应用。

### 传统运维是否因 LLM 变得更容易？

多位评论者表示，LLM 已经可以高效完成大多数系统管理任务，且 token 用量较少（相比复杂的编程任务）。

## 总结

这篇洞察揭示了云计算领域的一个深刻矛盾：现有的抽象模型阻碍了效率和创新能力。Crawshaw 的愿景是通过更接近金属的云服务，让开发者和 AI Agent 都能更高效地工作。这是否意味着云服务正在经历从"抽象优先"到"简化优先"的范式转变，值得持续关注。
