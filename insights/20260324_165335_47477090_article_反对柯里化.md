## 文章总结

### 主题概述

本文探讨了函数式编程中的柯里化（currying）问题，作者从理论角度分析了柯里化风格的优缺点，并提出了对这种广泛使用的技术的质疑。

### 三种多参数函数定义风格

文章介绍了编程语言中定义多参数函数的三种主要风格：

1. **参数列表风格**（imperative style）：多个参数作为函数的固有特性，如Rust中的`fn f(p1: P1, p2: P2, p3: P3) -> R`
2. **柯里化风格**（curried style）：将n参数函数定义为逐步应用的形式，如Haskell中的`f p1 p2 p3 = ...`
3. **元组风格**（tuple style）：函数只有一个参数，该参数是包含多个值的元组，如`f(p1, p2, p3) = ...`

理论上，这三种风格是同构的：类型`(P1, P2) -> R`和`P1 -> P2 -> R`之间存在一对一映射。

### 柯里化的优势与误区

**部分应用（Partial Application）**

文章指出，人们普遍认为柯里化风格的优势在于使部分应用变得简单直接。然而，作者认为这是一个误解。通过一些语法糖，元组风格的函数同样可以实现优雅的部分应用，而且可以支持非首个参数的固定，这在柯里化风格中默认是不支持的。

**"它很酷"**

作者承认柯里化风格确实很elegant，从感性层面让人感受到函数式编程的"矩阵"之美，但仅仅因为优雅并不意味着它就是最佳选择。

### 柯里化风格的问题

1. **性能开销**：每次调用多参数柯里化函数都会创建中间函数对象，虽然优化器可以消除大部分开销，但仍然存在一定成本。

2. **类型不对称**：函数的类型签名`P1 -> P2 -> P3 -> R`与标准的`In -> Out`形式不匹配，导致组合不顺畅。当函数需要多个输入时采用 stagger 形式，而多个输出时却返回元组，这种不对称性会带来问题。

3. **实际项目中的困扰**：作者在Rocq证明助手中处理谓词时发现，柯里化风格的函数类型无法与`In -> State Out`统一，导致类型问题。

### 元组风格的优势

- 更清晰的数据流展示：`foldr((+), 0, $) . map(const(1), $)`展示了数据流入的"流程"
- 支持任意位置的部分应用
- 类型签名更符合直觉：`In = (P1, P2, P3)`和`Out = R`

### 结论

作者认为，虽然函数式语言不会因为一篇文章就改变传统，但语言设计者可以考虑尝试元组风格配合替代的部分应用语法。作者同时承认，对于某些高阶函数（如`map`和`fix`），柯里化定义确实更优雅。

### 补充：依赖类型的例外

在依赖类型语言（如Coq/Agda）中，柯里化风格仍然是 superior 的，因为依赖元组的语法更为笨重。
