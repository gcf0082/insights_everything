## 核心观点

Ollama是运行本地LLM最流行的方式，但事实不应该如此。它最初通过成为第一个让不擅长编译C++或编写服务器配置的人能够使用llama.cpp的工具而获得关注，这是一个真正的贡献，但很短暂。该项目随后多年来系统性地掩盖其实际技术的来源，在用户关于他们运行的是什么的问题上误导用户，并背离了为其赢得信任的本地优先使命，同时吃着风投资金。

---

## 关键问题

### 1. 许可证归属问题

- Ollama的整个推理能力来自**llama.cpp**，这是Georgi Gerganov于2023年3月创建的C++推理引擎
- 一年多来，Ollama的README**没有提到llama.cpp**
- 项目二进制分发**没有包含llama.cpp代码所需的MIT许可证通知**
- GitHub issue #3185于2024年初提出许可合规请求，**超过400天没有收到维护者的回复**
- 2024年4月 issue #3697特别要求llama.cpp确认后，社区PR #3700在几小时内跟进
- Ollama联合创始人Michael Chiang最终只在README底部添加了一行："llama.cpp项目由Georgi Gerganov创立"

### 2. 性能问题

- 多个社区测试显示**llama.cpp运行速度比Ollama快1.8倍**（161 tokens/秒 vs 89 tokens/秒）
- CPU上差距为**30-50%**
- 最近Qwen-3 Coder 32B的比较显示llama.cpp**吞吐量高出约70%**
- 性能开销来自：Ollama的守护进程层、糟糕的GPU卸载启发式方法，以及落后于上游的 vendored 后端

### 3. 误导性模型命名

- DeepSeek R1发布后，Ollama将小型蒸馏版本简单地标记为"DeepSeek-R1"
- 实际上运行的是8B Qwen衍生蒸馏模型，与真正的6710亿参数R1完全不同
- GitHub issues #8557和#8698要求分离模型，均被关闭为重复项，没有修复

### 4. 闭源应用发布

- 2025年7月，Ollama发布了macOS和Windows的GUI桌面应用
- 应用在私人仓库开发，**没有发布许可证**，源代码不公开
- 网站将下载按钮放在GitHub链接旁边，给用户印象他们下载的是MIT许可的开源工具
- 社区成员发现二进制中可能有AGPL-3.0依赖

### 5. Modelfile：重新发明轮子

- GGUF格式设计原则：**单文件部署**，所有加载模型所需信息都包含在模型文件中
- Ollama添加了Modelfile作为配置层，实际上是重复发明轮子
- 用户报告：更改一个参数需要复制整个模型（30-60GB）
- 模板检测问题：Ollama只从硬编码列表中自动检测它已知的聊天模板

### 6. 注册表瓶颈

- 新模型发布后，GGUF通常在几小时内出现在Hugging Face上
- 使用llama.cpp可以立即运行：`llama-server -hf unsloth/Qwen3.5-35B-A3B-GGUF:Q4_K_M`
- 使用Ollama需要等待：需要有人为其注册表打包模型，选择量化选项，转换聊天模板
- **量化限制**：Ollama仅支持Q4_K_S、Q4_K_M、Q8_0、F16、F32量化
- 如果需要Q5_K_M、Q6_K或IQ量化，只能在Ollama之外自行量化

### 7. 云端转向

- 2025年末，Ollama在本地库旁边引入了云托管模型
- proprietary模型如MiniMax出现在模型列表中，**没有明确披露这会将数据发送出机器**
- 用户提出了关于数据路由的担忧
- **CVE-2025-51471**：令牌泄露漏洞，影响所有Ollama版本

### 8. VC模式

- Ollama是Y Combinator支持（W21）的初创公司
- 创始团队之前构建了被Docker Inc.收购的Docker GUI Kitematic
- 模式：**Launch on open source** → **Minimize attribution** → **Create lock-in** → **Launch closed-source components** → **Add cloud services**

## 替代方案

| 工具 | 描述 |
|------|------|
| **llama.cpp** | 引擎，有OpenAI兼容API服务器、内置Web UI、更好的吞吐量 |
| **llama-swap** | 多模型编排，热交换模型 |
| **LM Studio** | GUI，使用llama.cpp作为引擎，支持任何GGUF模型，无锁定 |
| **Jan** | 开源桌面应用，清晰的聊天界面，本地优先设计 |
| **Msty** | 精美的GUI，多模型支持，内置RAG |
| **koboldcpp** | Web UI和广泛配置选项 |
| **ramalama** | Red Hat的容器原生模型运行器，明确标注上游依赖 |

## 社区评论摘要

### 支持原文观点

- 许多用户认为Ollama解决了用户体验问题，但命名和文档令人困惑
- llama.cpp现在有默认安装的GUI， times have changed
- **UX是技术采用的重要因素**
- Ollama的哈希blob存储意味着切换工具需要重新下载所有内容
- llama.cpp性能显著更快

### 反对观点

- Ollama确实更容易使用，特别是对于普通用户
- 两者可以互补，取决于具体需求
- Ollama Cloud服务对测试各种开源模型很有用
- 许可问题属于法律范畴，不应混入道德考量

### 技术讨论

- `brew install llama.cpp` + `llama-server -hf ggml-org/gemma-4-E4B-it-GGUF --port 8000` 可以立即获得Web UI和OpenAI API
- llama.cpp现已支持MCP
- llama-swap可以处理多模型管理
- llama.cpp路由器模式支持热交换模型

---

## 结论

**本地LLM生态系统不需要Ollama。它需要llama.cpp。其余都是包装，而更好的包装已经存在。**

Georgi Gerganov在2023年3月的一个晚上开发了llama.cpp，开启了本地AI的革命。他和数百名贡献者多年来致力于使越来越强大的模型能够在消费级硬件上运行。这项工作真正重要，它是保持本地推理开放和可访问的基础。

Ollama用不错的CLI包装了这项工作，在其基础上筹集VC资金，花费一年多拒绝归功于此，然后叉分它弄糟，发布闭源应用，然后将其完全转向云服务。在每一个本可以成为良好开源公民的决策点，他们都选择了让自己在投资者面前看起来更自力更生的道路。
