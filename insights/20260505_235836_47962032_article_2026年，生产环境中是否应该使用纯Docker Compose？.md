## 摘要

本文探讨了在2026年是否仍应使用纯Docker Compose运行生产工作负载。作者Philip Miglince（Distr联合创始人）认为，纯Docker Compose仍然可以运行生产工作负载，但需要自行处理其留下的操作空白，包括：孤儿容器清理、健康检查自动恢复、镜像摘要固定、socket安全和更新推送等。

## 核心观点

### 简短答案

**是的**—纯Docker Compose在2026年仍可运行实际的生产工作负载，但前提是你自己处理它留下的操作空白。

### Docker Compose的适用场景

- 单节点部署
- 向客户环境推送多容器应用
- 不需要Kubernetes集群的内部长期服务
- 零售等边缘位置部署

### 主要操作挑战及解决方案

#### 1. 孤儿容器问题

**问题**：从docker-compose.yaml中移除服务后，运行`docker compose up -d`，旧容器仍会继续运行。

**解决方案**：
```bash
docker compose up -d --remove-orphans
docker compose down --remove-orphans
```

#### 2. 镜像和日志清理

**问题**：每次`docker compose pull`都会保留旧镜像；默认日志驱动无限制写入。

**审计命令**：
```bash
docker system df
docker system df -v
```

**清理命令**：
```bash
docker image prune -a --filter "until=168h" -f
docker container prune -f
docker builder prune -f
```

**日志限制配置**（/etc/docker/daemon.json）：
```json
{
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "10m",
    "max-file": "3"
  }
}
```

#### 3. 健康检查不会自动重启

**问题**：Docker报告容器状态为`unhealthy`，但不会自动重启。

**解决方案**：
- 使用autoheal sidecar（如`willfarrell/docker-autoheal`）
- 使用Docker Swarm（会自动重启不健康的任务）
- 使用Distr Docker agent

#### 4. 镜像摘要固定

**问题**：Docker标签是可变的，`myapp:1.4`今天和明天可能不同。

**解决方案**：使用摘要而非标签：
```yaml
services:
  app:
    image: myapp@sha256:9b7c0a3e1f...
```

#### 5. Docker Socket安全风险

**问题**：挂载`/var/run/docker.sock`的容器拥有主机的root权限。

**建议**：
- 盘点了挂载socket的容器
- 尽可能运行rootless Docker
- 考虑使用socket-proxy（如Tecnativa的docker-socket-proxy）
- 保持挂载socket的镜像最小化

#### 6. 更新推送

**问题**：Docker没有内置机制将新manifest推送到运行中的主机。

**常见方案**：
- Watchtower：轮询registry，容易设置但难以控制
- bastion + SSH + Ansible：适合少量客户
- Pull-based agent：推荐方案，如Distr agent

### 何时该考虑Kubernetes

当单节点Compose部署无法满足需求时，Kubernetes是更合适的选择：
- 生态系统庞大
- 操作模式成熟
- 人才库丰富

**Docker Swarm是另一个选项**：复用Compose YAML格式，但采用率已停滞，第三方工具较少。

## TL;DR 摘要

- 每次`compose up`和`compose down`时使用`--remove-orphans`
- 在daemon.json中限制容器日志，定期清理镜像
- 健康检查不会自动愈合，运行autoheal sidecar或使用Swarm
- 使用`@sha256:...`摘要固定镜像
- socket等同root权限，谨慎盘点和限制

- 当Compose不够用时，Kubernetes通常是更好的下一步

## 结论

纯Docker Compose在2026年仍可运行许多实际的生产工作负载，但需要接受"纯Compose"等同于"Compose加上操作实践"这一现实。这些问题都不难解决，但需要在2am发现它们之前就做好预防措施。
