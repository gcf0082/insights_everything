# 洞察报告：为什么我们仍在使用 Markdown？

**洞察链接：** https://bgslabs.org/blog/why-are-we-using-markdown/

**来源：** BGs Labs

**发布日期：** 2026年3月2日

**标签：** Markdown, CommonMark, Standards, Rant

**阅读时长：** 13分钟

---

## 核心摘要

本文是一位开发者对 Markdown 这一广泛使用的标记语言的深度"吐槽"。作者从形式语言理论的角度分析了 Markdown 的设计缺陷，认为 Markdown 既想作为简单的标记语言，又被期望具备编程语言的能力，导致其在两方面都做得不够好。文章最后探讨了可能的替代方案。

---

## 详细洞察

### 一、Markdown 的优点

1. **极低的学习曲线**：语法直观，普通人看一眼速查表就能上手
2. **可读性强**：语法即输出，`**加粗**` 永远输出 `<b></b>`
3. **最小化设计**：本意是将 Markdown 文件 1:1 转换为 HTML

---

### 二、Markdown 的问题

#### 问题一：语法歧义（Exhibit A）

同一种效果有多种写法：

- **加粗**：`**bold**`、`__bold__`、`<b>bold</b>`
- **斜体**：类似地有多种方式

甚至可以组合出混乱的嵌套写法，如 `***Peter* Piper** _Picked___a___Pack_ *of** Pick_led_* Peppers`，导致解析器产生 ReDoS（正则表达式拒绝服务）漏洞，如 CVE-2025-10666750（markdown-it 库）。

#### 问题二：内联 HTML 的灾难（Exhibit B）

Markdown 允许内联 HTML，但这引入严重问题：

- 解析 Markdown 还需要解析 HTML，形成双重复杂性
- 引入大量 XSS 安全漏洞（作者列出了多个 CVE）
- 攻击面随插件、钩子等功能扩展而增大

#### 问题三：过时且晦涩的语法（Exhibit C）

Markdown 诞生于 2000 年代，继承自 email 和 Usenet 的标记传统，导致：

- 标题有两种写法：Setext 风格和 ATX 风格
- 分割线有多种写法，其中部分与标题语法冲突
- 列表、数字有序列表的语法不直观
- 脚注语法使语言从上下文无关文法（CFG）退化为上下文敏感文法（CSG）

作者比喻：Markdown 是"标记语言中的 C++"——几乎所有东西都有两种写法，其中一些可能导致 XSS 和内存泄漏。

#### 问题四：解析简单，使用困难（Exhibit D）

- **解析容易**：标准 Markdown 只是简单的转译器
- **渲染困难**：现代需求要求footnotes、自定义 callouts、LaTeX 数学公式、PlantUML/Mermaid 图表、CSS 样式、shortcodes、标签、参考文献等

这意味着 Markdown 解析器需要变成一个完整的编译器。

---

### 三、形式语言理论视角

作者引用形式语言理论来论证观点：

| 类型 | 名称 | 生成器 |
|------|------|--------|
| 0 | 递归可枚举 | 图灵机 |
| 1 | 上下文敏感 | 线性有界自动机 |
| 2 | 上下文无关 | 下推自动机（大多数编程语言语法） |
| 3 | 正则 | 有限自动机（简单模式、Regex） |

- **正则语言**：Regex 可解析
- **上下文无关语言**：BNF/EBNF 可描述
- **上下文敏感语言**：如 C++ 语法

Markdown 的脚注等特性使其成为上下文敏感语言，失去了简单可解析的优势。

**图灵完备性**：作者指出，如果一个语言是图灵完备的，它就是编程语言。Markdown 本应是简单的标记语言，却因功能膨胀被当作编程语言使用。

---

### 四、可能的解决方案

| 方案 | 提出者 | 评价 |
|------|--------|------|
| 纯文本 | 资深汇编工程师 | 美观但不适用于普通人 |
| MDX | 创业者 | 试图成为 HTML，但失去了可读性 |
| ReStructuredText (.rst) | C 开发者 | 适合阅读但不适合编写 |
| 自定义构建系统 | 作者 | 需要明确定义语法、钩子，禁止内联 HTML |

**作者的核心观点**：我们需要的是一个带有构建系统的、明确的、可读性强的标记语言，而不是一个功能膨胀的怪物。

---

## 结论

Markdown 的问题在于它试图在简单标记语言和复杂文档系统之间两边不到岸。真正的解决方案不是继续修补 Markdown，而是构建一个具有以下特性的新系统：

1. **明确的、无歧义的语法**
2. **不允许内联 HTML**
3. **支持良好定义的 shortcodes 和函数**
4. **支持编译前/中/后的钩子**
5. **简单的构建系统**
