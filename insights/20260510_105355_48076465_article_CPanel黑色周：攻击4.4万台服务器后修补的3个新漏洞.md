# cPanel 安全周：勒索软件攻击后发现三个新漏洞

## 基本信息

- **洞察链接**: https://www.copahost.com/blog/cpanels-black-week-three-new-vulnerabilities-patched-after-ransomware-attack-on-44000-servers/
- **发布时间**: 2026年5月9日
- **作者**: Gustavo Gallas
- **来源**: Copahost Blog

## 事件概述

2026年5月8日，就在cPanel CVE-2026-41940身份验证绕过漏洞被用于入侵44,000台web托管服务器并部署勒索软件仅仅十天之后，cPanel悄然发布了第二个紧急安全补丁。此次更新涵盖三个新漏洞：CVE-2026-29201、CVE-2026-29202和CVE-2026-29203。

其中两个漏洞的CVSS评分为8.8，属于高严重性等级。这是十天内cPanel发布的第二个技术安全发布（TSR），两周内两次紧急补丁发布非同寻常。

## 三个新漏洞详情

### CVE-2026-29201 — 任意文件读取（CVSS 4.3）

**漏洞描述**: feature::LOADFEATUREFILE adminbin调用中对特性文件名验证不足，可能导致任意文件读取。

**实际影响**: 认证攻击者可以操控特性文件名参数读取托管服务器上无权访问的文件。虽然这不会直接授予root权限，但获取的信息（配置文件、凭据、内部路径）可用于策划更具破坏性的后续攻击。

**严重程度**: 中等（CVSS 4.3）

### CVE-2026-29202 — 任意Perl代码执行（CVSS 8.8）

**漏洞描述**: create_user API调用中对plugin参数验证不足，可能导致在已认证账户的系统用户上下文执行任意Perl代码。

**实际影响**: 这是三个漏洞中最危险的一个。认证用户（可能是共享服务器上的任何账户持有者）可以通过create_user API注入任意Perl代码。在共享托管服务器上，这可能允许一个租户运行影响整个服务器的代码。

**严重程度**: 高（CVSS 8.8）

### CVE-2026-29203 — 不安全符号链接导致的权限提升（CVSS 8.8）

**漏洞描述**: 不安全的符号链接处理漏洞，允许用户使用chmod修改任意文件的访问权限，导致拒绝服务或可能的权限提升。

**实际影响**: 通过创建指向敏感系统文件的符号链接并通过cPanel触发chmod操作，攻击者可以更改无权访问的文件的权限。这可能导致权限提升或拒绝服务（如果系统文件变得不可访问）。

**严重程度**: 高（CVSS 8.8）

## 背景上下文

为了理解这三个补丁为何比其单独CVSS评分更重要，有必要了解发生在此前十天的事件。

2026年4月28日，cPanel发布了CVE-2026-41940的紧急补丁，这是一个CVSS 9.8的身份验证绕过漏洞，允许未经身份验证的远程攻击者获得cPanel和WHM的管理访问权限。该漏洞作为零日漏洞被积极利用，利用尝试可追溯到2026年2月下旬——这意味着攻击者在修复可用之前有大约两个月的领先时间。

后果立即显现且严重。至少44,000个运行cPanel的IP地址在持续攻击中被入侵。黑客利用该漏洞入侵服务器并部署了一种名为"Sorry"的勒索软件Go语言Linux加密器。

十天内两次紧急技术安全发布反映了安全团队认可的一种集中修复周期：初始关键补丁触发对相邻代码路径的更深入审计，而该审计会发现以前未发现或被降级的问题。这在高调事件后并不罕见——实际上是对身份验证和会话处理代码加速重新检查的预期结果。

## 修复步骤

**标准更新**:
```
/scripts/upcp
```

**如果自动更新被禁用或处于固定层级**:
```
/scripts/upcp --force
```

**CloudLinux 6服务器**:
```
sed -i "s/CPANEL=.*/CPANEL=cl6110/g" /etc/cpupdate.conf
/scripts/upcp
```

**修补后重启cpsrvd**:
```
/scripts/restartsrv_cpsrvd
```

**验证修补版本**:
```
/usr/local/cpanel/cpanel -V
```

## 是否需要检查之前的入侵

如果您的服务器在2026年2月下旬至4月28日期间运行的是未修补的cPanel版本，应将其视为可能已被入侵，不仅需要打补丁，还需要调查。

建议的取证步骤包括：审计从2026年2月23日开始的访问日志——检查/usr/local/cpanel/logs/access_log和/usr/local/cpanel/logs/login_log中来自意外IP的异常会话身份验证模式。还要对用户主目录进行递归扫描，查找.sorry扩展名的文件。.sorry文件的存在确认了勒索软件部署，需要完整的 Incident Response，而不仅仅是打补丁。

## 更广泛的趋势

cPanel目前发生的情况是影响整个web托管安全格局的更广泛趋势的一部分。

近年来最严重的三个Linux内核漏洞——Copy Fail（CVE-2026-31431）和Dirty Frag（CVE-2026-43284/43500）——在4月下旬和5月初的八天内披露。cPanel勒索软件攻击暴露了44,000多台服务器。而现在三个cPanel CVE在首次紧急补丁发布后几天内就出现了。

这种披露的集中并非巧合。AI辅助的安全研究正在比协调披露流程更快地发现漏洞。漏洞被攻击者知晓与在生产环境中被利用之间的时间窗口正在从数周缩小到数天。在CVE-2026-41940的情况下，利用在补丁发布前数月就开始了。

对于运营cPanel服务器的人来说，直接的运营含义是：自动更新必须开启，补丁验证必须成为维护检查清单的一部分，并且在每次重大事件后进行日志审查不再是可选项。
