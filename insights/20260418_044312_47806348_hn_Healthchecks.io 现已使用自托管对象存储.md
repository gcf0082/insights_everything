# 洞察报告：Healthchecks.io 现使用自托管对象存储

**标题：** Healthchecks.io now uses self-hosted object storage

**来源：** Hacker News

**发布时间：** 2026年4月18日（约6小时前）

**得分：** 118 points

**评论数：** 61 comments

---

## 摘要

Healthchecks.io（一个监控服务）的开发者将对象存储从云服务迁移到了自托管的 Versity S3 Gateway。目前存储规模为 1400 万个对象，共 119GB，平均对象大小约 8.5KB。系统运行在 3 台 Web 服务器上以实现冗余，使用 Btrfs 文件系统。虽然成本比托管对象存储服务更高，但获得了更好的性能和可靠性。

---

## 主要讨论要点

### 1. 技术实现细节

- 使用 Versity S3 Gateway 提供 S3 API 接口
- 后端存储使用普通的 Btrfs 文件系统
- 有 3 个 Web 服务器共享访问同一个存储
- 监控工具使用 Netdata

### 2. 关于是否需要 S3 API 的讨论

有用户质疑既然是本地存储为何需要 S3 API。作者回应：
- 应用原本就基于 S3 API 构建，使用云存储
- 保持 S3 接口可以避免代码修改
- 便于在需要时切换回云服务或其他提供商

### 3. 替代方案讨论

评论中提到了多种自托管 S3 兼容存储方案：
- **MinIO** - 流行的自托管对象存储
- **Garage** - 由 NLnet 资助开发
- **Ceph (RGW)** - 企业级存储解决方案
- **SeaweedFS** - 适合处理大量小文件
- **Tigris** - 提供 S3 API 兼容性测试
- **CloudFlare R2** - 有用户迁移后费用降低 90%

### 4. 关于文件系统的担忧

部分用户对使用 Btrfs 表示担忧：
- 有人在生产环境中遇到过文件系统损坏问题
- 建议使用 ext4 或 ZFS
- 作者提到 Versitygw 没有使用 fsync，可能有数据丢失风险

### 5. 成本与规模讨论

- 119GB 的存储规模相对较小
- 租用专用服务器的成本高于托管对象存储（约100GB）
- 但自托管带来了更好的性能和可靠性
- 对于个人或小规模项目，几乎任何方案都能正常工作

### 6. 关于供应商锁定

讨论了 S3 API 是否导致供应商锁定：
- S3 API 已成行业标准，多个提供商支持
- AWS 公开了 API 规范，有多个开源实现
- 切换到自托管或云服务商都很方便

---

## 结论

这是一个有趣的小规模自托管案例展示了如何利用开源方案替代昂贵的云服务。虽然成本更高，但对于没有专业运维团队的个人项目来说，可靠性的提升和 S3 API 的兼容性使得这个方案值得考虑。
