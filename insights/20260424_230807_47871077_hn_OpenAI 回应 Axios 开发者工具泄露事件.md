# OpenAI 开发者工具供应链安全事件洞察报告

**洞察链接**：https://news.ycombinator.com/item?id=47871077

**来源**：Hacker News

**发布日期**：2026年4月22日

**标题**：OpenAI's response to the Axios developer tool compromise

---

## 事件概述

2026年3月31日，Axios npm包被攻击者注入恶意代码。OpenAI在其macOS应用签名流程的GitHub Actions工作中使用了受影响版本的Axios（1.14.1），导致其代码签名证书可能已被 compromisd。OpenAI于4月10日发布官方博客回应此事，并于4月21日向ChatGPT/Codex用户发送通知。

---

## 社区讨论要点

### 1. 响应延迟问题
- 博客发布于4月10日（事件发生10天后）
- 用户通知于4月21日发送（博客发布11天后）
- 社区质疑：在如此公开的事件后，依赖项审计、凭证轮换和公开沟通应该更加紧迫

### 2. 事件根因分析
- GitHub Actions工作流配置错误：使用了浮动标签（floating tag）而非特定提交哈希
- 未配置minimumReleaseAge来防止新包自动升级
- 这是一种bad practice，也是在转移责任（shifting blame）

### 3. 供应链安全最佳实践建议
1. **固定GitHub Actions到SHA**：使用提交哈希而非标签，避免自动更新
2. **使用npm ci而非npm install**：前者使用锁文件，后者可能获取最新版本
3. **设置min-release-age**：在.npmrc中设置min-release-age=7，大部分攻击可在数小时内被发现并从npm移除

### 4. 关于Axios的争论
- 部分开发者认为应停止使用Axios，改用原生fetch API
- 其他开发者认为Axios提供了更好的功能（如拦截器、错误处理、请求/响应转换等）
- 建议使用ky等轻量级fetch封装库作为替代

### 5. 替代HTTP库示例配置
- uv: `exclude-newer = "7 days"`
- npm: `min-release-age=7`
- pnpm: `minimum-release-age=10080` ( minutes )
- bun: `minimumReleaseAge = 604800` ( seconds )

### 6. 对未来的担忧
- GitHub已成为关键基础设施，类似WordPress成为攻击目标
- 建议使用pnpm代替npm（更好的速度、确定性、node_modules精简等）
- 依赖项审核和代码检查工具（如depsguard.com）可帮助自动配置安全设置

---

## 安全建议总结

1. 固定所有依赖项版本到具体提交哈希
2. 使用npm ci确保确定性构建
3. 配置包管理器的发布延迟设置
4. 考虑使用fetch或ky等轻量级替代方案替代Axios
5. 在CI/CD环境中使用 VM 隔离
6. 定期轮换代码签名证书
