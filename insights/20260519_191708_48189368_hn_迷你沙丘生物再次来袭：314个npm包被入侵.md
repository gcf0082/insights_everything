## 事件概述

2026年5月19日，npm生态系统遭受了新一轮大规模供应链攻击。攻击者通过入侵 `atool` npm账号，在22分钟内自动化发布了637个恶意版本，分布在317个npm包中。这次攻击被称为"Mini Shai-Hulud"事件，是继三周前SAP供应链攻击之后的又一次重大安全事件。

## 攻击技术分析

### 攻击载体

每个恶意版本都通过以下两种方式注入恶意代码：

1. **preinstall钩子**：在 `package.json` 中添加 `"preinstall": "bun run index.js"`
2. **可选依赖投递**：630/637个版本包含指向 `antvis/G2` GitHub仓库的伪造提交

### 恶意Payload特征

- **大小**：498KB混淆的Bun脚本
- **哈希**：SHA256: `a68dd1e6a6e35ec3771e1f94fe796f55dfe65a2b94560516ff4ac189390dfa1c`
- **混淆方式**：双层混淆（hex变量查找表 + AES加密字符串解码器）
- **与SAP事件的关联**：相同的Bun运行时要求、相同的扫描架构、相同的凭证正则集

### 攻击能力

Payload实现了以下攻击功能：

**凭证窃取**
- 环境变量扫描（80+种凭证类型）
- GitHub PAT、npm令牌、AWS密钥
- GCP、Azure服务账号
- 数据库连接字符串、Stripe密钥
- SSH密钥、Docker认证
- Kubernetes服务账号令牌、HashiCorp Vault令牌

**云基础设施探测**
- AWS完整凭证链：环境变量 → 配置文件 → EC2 IMDS → ECS容器元数据 → Secrets Manager
- Kubernetes API访问
- Docker socket检测及容器逃逸

**数据外传**
- 利用GitHub API作为C2通道（User-Agent伪装为python-requests）
- 创建Dune主题命名的公开仓库存储窃取数据

**持久化机制**
- CI/CD工作流注入（`.github/workflows/codeql.yml`）
- AI Agent劫持（Claude Code SessionStart钩子、VS Code自动运行任务）
- systemd服务/macOS LaunchAgent（`kitty-monitor`后门）
- 本地项目横向传播

### GitHub仿冒提交

攻击者利用GitHub的fork对象共享机制，在 `antvis/G2` 仓库中植入孤儿提交（无分支/标签依附），通过伪造 `Alexzjt`（真正的维护者）的作者身份来隐藏payload来源。这是一种已知的技术，已被Chainguard文档记录。

## 社区反应

### 核心观点

1. **对npm生态系统的批评**
   - "Node是当代的Visual Basic——如果有自我更新能力的话"
   - npm缺乏构建脚本白名单机制（与pnpm的allowBuilds对比）
   - 任何包管理器都存在同等风险（AUR模式）

2. **防御建议**
   - 使用 `npm config set min-release-age=2` 避免新发布版本
   - 在隔离VM/容器中运行开发环境
   - 使用podman替代docker（rootless）
   - 使用OpenSnitch监控出站连接

3. **对AI Agent的担忧**
   - 攻击者已针对Claude Code和Codex部署持久化钩子
   - AI模型可能成为新的攻击向量

4. **行动呼吁**
   - 轮换所有可能暴露的凭证
   - 检查本地项目中的恶意钩子文件
   - 使用pmg等安装代理进行实时防护

### 热门评论

- "无法阻止这种攻击，唯一经常发生这种情况的开发社区"
- "我已卸载所有包管理器，只在devcontainers/VM中使用"
- "使用六年前依赖的企业更安全"

## 建议措施

### 立即行动

1. 检查package-lock.json中是否存在2026-05-19发布的atool账号包版本
2. 轮换构建环境中的所有凭证
3. 检查是否存在未经授权的公开仓库
4. 查找本地文件：`~/.claude/settings.json`、`.vscode/tasks.json`、`.claude/setup.mjs`
5. 检查systemd服务 `kitty-monitor` 和 `gh-token-monitor`

### 长期防御

1. 使用依赖版本锁定而非semver范围
2. 使用pmg等安装代理在安装时拦截恶意包
3. 配置依赖冷却期（Dependency Cooldown）
4. 在隔离环境中进行开发
5. 定期审计CI/CD配置
