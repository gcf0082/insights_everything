## 核心讨论内容

### 1. 计算几何中的浮点数问题

讨论首先延伸到计算几何领域的特殊挑战。评论指出，计算几何通常是"他人的几何"——即开发者无法控制输入数据的质量或意图。在这种情况下，两个点或平面是否真正对齐已不再是数学问题，而是用户意图问题。

**关键观点**：
- 几何内核（如 Open Cascade）采用"模糊布尔运算"，允许容差来处理不精确的网格输入
- 几何内核会进行多达7次的容差扩展步骤来处理点分类问题
- 这是一个在2026年仍在被积极研究的问题，目前没有完美的解决方案

### 2. IEEE Epsilon 的误用

多个评论指出 `f64::EPSILON` 被严重误用：

**技术细节**：
- `EPSILON` 是1.0与下一个可表示浮点数之间的差值
- 它是 \[1, 2) 范围内相邻浮点数的间距
- 对于 ≥2.0 的数字，`abs(a - b) <= epsilon` 等同于 `a == b`
- 对于极小数（如 1e-50），epsilon 比较会认为 1e-50 和 2e-50 相等

**核心结论**：在99.9%的情况下，epsilon 是错误的工具。

### 4. 绝对容差与相对容差

标准做法是结合两者：

```python
# Python math.isclose 的实现
abs(a - b) <= max(rel_tol * max(abs(a), abs(b)), abs_tol)
```

- **相对容差**：处理大数值
- **绝对容差**：处理接近零的情况

### 5. Rust assert_f64_eq 宏的讨论

原帖子作者展示了他们的 Rust 宏实现，收到大量改进建议：

**当前实现的问题**：
```rust
if (a >= b && a - b < f64::EPSILON) || (a <= b && b - a < f64::EPSILON)
```

**建议改进**：
- 让用户指定 epsilon 值
- 支持多种精度模式（Steps、Digits、Rel）
- 支持 NaN 处理
- 使用 ULP 而不是固定的 epsilon

### 6. 浮点数精度与原点距离

讨论还涉及了浮点数的精度分布特性：
- 离原点越近，精度越高
- 使用固定点坐标可能比浮点数更适合某些几何应用
- 大型游戏（如 Kerbal Space Program）曾因浮点精度问题需要重新设计坐标系统

### 7. x87 浮点协处理器问题

有评论指出 Intel x87 的"过度精度"问题：
- 寄存器中的值可能与存储到内存后的值不同
- 这可能导致比较结果失效
- 虽然 Intel 已将此标记为"遗留"技术，但暂无移除计划

## 关键结论

1. **浮点数相等性比较并非总是错误的**：在确定性计算场景下，精确相等比较是正确且必要的

2. **选择正确的比较方法取决于场景**：
   - 需要完全确定性 → 使用精确相等 (==)
   - 需要容差 → 使用绝对/相对容差组合
   - 需要自适应 → 使用 ULP 方法

3. **没有通用的完美解决方案**：容差值的选择取决于具体应用场景、物理量精度需求和数值特性

4. **理解浮点数底层表示至关重要**：了解 ULP、epsilon 含义和精度分布是正确处理浮点数比较的前提
