## 事件概述

**Trivy** 是一个流行的开源安全漏洞扫描工具，2026年3月22日再次遭受供应链攻击。攻击者使用泄露的凭据发布了恶意的 Trivy v0.69.5 和 v0.69.6 DockerHub 镜像。这是 Aqua Security（Trivy 维护公司）在短短一个月内遭受的**第三次**攻击。

### 攻击时间线

1. **2026年2月** - 首次攻击，攻击者获取了 Trivy 的 GitHub 仓库访问权限
2. **2026年3月19日** - 第一次供应链攻击事件曝光
3. **2026年3月22日** - 攻击者再次行动，篡改 DockerHub 账户，重新发布了75个标签中的75个

---

## 关键洞察

### 1. 版本标签可变性带来的安全隐患

- GitHub 允许用户覆盖已有标签（tag），攻击者利用这一特性将恶意代码注入到已有的安全版本中
- 即使开发者使用了版本锁定（如 `v0.69.5`），攻击者仍可在同一标签下更新恶意内容
- 建议：使用完整 commit SHA 进行版本 pinning，这是唯一真正不可变的方式

### 2. 凭证轮换不彻底

- Trivy 团队声称在第一次攻击后已"轮换"了凭据
- 但轮换过程**非原子性**，攻击者可能获取到了新刷新的令牌
- 这是同一攻击者在短时间内能够发动第二次攻击的根本原因

### 3. GitHub Actions 生态的安全困境

- 即使用户在 workflow 中 pinning 了 action 的 SHA，该 action 内部的**传递依赖**（transitive dependencies）可能仍然指向可变引用
- 许多 GitHub Action 内部会 clone 其他仓库、curl 二进制文件或执行安装脚本，这些操作基本不受外部 pinning 保护

### 4. 安全工具自身的安全风险

- 评论者指出 Trivy 的 `setup-trivy` Action 曾直接 clone 主分支并执行 shell 脚本
- 这意味着安全公司实际上给予了**任意代码执行**权限给所有使用该工具的 CI 工作流
- 安全工具本身可能成为攻击面，而非防护手段

### 5. 中心化供应链的风险

- 单一供应商（GitHub）的安全问题会产生巨大的爆炸半径
- 有评论建议考虑使用去中心化的代码托管方案来减小供应链攻击的影响

## 社区讨论焦点

| 话题 | 观点 |
|------|------|
| **强制不可变版本** | 部分开发者呼吁 GitHub 强制要求 Actions 使用不可变版本标签 |
| ** pinning vs 自动更新** | 存在" pinning 悖论"：安全团队希望锁定版本以防止引入恶意代码，但也希望自动获取安全更新 |
| **沙箱隔离** | 有开发者建议将扫描工具放入严格受限的沙箱中运行 |
| **安全工具信任问题** | 社区反思不应盲目信任安全工具，需对其进行代码审查 |

---

## 建议措施

1. **使用完整 SHA 进行版本 pinning**，而非版本号
2. **审计 CI 工作流中的所有依赖**，包括间接依赖
3. **对安全工具进行代码审查**，再授予生产环境访问权限
4. **使用只读镜像数据**进行扫描，避免给予工具过多权限
5. **实施不可变发布**（GitHub 已有此功能）
6. **分离敏感权限**，避免单一凭据拥有过多权限
