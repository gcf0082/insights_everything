## 摘要

本文是"Claude Code 规模化"系列文章的一部分，介绍了在大型代码库中成功部署 Claude Code 的最佳实践。Claude Code 已在生产环境中运行于数百万行代码的单体仓库、数十年历史的遗留系统、数十个微服务仓库以及拥有数千名开发者的组织中。

## 核心内容

### Claude Code 如何导航大型代码库

Claude Code 导航代码库的方式与软件工程师类似：遍历文件系统、读取文件、使用 grep 查找所需内容、跟踪代码库中的引用。它在开发者的本地机器上运行，不需要构建、维护或上传代码库索引。

与依赖 RAG 检索的 AI 编码工具不同，Claude Code 使用**代理搜索**（agentic search），无需维护嵌入管道或集中式索引，每个开发者的实例都基于实时代码库工作。但这种方法有代价：当 Claude 有足够的起始上下文来知道从哪里开始时效果最好。

### 扩展层：模型的重要性不亚于模型本身

最常见的误解是 Claude Code 的能力仅由使用的模型决定。实际上，围绕模型构建的**工具链**（harness）比模型本身更能决定 Claude Code 的表现。

工具链由五个扩展点构建：

1. **CLAUDE.md 文件** - 优先级最高，在每个会话开始时自动读取，用于提供项目特定的约定和代码库知识
2. **Hooks** - 在关键时刻运行的脚本，最有价值的用途是持续改进，而非仅防止错误
3. **Skills** - 按需加载的打包指令，避免在每个会话中加载所有专业知识
4. **Plugins** - 将技能、hooks 和 MCP 配置打包为可安装的包，便于在新工程师入职时立即获得相同的设置
5. **MCP 服务器** - 连接内部工具、数据源和 API

两个补充能力：
- **LSP 集成** - 提供符号级精确度，可跟踪函数调用到定义，追踪跨文件引用
- **Subagents** - 将探索与编辑分离，使用独立的 Claude 实例

### 三种成功部署的配置模式

#### 1. 使代码库在规模上可导航

- **保持 CLAUDE.md 文件精简且分层** - 根文件提供全景，子目录文件提供本地约定
- **在子目录而非仓库根目录初始化** - Claude 会自动向上遍历目录树并加载沿途的每个 CLAUDE.md 文件
- **按子目录限定测试和 lint 命令** - 避免为一个小改动运行完整套件导致超时
- **使用 .claude/ignore 文件排除生成的文件** - 在 settings.json 中提交权限拒绝规则，使每位开发者获得相同的噪音减少
- **当目录结构不工作时构建代码库地图** - 对于没有常规目录结构的组织，在仓库根目录添加轻量级 markdown 列出每个顶级文件夹及其功能
- **运行 LSP 服务器以便 Claude 按符号而非字符串搜索** - Grep 常见函数名会返回数千个匹配，而 LSP 只返回指向同一符号的引用

#### 2. 积极维护 CLAUDE.md 文件

随着模型演进，为当前模型编写的指令可能会与未来模型产生冲突。团队应每三到六个月进行一次有意义的配置审查，在重大模型发布后性能停滞时也值得进行一次。

#### 3. 为 Claude Code 管理和采用分配所有权

技术配置本身并不能推动采用。成功的组织也在组织层面投入资源：

- **提前进行基础设施投资** - 在广泛访问之前，一个小团队（甚至一个人）先完成工具配置，使 Claude 在开发者首次使用时就已经融入工作流程
- **指定负责人 (DRI)** - 一个人拥有 Claude Code 配置的所有权、设置和权限策略的决策权，以及保持它们更新的责任
- **建立跨职能工作组** - 尽早将工程、信息安全和治理代表聚集在一起，共同定义需求并制定推广路线图

## 关键要点

1. **工具链比模型更重要** - 配置和扩展层决定实际表现
2. **CLAUDE.md 文件需要精心维护** - 保持精简、分层，并根据模型演进积极更新
3. **投资代码库可导航性** - 使 Claude 能够快速找到正确的上下文
4. **组织层面的准备同样重要** - 提前完成基础设施配置，指定明确的责任人
5. **LSP 集成对多语言代码库至关重要** - 提供符号级导航能力

## 适用场景

Claude Code 设计用于 conventional 软件工程环境：工程师是主要的代码库贡献者，使用 Git，代码遵循标准目录结构。大多数大型代码库符合这一模式，但游戏引擎、大型二进制资产、非传统版本控制等非传统设置需要额外的配置工作。
