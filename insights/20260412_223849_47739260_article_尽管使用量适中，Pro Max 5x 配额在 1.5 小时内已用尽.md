## 问题摘要

在 Pro Max 5x 配额重置后，用户仅进行了**适度使用**（主要是问答和轻度开发），但配额在 **1.5 小时内**就耗尽了。在此之前的 5 小时**重度开发**（多文件实现、graphify 管道、多代理生成）消耗了之前的配额窗口——这是预期的。但重置后的耗尽情况不符合预期。

经调查发现可能的根本原因：**cache_read tokens 以全价计入速率限制**，导致提示缓存的成本优势在配额方面完全失效。

## 核心问题分析

### 问题 1：Cache Read Token 配额计算方式

**预期行为**: cache_read tokens 应以降低的费率（1/10）计入速率限制，与降低的成本一致。

**实际观察**: 配额耗尽率与 cache_read 以全价计算一致。

**计算对比**:

- **如果 cache_read 按 1/10 计数**: 窗口 2 总计 13.1M 有效 token，1.5 小时 = 8.7M/小时。这**不应该**耗尽 Pro Max 5x 配额。
- **如果 cache_read 按全价计数**: 窗口 2 总计 103.9M + 1.4M + 387k = 105.7M tokens，1.5 小时 = 70.5M/小时。这会解释配额耗尽的原因，但意味着**提示缓存对速率限制毫无作用**。

### 问题 2：后台会话消耗共享配额

在其他终端中保持打开的会话会继续进行 API 调用（压缩、回顾、处理钩子），即使用户并未主动交互。这些调用消耗同一配额池。

在本案例中，`token-analysis`（296 次调用）和 `career-ops`（173 次调用）在用户未主动交互的情况下仍消耗了大量配额。

### 问题 3：自动压缩产生昂贵峰值

每次自动压缩事件导致一次 API 调用，该调用以**压缩前的完整上下文**（约 966k tokens）作为 cache_creation，随后重新开始。这意味着最昂贵的单次调用是自动发生的，无需用户操作。

### 问题 4：1M 上下文窗口放大问题

更大的上下文窗口 = 每次调用更多 tokens = 更快的配额耗尽。1M 窗口被宣传为一个功能，但当 cache_read tokens 以全价计入配额时反而适得其反。

## 复现步骤

1. 在 Pro Max 5x 上使用 Opus 启动 Claude Code
2. 在 `~/.claude/rules/` 中放置约 30 个规则文件（约 19k tokens 固定开销）
3. 在项目中进行工具密集型操作（文件读取、构建、测试）
4. 通过 `/context` 命令观察上下文增长
5. 200-300 次工具调用后检查配额——配额将大幅耗尽
6. 在其他终端中保持 2-3 个其他 Claude Code 会话打开
7. 配额重置后，观察即使适度使用也会耗尽配额

---

## 预期行为

- cache_read tokens 应以降低的费率（1/10）计入速率限制
- 后台/空闲会话不应消耗大量配额
- 自动压缩不应产生过大的配额峰值
- Pro Max 5x 应能维持每个配额窗口至少 2-3 小时的适度 Opus 使用

---

## 实际行为

- 配额在 1.5 小时内耗尽（8.7M 有效 tokens/小时）
- 后台会话消耗了重置后配额的 78%
- 发送的总原始 tokens（105.7M）与 cache_read 以全价计算一致

---

## 建议改进

1. **澄清 cache_read 配额计算**: 文档说明 cache_read tokens 是按全价还是降低的费率计入速率限制
2. **按有效 tokens 限速**: 对速率限制按 1/10 费率计算 cache_read，与成本降低一致
3. **会话空闲检测**: 不将空闲会话开销计入配额，或警告用户有关打开的会话
4. **配额可见性**: 在 Claude Code 中显示实时 token 消耗明细（cache_read vs cache_create vs input vs output）
5. **上下文感知配额估算**: 在操作前根据当前上下文大小估算配额成本

---

## 结论

用户无法以这种速度消耗 token 使用量。此问题严重影响了 Pro Max 5x 计划的用户体验，特别是在使用高上下文窗口和后台会话时。
