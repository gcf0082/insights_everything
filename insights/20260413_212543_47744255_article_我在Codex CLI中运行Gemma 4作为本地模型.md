## 摘要

Daniel Vaughan花费一整天时间在两台机器上测试Google最新的Gemma 4模型，探索其是否能取代云端模型进行日常的agentic编程工作。他设置了两种配置：24 GB的M4 Pro MacBook Pro（运行26B MoE变体）和Dell Pro Max GB10（128 GB统一内存，运行31B Dense变体）。测试结果显示，Gemma 4的86.4%函数调用得分并非营销噱头，而是真正可用的工具调用能力。令人惊讶的发现是：对于agentic编程，模型质量比token生成速度更重要。

## 核心发现

### 1. 从"不可用"到"可用"的质变

- **Gemma 3**: 仅有6.6%的工具调用得分，几乎无法使用
- **Gemma 4**: 达到86.4%的工具调用得分
- 这个差距是从"完全损坏"到"可以工作"的关键一步

### 2. 速度与质量的权衡

| 指标 | MacBook Pro (26B MoE) | Dell GB10 (31B Dense) | GPT-5.4 云端 |
|------|----------------------|-------------------|--------------|
| Token生成速度 | 5.1x更快 | 基准 | 最快 |
| 总完成时间 | 4分42秒 | 6分59秒 | 65秒 |
| 首次尝试成功 | 否 | 是 | 是 |
| 代码质量 | 需5次尝试 | 1次成功 | 最佳 |

### 3. 关键洞察：质量比速度更重要

- Mac生成token的速度是Dell的5.1倍
- 但实际完成时间只快了30%（4分42秒 vs 6分59秒）
- MoE变体留下死代码，需要5次尝试才能写出测试套件
- Dense变体一次就产生可工作的代码
- **结论**：一次就做对的模型比迭代更快但需要更多迭代的模型更有价值

### 4. 云端模型的优势

GPT-5.4云端模型表现最佳：
- 完成时间：65秒
- Token数：最少
- 质量：最佳
- 首次尝试：5/5成功

## 设置过程中的经验教训

### Apple Silicon设置（MacBook Pro）

- **不要使用Ollama**：在Apple Silicon上，Ollama的Flash Attention在提示词超过500 tokens时会冻结
- **使用llama.cpp**：需要六个仔细配置的标志位来避免内存溢出
- Codex CLI的系统提示需要约27,000 tokens
- 建议配置：
  ```
  llama-server -m /path/to/gemma-4-26B-A4B-it-Q4_K_M.gguf \
    --port 1234 -ngl 99 -c 32768 -np 1 --jinja \
    -ctk q8_0
  ```

### NVIDIA设置（Dell GB10）

- **vLLM失败**：由于Blackwell ARM架构的PyTorch版本不兼容
- **Ollama v0.20.5**：是唯一可用的选项
- 需要通过SSH隧道连接端口11434
- `stream_idle_timeout_ms`至少设置为1,800,000

## 实用建议

### 如果要尝试本地Gemma 4

1. **在Apple Silicon上**：完全跳过Ollama，使用llama.cpp并加`--jinja`标志
2. **上下文设置**：32,768 tokens（Codex CLI系统提示至少需要27,000 tokens）
3. **量化KV缓存**：使用`-ctk q8_0 -ctv q8_0`
4. **禁用网络搜索**：在Codex CLI配置文件中设置`web_search = "disabled"`
5. **在NVIDIA上**：使用Ollama v0.20.5，通过SSH隧道连接远程机器

### 模型选择建议

- 如果有足够内存，使用31B Dense变体
- 31B模型的首次尝试准确率比26B MoE变体的原始token速度更重要
- 当agent在执行工具调用时，一次成功比迭代速度更重要

### 工作流建议

混合工作流程：
- `codex --profile local`：用于迭代和隐私敏感的工作
- 云端模型：用于追求最佳质量和速度的任务

## 基准测试环境

- 测试日期：2026年4月12日
- Codex CLI版本：v0.120.0
- Mac：llama.cpp ggml 0.9.11 (build 8680)，24 GB M4 Pro MacBook Pro
- Dell：Ollama v0.20.5，Dell Pro Max GB10，128 GB统一内存，NVIDIA Blackwell

## 结论

Gemma 4现在已经足够用于本地agentic编程。选择更大的模型（如果内存允许）会带来更好的首次尝试准确率，这比原始token速度更重要。对于privacy敏感或需要离线工作的场景，本地模型是一个可行的选择，但对于追求最佳质量和速度的场景，云端模型仍然是最佳选择。
