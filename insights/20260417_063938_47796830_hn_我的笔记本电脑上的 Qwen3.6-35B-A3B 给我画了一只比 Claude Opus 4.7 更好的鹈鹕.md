## 内容摘要

这是一篇来自 Simon Willison 的实验报告，作者在笔记本电脑上运行了 Qwen3.6-35B-A3B（一个仅30亿参数的本地模型），并将其生成的 SVG 图像与 Claude Opus 4.7 进行对比。测试题目是"一只鹈鹕骑自行车"和"一只火烈鸟骑单轮车"。结果显示，Qwen 生成的鹈鹕比 Opus 更好，这让作者感到惊讶，因为 Qwen 只是一个可以在本地运行的较小模型。

## 主要讨论观点

### 1. 对测试结果的看法

- **支持 Qwen 的观点**: Qwen 生成的火烈鸟更有艺术感——带有太阳镜和领结，甚至还在吸烟。而 Opus 生成的火烈鸟看起来很普通。
- **支持 Opus 的观点**: Opus 的火烈鸟更符合物理现实——正确坐在座位上，脚踩在踏板上。而 Qwen 的版本缺少喙，翅膀有三条尾巴，腿被截断。
- **关键事实**: 这是一个仅30亿参数的模型，能够与昂贵的前沿模型接近，这本身就是值得注意的。

### 2. 关于测试有效性的讨论

- **批评意见**: 这个"鹈鹕测试"已经被各大AI实验室针对训练，人们用它来测试每个模型发布已经变得毫无意义。
- **支持意见**: 尽管实验室可能针对此测试优化，但如果它们真的在测试，为什么大多数尝试仍然表现不佳？
- **替代方案**: 建议使用不同的动物和活动组合（如"鲸鱼踩滑板"）来测试模型对分布外情境的适应能力。

### 3. 基准测试对比

- 在 Power Ranking 编码任务中，Qwen3.6-35B-A3B 解决了 11/98 个任务，而 Claude Opus 4.7 解决了 95/98 个任务。
- 有评论指出这些基准可能存在训练数据污染问题，因为模型可能在相同的开源项目上训练。
- 建议将本地模型与同类定位的模型比较（如 Haiku、Flash 或 GPT nano），而不是与昂贵的前沿模型比较。

### 4. Claude 性能回归

- 有用户指出 Opus 和 Sonnet 自 4.1 版本以来在许多非编码任务上表现有所下降。
- 在尝试让模型更新幻灯片中的图表时遇到了困难，显示出"玩具"和"工具"之间的差距。

### 5. 本地模型的应用

- 用户报告在 M5 Max MacBook Pro（128GB RAM）上通过 OpenCode 和 oMLX 使用 Qwen3.5-35B-A3B，工具调用质量和代理工作流处理能力令人印象深刻。
- 讨论了可用于驱动本地 LLM 的代理工具，包括 OpenCode、Pi、Qwen Code 等。

### 6. 关于测试价值的争论

- 有人认为这种"有趣"的测试是浪费时间，因为AI边界被推得越远，越难定义什么是智能。
- 另一方认为，测试模型"不能做什么"比测试"能做什么"更有用，随机任务可以展示模型的通用智能程度。

## 总结

这个讨论反映了对AI模型评估方法的深入思考。一方面，Qwen 这样的本地小模型能在某些任务上接近前沿模型，展示了模型压缩和优化的进展；另一方面，仅凭单一的创意图像生成测试来评判模型能力存在明显局限。讨论还涉及了基准测试的可靠性、模型是否针对公开测试进行训练，以及本地模型在实际开发中的可用性等重要话题。
