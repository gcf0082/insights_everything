## 摘要

本文详细介绍了 Addy Osmani 创建的 Agent Skills 项目——一个旨在将资深工程师的工作流程规范编码到 AI 编程代理中的开源工具集。该项目已在 GitHub 上获得超过 26,000 颗星标。本文深入解释了每个设计选择背后的原因、如何映射到标准软件开发生命周期（SDLC）和 Google 公开的工程实践，以及用户即使不安装任何技能也可以"窃取"哪些最佳实践。

---

## 核心观点

### 1. AI 代理的默认行为是走捷径

任何 AI 编程代理的默认行为都是走最短路径到"完成"。你要求一个功能，它就写出功能实现代码。它不会询问是否有规范、不会先写测试、不会考虑变更是否跨越信任边界、也不会检查 PR 看起来怎么样。代理生成代码，宣布胜利，然后转向下一个任务。

这与每位资深工程师职业生涯中学会了避免的失败模式相同。任何任务的资深版本都包含不会出现在 diff 中的工作： surfaced assumptions（暴露假设）、 writing the spec（编写规范）、 breaking the work into reviewable chunks（将工作分解为可审查的块）、 choosing the boring design（选择无聊的设计）、 leaving evidence that the result is correct（留下证明结果是正确的证据）、 sizing the change so a human can actually review it（调整变更大小以便人类可以实际审查）。这些步骤是区分能够规模化发布可靠软件的工程师和推送破坏性代码的人的大部分因素。

代理跳过这些步骤的原因与任何初级工程师一样：它们是看不见的。奖励信号指向"任务完成"而不是"任务完成且设计文档存在"。所以我们必须将资深工程师的脚手架 bolt back on（ bolted on 意思是"附加上去"）。

### 2. 什么是"skill"（技能）

"skill"一词在 Claude Code / Anthropic 的术语中承担了很多工作，需要精确理解。技能是一个带有 frontmatter 的 markdown 文件，当情况需要时会被注入到代理的上下文中。介于 system-prompt 片段和 runbook（操作手册）之间。

技能**不是**参考文档。它不是"你应该了解的所有关于测试的知识"。它是一个 workflow（工作流）：代理遵循的一系列步骤，带有产生证据的检查点，以定义的退出标准结束。

这个区别就是整个游戏。如果你把一篇关于测试最佳实践的 2000 字论文放入代理的上下文，代理会阅读它，生成看起来合理的文本，然后跳过实际的测试。如果你把一个**工作流**放在那里（先写失败的测试，运行它，观察它失败，写最少通过它的代码，观察它通过，重构），代理有事情要做，你也有东西可以验证。

**Process over prose（过程胜于论述）。Workflows over reference（工作流胜于参考）。 Steps with exit criteria over essays without them（带退出标准的步骤胜于无退出标准的论文）**。这个单一的区别将有用的技能与漂亮的 markdown 文件分开。这也解释了为什么那么多"AI rules"repo 在实践中最终什么也做不了。那些规则是论文。

### 3. 技能编码的 SDLC

Repo 中的二十个技能围绕六个生命周期阶段组织，有七个斜杠命令 sitting on top（在顶部）。Define（`/spec`）是你决定实际构建什么的地方。Plan（`/plan`）将工作分解。Build（`/build`）以垂直切片实现。Verify（`/test`）证明它有效。Review（`/review`）捕获遗漏的内容。Ship（`/ship`）将其安全地送到用户手中。`/code-simplify` sitting across the whole thing（ sits across 意思是"横跨"或"贯穿"）。

这不是巧合。它是每个正常运行的工程组织运行的相同 SDLC，只是用不同的词汇。Google 称它为 design doc → review → implementation → readability review → launch checklist。Amazon 称它为 working-backwards memo 和 bar raiser。每个健康的团队都有这个循环的某个版本。

AI 编程代理的新鲜之处在于*大多数代理默认跳过大多数这些阶段*。你要求一个功能，你得到一个实现，而规范、计划、测试、审查和启动检查表都没有发生。**技能推动代理经历与资深工程师强制自己经历相同的阶段，因为跳过它们 shipping code 就是你制作 incident（事故）的方式。**

一个复杂的特性可能会按顺序激活十一个技能。一个小的 bug 修复可能使用三个。router（`using-agent-skills`）决定哪个适用。关键是工作流缩放到实际范围，而不是假设的范围。

### 4. 五大原则

项目中有五个设计决策是承重的。其他一切都源于它们。

#### 原则 1：Process over prose

已经涵盖。工作流是 agent-actionable（代理可操作）的；论文不是。对人类团队也是如此。如果你的团队手册在时间压力下是 200 页，没人会读它。如果它是一小套带检查点的工作流，人们实际上会运行它们。

#### 原则 2：Anti-rationalization tables（反合理化表格）

这是项目中最独特的设计决策，也是我最想让其他团队窃取的。

每个技能都包含一个表格，列出了代理（或疲倦的工程师）可能用来跳过工作流的常见借口，配有书面反驳。几个接近原件的例子：

- *"这个任务太简单，不需要规范。"* → 接受标准仍然适用。五行很好。零行不行。
- *"我稍后会写测试。"* → Later（稍后）是承重词。没有稍后。先写失败的测试。
- *"测试通过，ship it。"* → 通过的测试是证据，不是证明。你检查了运行时吗？你验证了用户可见的行为吗？人类阅读了 diff 吗？

这样做的原因是 LLMs 擅长合理化。它们会生成一个听起来合理的段落，解释为什么*这个特定的*任务不需要规范，或者为什么*这个特定的*变更 fine to merge without review（值得不经过审查就合并）。**反合理化表格是代理尚未说出的谎言的预先写好的反驳。**

这个模式对人类团队也一样好。大多数工程衰退不是任何人选择做坏事。是人们接受听起来合理的借口来跳过他们不想做的部分。一个写下反合理化的团队会减少这种情况。

#### 原则 3：Verification is non-negotiable（验证是不可协商的）

每个技能都以具体证据终止。测试通过。构建输出干净。运行时跟踪显示预期的行为。审查者签署。*"Seems right"* 永远不够。

这与使 Anthropic 的 harness 从失败中恢复、使 Cursor 的 planner/worker/judge split 实际上 catch bugs、以及使任何 long-running agent 可恢复的原则相同。代理是生成器。你需要一个 separate signal（单独的信号）表明工作完成。技能将那个信号烘焙到每个工作流中。

#### 原则 4：Progressive disclosure（渐进式披露）

不要在会话开始时将所有二十个技能加载到上下文中。根据阶段激活它们。一个小的 meta-skill（`using-agent-skills`）作为 router（路由器），决定哪个技能适用于当前任务。

这是 applied at skill granularity（应用于技能粒度）的 harness engineering 教训。加载到上下文中的每个 token 都会在某处降低性能，所以你加载相关的，把剩下的留在磁盘上。**渐进式披露是如何将二十个技能的库装入 5K-token 槽而不毒害 wells（的意思是"不污染上���文"）的方法。**

#### 原则 5：Scope discipline（范围纪律）

meta-skill encode（编码）了一个如果我可以，我会 staple（钉）在每个代理上的 non-negotiable（不可协商的）：*"touch only what you're asked to touch."* 不要重构相邻的系统。不要删除你不完全理解的代码。不要刷到 TODO 并决定重写文件。

这听起来显而易见，直到你观看代理决定修复一个 bug 需要现代化三个不相关的文件。**范围纪律是决定代理的 PR 是可合并还是必须 unwound（卷回）的 single biggest determinant（单一最大决定因素）**。这也是映射最清晰地到 Google 的代码审查规范的原则，审查者会阻止做不止一件事的 PR。

### 5. Google DNA

技能饱和了来自 *Software Engineering at Google* 和 Google 公开的工程文化的实践。这是故意的。Google 规模软件工作的大部分内容是记录在案且公开的，而且它正是代理最有可能跳过的部分。

部分映射——哪个技能编码了哪个实践：

- **Hyrum's Law** 在 `api-and-interface-design` 中。你的 API 的每个可观察行为最终都会被某人依赖，所以考虑到这一点进行设计。
- **测试金字塔（~80/15/5）和 Beyoncé Rule** 在 `test-driven-development` 中。*"If you liked it, you should have put a test on it."* 基础设施变更 catch不了 bugs；测试能。
- **DAMP over DRY in tests.** Google 的测试哲学明确表示测试代码应该像规范一样阅读，即使以一些重复为代价。过度抽象的测试是一个已知的反模式。
- **~100 行 PR sizing，with Critical / Nit / Optional / FYI severity labels** 在 `code-review-and-quality` 中。直接来自 Google 的代码审查规范。大的 PR 不会被审查；它们会被 rubber-stamped（章戳）。
- **Chesterton's Fence** 在 `code-simplification` 中。不要移除你理解为什么它被放在那里之前的事物。
- **Trunk-based development and atomic commits** 在 `git-workflow-and-versioning` 中。
- **Shift Left and feature flags** 在 `ci-cd-and-automation` 中。尽早 catch 问题，将 deploy 与 release 解耦。
- **Code-as-liability** 在 `deprecation-and-migration` 中。每一行你保留的都是你必须永远维护的，所以 prefer the smaller surface（选择更小的表面）。

这些都不是新想法。**关键是它们默认不在代理中。** 前沿模型在训练数据中读过"HyHyrum's Law"这个词，但当它在凌晨 3 点设计你的 API 时不会应用 Hyrum's Law。技能是确保它应用的方法。

### 6. 如何实际使用它

三种模式，大致按增加承诺排序。

**模式 1：通过 marketplace 安装。** 如果你使用 Claude Code：

```
/plugin marketplace add addyosmani/agent-skills
/plugin install agent-skills@addy-agent-skills
```

你得到斜杠命令（`/spec`, `/plan`, `/build`, `/test`, `/review`, `/ship`, `/code-simplify`），代理根据上下文自动激活相关技能。这是我推荐大多数人开始的路径。

**模式 2：将 markdown 放入你的工具中。** 技能是带有 frontmatter 的纯 markdown。Cursor 用户将它们放在 `.cursor/rules/` 中。Gemini CLI 有自己的安装路径。Codex, Aider, Windsurf, OpenCode，任何接受 system prompt 的东西都可以阅读它们。工具下面的工作流比工具本身更重要。

**模式 3：将它们作为规范阅读。** 即使你从不安装任何东西，技能也是对 AI 代理良好工程*文档化描述*。阅读 `code-review-and-quality.md` 并将五轴框架应用于你团队的审查流程。阅读 `test-driven-development.md` 并用它来解决与初级工程师下一次"我们是否需要先写测试"的争论。阅读 meta-skill 并窃取五个不可协商项用于你自己的 AGENTS.md。

这第三种模式是我实际上会开始的。选择最接近你当前痛苦的四到五个技能。决定你想要强制执行哪些工作流��然��安装运行时，或者推出你自己的来做强制执行。

### 7. 即使不安装也能窃取的东西

无论你是否完全使用 AI 代理，我都建议你窃取的几个模式。

**作为团队实践的反合理化。** 写下你的团队告诉自己的谎言。*"我们将在 launch 后修复测试。"* *"这个变更太小，不需要设计文档。"* *"没关系，我们有监控。"* 将每个与反驳配对。将它放在你的 AGENTS.md 或工程 wiki 中。它会为你保存争论，并 catch 下一个疲倦的周五下午的捷径。

**Process over prose for anything you write internally.** 如果你发现自己写了一个标题为"我们如何处理 X"的 2000 字文档，你已经写了参考材料。将其转换为带检查点的工作流。文档缩减到 400 字，人们实际上会运行它。对于入门指南和 runbooks 来说，这与代理技能一样适用。

**Verification as a hard exit criterion.** 使"produce evidence"成为每个任务的退出步骤。对于代理、对于工程师、对于你自己。证据是证明工作完成的任何东西：green test run（一组通过的测试）、 screenshot（截图）、 log（日志）、 review approval（审查批准）。没有它，任务就没有完成。*"Seems right"* 永远不会 close the loop（闭环）。

**Progressive disclosure for any rulebook.** 不要写 50 页的手册。写一个指向正确小章节的小路由器。这适用于 AGENTS.md、runbooks、incident playbooks 以及任何人在时间压力下会阅读的东西。

**五个不可协商项，从 meta-skill 中提升，我会放在任何 AGENTS.md 中：**

1. Surface assumptions before building. Wrong assumptions held silently are the most common failure mode.
2. Stop and ask when requirements conflict. Don't guess.
3. Push back when warranted. The agent (or engineer) is not a yes-machine.
4. Prefer the boring, obvious solution. Cleverness is expensive.
5. Touch only what you're asked to touch.

这值得在五行中建立一个有价值的工程文化，你不需要安装任何东西来采用它。

## 技术实现细节

### 技能格式

技能是带有 YAML frontmatter 的 markdown 文件。例如：

```yaml
---
name: "Test-Driven Development"
description: "Write failing tests first, then implement to pass"
triggers:
  - "writing new functionality"
  - "bug fix that needs test coverage"
steps:
  - "Write a failing test that describes the expected behavior"
  - "Run the test and verify it fails"
  - "Write minimum code to make the test pass"
  - "Run the test and verify it passes"
  - "Refactor while keeping tests green"
exit_criteria:
  - "All tests pass"
  - "Code follows project style"
  - "No TODO comments left behind"
---
```

### 核心技能列表

- **using-agent-skills** - 元技能，路由和协调
- **spec-generation** - 规范生成
- **planning** - 任务规划和分解
- **test-driven-development** - 测试驱动开发
- **code-review-and-quality** - 代码审查和质量
- **git-workflow-and-versioning** - Git 工作流和版本控制
- **ci-cd-and-automation** - CI/CD 和自动化
- **api-and-interface-design** - API 和接口设计
- **deprecation-and-migration** - 弃用和迁移
- **code-simplification** - 代码简化

### Slash 命令

- `/spec` - 定义规范
- `/plan` - 规划实现

## 结论和启示

作者最想让人们从这个项目中带走的东西，比技能本身更重要的，是这个框架。

AI 编程代理是非常能干的初级工程师，没有对工作中不会出现在 diff 中的部分的直觉。资深工程工作（在 diff 中不会显示的部分：surfacing assumptions、 sizing changes、 writing the spec、 leaving evidence、 refusing to merge what can't be reviewed）是代理会跳过的部分，除非你让它不可能跳过。工作是越来越多地将这种 disciplina 编纂成代理无法-talk itself out of（说服自己不做）的东西。

技能是那种形式。反合理化表格。渐进式披露。过程胜于论述。验证作为承重退出标准。已经 work 的 Google 实践，使其可移植。

关键要点：

1. **AI 代理默认走捷径** - 需要显式强制执行资深工程师的工作流程
2. **工作流优于文档** - 将规范转化为可执行的步骤
3. **反合理化是关键** - 预先阻止常见的借口
4. **验证不可协商** - 结果必须有证据支持
5. **渐进式披露** - 只加载当前需要的技能
6. **范围纪律** - 只做被要求的事
7. **Google 工程实践** - 已经过验证的最佳实践
