## 洞察摘要

### 核心事件

Python 3.14 引入了一个新的**增量垃圾回收器（incremental GC）**，但在实际生产环境中出现了严重的**内存压力**问题。Python 核心团队决定在 Python 3.14 和 3.15 中将其回退，恢复到 Python 3.13 中的**代际垃圾回收器（generational GC）**。

### 技术背景

- **增量GC的设计目标**：减少"stop-the-world"长暂停时间
- **实际效果**：成功降低了最大GC暂停时间（例如从26ms降至1.3ms）
- **代价**：当存在大量循环垃圾时，内存使用可能大幅增加（最坏情况下达5倍），运行时间变慢

### 回退原因

1. **生产环境问题**：大量用户报告严重的内存压力
2. **未经过PEP流程**：增量GC没有经过标准的PEP评估流程
3. **历史因素**：在3.13最终发布前曾被回退过一次
4. **维护成本**：同时维护两个GC版本会增加复杂性

### 关键数据（来自测试）

| 指标 | 增量GC | 代际GC |
|------|--------|--------|
| 最大GC暂停 | 1.3ms | 26ms |
| 峰值内存(RSS) | 2.7x | 1x |
| 内存使用(最坏情况) | 5x | 1x |

### 时间安排

- **Python 3.15**：第一个测试版计划于2026年5月5日发布，如果回退准备就绪，可能会发布额外的alpha 9
- **Python 3.14.5**：原计划2026年6月9日发布，将提前发布以包含GC回退

### 未来计划

如果未来要重新引入增量GC，需要通过标准的PEP流程进行更彻底的评估。有开发者建议可以保留两个GC版本，让用户可以选择。

## 社区讨论要点

1. **双GC方案的可行性**：核心开发者讨论是否可以在同一版本中保留两个GC，让用户通过启动参数选择
2. **Sphinx减速问题**：这是触发回退的关键案例，涉及创建大量容器对象和引用循环
3. **缺乏真实世界基准测试**：目前缺少能真实反映GC行为的生产环境测试用例
4. **补丁发布的风险**：对于补丁版本来说，改变GC这样的核心组件被认为风险较高
