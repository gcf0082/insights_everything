## 核心洞察内容

### 1. API Key的标准格式

API Key通常由三部分组成：

- **前缀（Prefix）**: 用于标识API Key类型，如GitHub使用`gh(o|p)`，Stripe使用`sk_live`或`sk_test`
- **随机字符串**: 随机生成的十六进制字符串，长度可变
- **校验和（Checksum）**: 用于在访问数据库前快速验证Key有效性

存储时需哈希处理，与密码处理方式相同，创建后需立即复制否则永久丢失。

### 2. 多租户分片系统中的路由问题

在多租户系统中，API Key需要路由到正确的数据库分片。作者探索了三种方案：

| 方案 | 描述 | 性能 |
|------|------|------|
| 方案1 | 直接将API Key哈希映射到账户ID | 良好（2000万条记录测试通过） |
| 方案2 | 使用前缀作为唯一标识符在元分片中映射 | 可比较，但使前缀可预测 |
| 方案3 | 编码哈希（如Base64）并提取前几位 | 存在性能问题 |

### 3. 编码方案的性能陷阱

作者对Base-62和Base-70编码进行了基准测试，发现BigInt操作在JavaScript中非常缓慢，因为：

- BigInt使用软件实现的任意精度数学
- 相比直接哈希比较，堆分配更多
- 多次除法操作（43次）导致性能下降

### 4. 最终方案：SHAKE算法

采用SHA3家族的SHAKE算法：

- 海绵式算法可输出任意长度
- 使用Base64url编码替代十六进制
- 减少索引大小和计算成本
- 10字符编码提供约72 Quintillion种可能性，碰撞概率接近零

**性能结果（100万记录，1000个API Key，100次运行）：**

| 方案 | RTT中位数(ms) | PG中位数(ms) |
|------|---------------|--------------|
| Full SHA-256 | 0.222 | 0.006 |
| SHAKE256 32B→b64url | 0.189 | 0.005 |
| SHAKE256 32B + 8字符前缀 | 0.113 | 0.004 |

## 关键经验教训

1. **B-Tree索引对字符串同样高效**：PostgreSQL的B-Tree索引对任何可排序数据都有效，字符串大小的假设对查询性能影响不大

2. **避免过早优化**：不应假设短字符串一定比长字符串快

3. **BigInt操作的代价**：JavaScript中BigInt操作比普通数字操作慢得多

4. **POC测试的重要性**：基准测试必须公平比较，排除连接时间等干扰因素
