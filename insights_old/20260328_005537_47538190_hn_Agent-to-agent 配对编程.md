## 核心内容

这篇文章探讨了如何使用AI智能体进行结对编程（pair programming）。作者创建了一个名为"loop"的工具，让Claude和Codex两个AI智能体可以直接相互通信，进行代码编写和审查工作。

### 主要发现

1. **多智能体验证效果显著**：当Claude和Codex同时审查同一个PR并发现相同问题时，团队100%会修复该问题。

2. **多个LLM一致性**：2024年有研究表明，当4个LLM给出相同答案时，有95-99%的概率是正确的。

3. **实际应用场景**：作者使用Codex App Server和Claude Code Channels实现两个智能体之间的直接连接，比让一个智能体生成另一个智能体或读写共享文件的效果更好。

## 社区讨论要点

### 支持观点

- **Pivotal Labs经验**：有用户分享在Pivotal Labs的经历，每天全员结对编程确实有效，关键是要学会跳出固有思维进行沟通。

- **能力边界**：结对编程最适合处理超出当前能力范围的问题，可以减少"一直在脑中思考"的情况。

- **代码审查效率**：Claude负责生成代码，Codex负责审查错误。Claude完成后，Codex几乎总能发现问题。

### 质疑观点

- **缺乏科学验证**：社区普遍认为多智能体设置目前仍缺乏系统性测量证据，大部分是轶事性质的。

- **成本问题**：使用多个智能体成本较高，类似"按需付费"产品的商业模式。

- **价值疑问**：有评论认为如果两个智能体意见一致，结对编程的意义何在？

### 展望与建议

- **不同配置的智能体**：有观点认为关键可能不在于多个真实智能体，而是同一个智能体的两种不同配置（系统提示、模型、温度等）轮换使用。

- **PLAN.md的重要性**：计划文档应保留在git或PR中，以便追踪意图与实现的偏差。

- **人类参与**：未来可能让智能体网络在需要时联系人类作为真理来源。

## 技术实现

- 使用Codex App Server（开源）
- 使用Claude Code Channels
- 支持运行交互式TUI
- 可以同时在两个会话中引导和回答问题

---

## 结论

Agent-to-agent结对编程是一个新兴领域，通过多个AI智能体的协作来提高代码质量。虽然目前还缺乏严格的科学验证，但实践表明这种方法在代码审查和bug发现方面具有显著效果。随着模型和智能体框架的改进，这一领域仍有很大的探索空间。
