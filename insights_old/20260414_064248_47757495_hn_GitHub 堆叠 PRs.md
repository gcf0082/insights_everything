## 内容摘要

GitHub Stacked PRs 是 GitHub 推出的一个新功能，旨在解决大型代码审查项目中面临的挑战。该功能允许开发者创建一系列相互依赖的 Pull Request，形成"堆叠"结构，每个 PR 可以独立审查和合并。

### 核心功能

1. **链式 PR 管理**：将大型变更拆分成一系列小型、专注于单一职责的 PR
2. **独立审查**：每个 PR 可以独立审查，无需等待整个堆栈合并
3. **自动变基**：自动化处理 PR 之间的依赖关系和变基操作
4. **CLI 工具支持**：提供 `gh stack` 命令行工具
5. **UI 支持**：也可以通过 Web UI 创建和管理堆叠 PR

### 使用场景

- 大型代码库的单体迁移（如升级 react-native）
- 需要并行开发多个相关功能的场景
- 审查速度跟不上开发速度的团队
- 需要将大型变更拆分成小块以便审查

## 社区讨论要点

### 正面观点

1. **提高审查效率**：小型 PR 更容易审查，审查者可以快速完成
2. **并行工作流**：开发者可以在等待审查的同时继续在上一层 PR 上工作
3. **增量交付**：可以单独合并部分 PR，实现增量价值交付
4. **类似 Phabricator/Gerrit**：这种工作流在大型科技公司（Google、Meta）已使用多年

### 质疑观点

1. **Git 本身是否足够**：有用户认为直接使用 Git 分支和变基即可实现类似功能
2. **复杂度增加**：需要学习新工具和工作流
3. **破坏 Git 历史**：频繁变基和强制推送可能带来风险
4. **PR 模式问题**：有用户认为问题根源在于 GitHub 的 PR 抽象层本身

### 技术讨论

1. **历史渊源**：堆叠式代码审查最初用于 Linux 内核通过邮件发送补丁，后来被 Gerrit 和 Phabricator 发扬光大
2. **Git vs Mercurial**：讨论了 Mercurial API 更好但未能战胜 Git 的网络效应
3. **jj 工具**：有用户提到使用 jj (jujutsu) 工具实现类似工作流
4. **Squash 合并问题**：Squash 合并后会导致依赖的 PR 出现冲突

### 官方回应

GitHub Stack 团队表示功能正在 private preview 阶段，将在未来几周内向等待列表上的用户推出，欢迎反馈。

---

## 总结

GitHub Stacked PRs 为大型项目的代码审查工作流提供了新的解决方案，特别适合需要将大型变更拆分为小型独立 PR 的场景。该功能借鉴了 Phabricator 和 Gerrit 的堆叠式代码审查经验，并将其带入 GitHub 生态。虽然有用户质疑其必要性，但更多人对其简化大型项目代码审查的潜力表示期待。

该功能目前处于 private preview 阶段，需要通过等待列表申请使用。
