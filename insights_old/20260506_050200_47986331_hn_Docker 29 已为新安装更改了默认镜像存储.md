## 内容摘要

Docker Engine v29（2025年11月发布）对于新安装，将默认镜像存储从传统的存储驱动切换为 containerd。这一变更导致镜像和容器快照现在存储在 `/var/lib/containerd` 而不是 `/var/lib/docker`。

### 主要问题

1. **磁盘空间占用增加**：containerd 镜像存储同时保留压缩和非压缩格式的镜像层，而传统驱动只存储非压缩层，导致占用更多磁盘空间。

2. **层共享失效**：使用 containerd 后，共享基础层的多个镜像会为每个镜像存储各自压缩版本的共享层，即使底层未压缩层已经通过快照程序去重。这对于需要维护大量相似基础镜像的用户来说是一个重大问题。

3. **ML/AI 容器影响严重**：机器学习容器通常很大（单个 PyTorch 容器可达数GB），每次代码修改都会重复存储数GB的相同内容，开发体验极差。

4. **构建导出速度下降**：有用户报告导出镜像到文件从4.3秒增加到71.5秒（5.76GB镜像）。

### 社区反应

- 许多用户认为这是一个"非常奇怪的决定"
- 有人指出这对于使用 userns-remap 的 Docker 安装会引发兼容性问题
- 部分用户考虑改用 Podman 作为替代方案
- 已有人提交 Issue 尝试解决这个问题：containerd/containerd#13307

## 结论

Docker 29 的这一重大变更引发了社区的广泛不满。主要争议点在于磁盘空间占用大幅增加、层共享机制失效，以及对大型镜像（尤其是机器学习容器）的负面影响。许多开发者正在考虑迁移到 Podman 等替代方案。
