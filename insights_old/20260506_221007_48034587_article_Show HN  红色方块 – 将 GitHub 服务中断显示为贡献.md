## 项目概述

Red Squares 是一个讽刺性的网络工具，它将 GitHub 平台的宕机事件以类似 GitHub 贡献图表（contribution graph）的形式呈现。正常情况下，GitHub 的贡献图表显示用户每天的代码提交情况，绿色方块越多表示当天提交越多。而这个项目则用红色方块来表示 GitHub 平台发生宕机的日子，从而形成了一种独特的数据可视化效果。

## 数据来源

该项目的数据来自 `github-outages` 仓库（https://github.com/outages/github-outages），这是一个非官方的 Git 历史记录项目，用于追踪 GitHub.com 平台的宕机历史。

## 主要发现

### 1. 周末宕机率明显较低

根据用户分析和评论中的数据，GitHub 在工作日和周末的可用性存在显著差异：

- **工作日可用性**：约 89.3%
- **周末可用性**：约 96.5%
- **工作日受影响天数比例**：62%
- **周末受影响天数比例**：11%

这表明 GitHub 的宕机事件主要集中在工作日，周末的稳定性明显更好。

### 2. 周二是最危险的一天

根据分析数据，周二到周四是最容易出现宕机的时段，其中周二被认为是"危险区"。这一现象可能与工作日的部署活动和代码变更有关。

### 3. 用户反应与讨论

该项目的 Hacker News 讨论引发了广泛的技术社区讨论，主要话题包括：

- **微软收购后的变化**：有用户指出 GitHub 自 2018 年被微软收购后，可用性问题明显增加
- **负载与容量问题**：大多数宕机与容量问题相关，而非纯粹的代码部署问题
- **官方与第三方状态页面的差异**：官方状态页面与独立追踪的宕机数据存在显著差异
- **GitHub Actions 最受影响**：在过去的 12 个月中，GitHub Actions 经历了 57 次宕机，是受影响最严重的服务

### 4. 2025-2026 年宕机数据

根据独立追踪数据（2025年5月至2026年4月）：

- **总事件数**：257 起
- **重大宕机**：48 起
- **受影响最严重月份**：2026年2月（37 起事件）
- **最常受影响服务**：GitHub Actions（57 次宕机）

## 技术实现

该项目采用了现代化的前端技术构建，使用了模块化的 JavaScript 和 CSS 资源。从网页结构来看，它是一个单页应用（SPA），能够动态加载和渲染宕机数据。

## 社区意义

Red Squares 项目的出现反映了开发者社区对 GitHub 可靠性问题的关注和不满。通过这种讽刺性的可视化方式，项目作者以幽默但深刻的方式揭示了 GitHub 平台在可用性方面的挑战。这种做法与近年来其他类似的"民间"GitHub 状态追踪项目（如 "The Missing GitHub Status Page"）形成了呼应，它们共同提供了一种不同于官方数据的独立视角。

## 结论

Red Squares 作为一个数据可视化项目，不仅展示了 GitHub 近年来的宕机历史，也成为了技术社区对云服务可靠性讨论的催化剂。它提醒用户即使是大型科技公司提供的服务，也需要关注其实际可用性而非仅仅依赖官方声明。
