# 洞察报告：Stop Using Ollama

## 基本信息

- **洞察链接**: https://sleepingrobots.com/dreams/stop-using-ollama/
- **发布日期**: 2026年4月15日
- **作者**: Zetaphor
- **标签**: llm, local, ai, open-source, opinion

## 摘要

本文深入分析了Ollama为何不应该成为运行本地大语言模型的首选工具。作者认为，Ollama虽然凭借便捷的CLI界面获得了广泛使用，但它建立在llama.cpp的基础上，却多年拒绝给予适当归属，后来又转向闭源应用和云服务，偏离了最初的开源使命。

## 核心观点

### 1. 依赖llama.cpp却回避归属

Ollama的整个推理能力都来自Georgi Gerganov于2023年3月创建的llama.cpp项目。llama.cpp是让在消费级笔记本上运行大语言模型成为可能的关键引擎。然而，Ollama在成立后的**一年多时间里**，从未在其README、网站或营销材料中提及llama.cpp。社区在2024年初发现了这个问题（GitHub issue #3185），**超过400天**没有得到维护者的回应。直到社区提出PR后，Ollama联合创始人Michael Chiang才在README底部添加了一行说明。

### 2. 糟糕的Fork

2025年中期，Ollama试图远离llama.cpp，直接基于ggml（llama.cpp使用的底层张量库）构建自定义后端。结果适得其反：Ollama的自定义后端**重新引入了llama.cpp多年前已解决的问题**，包括结构化输出支持损坏、视觉模型失败、GGML断言崩溃等。Georgi Gerganov本人也指出Ollama对GGML的fork存在严重问题。

**性能对比**：
- llama.cpp比Ollama快**1.8倍**（161 tokens/s vs 89 tokens/s）
- CPU环境下差距达**30-50%**
- Qwen-3 Coder 32B上，llama.cpp吞吐量高出约**70%**

### 3. 误导性的模型命名

2025年1月DeepSeek发布R1系列模型时，Ollama将较小的蒸馏版本（如DeepSeek-R1-Distill-Qwen-32B，这实际上是基于Qwen的微调模型）简单地标记为"DeepSeek-R1"。用户运行`ollama run deepseek-r1`实际拉取的是一个8B的Qwen衍生模型，与真正的6710亿参数R1模型完全不同。这种做法误导了用户，也损害了DeepSeek的声誉。

### 4. 闭源桌面应用

2025年7月，Ollama发布了macOS和Windows的GUI桌面应用，但该应用开发在私有仓库中，源代码未公开，没有附带许可证。这对于一个声称开源的项目来说是一个令人困惑的举动。社区立即提出担忧，维护者沉默了数月。

### 5. Modelfile：重复造轮子

GGUF格式（由Georgi Gerganov创建）的核心理念是**单文件部署**——所有加载模型所需的信息都包含在模型文件中。Ollama的Modelfile是一个额外的配置文件，需要指定基础模型、聊天模板、系统提示、采样参数等——而这些信息大多已经存在于GGUF文件中。

**问题**：
- Ollama只能自动检测其已知的聊天模板列表
- 如果GGUF文件有有效的Jinja聊天模板但不匹配Ollama的已知模板，会回退到损坏的模板
- 修改参数需要导出整个模型（30-60GB）进行复制

### 6. 注册表瓶颈

新模型发布时，GGUF通常在几小时内出现在Hugging Face上。但通过Ollama，用户需要等待Ollama团队打包模型、选择量化方式、转换聊天模板并推送。Ollama通常只提供Q4_K_M和Q8_0量化，**不支持**Q5、Q6或IQ量化。

### 7. 云端转向

2025年末，Ollama引入了云托管模型，开始将提示路由到第三方云提供商。用户运行MiniMax等闭源模型时，数据可能会被转发到外部提供商。此外，发现了CVE-2025-51471安全漏洞——恶意注册表服务器可诱使Ollama在模型拉取过程中向攻击者控制的端点发送认证令牌。

### 8. VC模式

Ollama是Y Combinator支持的初创公司（2021年冬季批次），由之前构建Docker GUI（后被Docker收购）的工程师创立。其发展路径遵循典型模式：
1. 以开源起步，基于llama.cpp建立社区信任
2. 最小化归属，让产品看起来独立
3. 创建锁定——专有模型注册表格式、哈希文件名
4. 推出闭源组件
5. 添加云服务作为盈利手段

## 替代方案

| 工具 | 描述 |
|------|------|
| **llama.cpp** | 真正的引擎，有OpenAI兼容API服务器，内置Web UI，持续优于Ollama |
| **llama-swap** | 多模型编排，可热交换模型 |
| **LM Studio** | GUI应用，使用llama.cpp，无锁定 |
| **Jan** | 干净的聊天界面，本地优先设计 |
| **koboldcpp** | Web UI和广泛配置选项 |
| **ramalama** | Red Hat的容器化模型运行器，明确标注上游依赖 |

## 结论

Georgi Gerganov在2023年3月的一个晚上创造了llama.cpp，掀起了本地AI的革命。Ollama则在这一工作基础上包装了一个便捷的CLI，筹集VC资金，多年拒绝给予归属，糟糕地fork了代码，推出了闭源应用，然后转向云服务。在每个本可以成为良好开源公民的决策点，它们都选择了让自己在投资者面前看起来更独立的道路。

**本地LLM生态系统不需要Ollama，需要的是llama.cpp。剩下的只是包装，而更好的包装已经存在。**
