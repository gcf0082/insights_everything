## 核心洞察

本文作者Jonathan Vogel（AWS开发者倡导者）分享了他在Java应用性能优化方面的实战经验。通过修复代码中的常见反模式，他将一个Java订单处理应用的性能提升了约**5倍**：

- **执行时间**：1198ms → 239ms
- **吞吐量**：85,000订单/秒 → 419,000订单/秒
- **堆内存**：1GB+ → 139MB
- **GC暂停**：19次 → 4次

## 八大性能反模式

### 1. 循环中的字符串拼接

**问题代码**：
```java
String report = "";
for (String line : logLines) {
    report = report + line + "\n";
}
```

**问题**：String是不可变的，每次`+`操作都会创建新的String对象并复制所有内容，时间复杂度O(n²)

**修复方案**：
```java
StringBuilder sb = new StringBuilder();
for (String line : logLines) {
    sb.append(line).append("\n");
}
String report = sb.toString();
```

---

### 2. 循环内的Stream迭代（意外的O(n²)）

**问题代码**：
```java
for (Order order : orders) {
    int hour = order.timestamp().atZone(ZoneId.systemDefault()).getHour();
    long countForHour = orders.stream()
        .filter(o -> o.timestamp().atZone(ZoneId.systemDefault()).getHour() == hour)
        .count();
    ordersByHour.put(hour, countForHour);
}
```

**问题**：对每个订单都遍历整个列表，导致O(n²)复杂度。在演示应用中，这个模式占据了**71%的CPU**！

**修复方案**：
```java
for (Order order : orders) {
    int hour = order.timestamp().atZone(ZoneId.systemDefault()).getHour();
    ordersByHour.merge(hour, 1L, Long::sum);
}
```

---

### 3. 热路径中使用String.format()

**问题**：`String.format()`每次调用都需要解析格式字符串，性能最差

**修复方案**：使用`String.format()`仅用于数字格式化，其他部分用字符串连接或StringBuilder

---

### 4. 热路径中的自动装箱（Autoboxing）

**问题代码**：
```java
Long sum = 0L;
for (Long value : values) {
    sum += value;
}
```

**问题**：每次迭代都会创建新的Long包装对象，造成大量堆内存开销

**修复方案**：使用基本类型`long`

```java
long sum = 0L;
for (long value : values) {
    sum += value;
}
```

---

### 5. 使用异常进行流程控制

**问题**：`Throwable.fillInStackTrace()`会遍历整个调用栈，创建StackTraceElement对象，开销巨大

**修复方案**：预先验证输入
```java
public int parseOrDefault(String value, int defaultValue) {
    if (value == null || value.isBlank()) return defaultValue;
    for (int i = 0; i < value.length(); i++) {
        char c = value.charAt(i);
        if (i == 0 && c == '-') continue;
        if (!Character.isDigit(c)) return defaultValue;
    }
    try {
        return Integer.parseInt(value);
    } catch (NumberFormatException e) {
        return defaultValue;
    }
}
```

---

### 6. 过度宽泛的同步

**问题**：整个方法使用synchronized会成为并发瓶颈

**修复方案**：使用`ConcurrentHashMap`和`LongAdder`
```java
private final ConcurrentHashMap<String, LongAdder> counts = new ConcurrentHashMap<>();

public void increment(String key) {
    counts.computeIfAbsent(key, k -> new LongAdder()).increment();
}
```

---

### 7. 重复创建可复用对象

**问题代码**：
```java
public String serializeOrder(Order order) throws JsonProcessingException {
    return new ObjectMapper().writeValueAsString(order);
}
```

**问题**：ObjectMapper构建成本高，每次调用都重复初始化

**修复方案**：
```java
private static final ObjectMapper MAPPER = new ObjectMapper();

public String serializeOrder(Order order) throws JsonProcessingException {
    return MAPPER.writeValueAsString(order);
}
```

---

### 8. 虚拟线程 pinning（JDK 21-23）

**问题**：虚拟线程在`synchronized`块中执行阻塞I/O时会pin住carrier线程

**修复方案**：使用`ReentrantLock`替代`synchronized`（JDK 24已修复）

## 关键结论

1. **单独来看这些反模式都不致命**：它们不会导致崩溃或错误结果，只是让代码变慢

2. **组合效应惊人**：多个反模式叠加导致性能下降5倍、堆内存增加7倍、GC暂停增加近5倍

3. **必须使用性能分析工具**：这些模式在代码审查中很难发现，需要借助JFR等工具定位热点

4. **生产环境影响巨大**：在分布式系统中，优化效果会通过fleet实例数量放大，成本节省显著
