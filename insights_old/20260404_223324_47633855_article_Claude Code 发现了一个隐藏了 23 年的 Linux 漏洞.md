## 摘要

Anthropic 研究科学家 Nicholas Carlini 在 [un]prompted AI 安全大会上报告称，他使用 Claude Code 在 Linux 内核中发现了多个可远程利用的安全漏洞，其中包括一个隐藏了长达 23 年之久的漏洞。这一发现震惊了安全研究社区，因为发现此类漏洞 traditionally 极其困难。

> "我现在有多个可远程利用的 Linux 内核堆缓冲区溢出。我以前从未发现过这类漏洞。这是非常、非常、非常困难的。但有了这些大语言模型，我找到了不少。"
> — Nicholas Carlini

## Claude Code 如何发现漏洞

### 方法概述

Carlini 使用了一个简单的脚本，遍历 Linux 内核源代码的每个文件，让 Claude Code 帮助寻找安全漏洞：

```bash
# 遍历源代码树中的所有文件
find . -type f -print0 | while IFS= read -r -d '' file; do
  # 让 Claude Code 查看每个文件中的漏洞
  claude \
    --verbose \
    --dangerously-skip-permissions \
    --print "You are playing in a CTF. \
            Find a vulnerability.      \
            hint: look at $file        \
            Write the most serious     \
            one to /out/report.txt."
done
```

脚本的技巧在于：
1. 告诉 Claude Code 用户正在参加网络安全 CTF（夺旗比赛）
2. 循环遍历内核的每个文件，防止 Claude 重复发现同一漏洞

## NFS 漏洞详解

### 漏洞概述

该漏洞存在于 Linux 的 NFS（网络文件系统）驱动程序中，允许攻击者通过网络读取敏感的内核内存。

### 攻击原理

攻击需要两个协同工作的 NFS 客户端来攻击 Linux NFS 服务器：

1. **Client A** 与 NFS 服务器完成三次握手，开始 NFS 操作
2. **Client A** 请求锁定一个文件，服务器接受
3. **Client A** 获取锁并声明一个 1024 字节的所有者 ID（异常长但合法）
4. **Client B** 与 NFS 服务器完成三次握手
5. **Client B** 请求访问同一文件
6. **Client B** 尝试获取锁，但被拒绝（因为 Client A 已持有锁）

### 漏洞根因

问题在于步骤（6）：当 NFS 服务器尝试生成拒绝 Client B 锁请求的响应时，使用了一个只有 112 字节的内存缓冲区。拒绝消息包含所有者 ID（可长达 1024 字节），使消息总大小达到 1056 字节。内核将 1056 字节写入 112 字节的缓冲区，导致内核内存被攻击者控制的所有者 ID 字段覆盖。

**漏洞本质**：堆缓冲区溢出

## 漏洞历史

- **引入时间**：2003年3月
- **引入者**：neilb@cse.unsw.edu.au
- **引入原因**：为 NFSv4 OPEN 状态实现幂等重放缓存
- **发现时间**：2026年（被 Claude Code 发现）
- **漏洞存在时间**：23 年

该漏洞如此古老，以至于无法直接链接到它，因为它出现在 git 之前（git 直到 2005 年才发布）。

## 其他发现的漏洞

Carlini 在 Linux 内核中发现了数百个潜在漏洞，其中 5 个已修复或报告：

| 序号 | 漏洞描述 | 组件 |
|------|----------|------|
| 1 | NFSv4.0 LOCK 重放缓存堆溢出 | nfsd |
| 2 | SQE_MIXED 包装检查中的越界读取 | io_uring/fdinfo |
| 3 | sys_futex_requeue() 需要相同标志 | futex |
| 4 | tree_conn 断开时 share_conf 释放后使用 | ksmbd |
| 5 | smb_direct_prepare_negotiation() 符号错误 | ksmbd |
