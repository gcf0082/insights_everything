# 洞察报告：Linux 7.0 导致 PostgreSQL 性能下降

## 基本信息

- **原始链接**: https://news.ycombinator.com/item?id=47949585
- **标题**: Linux 7.0 Broke PostgreSQL: The Preemption Regression Explained
- **发布时间**: 5小时前
- **点数**: 115
- **评论数**: 61
- **来源**: thecoder.cafe

## 文章摘要

文章解释了Linux 7.0内核中引入的PREEMPT_LAZY选项如何导致PostgreSQL性能下降约50%。问题的核心在于：

1. Linux 7.0改变了内核的抢占行为
2. 当PostgreSQL持有自旋锁时，如果被抢占，其等待的锁路径会导致所有其他后端阻塞
3. 这是由于页面错误处理过程中发生的抢占导致的

## 关键洞察

### 1. 问题覆盖面有限

- 多数评论者指出这个问题**仅影响特定配置**（如禁用huge pages的120GB内存PostgreSQL实例）
- 在Graviton 4设置上测试，没有使用huge pages
- 许多评论者认为这是边缘案例，不是普遍问题

### 2. 关于huge pages的讨论

- 启用huge pages可以将PostgreSQL性能提升10-20%
- 透明huge pages与预分配huge pages是不同的概念
- 大型PostgreSQL实例应该配置huge pages
- 这是性能优化的推荐做法，但配置较繁琐

### 3. 与其他内核问题的关联

- MongoDB在Linux 7.0上因tcmalloc问题完全无法运行
- Linux 6.19曾破坏了tcmalloc和MongoDB

### 4. 技术细节

- 问题的真正解决方案可能是rseq（Restartable Sequences）的时间片扩展功能
- PREEMPT_NONE选项意味着内核几乎不会中断运行线程
- 这些选项控制内核的可抢占性，用户空间始终可以被抢占

### 5. 关于桌面Linux的讨论

- 评论者指出Linux在低内存情况下的桌面体验不佳
- 当RAM耗尽时，桌面环境会变得无响应
- 建议启用swap或使用earlyoom

### 6. 关于Linus Torvalds的讨论

- 有人提到Linus对贡献者的批评风格
- 讨论这种反馈方式是否合适
- 多数人认为这是针对经验丰富的开发者，不是新手

## 技术背景

### 页面元数据开销

- Linux中每个struct page占64字节
- 128GB RAM服务器使用4KB页面：超过3000万个页面
- 仅元数据就需要约2GB内存

### 解决方案

- 启用huge pages可以减少页面数量
- 使用rseq检测并处理抢占
- 配置内核参数调整调度行为

## 结论

这次讨论揭示了Linux内核更新可能对特定工作负载产生意外影响。虽然PostgreSQL性能下降50%看起来很严重，但这个问题主要影响使用大内存但未启用huge pages配置的系统。对于生产环境，建议：

1. 为大型PostgreSQL实例启用huge pages
2. 监控系统在不同内核版本上的性能
3. 关注内核文档和社区关于preemption的讨论
