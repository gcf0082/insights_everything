## 摘要

本文介绍了 `safe-gc` crate——一个完全不使用 `unsafe` 代码实现的 Rust 垃圾回收库。该库提供了安全的 API 接口，内部实现也不包含任何 `unsafe` 代码，甚至使用了 `forbid(unsafe_code)` pragma。不过，该垃圾回收器的性能并不是特别高。

## 核心内容

### 背景

此前，所有 Rust 的垃圾回收库都在内部使用了大量 `unsafe` 代码。`safe-gc` 是首个完全不使用 `unsafe` 的实现。

### 实现方式

与直接解引用 `Gc<T>` 指针不同，`safe-gc` 通过索引堆来访问对象，这使其能够遵守 Rust 的所有权和借用规则。

两种引用类型：
- `Gc<T>`：可复制，用于从 GC 管理对象内部引用其他对象，但不 root 对象
- `Root<T>`：会 root 对象，防止被回收，不可复制

### 算法

采用简单的 mark-and-sweep（标记-清除）算法：
1. 重置所有 mark bits
2. 标记阶段：从 root set 开始标记所有可达对象
3. 清除阶段：回收不可达对象

### 安全性保证

- 未正确实现 `Trace` 或持有未 root 的 `Gc<T>` 跨垃圾回收操作会导致 panic 或返回错误对象，而非内存安全问题
- ABA 问题可通过添加 generation counter 解决

## HN 评论热点

1. **是否需要 Rust GC**：评论者对 Rust 是否应该添加 GC 存在分歧
   - 支持者：实现解释器、 cyclic 数据结构时有用
   - 反对者：Rust 的设计理念就是无需 GC，添加 GC 是走回头路

2. **与 unsafe 的比较**：索引方式比 unsafe 代码更安全——错误是逻辑 bug 而非 UB

3. **性能讨论**：GC 代码通常比非 GC 代码慢，但在某些场景下可以接受

4. **CVE-rs 讨论**：有人指出 Rust 仍有内存安全问题，但被反驳 unsafe blocks 是设计的一部分
