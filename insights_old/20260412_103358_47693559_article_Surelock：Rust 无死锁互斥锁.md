## 概述

Surelock 是一个 Rust 库，旨在通过编译时检查在静态层面预防死锁。如果代码能够编译，就永远不会发生死锁。该库不需要 `Result`、`Option`，也不需要在锁路径上处理运行时 panic。

---

## 核心概念

### 死锁的四个必要条件（Coffman 条件）

| 条件 | 含义 |
|------|------|
| 互斥 | 至少有一个资源被独占持有 |
| 持有并等待 | 线程持有锁的同时等待另一个锁 |
| 不可抢占 | 锁无法被强制夺走 |
| 循环等待 | 等待图中存在环路 |

**关键洞察**：打破任一条件即可防止死锁。互斥是 mutex 的本质目的，不可抢占会引入其他问题。因此，解决方案集中在打破"持有并等待"和"循环等待"。

### 设计原理

作者借用了物理世界的类比：获取锁就像获得一把钥匙（`MutexKey`），钥匙在锁使用期间被保留，解锁时才归还。这个"钥匙"是一个**线性作用域令牌**，记录了当前已持有的锁，编译器据此判断是否允许获取更多锁。

## 双重机制设计

### 1. 同层级锁：LockSet

- **场景**：多个锁位于同一层级
- **实现**：每个 `Mutex` 在创建时获得唯一的递增 `LockId`，`LockSet` 在构造时按 ID 预排序，无论线程以何种顺序请求锁，最终获取顺序都是确定性的
- **优势**：原子性获取（全部成功或全部失败），无需运行时追踪

### 2. 不同层级锁：Level<N>

- **场景**：锁代表不同类型的资源（如配置锁 vs 账户锁）
- **实现**：通过 `Level<N>` 类型和 `LockAfter` trait 约束，在编译时强制执行严格的升序获取
- **优势**：编译时错误而非运行时崩溃

### 选择总序而非 DAG 的原因

`lock_tree` 使用 DAG（允许声明分支独立），但这会导致问题：如果线程1获取 A→B，线程2获取 B→A，且两者在 DAG 中都合法，仍会产生死锁。Surelock 采用严格的**总序**，更受限但**真正安全**。

## API 设计

### MutexKey 的生命周期

```
Locksmith (程序级单例) → KeyVoucher (可转移到其他线程) → KeyHandle (线程级) → MutexKey<'scope> (带不变生命期) → MutexGuard
```

### 两种获取方式

1. **隐式**：`lock_scope`（推荐）
   - 在 `std` 上自动处理 `thread_local!`
   - 示例：
     ```rust
     lock_scope(|key| {
         let (guard, key) = data.lock(key);
     });
     ```

2. **显式**：`Locksmith` + `KeyVoucher`（适用于 `no_std`、嵌入式）
   - 更细粒度的控制
   - 适合裸机环境

## 特色功能

- **零运行时依赖**：默认仅需 `alloc`，`thiserror` 是唯一运行时依赖（零成本）
- **`no_std` 支持**：可配合 `portable-atomic` 和 `critical-section` 用于 Cortex-M0 等无 CAS 的目标
- **逃生舱**：`Mutex::unchecked_lock()` 隐藏在 `escape-hatch` feature flag 下，可供特殊情况使用

## 与现有方案的对比

| 特性 | Surelock | happylock | lock_tree |
|------|----------|-----------|-----------|
| 多锁原子获取 | ✓ | ✓ | ✗ |
| 编译时层级排序 | ✓ | ✗ | ✓ |
| 实例级层级分配 | ✓ | ✗ | ✗ |
| DAG 排序 | ✗（总序） | ✗ | ✓ |
| 作用域唯一性 | ✓ | ? | ? |

### happylock 的局限
- 打破"持有并等待"条件，必须一次性获取所有锁，无法增量获取
- 按内存地址排序，地址不稳定需要额外 `unsafe` 处理

### lock_tree 的局限
- DAG 排序可能导致编译通过但运行时死锁（见上文分析）

## 总结

Surelock 是作者对 Rust 类型系统潜力的探索——利用类型检查器在编译期强制锁获取顺序，让正确的做法成为最简单、最自然的选择。死锁在理论上早已被"解决"（自1971年起），挑战在于让预防机制足够 ergonomics 以至于人们真正愿意使用它。
