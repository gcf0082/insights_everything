## 事件概述

2026年3月30日，著名的JavaScript HTTP客户端库**Axios**在npm上发布了两个恶意版本：`axios@1.14.1` 和 `axios@0.30.4`。攻击者入侵了一个主要维护者（jasonsaayman）的npm账户，绕过了项目的正常GitHub Actions CI/CD流程，手动发布了带有恶意代码的包。

### 攻击方式

恶意版本并没有在Axios本身代码中注入任何恶意代码，而是通过引入一个虚假依赖包 `plain-crypto-js@4.2.1`（该包从未在Axios源代码中被导入使用）来执行 `postinstall` 脚本。这个脚本作为一个**跨平台远程访问木马（RAT）**，针对macOS、Windows和Linux系统。

## HN 讨论热点

### 1. 防护措施建议

社区建议所有包管理器设置**最低发布年龄**（minimum release age）：

- **npm**: `min-release-age=7` (天) + `ignore-scripts=true`
- **pnpm**: `minimum-release-age=10080` (分钟)
- **bun**: `minimumReleaseAge = 604800` (秒)
- **uv**: `exclude-newer = "7 days"`

`ignore-scripts=true` 配置可以完全阻止此类攻击。pnpm 和 bun 默认不执行生命周期脚本。

### 2. Axios 为何仍被广泛使用

- Node.js 18 才将 fetch 作为标准内置，之前版本不稳定
- Axios 拥有**拦截器**（interceptors）功能，方便添加 token、处理401、错误重试等
- 支持代理（proxy），对企业场景很重要
- 测试时容易 mock 不同响应

### 3. 供应链安全担忧

社区对npm生态系统的供应链安全表达了深切担忧：

> "If everyone avoids using packages released within the last 7 days, malicious code is more likely to remain dormant for 7 days."

- 有人提议对发布流程增加**人工审批**或**冷却期**
- 建议使用 **bwrap**（Linux沙箱）限制包管理器的系统访问权限
- 有人提到最终可能需要企业化的**依赖托管服务**
- 建议将依赖代码复制到自己的代码库中审查

### 4. 攻击的复杂性

这是有记录以来针对top-10 npm包**最复杂的供应链攻击**之一：

- 恶意依赖提前18小时部署（避免触发"新包"警报）
- 三个平台（macOS/Windows/Linux）分别构建不同的 payload
- 在39分钟内同时覆盖两个发布分支
- 攻击后自动清除所有痕迹（删除自身、替换package.json）

### 5. 检测方法

```bash
# 检查是否安装了恶意版本
npm list axios | grep -E "1\.14\.1|0\.30\.4"

# 检查是否有 plain-crypto-js
ls node_modules/plain-crypto-js && echo "可能受影响"

# 检查 RAT 文件是否存在
ls /Library/Caches/com.apple.act.mond  # macOS
ls /tmp/ld.py  # Linux
```

### 6. 修复建议

1. 降级到安全版本：`npm install axios@1.14.0` 或 `axios@0.30.3`
2. 使用 `overrides` 阻止恶意版本被传递依赖解析
3. 使用 `--ignore-scripts` 阻止 postinstall 脚本执行
4. 如果发现RAT artifacts，需将系统视为完全泄露，重新构建

这次事件再次凸显了npm生态系统的供应链安全问题。由于Axios每周下载量超过3亿，任何细微的安全漏洞都可能影响巨大。社区共识是：**设置最低发布年龄、锁定依赖版本、禁用postinstall脚本**是当前最有效的防护措施。
