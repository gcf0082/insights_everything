## 一、版本发布概述

Erlang/OTP 29.0 于2026年5月13日发布，这是一个主要版本更新，包含新功能、改进以及一些不兼容的变更。该版本带来了多项重要更新，特别关注安全性和开发者体验。

---

## 二、主要新特性

### 2.1 安全改进
- **SSH daemon 默认禁用**：Shell 和 exec 服务现在默认禁用，实现了“默认安全”原则，防止认证用户执行任意 Erlang 代码
- **SFTP 默认禁用**：SSH daemon 启动时不再默认启用 SFTP 子系统
- **默认密钥交换算法更新**：默认密钥交换算法现在是 mlkem768x25519-sha256，这是一种混合抗量子算法，结合了 ML-KEM-768 和 X25519

### 2.2 语言新特性
- **原生记录（Native Records）**：根据 EEP-79 实现的实验性功能，原生记录是类似传统元组记录的数据结构，但是一种真正的数据类型
- **is_integer/3 守卫**：新增的守卫 BIF 可以轻松验证值是否为整数并在特定范围内，例如 `is_integer(I, 0, 100)`
- **多值推导式（Multi-valued Comprehensions）**：根据 EEP 78 支持，例如 `[-I, I || I <- [1, 2, 3]]` 将产生 `[-1,1,-2,2,-3,3]`
- **unsafe 属性支持**：新增 `-unsafe` 属性用于标记不安全的函数

### 2.3 标准库更新
- **io_ansi 模块**：新增模块，允许用户发出虚拟终端序列（ANSI 序列），用于为文本添加颜色/样式或创建完整的终端应用程序
- **ct_doctest 模块**：新增模块，允许用户测试 Erlang 模块文档和文档文件中的示例
- **rand 模块**：新增 `rand:shuffle/1` 和 `rand:shuffle_s/2` 函数用于随机排列列表

## 三、社区讨论焦点

### 3.1 OTP 含义讨论
有用户询问 OTP 的含义，有回复指出 OTP 原指 "Open Telecom Platform"（开放电信平台），但现在 OTP 只是一个独立术语。不过官方文档中仍保留了对 OTP 原始含义的引用。

### 3.2 Erlang 现状与未来
- **新项目使用情况**：有用户询问是否还有人在新项目中使用纯 Erlang，回复显示仍有一些开发者今年在 IoT、应用服务器和 TUI 框架等项目中使用 Erlang
- **选择 Erlang 的理由**：部分开发者认为 Elixir 相比 Erlang 没有明显优势，Erlang 更符合他们的思维习惯
- **WhatsApp 仍在使用**：多位知情人士确认 WhatsApp 仍在使用 Erlang，并且还在增加 Rust 的使用

### 3.3 与其他语言的比较
- **Erlang vs Elixir**：讨论了原生记录对 Elixir 的潜在影响，Elixir 的 struct 有可能编译为原生记录
- **Elixir/Phoenix vs Rails/Django**：社区进行了热烈讨论，支持者认为 Phoenix 在长期维护、并发处理、生产稳定性方面优于 Rails 和 Django
- **Elixir vs Go**：有用户认为 Elixir 的并发更符合人体工学，例如并行 map 示例比 Go 更简洁

### 3.4 安全漏洞关注
有用户报告在生产环境中发现多个高危 CVE：
- CVE-2025-32433
- CVE-2026-28808
- CVE-2026-23941
- CVE-2026-32144
- CVE-2025-48041
- CVE-2025-30211

建议尽快升级到最新版本以修复这些安全漏洞。

## 四、总结

Erlang/OTP 29.0 是一个重要的版本更新，重点关注安全性和开发者体验。SSH 默认禁用的安全改进、多值推导式和原生记录等语言特性的增强，以及新的 io_ansi 模块，都显示出 Erlang 生态系统在持续演进。

社区讨论反映出 Erlang/OTP 在电信领域和需要高可靠性的应用中仍然具有重要地位，WhatsApp 等大型案例证明了其在生产环境中的可靠性。同时，与 Elixir、Go、Rails 等其他语言的比较也显示出不同技术栈各有其适用场景。
