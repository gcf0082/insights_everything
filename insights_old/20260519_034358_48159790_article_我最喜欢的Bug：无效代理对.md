# 洞察报告：无效代理对（Invalid Surrogate Pairs）

**洞察链接**: https://george.mand.is/2026/05/my-favorite-bugs-invalid-surrogate-pairs/

**作者**: George Mandis

**发布日期**: 2026年5月14日

**标签**: JavaScript, Unicode, Bug, Emoji, TipTap, Yjs

---

## 摘要

本文作者分享了他最喜欢的Bug故事——一个关于无效代理对（Invalid Surrogate Pairs）的技术问题。该Bug导致协作编辑器在特定情况下静默停止同步用户内容，原因是一个CRDT操作恰好落在emoji代理对的中间，导致`.slice()`方法产生孤立的代理字符，进而触发`URIError`异常。

---

## 问题背景

作者在迁移一个遗留编辑器到更协作的体验时，使用了以下技术栈：

- **TipTap**: 基于ProseMirror的富文本编辑器
- **Yjs**: 处理实时同步的CRDT库

在Alpha版本期间，编辑器有时会静默停止保存内容。用户继续输入时一切看起来正常，但编辑内容停止同步到Yjs文档。下次打开页面时，失败点之后的所有内容都消失了。

---

## 问题诊断

经过产品经理的深入调查，发现问题只在特定操作下出现：

1. 使用两个需要代理对的emoji（如🟢和🔴）
2. 在两个emoji之间插入一个字符

这个特定操作触发了底层CRDT库中的一个splice操作，将一个代理对从中间分割。

**关键条件**：
- 只有超过U+FFFF的emoji才会触发（需要代理对）
- 只有导致在错误字节偏移处进行splice的操作才会触发

---

## 技术原理解析

### 三个核心概念

| 概念 | 定义 |
|------|------|
| **码元（Code Units）** | JavaScript内部用于存储字符串的16位值，`.length`、`.slice()`和`.charCodeAt()`都基于此操作 |
| **码点（Code Points）** | Unicode定义的单个字符，如U+1F920（🤠） |
| **字素簇（Grapheme Clusters）** | 人类感知的"一个字符"，如👩‍🚀实际由3个码点和5个码元组成 |

### 示例对比

| 字符 | 码元数 | 码点数 | 字素数 |
|------|--------|--------|--------|
| A | 1 | 1 | 1 |
| 🤠 | 2 | 1 | 1 |
| 👩‍🚀 | 5 | 3 | 1 |
| 👨‍👨‍👧‍👧 | 11 | 7 | 1 |

### `.slice()`的问题

当使用`.slice()`在代理对中间切割时：

```javascript
"🤠".slice(0, 1); // → '\uD83E'  (孤立的高代理)
"🤠".slice(1, 2); // → '\uDD20'  (孤立的低代理)
```

这些片段不是有效字符，尝试编码时会抛出错误：

```javascript
encodeURIComponent("🤠".slice(0, 1));
// URIError: URI malformed
```

---

## 问题根源

Yjs依赖一个名为lib0的工具库。lib0的`splice`方法内部使用JavaScript的`.slice()`。当CRDT操作恰好落在emoji代理对的两个半部分之间时，lib0会产生一个带有孤立代理的字符串。该字符串最终在同步过程中被传递给`encodeURIComponent`，抛出未捕获的`URIError`。

由于错误未被捕获，同步直接停止。编辑器在本地继续工作，给用户一切正常的假象，而编辑内容却静默地丢失了。

---

## 临时解决方案

由于无法直接修复lib0（尽管后来被修复），作者团队采取了以下措施：

1. **添加离线支持**: 虽然最初不需要，但添加离线支持可以在用户断开连接时继续在本地更新CRDT，下次重新连接时再同步。

2. **全局错误监听器**: 捕获`URIError: URI malformed`错误，显示模态框提示用户刷新页面。

---

## 最终解决方案

1. **lib0被修复**: 上游修复检测切片字符串的第一个字符是否是没有匹配低代理的高代理，并将其替换为U+FFFD（Unicode替换字符）。

2. **将emoji设为原子节点类型**: 在ProseMirror/TipTap中定义自定义节点类型，使每个emoji成为一个不可分割的单元，编辑操作无法将emoji分成两半。

---

## 现代最佳实践

对于需要正确分割字符的场景，应使用`Intl.Segmenter`：

```javascript
const seg = new Intl.Segmenter(undefined, { granularity: "grapheme" });
const segments = [...seg.segment("👩‍🚀A👍")].map((s) => s.segment);
// → ['👩‍🚀', 'A', '👍']
```

这按字素簇而非码元进行分割，不会产生孤立的代理或被分割的emoji。

---

## 启示

任何使用`str.slice(0, 1)`或`str[0]`来获取"第一个字符"的代码都可能存在此问题。最常见的场景是生成用户名字首字母——当用户名字以emoji开头时，大多数应用会失败。
