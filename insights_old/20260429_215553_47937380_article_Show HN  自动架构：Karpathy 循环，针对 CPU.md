## 项目概述

本项目探索了 Andrej Karpathy 的自主研究循环方法（autoresearch）是否能泛化到 CPU 架构领域。autoresearch 原本在 Python/梯度下降领域取得成功，研究者将其应用于 RISC-V CPU 微架构优化，这是一个完全不同的领域。

### 技术栈

- **目标硬件**：5级流水线顺序 RV32IM 处理器（SystemVerilog 实现）
- **评估平台**：Gowin GW2A-LV18 FPGA（Tang Nano 20K）
- **基准测试**：CoreMark

---

## 实验设置

### 运行机制

每轮三个并行槽：

1. **假设生成**：代理提出微架构假设（YAML 格式），通过 schema 验证
2. **实现**：在隔离的 git worktree 中编辑 RTL 文件
3. **评估**：
   - riscv-formal：53个符号 BMC 检查
   - Verilator 协同仿真：RVFI 字节级比对
   - 3种子布局布线：Fmax × CoreMark 迭代/周期 = 适应度
   - CoreMark CRC 验证

### 多样性策略

强制每个槽选择不同类别（micro_opt | structural | predictor | memory | extension），避免固守单一想法。

## 关键发现

### 验证器的重要性

73个假设中63个是错误的。验证器捕获了多种失败模式：

1. **相同想法，两次失败**：DIV/REM 首次实现破坏了仿真，第二次修正后成功
2. **沙箱违规**：代理试图编辑 rtl/** 和 test/test_*.py 外的文件被拒绝
3. **73% 回归**：某轮提议的 JALR 预测器导致性能暴跌， orchestrator 捕获
4. **Schema 错误**：类型错误在代码生成前被拦截

### 验证器组件

- 形式检查：ill/unique/liveness/cover 属性
- 路径沙箱：限制可编辑文件范围
- 3种子 P&R：中位数 Fmax
- CRC 验证：重新验证 CoreMark 输出
- MMIO 计时：排除热身和 printf 影响

## 核心洞察

### 1. 验证器是核心竞争力

> 自主循环是生产者。验证器是唯一站在你和"自信的错误"之间的事物。

**关键结��**：
- 自主研究循环已商品化（模型 + 提示 + 工具 + 分数板）
- 验证器不是商品，它编码了业务真正的"正确"含义
- 未来赢家公司不是拥有最聪明规划器的公司，而是验证器最清晰的公司

### 2. CPU 优化的自动化潜力

- 10小时内自动找到10项优化
- 性能提升 +92% 超越基线
- 超越人类调优的 VexRiscv（+56%）
- 资源使用减少 40%

### 3. 搜索效率

- 关键突破来自第3次迭代（冷多周期 DIV/REM）
- 代理不知道这会同时减少 LUT 使用
- 通过实践和观察发现

## 未来方向

1. **种群搜索**：从每轮保留 top-K，而非顺序搜索
2. **工作负载泛化**：在 Embench 上测试，验证 CoreMark 成果是否泛化
3. **Verifier 优先**：找到业务中足够清晰的验证器来指向循环

---

## 结论

本项目证明：
1. 自主研究循环可以成功泛化到硬件领域
2. 验证器质量决定项目成败
3. 自动优化可在10小时内超越人类多年调优成果
4. **未来属于拥有最清晰验证器的团队**，而非最聪明的规划器
