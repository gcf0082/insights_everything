## 核心洞察摘要

本文揭示了关于 Linux 系统中 Zswap 和 Zram 的常见误解，提供了关于何时使用这两种压缩交换技术的专业建议。

### 主要观点

**1. 优先使用 Zswap**
- 如果不确定，应该优先使用 Zswap
- Zswap 作为磁盘交换前端的压缩层，自动将冷数据分层到磁盘
- 与内核内存管理子系统紧密集成，能够优雅地分配压力

**2. Zram 的问题**
- Zram 是压缩的 RAM 块设备，有硬容量限制
- 当 Zram 满时，没有自动驱逐机制
- 会导致 LRU 反转：快速 RAM 中充满冷数据，而热数据被推到慢速磁盘
- 仅适用于内存极度受限的嵌入式系统、无磁盘设置或特定安全要求场景

**3. LRU 反转问题**
- 当 Zram 与磁盘交换配合使用时，Zram 会被冷页填满
- 活跃工作集被迫进入慢速磁盘
- 这实际上会使情况变得更糟，比没有压缩交换还要差

**4. 性能数据**
- Instagram 启用 Zswap 后，磁盘写入减少高达 25%
- Django 工作负载实现约 5:1 的压缩比
- Cloudflare 报告 Zram 下存在 20-30 分钟的brownout问题

## 社区讨论要点

### 1. 关于消费级 SSD 的重要讨论

用户 **patrakov** 提出了一个重要反驳：

- 假设"每个有 SSD 的人都有一个可以用作交换的设备"是错误的
- 许多消费级 SSD（特别是无 DRAM 的，如 Apacer AS350 1TB）在同步写入时会产生超过 10 秒的延迟峰值
- 这比任何 HDD 都差
- 如果只有无 DRAM 的消费级 SSD，最好使用 Zram

### 2. 作者的回应

文章作者 **cdown** 回应指出：

- 拥有磁盘交换实际上可以**减少**磁盘 I/O
- 没有交换时，Zram 满后匿名页无处可去
- 内核只能回收文件缓存，导致更多磁盘读写
- Zswap 可以让内核在匿名页和文件缓存之间选择更冷的页面

### 3. NVMe vs SATA

**Numerlor** 指出：

- 无 DRAM 在 NVMe 上问题较小，因为可以使用主机内存缓冲区
- 在 SATA 上，无 DRAM 在各方面都更差

### 4. Zswap 无后端支持

**cdown** 提到：

- 正在开发"虚拟交换空间"功能，允许 Zswap 完全在 RAM 中运行，无需磁盘后端
- 由 Nhat Pham 领导，预计在未来几个月推出

### 5. 桌面发行版的态度

社区观察到：

- Fedora、PopOS、Ubuntu 等发行版默认使用 Zram
- 原因包括：消除磁盘交换、简化配置、隐私考虑
- 但专业人士认为这不是内存管理的最优解

### 6. 压缩算法比较

**MaxCompression** 指出：

- LZ4（默认）：速度优先，压缩比 2-2.5x
- zstd：压缩比 3-3.5x，但 CPU 成本更高
- 内存页面数据模式简单，LZ 变体表现意外地好

## 实践建议

### 对于大多数用户

1. **使用 Zswap + 磁盘交换**
2. **不要将 Zram 与磁盘交换配合使用**
3. **如果必须使用 Zram**，配合 systemd-oomd 或 earlyoom 等用户空间 OOM 管理器

### 对于嵌入式/无磁盘系统

- Zram 是合理的选择
- 需要配合用户空间 OOM 守护进程

### 发行版现状

- 大多数 Linux 发行版在此问题上配置各异
- Windows 和 macOS 多年来已默认启用内存压缩

## 技术细节补充

### Zswap 工作原理

- Zswap 集成到内存管理子系统
- 当需要交换页面时，Zswap 先尝试在 RAM 中压缩存储
- 当池满时，自动将冷页驱逐到磁盘
- 提供动态分层：热数据在压缩 RAM 中，冷数据在磁盘

### Zram 限制

- 是块设备，内核无法区分其与普通磁盘
- 无自动 LRU 跟踪
- 页面集群等默认优化反而对 Zram 不利
- 需要手动配置写回，且效果不佳

### 未来发展

- 内核开发者正在推动 Zswap 成为唯一的压缩交换实现
- 无后端 Zswap 正在开发中，将进一步削弱 Zram 的使用场景
