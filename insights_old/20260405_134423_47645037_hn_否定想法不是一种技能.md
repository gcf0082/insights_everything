## 文章核心观点

文章作者批评了工作中常见的一种现象：员工轻易否决新想法，却认为这是一种技能。作者认为这些看似合理的反对意见实际上并没有为团队增加任何价值。

### 常见的否定意见类型

| 否定意见 | 作者的回应 |
|---------|-----------|
| "没有客户要求这个" | 证明存在市场需求 |
| "Python太慢，不适合" | 展示快速的概念验证 |
| "引入太多复杂性" | 证明这是满足需求的最小复杂性 |
| "以前尝试过但失败了" | 解释发生了什么变化 |
| "DevOps不想支持另一个服务" | 重新评估DevOps的定义 |
| "人们习惯了现有的工作方式" | 人们不拒绝好的改变，需要证明改变的好处 |

### 关键结论

如果无法说服这些"正确且聪明"的反对者，应该放弃这个想法，第二天再想一个新主意。

## 社区评论要点

### 支持作者观点

- **dsr_**: 既然这些人都正确且聪明，你的任务是给出好的答案来回应他们的反对。
- **phil21**: 团队抵制改变是一种自保技能，因为好的改变可能让他们失去存在的价值。
- **JumpCrisscross**: 最好的团队是有人愿意指出你在说废话的团队。

### 反对/质疑观点

- **bawolff**: 如果随意提出想法而不考虑基本计划，是在浪费大家时间。涉及其他团队劳动力的计划需要得到相关方同意。
- **sigseg1v**: 如果想法不好，它就不会发生。提出者需要培育想法并为之辩护。
- **ceejayoz**: 知道什么时候一个想法是坏想法也是一种技能。
- **notatoad**: 在正式提出想法前，应该先自己考虑可能的反对意见并准备好答案。

### 中立/补充观点

- **loose-cannon**: 一方面批判性评估想法有价值，另一方面对每个小的不确定性都要求答案会阻碍探索。
- **wakamoleguy**: 这些都是风险，不是所有风险都需要被化解，但可以接受。
- **analog31**: 会议类型很重要——是同级会议还是有管理层参与的会议？

## 文章启示

1. **批评 vs 建设性反馈**：反对一个想法很容易，但提供有价值的替代方案才是真正的技能。

2. **想法的演化**：想法需要时间探索和迭代，不应该在最初期就被一堆问题围攻。

3. **文化因素**：默认"这为什么不行"的团队会非常善于制造空无一物的结果。

4. **责任分工**：想法提出者需要做功课，但环境也需要给他们足够的机会来完成任务。
