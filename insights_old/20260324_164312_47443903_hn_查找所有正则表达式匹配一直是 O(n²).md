## 核心洞察

### 问题背景

每个承诺线性时间匹配的regex引擎，在要求"查找所有匹配"时都会打破这个承诺。这个问题自1970年代以来一直存在，隐藏在迭代循环中。

**现象：** 在文档中搜索一个模式需要1秒。搜索一个大100倍的文档不是花费100秒，而是接近**3小时**。

### 问题根源

以模式 `.*a|b` 和包含n个'b'的文本为例：
- 在每个位置，引擎首先尝试 `.*a`：扫描整个剩余文本寻找'a'，找不到，失败
- 然后'b'分支匹配一个字符
- 前进一个位置，重复

报告n个单字符匹配需要 `n + (n-1) + (n-2) + ... = O(n²)` 工作。

### 学术界的盲区

几乎所有学术regex论文都只关注单匹配问题，然后简单地说"只要迭代"来处理全匹配。regex理论将问题简化为是/否问题——这完全忽略了实践中真正重要的东西：匹配在哪里、匹配有多长、有多少个匹配。

### 现有解决方案的局限

1. **Aho-Corasick (1975)**：只适用于固定字符串，不适用于正则表达式
2. **Hyperscan/Vectorscan**：使用"最早匹配"语义，会改变结果
3. **RE2、Go regexp、rust regex crate**：都只保证单匹配的线性时间

### RE#的解决方案

RE#是第一个能在**两遍扫描**内找到所有匹配，且不改变语义的regex引擎：
- 反向DFA标记可能开始匹配的位置
- 正向DFA从每个标记位置解析最长匹配
- 一万次匹配和一次匹配，耗时相同

### Hardened模式

RE#还提供hardened模式，在对抗性输入上保证线性时间，但对正常模式有3-20x的常数因子开销。

## 关键评论摘要

1. **@conartist6**：提到bablr/regex是一个NFA系统，完全避免了这种复杂度爆炸，但会在非常长的输入上耗尽堆内存

2. **@10000truths**：认为最好的解决方案是像运行不受信任代码一样沙盒化regex，.NET和Python的regex包正确实现了这一点

3. **@nine_k**：认为实际使用中的regex有明确的边界，可以限制病理性的回溯行为

4. **@zahlman**：指出即使没有灾难性回溯，单次搜索也可能出现二次方行为，例如 `\d+\s+` 在只包含数字的字符串上

5. **@ieviev（作者）**：RE#使用leftmost-longest (POSIX)语义，这是数学上无歧义的解释

## 技术细节

| 特性 | 说明 |
|------|------|
| **两遍算法** | 反向DFA + 正向DFA |
| **匹配语义** | leftmost-longest (POSIX) |
| **性能** | 在产生大量匹配的模式上，从O(n²)变为O(n) |
| **Hardened模式** | 额外保证O(n×S)时间复杂度 |

### RE#不支持的特性

- 无捕获组
- 无懒惰量词 `.*?`

### 性能对比

在rebar基准测试中，RE#在多种场景下表现优于rust/regex：
- 英文literal: 34.8 GB/s vs 33.7 GB/s
- 不区分大小写英文: 17.1 GB/s vs 9.7 GB/s

## 结论

这篇文章揭示了regex引擎中一个长期被忽视的根本性问题：即使是最先进的线性时间regex引擎，在查找所有匹配时也会退化为二次方时间。RE#通过两遍扫描的创新方法首次解决了这个问题，同时保持语义不变。
