# 洞察报告：Rust 无法 catch 的 Bug

**洞察链接**：https://corrode.dev/blog/bugs-rust-wont-catch/

**来源**：Corrode - Rust Consulting

**作者**：Matthias Endler

**发布日期**：2026-04-29

---

## 摘要

2026年4月，Canonical 披露了 uutils（GNU coreutils 的 Rust 重实现）在 Ubuntu 26.04 LTS 外部安全审计中发现的44个 CVE。这些 bug 全部出现在生产级 Rust 代码库中，作者都是经验丰富的开发者，但没有一个被 Rust 的安全机制（borrow checker、clippy、cargo audit）捕获。

本文深入分析了这些漏洞，揭示了 Rust 安全边界之外的常见错误模式。

---

## 核心洞察

虽然 Rust 避免了以下传统 C 语言安全问题：
- 无缓冲区溢出
- 无 use-after-free
- 无双重释放
- 无共享可变状态的数据竞争
- 无空指针解引用
- 无未初始化内存读取

**但仍有大量 bug 存在于 Rust 程序与外部系统（文件系统、操作系统调用、字节序列）的边界处。**

---

## 主要漏洞分类

### 1. TOCTOU（Time Of Check To Time Of Use）路径漏洞

**问题**：在同一路径上执行两次系统调用，中间可能被攻击者替换为符号链接。

**案例**（CVE-2026-35355）：
```rust
fs::remove_file(to)?;
let mut dest = File::create(to)?;  // 路径在此处被重新解析
```

攻击者可在两次调用之间将 `to` 替换为指向 `/etc/shadow` 的符号链接，导致特权进程覆盖系统文件。

**修复**：使用 `OpenOptions::create_new(true)` 确保文件不存在（包括悬空符号链接）。

**规则**：基于文件描述符而非路径进行操作。

---

### 2. 创建后设置权限

**问题**：先创建文件/目录，再设置权限，期间存在权限窗口。

```rust
fs::create_dir(&path)?;
fs::set_permissions(&path, Permissions::from_mode(0o700))?;  // 窗口期
```

**规则**：使用 `OpenOptions::mode()` 或 `DirBuilderExt::mode()` 在创建时直接设置正确权限。

---

### 3. 路径字符串比较 vs 文件系统身份

**问题**：
```rust
if file == Path::new("/") { ... }  // 绕过：/../, /./, symlink
```

**修复**：使用 `fs::canonicalize()` 解析路径后再比较。

**规则**：比较路径前先解析为规范形式，比较两个任意路径时应打开文件比较 `(dev, inode)` 对。

---

### 4. Unix 边界处的字节处理

**问题**：Rust 的 `String` 强制 UTF-8，但 Unix 路径、参数、管道数据是原始字节。

**案例**（CVE-2026-35346）：
```rust
print!("{}", String::from_utf8_lossy(ra));  // 静默损坏非UTF-8数据
```

GNU `comm` 处理二进制文件，但 uutils 版本用 U+FFFD 替换无效字节。

**修复**：使用 `write_all()` 直接写入字节，而非 `print!`。

**规则**：
- 路径：`Path`/`PathBuf`
- 环境变量：`OsString`
- 流内容：`Vec<u8>` 或 `&[u8]`

---

### 5. Panic 即 DoS（拒绝服务）

**问题**：每个 `unwrap`、`expect`、索引操作在处理恶意输入时都会导致进程崩溃。

**案例**（CVE-2026-35348）：
```rust
let path = std::str::from_utf8(bytes).expect("...");
```

文件名含非 UTF-8 字符时，整个 `sort --files0-from` 进程中止。

**规则**：
- 将错误转换为错误值，而非 panic
- 在 CI 中启用 clippy 警告：
  ```toml
  [lints.clippy]
  unwrap_used = "warn"
  expect_used = "warn"
  panic = "warn"
  indexing_slicing = "warn"
  arithmetic_side_effects = "warn"
  ```

---

### 6. 丢弃错误信息

**问题**：使用 `.ok()`、`.unwrap_or_default()` 丢弃有意义的错误。

**案例**：
- `chmod -R` 返回最后文件的退出码，而非最严重错误
- `dd` 对磁盘满静默处理

**规则**：保留最严重错误，不要随意丢弃 Result。

---

### 7. 与原工具行为不一致

**问题**：实现与 GNU 行为差异导致脚本依赖破裂。

**案例**（CVE-2026-35369）：
- GNU `kill -1` 读取为"信号1"
- uutils 读取为"向 PID -1 发送默认信号"，在 Linux 上杀掉所有可见进程

**规则**：Bug-for-bug 兼容性是安全特性。运行 GNU 测试套件验证行为一致。

---

### 8. 跨越信任边界时未解析输入

**问题**（CVE-2026-35368，最严重）：
```rust
chroot(new_root)?;
// 仍在 uid 0，但已在攻击者控制的文件系统中
let user = get_user_by_name(name)?;  // 加载新根目录中的动态库！
```

**修复**：在 `chroot` 之前解析用户名。

**规则**：跨越信任边界前完成所有解析，跨边界后任何库调用都可能执行攻击者代码。

---

## 总结与建议

**Rust 确实防止了大多数传统内存安全问题**，这是巨大成就。但剩下的漏洞类别更加有趣——它们位于受控 Rust 环境与混乱外部世界的边界处。

**检查清单**（grep 你的代码库）：
- `from_utf8_lossy`（数据损坏）
- 散落的 `unwrap()` 调用
- 被丢弃的 Result
- `File::create`（而非 `OpenOptions`）
- 字符串与 `"/"` 的比较

**idiomatic Rust** 不仅是被 borrow checker 通过或 clippy 无警告的代码，而是其类型、名称和控制流能准确描述系统真实行为的代码。这意味着使用文件描述符而非路径、`OsStr` 而非 `String`、? 而非 unwrap、bug-for-bug 兼容性优先于干净语义。
