## 内容摘要

本文讨论了作者在使用 Lean 形式化验证工具证明一个压缩程序 (lean-zip) 正确后，发现了两个 bug，但这些 bug 实际上都在验证边界之外。

**发现的两个 bug:**
1. **OOM 拒绝服务:** 归档解析器的部分从未被验证
2. **堆缓冲区溢出:** 位于 Lean 运行时本身，而非被证明的代码中

---

## 主要讨论观点

### 1. 标题具有误导性

- 读者普遍认为标题暗示在已证明的代码中发现了 bug，但实际情况是 bug 位于证明边界之外
- 发现的 bug 不在 Lean 内核（负责验证的部分），而在 Lean 运行时
- 这类似于"JRE 有 bug"意味着使用 JRE 的 Java 应用有 bug

### 2. 验证边界的重要性

- 形式化验证只覆盖代码的特定部分
- 未验证的部分（如解析器、运行时）可能存在漏洞
- 理解"什么被验证"和"什么未被验证"至关重要
- sel4 微内核的做法是将整个二进制都纳入证明

### 3. 规格缺陷 vs 实现缺陷

- 即使代码符合规格编写，如果规格本身有问题，仍会出错
- 形式化验证无法保证解决了正确的问题
- 规格捕获人类意图的准确性是验证中最困难的部分之一
- AI 可以帮助"氛围编码"证明，这是一个 killer app

### 4. 形式化验证的价值

- 形式验证大幅减少了可能出现错误的表面积
- AWS 案例：形式验证使 IAM ACL 评估代码的性能提高了一倍
- 业界认识到形式验证比传统测试更有效
- 但验证不是万能的，需要理解其局限性
- 将形式验证视为"比什么都不做更好"的理解更为实际

### 5. 关于"无法证明不存在 bug"的争论

- 形式方法 practitioners 指出，形式验证可以明确地证明特定 bug 或类别的 bug 不存在
- 所有证明最终都基于公理，这是正常的
- 如果能够到达"软件中唯一的 bug 在规格、证明助手中"的状态，那将是一个巨大的进步

---

## 关键结论

1. **验证之外仍有风险:** 形式验证是有价值的，但它不能覆盖整个系统。未验证的运行时、解析器和库组件仍然可能存在 bug

2. **标题需谨慎:** 文章的标题具有误导性，实际发现的是 Lean 运行时中的 bug，而非被证明代码中的 bug。这提醒我们在宣传形式验证结果时需要更加精确

3. **规格的重要性:** 形式验证的有效性取决于规格本身的质量。如果规格有问题，即使代码通过了验证，也可能无法满足用户的实际需求

4. **逐步推进的方法:** 验证工作应逐步扩展到更多组件，同时明确标注已验证和未验证的部分
