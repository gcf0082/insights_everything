## 原文摘要

作者 Andrew Stuart 是一名曾为 AWS 早期布道者的开发者，他回顾了自己从"铁杆粉丝"到"彻底离开"的转变历程，并在最近因测试需求重新登录 AWS 时，再次遭遇了令人沮丧的体验。

**作者列出的离开 AWS 的主要原因：**

1. **社区客户端库问题**：AWS 最初6年不自己构建客户端库，让社区免费为其编写 Python 等语言的库。
2. **Python 2 迁移缓慢**：AWS 在 Python 2 到 3 的迁移上拖延了极长时间。
3. **DynamoDB 体验极差**：一天内就产生了 $75 账单，系统在各方面都是"垃圾"。
4. **出口流量费用高昂**：从 20 美分/GB 降到 9 美分/GB，依然极其昂贵。
5. **复杂且欺骗性的计费**：在内部系统间数据传输时重复计费，陷阱重重。
6. **IAM 复杂度惊人**：作者形容这是"路西法坐在地狱第九层发明的最糟糕的访问控制折磨"。
7. **整体复杂度失控**：表面说 AWS 能简化运维，实际每样东西都极其复杂，需专家团队维护。
8. **Lambda 的问题**：启动慢、开发复杂度高、供应商锁定严重。
9. **对开源项目的侵蚀**：通过 OpenSearch、Valkey、DocumentDB 等克隆开源项目，迫使 Elastic、Redis、MongoDB 改用 SSPL 等许可证来防御。

**最近的最后一根稻草：** 作者因测试高性能 EC2 实例（192核、1TB内存），被 AWS 标记为"疑似安全漏洞"，导致整个账户被暂停——包括他的 Business Email（WorkMail）也无法使用。4天后仍未得到支持响应，业务受到严重影响。

## 社区讨论洞察

### 1. 关于 AWS 定价不透明的用户体验

社区普遍认同原帖中关于 AWS 不在 UI 中显示价格的问题。有评论指出：
- Azure 在这方面做得更好，会在可能产生高费用的地方显示价格。
- AWS 的定价计算器是独立应用，增加了使用摩擦。
- 有评论用"虐待性关系"来形容这种设计。

### 2. 关于开源与商业化的辩论

关于 AWS 是否"掠夺"了开源项目，社区存在分歧：
- 一派认为 AWS 利用开源许可证的宽松性，这是合法的商业行为，项目方应改进许可证。
- 另一派认为 AWS 凭借生态垄断地位，用别人的开源代码抢走客户关系，是不公平竞争。

### 3. 简化 vs 复杂性的对比

- 有评论指出 Cloudflare Workers 等现代平台提供了更好的全栈应用部署体验，成本合理。
- 有人怀念 Heroku 的简洁设计，批评 AWS 将简单的事情复杂化。
- Render 被提及作为 Heroku 的优秀替代品。

### 4. 供应商锁定的担忧

- AWS Lambda 被广泛批评为供应商锁定的典型案例。
- 从 AWS 迁移出来的成本极高，尤其是 Lambda 函数。

## 核心洞察结论

1. **AWS 的核心问题不是功能，而是用户体验和信任**：从价格透明度到账户支持响应速度，AWS 的设计似乎更多服务于自身而非用户。

2. **复杂性是 AWS 的双刃剑**：AWS 用"简化运维"作为卖点，但实际上引入了更多的复杂性，需要专家团队维护。

3. **开源社区正在反击**：Elastic、Redis、MongoDB 等项目通过修改许可证来防御 AWS 的商业化行为，这是技术行业格局变化的重要信号。

4. **替代方案正在成熟**：Cloudflare、Render、DigitalOcean 等平台提供了更简洁、更友好的选择，开发者正在用脚投票。

5. **供应商锁定风险需要认真对待**：Lambda 等服务虽然看似方便，但迁移成本极高，选择时需慎重考虑长期影响。
