## 文章摘要

本文作者对比了NVIDIA官方的NemoClaw教程与自己构建的Wirken.AI网关在构建安全本地AI代理方面的架构差异。作者认为，当前的AI代理网关设计正在重蹈MS-DOS时代的安全覆辙，需要借鉴Unix的历史经验来构建更安全的代理架构。

## 核心观点

### 1. MS-DOS的历史教训

作者回顾了MS-DOS时代的安全问题：
- 任何程序都可以查看和修改内核、钩子中断、写入磁盘任何位置
- 没有安全防护机制
- 所有数据使用单一密码共享
- 2006年沃尔玛数据泄露事件直到2009年才通知用户

作者认为，当前的AI代理网关设计正在重蹈MS-DOS的覆辙，整个代理被沙箱包裹，缺乏细粒度的权限控制。

### 2. NVIDIA NemoClaw教程的方法

| 步骤 | NemoClaw方案 |
|------|-------------|
| 运行时 | 使用Docker注册NVIDIA容器运行时，设置cgroup命名空间模式为主机 |
| Ollama | 绑定到0.0.0.0以便沙箱代理跨网络命名空间访问推理服务 |
| 安装 | curl管道bash从NVIDIA URL安装 |
| 模型 | 使用ollama pull加载模型，预热到GPU内存 |
| 入职 | 向导生成包含策略和推理的沙箱镜像 |
| Telegram | 通过聊天渠道发送配对代码，用户从沙箱内部批准 |
| Web UI | 本地URL带能力令牌 |
| 远程访问 | 通过OpenShell启动主机端端口转发，然后建立SSH隧道 |
| 策略 | 在netns边界强制执行，阻止的出站连接在TUI中显示 |

### 3. Wirken.AI的替代方案

| 步骤 | Wirken.AI方案 |
|------|-------------|
| 运行时 | 无等效步骤，网关作为主机进程运行，Docker仅用于per-tool-call沙箱 |
| Ollama | 保持在127.0.0.1，代理是主机进程，loopback足够 |
| 安装 | curl管道sh，安装程序使用ssh-keygen验证发布签名 |
| 模型 | 相同模式，委托推理给Ollama |
| 入职 | 向导写入provider配置和通道注册，权限模型在二进制文件中 |
| Telegram | Bot令牌到加密保管库，为适配器生成新Ed25519密钥对，无内聊配对 |
| Web UI | 本地URL，loopback绑定，无需令牌 |
| 远程访问 | 仅SSH隧道，WebChat监听器已在主机loopback上 |
| 策略 | 在工具调度层强制执行，16个高风险命令前缀始终提示，其他首次使用记住30天，批准的命令在hardened Docker容器中运行（cap_drop ALL, no-new-privileges, 只读rootfs, 64MB tmpfs） |

### 4. 关键架构差异

**NemoClaw的妥协：**
- 绑定到0.0.0.0因为沙箱无法访问loopback
- 通过聊天渠道配对因为没有独立的身份平面
- 将整个代理包装在容器中因为代理本身还不受信任
- 在netns边界批准因为工具层没有权限概念

**Wirken的设计原则：**
- 每个通道是具有自己Ed25519身份的独立进程
- 保管库在进程外运行
- 推理保持在loopback因为代理在主机上
- Shell执行在工具层配置的hardened容器中运行，而不是包装整个代理
- 16个高风险命令前缀每次调用都提示，记住30天

### 5. 审计日志示例

作者展示了审计日志中的两种场景：

**Tier 3 拒绝curl：**
```
[4] assistant_tool_calls
     call: exec({"command":"curl https://httpbin.org/get"})
[5] permission_denied
     action_key='shell:curl'  tier=tier3
[6] tool_result
     tool=exec success=False
     output: Permission denied: 'exec' requires tier3 approval.
```

工具调用从未到达沙箱，拒绝被记录为审计链中的类型化事件。

**Hardened沙箱on sh：**
```
[14] assistant_tool_calls
     call: exec({"command":"sh -c \"touch /cannot_write_here 2>&1; ...\""})
[15] tool_result
     tool=exec success=True
     output:
       touch: cannot touch '/cannot_write_here': Read-only file system
       ws_ok=1
       tmp_ok=1
```

"Read-only file system"是内核拒绝在只读挂载上打开新文件，不是DAC检查。`ws_ok=1`确认工作区bind-mount保持可写，`tmp_ok=1`确认/tmp上的tmpfs也可写。

## 结论

作者认为AI代理空间仍然处于早期阶段，需要借鉴计算机历史已经知道的安全经验来构建日常使用的安全代理。NVIDIA教程中的变通方案是在试图充分利用一个不能像工程师通常喜欢的那样分离关注点的基础设施。而Wirken通过在工具层实现细粒度的安全控制，提供了更优的架构选择。

作者强调，像Unix在1973年获得进程分离、用户分离、文件权限和小程序之间的管道一样，我们应该将同样的安全原则应用到AI代理中，确保日常使用的安全性。
