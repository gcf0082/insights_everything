## 摘要

这是一个关于 Mercurial 版本控制系统生存现状的讨论。Mercurial 创建于2005年，曾与 Git 并驾齐驱，但在2010年代的 popularity contest 中败北。然而，这个"被遗忘"的项目至今仍在活跃开发，并在2026年FOSDEM大会上进行了演讲，探讨其生存之道和对版本控制未来的展望。

## 核心讨论话题

### 1. 当年选择 Mercurial 是"错误"的吗？

多位用户分享了近20年前在工作中选择 Mercurial 而不是 Git 的经历。普遍观点是：从技术角度看，Mercurial 确实是更好的选择——更用户友好、性能足够、跨平台支持更好。但显然 Git 赢得了战争。

> "I still think I made the right choice from a technical perspective; I thought Mercurial was way more user-friendly while providing all the features and performance needed."

### 2. Mercurial 的优势

- **更低的上手门槛**: 不需要理解内部数据结构就能使用
- **更安全的默认行为**: 不容易酿成灾难性错误
- **更好的命令行设计**: 命令命名清晰、易于记忆
- **优秀的 GUI 工具**: TortoiseHg (THG) 在2009年远超 Git 的图形界面
- **稳定的提交标识符**: 即使在 rebase 或 amend 后也能保持不变

### 3. 为什么 Mercurial 输掉了？

- **GitHub 的崛起**: GitHub 大幅改善了 Git 生态，而不支持 Mercurial
- **网络效应**: 几乎所有人都在用 Git，新人自然会选 Git
- **性能问题**: Mercurial 最初用 Python 编写，在性能上落后于 C 编写的 Git
- **BitBucket 的转变**: 虽然 BitBucket 最初支持 Mercurial，但最终因需求而添加了 Git 支持

### 4. Mercurial 的现状与未来

- 项目已持续用 Rust 重写核心操作，性能大幅提升（100ms 内完成百万文件仓库的 `hg status`）
- Python 2 到 Python 3 的迁移实际上非常平滑
- Heptapod（GitLab 的 Mercurial 分支）提供了自托管选择
- Jujutsu (jj) 被认为是"继承 Mercurial 最佳特性"的新一代版本控制工具，支持 Git 作为后端

### 5. 关于 Git vs Mercurial 的争议

用户对两者 UX 的评价截然不同：

- **支持 Mercurial**: Mercurial 命令更直观，`hg rebase -s -d` 比 `git rebase --onto` 更容易理解；不需要理解底层数据结构
- **支持 Git**: Git 的数据模型更简单直接，用户能完全控制自己的数据；分支操作更灵活

> "Git has the right data model and has always been about empowering users to edit their data."

## 关键启示

1. **技术优秀不等于市场胜利**: Mercurial 在很多方面确实比 Git 更好，但社区效应和生态系统最终决定了胜负

2. **用户友好性很重要但不够**: Mercurial 的学习曲线更平缓，但这不足以抵消网络效应

3. **开源项目的生存之道**: 即使失去主导地位，只要持续满足一部分用户的需求，项目仍能继续生存

4. **新的竞争者正在出现**: Jujutsu (jj) 试图结合 Mercurial 的用户体验和 Git 的生态优势
