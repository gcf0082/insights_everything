## 项目概述

Craft 是一个轻量级的 C/C++ 构建和工作流工具，旨在简化项目设置流程。用户无需编写复杂的 CMakeLists.txt，只需使用简单的 `craft.toml` 配置文件：

```toml
[project]
name = "my_app"
version = "0.1.0"
language = "c"
c_standard = 99

[build]
type = "executable"
```

**主要功能**:
- `craft build` - 自动生成 CMakeLists.txt 并构建项目
- `craft add` - 添加依赖 (支持 Git、本地路径、包管理器)
- `craft init` - 采用现有 C/C++ 项目或初始化空目录
- `craft template` - 保存项目结构为模板
- `craft gen` - 生成头文件和源文件 (含模板代码)
- `craft upgrade` - 自动更新自身
- 支持通过 `CMakeLists.extra.cmake` 扩展功能
- 跨平台支持: macOS、Linux、Windows

## 主要反馈与讨论主题

### 1. 构建系统专业建议 (高票评论)

一位管理大型 C/C++ 软件栈 (1500+ 组件) 的开发者提出了关键建议:

| 建议 | 说明 |
|------|------|
| **离线构建支持** | 必须支持无互联网环境构建，手动指定依赖路径，这对安全性和可重现性至关重要 |
| **优化标志默认值** | 绝对不要默认使用 `-O3 -march=native`，应使用 CMake 的 "RelWithDebInfo" 等效配置 (`-O2 -g -DNDEBUG`) |
| **自构建能力** | 必须能被其他构建工具 (如 CMake) 构建，以确保可重现性和跨编译 |
| **pkg-config 兼容** | 支持 pkg-config 和 CPS 规范，实现与其他构建系统的互操作性 |
| **交叉编译支持** | 必须支持交叉编译，否则无法进入嵌入式领域 |

> 交叉编译应该是默认设计假设，而非特殊情况。

### 2. -O3 与 -march=native 争论

关于编译器优化标志引发激烈辩论:

- **反对方**: `-march=native` 会锁定程序到特定 CPU，无法在不同硬件上运行
- **支持方**: 在构建和部署环境完全一致的情况下使用是合理的
- **折中建议**: 使用微架构级别 (如 x86-64-v2) 或完全通用构建

> `-O3` 有时甚至比 `-O2` 慢，建议仅在有基准测试证明提速时才使用。

### 3. 与现有工具对比

| 工具 | 特点 |
|------|------|
| **xmake** | 支持生成 CMakeLists.txt 和 compile_commands.json，支持 Conan/vcpkg，对 C++20 模块支持较好 |
| **Meson** | 类似 xmake，使用 Python 层 + Ninja 构建器，获得较多开源项目采用 |
| **Conan2** | 功能最全面的 C/C++ 包管理器，支持主机/目标配置、ABI 兼容、二进制缓存、离线模式、交叉编译 |
| **CMake** | 行业标准但学习曲线陡峭，DSL 设计不佳 |

### 4. C/C++ 构建系统的根本挑战

- **标准化缺失**: C++ 委员会拒绝对构建工具、依赖管理进行标准化
- **遗留问题传播**: 现有构建系统的缺陷会传递给下游用户
- **生态碎片化**: 众多不兼容的包管理器导致互操作性差
- **边缘案例复杂性**: 不同平台 (Linux、Windows、macOS、BSD、嵌入式) 的特殊需求难以全面覆盖

> 一个新的 C/C++ 构建/包管理系统在 AI 辅助下实际上是可解决的问题。

### 5. AI 时代构建工具的必要性

有评论认为 AI 工具 (如 Claude) 可以快速生成任何构建系统，因此新建构工具意义不大:

- AI 可以用自然语言生成 Makefile、CMake、Ninja 等
- 学习 CMake 等工具需要数天，而 AI 几分钟就能生成可工作的配置

**对此的反驳**:
- AI 仍需处理库路径查找、版本管理、条件编译标志等复杂问题
- 使用简单命令比每次写提示词更高效
- 开源免费工具始终有市场价值

### 6. 安装方式安全性

安装脚本使用 `curl | sh` 写入 bashrc 的方式引发安全担忧:

- 批评: 这是供应链风险，比直接下载二进制更危险
- 辩护: 脚本可审查，比执行未知二进制更透明
- 建议: 希望有 TUI 安装程序，支持自托管安装规范

## 项目优缺点分析

### 优点
1. 极简的 TOML 配置格式，降低入门门槛
2. 自动生成 CMake，减少样板代码
3. 集成依赖管理 (git、路径、包名称)
4. 跨平台支持
5. 模板和代码生成功能
6. 开源且免费

### 缺点/需改进
1. 默认优化标志不适合生产环境
2. 缺少离线构建和手动依赖路径支持
3. 无交叉编译支持
4. 无 pkg-config 兼容性
5. 生态刚起步，边缘案例覆盖不足
6. 无系统级依赖管理 (如 apt 包)
7. 包管理器功能 (如版本锁定、二进制缓存) 缺失

## 结论

Craft 代表了 C/C++ 社区对简化构建流程的持续探索。作为类似 Cargo 的尝试，它在易用性方面取得了进展，但在专业开发环境所需的成熟度方面还有很长的路要走。

社区反馈表明，构建系统的核心挑战不仅在于语法简化，更在于:
- 支持复杂的企业级需求 (离线构建、交叉编译、ABI 兼容)
- 与现有生态系统 (CMake、Conan、vcpkg) 的互操作性
- 边缘案例处理能力

对于个人项目和小规模团队，Craft 提供了友好的入门体验。但对于需要生产级可靠性的项目，Conan2 等成熟方案仍是更安全的选择。
