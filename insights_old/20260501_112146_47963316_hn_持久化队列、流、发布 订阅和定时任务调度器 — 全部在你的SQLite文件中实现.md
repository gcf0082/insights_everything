## 内容摘要

本文介绍了一个名为 **Honker** 的项目，它允许在 SQLite 文件内部实现持久化队列、流、发布/订阅以及 cron 调度器。项目声称可以通过 SQLite 实现分布式系统的常见 primitives，无需额外依赖 Redis 或其他消息队列。

核心实现机制是：**每毫秒轮询 SQLite 的 PRAGMA data_version**，这是一个单调递增的计数器，SQLite 在每次提交时都会递增，读取大约需要 3 微秒。

## 主要讨论观点

### 批评观点

1. **轮询开销问题**：每毫秒进行一次 SELECT 查询被认为是一种低效的设计。有评论指出，每秒消耗 3 毫秒 CPU 时间意味着 0.3% 的 CPU 浪费。

2. **不如内核文件监视器**：评论认为 inotify 或 kqueue 等内核文件监视机制比轮询更优。

3. **不如 Redis 有吸引力**：多位评论者认为直接使用 Redis 会更简单高效。

4. **跨进程问题**：SQLite WAL hook 只能捕获本连接的更改，无法可靠地捕获其他进程的更改。

### 正面观点

1. **简单易用**：对于不需要大规模分布式的场景，这是一个简单的解决方案。

2. **已有先例**：k3s 的 Kine 项目已经使用 SQLite 轮询来实现 etcd watch 功能。

3. **足够快**：有评论指出在 /dev/shm 上可以达到约 15 万次写入/秒。

4. **适合中间场景**：对于"太认真对待内存队列，不够大需要 Kafka"的应用场景很合适。

## 技术细节

- **轮询机制**：PRAGMA data_version 是 SQLite 的元数据，而非数据页
- **现有替代方案**：pg-boss 和 Oban 是 Postgres 端的标准方案，现在 Oban 也支持 SQLite
- **作者改进方向**：计划使用 inotify、kqueue 或 mmap'd shm 文件检查来消除轮询

## 结论

Honker 是一个有趣的概念验证项目，它将分布式系统的常见 primitives 引入 SQLite。然而，每毫秒轮询的设计引发了关于效率的广泛讨论。尽管作者表示正在改进，但目前来看，它更适合不需要强实时性的小规模应用场景。
