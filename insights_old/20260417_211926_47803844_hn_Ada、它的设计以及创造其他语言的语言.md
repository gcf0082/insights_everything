## 摘要

本报告总结了 Hacker News 上关于 Ada 编程语言设计及其对现代编程语言影响的讨论。讨论涵盖了 Ada 为何未能成为主流语言、其设计特点、语言冗长性争议以及与 Rust、C++ 等现代语言的比较等话题。

---

## 主要讨论观点

### 1. Ada 未成为主流的原因

**昂贵的编译器成本**
- 典型编译器要花费数万美元
- 在流行语言可以免费获取的数十年间，没有任何开源或免费编译器存在
- GNU ADA 编译器 (GNAT) 直到 1995 年才发布

**硬件限制**
- Ada 语言和当时的编译器技术过于重量级，无法在 1980 年代的微型计算机上良好运行
- 即使 Intel 打造了融入 Ada 概念的 i432 "芯片上的大型机"，性能仍然很差
- 微型计算机后来改变了世界，带来了 C 和汇编语言的遗留问题

**生态系统的局限性**
- 字符串处理不太好
- 并发机制不使用操作系统线程

### 2. 语言特性冗长性争议

**支持冗长的观点**:
- 编程是人类活动，应该使用人类语言
- 避免使用需要解码的编码形式
- Ada 专门为读者设计，而非为写作者设计

**反对冗长的观点**:
- 源代码应该尽可能在屏幕上显示更多内容
- 可以用符号替换冗长的关键字

**建议**: 理想的编程语言应该支持两种等效表示形式

### 3. Ada 的复杂度

- Ada 曾经被批评过于复杂和冗长
- 如今的批评显得天真，因为许多后续语言变得更加复杂
- Ada 复杂度与 C++、C# 和 Rust 相当
- Ada 拥有设计原理文档，这是其他语言应该学习的

### 4. Ada 对现代语言的影响

**Sum Types (联合类型)**:
- 过去二十年中添加 sum types 的语言都采用了 Ada 原始标准中的特性
- Sum types 的起源可以追溯到 John McCarthy 1964 年的提案

**参数传递机制**:
- Ada 的 "in"、"out" 和 "in-out" 参数规范优于 C/C++
- C++ 无法区分 "out" 和 "in-out" 参数是许多缺陷的根源

### 5. 关于文章质量的讨论

**AI 生成质疑**:
- 文章被怀疑由 AI 生成
- Twitter 账户 (Iqiipi_Essays) 发布速度很快
- 写作质量被批评为"糟糕"，重复使用相同的修辞手法

### 6. 与其他语言的比较

**与 Rust 的比较**:
- Rust 有 affine types，Ada 没有完全对应的特性
- Ada 的 limited types 可以部分类比
- SPARK (Ada 的可验证子集) 有类似的移动语义

**与 Modula 的关系**:
- Ada 没有从 Modula 借用任何东西
- 两者共有的特性都来自 Xerox Mesa

## 历史背景

### Ada 开发时间线

- **1975年4月**: STRAWMAN 需求
- **1975年8月**: WOODENMAN 需求
- **1976年1月**: TINMAN 需求
- **1977年1月**: IRONMAN 需求
- **1978年6月**: STEELMAN 需求
- **1979年6月**: 初步 Ada 参考手册

### JOVIAL 语言

- 美国空军在 Ada 之前使用 JOVIAL
- JOVIAL 源自 IAL (1958年12月)，即 ALGOL 60 的前身
- 空军原本计划用 Ada 替代 JOVIAL，但因 Ada 开发耗时过长而继续使用 JOVIAL

## 社区评价

**正面评价**:
- Ada 被低估
- 类型系统严格但不经常"大喊大叫"
- 语言鼓励明确表达程序意图
- 适合编写更安全、更清晰的代码

**负面评价**:
- 编译器昂贵
- 语法冗长
- 某些特性在现代硬件上才可行

---

## 结论

Ada 作为一门在 1980 年代设计的高级编程语言，具有许多当时革命性的特性，包括类型安全、并发支持和模块化。虽然由于商业和技术因素未能广泛采用，但其设计理念对后世语言产生了深远影响。许多现代语言的特性都可以追溯到 Ada 的设计，如参数模式、类型系统和包机制。
