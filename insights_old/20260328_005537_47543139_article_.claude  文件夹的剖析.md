## 核心洞察总结

### 1. .claude/ 文件夹概述

`.claude` 文件夹是 Claude Code 的控制中心，包含指令、自定义命令、权限规则和跨会话记忆。项目级 `.claude/` 需提交到 git，全局级 `~/.claude/` 存储个人偏好。

### 2. CLAUDE.md：核心指令文件

Claude 启动时首先读取 `CLAUDE.md`，将其加载到系统提示中全程参考。

**应包含内容**：
- 构建、测试和 lint 命令
- 关键架构决策
- 非显而易见的注意事项
- 导入规范、命名模式、错误处理风格
- 主要模块的文件和文件夹结构

**建议**：
- 保持在 200 行以内
- 过长会导致上下文膨胀，指令遵循度下降

### 3. CLAUDE.local.md：个人覆盖文件

用于个人偏好设置（如不同的测试运行器），自动被 gitignore，不会提交到仓库。

### 4. rules/ 文件夹：模块化指令

解决团队增大后 `CLAUDE.md` 过于庞大的问题。每个 markdown 文件自动加载，可通过 YAML frontmatter 实现路径作用域：

```yaml
---
paths:
  - "src/api/**/*.ts"
---
# 规则内容
```

### 5. commands/ 文件夹：自定义斜杠命令

每个 markdown 文件对应一个斜杠命令，文件名即为命令名（如 `review.md` → `/project:review`）。使用反引号语法 `!` 执行 shell 命令并嵌入输出。

支持传递参数：`$ARGUMENTS`

### 6. skills/ 文件夹：可重用的工作流

与命令的区别：
- **命令**：等待用户手动触发
- **技能**：当任务匹配技能描述时，Claude 可自动调用

技能放在独立子目录中，可包含支持文件（如 `DETAILED_GUIDE.md`）。

### 7. agents/ 文件夹：专业子代理

定义专门化子代理角色，每个代理有独立的系统提示、工具访问权限和模型偏好：

```yaml
---
name: code-reviewer
description: 代码审查专家
model: sonnet
tools: Read, Grep, Glob
---
```

优势：隔离上下文窗口，避免主会话被大量中间过程 tokens 污染。

### 8. settings.json：权限控制

定义 Claude 允许和拒绝的操作：

```json
{
  "permissions": {
    "allow": ["Bash(npm run *)", "Bash(git status)", "Read", "Write", "Edit"],
    "deny": ["Bash(rm -rf *)", "Bash(curl *)", "Read(./.env)"]
  }
}
```

也支持 `settings.local.json` 用于个人覆盖。

### 9. 全局 ~/.claude/ 文件夹

- `CLAUDE.md`：全局指令，跨所有项目
- `projects/`：存储会话记录和自动记忆
- `commands/`、`skills/`、`agents/`：个人专属，跨项目可用

---

## 实践建议

1. **从 /init 开始**：让 Claude 生成初始 `CLAUDE.md`
2. **设置 permissions**：至少允许运行命令，禁止读取 .env
3. **创建常用命令**：代码审查、问题修复是好的起点
4. **逐步拆分**：当 CLAUDE.md 变拥挤时，拆分成 rules/
5. **添加全局偏好**：在 `~/.claude/CLAUDE.md` 中设置个人编码原则

---

## 关键结论

`.claude` 文件夹本质上是一个协议，用于向 Claude 明确你是谁、你的项目做什么、应遵循什么规则。定义得越清晰，纠正 Claude 的时间就越少，它做有用工作的时间就越多。

**最高杠杆文件是 CLAUDE.md，先把它做好，其他都是优化。**
