## 概述

本洞察讨论了 GitHub 的一项用户体验（UX）更改：将跨仓库的 issue 链接改为在弹出窗口中打开，而不是导航到新页面。这一更改引发了社区的广泛讨论和不满。

## 核心事件

### GitHub 官方的回应

GitHub 员工 Matt138 在讨论中解释了这项更改的动机：

> "这是出于性能考虑的更改。我们添加这个功能是因为加载跨仓库的 issue 比加载同仓库的 issue 慢得多，这是由于 header 的加载方式造成的（我们正在解决这个问题）。"

他进一步补充道：

> "页面加载成本约为 500-800ms p50，而同仓库内导航不到 100ms（或者不使用 header 时的加载时间）"

**更新**：GitHub 已经回滚了这项更改，恢复了之前的行为。

## 社区讨论要点

### 1. GitHub 性能下降的原因

- **AI 驱动**：GitHub 使用量因代码生成的便利性而激增
- **AI 爬取和检查**：不仅仅是代码生成，AI 代码爬取和检查也增加了负载

### 2. 大型科技公司 UX 问题

社区讨论了为什么像 GitHub、LinkedIn 这样的价值数十亿美元的公司 网站 UX 却很差：

- **团队割裂**：各团队各自为政，每个团队负责自己的小功能，缺乏端到端的整体视图
- **内部激励**：产品越来越多地根据内部激励而非用户实际需求来设计
- **业务导向**：UX 决策往往被业务指标驱动，而非用户实际体验

### 3. UX 作为一个职业的问题

- **设计与实用的冲突**：许多 UX 设计师更关注"展示设计技能"而非"实用性"
- **指标化问题**：现代 UX 往往被指标驱动，点击率不等同于用户想要的东西
- **流行趋势**：UX 工程师常常推出最新的设计潮流（fads），而非用户真正需要的

### 4. 开发者 vs 设计师

社区普遍认为：

- **GitHub 应该由开发者主导**：GitHub 是为软件开发者存在的网站，应该由开发者而不是设计师来主导
- **工具复杂性**：随着时间推移，软件变得越来越复杂，从简洁的工具变成臃肿的企业级产品

### 5. 其他观点

- **链接应该是链接**：用户强调"链接应该就是链接"，不应该被改变行为
- **弹出窗口的缺点**：打破用户的浏览器使用直觉，让页面加载更慢
- **浏览器分屏功能**：讨论了 Firefox 和 Chrome 最新的分屏功能

## 结论

这个讨论反映了开发者社区对 GitHub UX 更改的强烈反应。虽然 GitHub 以性能为由来解释这一更改，但用户普遍认为这种弹出窗口式的交互方式既不直观又降低了效率。值得肯定的是，GitHub 迅速听到了社区的反馈并回滚了这项更改。

这一事件也引发了对整个科技行业 UX 问题的广泛讨论：大型科技公司往往因为内部激励结构、团队割裂等原因，难以提供真正以用户为中心的产品体验。
