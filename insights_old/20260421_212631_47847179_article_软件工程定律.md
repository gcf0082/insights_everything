## 概述

该网站收录了 **56条软件工程定律**，汇集了塑造软件系统、团队和决策的原则与模式。这些定律并非严格的法律条款，而是经过实践验证的经验法则和心智模型，可帮助工程师和管理者做出更好的决策。

---

## 分类总结

### 一、架构 (Architecture)

| 定律 | 描述 |
|------|------|
| **Conway定律** | 组织设计系统时会镜像其自身的沟通结构 |
| **Gall定律** | 复杂系统能够工作，必然是从简单系统演化而来 |
| **抽象渗漏定律** | 所有非平凡的抽象在某种程度上都是渗漏的 |
| **复杂性守恒定律** | 每个应用都有固有的不可简化的复杂性，只能转移，不能消除 |
| **意外后果定律** | 改变复杂系统时，预期会有意外发生 |
| **Zawinski定律** | 每个程序都会扩展到能读取邮件为止 |
| **分布式系统谬误** | 新分布式系统设计者常犯的八个错误假设 |

### 二、团队 (Teams)

| 定律 | 描述 |
|------|------|
| **Dunbar数字** | 一个人能维持约150个稳定关系 |
| **Putt定律** | 技术能力倾向两极分化，无能者倾向于走到一起 |
| **90-90法则** | 前90%的代码占开发时间的90%，剩余10%占另外90% |
| **Ringelmann效应** | 团队越大，人均贡献越小 |

### 三、规划 (Planning)

| 定律 | 描述 |
|------|------|
| **过早优化定律** | 过早优化是万恶之源 |
| **Hofstadter定律** | 事情总是比你预期的花费更长时间 |
| **Parkinson定律** | 工作会扩展填满可用时间 |
| **Brooks定律** | 向延误的项目添加人员会使其更加延误 |
| **Goodhart定律** | 当一个指标成为目标时，它就不再是好指标 |
| **Gilb定律** | 任何需要量化的东西都可以用某种方式测量 |
| **软件估算五定律** | 估算是浪费、不可转移、错误、临时、必要 |

### 四、质量 (Quality)

| 定律 | 描述 |
|------|------|
| **Murphy定律** | 任何可能出错的事都会出错 |
| **破窗理论** | 看到失修迹象时，人们会更随意地对待环境 |
| **童子军规则** | 让代码比你发现时更好 |
| **Postel定律** | 对自己要做的事保守，对别人做的事宽容 |
| **Linus定律** | 足够多的眼睛，所有bug都是浅显的 |
| **Lehman软件演化定律** | 反映现实世界的软件必须演化，否则会变得越来越无用 |

### 五、设计 (Design)

| 定律 | 描述 |
|------|------|
| **YAGNI原则** | 除非必要，否则不要添加功能 |
| **KISS原则** | 保持简单愚蠢 |
| **DRY原则** | 不要重复自己 |
| **惊讶最小化原则** | 系统行为应符合用户预期 |

### 六、决策 (Decisions)

| 定律 | 描述 |
|------|------|
| **Hyrum定律** | 足够多的API用户会依赖系统的所有可观察行为 |
| **Cunningham定律** | 正确答案的错误回答出人意料地有效 |
| **Hanlon定律** | 假设愚蠢而不是恶意 |
| **Sturgeon定律** | 90%的一切都是垃圾 |

### 七、规模 (Scale)

| 定律 | 描述 |
|------|------|
| **Amdahl定律** | 并行计算机加速受限于无法并行的部分 |
| **Moore定律** | 组件成本的复杂性每两年翻一番 |
| **Pareto原理** | 80%的效果来自20%的原因 |

## 核心原则汇总

### 最重要的5条定律

1. **过早优化是万恶之源** - 在明确需要之前不要优化
2. **Brooks定律** - 向延误项目加人会使其更延误
3. **Murphy定律** - 任何可能出错的事都会出错
4. **Conway定律** - 系统会反映组织结构
5. **Gall定律** - 复杂系统从简单系统演化而来

### 项目管理相关

- **Parkinson定律**：工作填满可用时间
- **Hofstadter定律**：花费时间总比预期长
- **Goodhart定律**：指标成为目标后就失去意义
- **软件估算五定律**：估算是浪费、不可转移、错误、临时、必要

### 代码质量相关

- **童子军规则**：保持代码比发现时更好
- **Linus定律**：众目睽睽，bug无处遁形
- **抽象渗漏定律**：所有抽象都有渗漏
- **复杂性守恒定律**：复杂性只能转移，不能消除

## 实践建议

### 给工程师的建议

1. 遵循YAGNI原则，避免过度设计
2. 使用童子军规则持续改进代码
3. 理解并应用Conway定律设计团队结构
4. 警惕过早优化，先让代码工作再优化
5. 记住Murphy定律，做好错误处理

### 给管理者的建议

1. 不要向延误项目盲目加人
2. 估算要有心理准备，实际时间总是更长
3. 警惕"指标成为目标"的陷阱
4. 理解Price定律：一个少数人做大部分工作
5. 考虑Ringelmann效应，控制团队规模

### 给团队的建议

1. 保持团队规模小于Dunbar数字（150）
2. 遵循Postel定律，对内保守对外宽容
3. 应用破窗理论，保持代码整洁
4. 使用Gall定律，从简单开始，逐步演化

## 结论

这56条软件工程定律是软件工程领域的智慧结晶，它们不是魔法公式，而是强大的心智工具，能帮助诊断问题并在压力下做出更好的决策。理解这些原则不会保证成功，但会让从业者在软件工程的复杂世界中更有准备。
