# 洞察报告：尽管存在疑虑，联邦网络安全专家仍批准了微软云服务

## 文章摘要

ProPublica 的一项调查发现，联邦政府的网络安全评估机构 FedRAMP 在 2024 年底批准了微软的 Government Community Cloud High (GCC High) 云服务，尽管该产品存在严重的安全隐患。

### 关键发现

1. **内部评价极差**：联邦审查人员将微软的产品描述为"一堆垃圾"（a pile of shit），并表示"无法自信地评估系统的整体安全状况"。

2. **审查过程长达五年**：FedRAMP 从 2020 年开始审查 GCC High，期间多次要求微软提供详细的数据流向图和加密文档，但微软始终未能提供完整的资料。

3. **既成事实的批准**：由于联邦机构在审查期间就被允许部署该产品，到 2024 年底，GCC High 已在政府和国防行业广泛使用。审查人员最终表示，他们"几乎没有选择，只能授权该技术"——不是因为问题已解决，而是因为产品已被广泛采用。

4. **安全历史糟糕**：微软产品在过去三年中涉及两起重大网络攻击——一次是俄罗斯黑客利用微软漏洞攻击联邦机构，另一次是中国黑客渗透了 GCC 云并窃取了商务部长和美国驻华大使的电子邮件。

5. **FedRAMP 资源被大幅削减**：特朗普政府的政府效率部门大幅削减了 FedRAMP 的员工和预算，目前年度预算仅为 1000 万美元，为十年来最低水平。

## Hacker News 讨论焦点

### 1. FedRAMP 审批机制的根本问题

讨论中最受关注的焦点是 FedRAMP 审批流程的制度性缺陷：

- **"先上车后补票"问题**：由于产品可以在审查期间部署，一旦产品被广泛使用，审查人员就面临"是否愿意与众多政府机构对抗以移除它"的艰难选择。

- **成本高昂**：有评论指出，FedRAMP 认证成本高达 50-100 万美元，仅实施费用就需 2-300 万美元，加上漫长的等待时间（2-3 年），这实际上是对小企业的排斥。

- **与 SOC 2 对比**：SOC 2 认证只需约 2 万美元（含工具和审计），而 FedRAMP 成本高出 20-50 倍。

### 2. Palantir 在 FedRAMP 中的角色争议

讨论中出现了关于 Palantir 主导地位的争论：

- 有评论称 FedRAMP 实际上是"Palantir 税"，几乎所有希望向政府销售产品的初创公司（包括 Anthropic、xAI、OpenAI）都不得不使用 Palantir 的 FedRAMP 服务。

- 另一方反驳称这是"绝对虚构"，表示可以自行获得 ATO（授权操作）。

### 3. 微软产品的用户体验问题

大量评论集中批评微软产品的使用体验：

- **Entra ID (原 Azure AD) 的复杂性**：有评论指出 Entra ID 有 12 种强制 MFA 的方式、20 种禁用用户的方法、4 种身份验证方式，以及带有 50 个变量和模板的条件访问配置。

- **登录问题**：微软的 SSO 登录流程被广泛批评为"最糟糕"的——有大量重定向、"记住我"复选框从不生效、多账户情况下无法正常工作。

- **产品迷宫**：评论指出微软同时做三件相互矛盾的事情：快速推出新产品、保持向后兼容性、永不淘汰过时产品。这导致每个微软产品都是重叠 API 的迷宫。

### 4. 微软的安全问题

- **Storm-0558 事件**：有评论提到微软一个被盗的签名密钥能够为任何 MSA/Azure AD/政府 AD 用户伪造身份验证令牌。

- **Entra ID 漏洞**：还有报道称 Entra ID 存在允许劫持任何公司租户的漏洞。

- **内部安全文化**：有自称微软员工的评论者透露，公司内部现在"大量"使用 AI 生成代码然后直接发送 PR。

### 5. 微软内部的混乱

一位自称在微软工作超过十年的员工详细描述了公司内部的问题：

- 有多个不同的大型数据处理系统，但互不通信
- 多个作业编排系统，但直到 Azure Data Factory 才向外发布
- 内部安全环境复杂，需要使用多台电脑和三个账户
- 团队之间不协调，不同产品使用不同的日志格式和约定
- 员工使用 MacBooks 开发（据报道约占 40%）

## 结论

这次讨论揭示了以下几个层面的问题：

1. **联邦认证机制的失败**：FedRAMP 在资源不足和既成事实的压力下，难以执行严格的安全审查。

2. **微软的市场主导地位**：微软在政府云服务市场的主导地位使其即使在存在明显安全问题的情况下也能获得认证。

3. **产品体验问题**：微软企业产品的复杂性不仅影响安全评估，也给日常使用带来极大困扰。

4. **安全文化的缺失**：从内部反馈来看，微软在快速开发和安全之间似乎选择了前者。
