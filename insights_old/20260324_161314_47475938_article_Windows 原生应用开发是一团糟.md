## 内容摘要

本文作者结合自身经历，阐述了一个简单的Windows工具应用的开发过程，并深刻揭示了现代Windows原生应用开发领域存在的系统性混乱问题。

### 一、作者背景与开发动机

作者是一位Windows老用户，从10岁开始学习Visual C++，后来从事.NET开发工作，但从未真正编写过原生Windows应用。出于怀旧情结，他决定开发一个名为"Display Blackout"的小工具——用于在游戏时将OLED显示器屏幕显示为纯黑色（实现类似关闭屏幕的效果）。然而，这趟开发之旅却让他深刻体会到Windows原生开发环境的糟糕现状。

### 二、应用程序的功能需求

该应用需要实现以下核心功能：
- 枚举机器显示器及其边界
- 放置无边框、无标题栏、不激活的黑色窗口
- 拦截全局键盘快捷键
- 可选开机自启动
- 存储持久化设置
- 显示托盘图标及菜单

### 三、Windows编程框架的演变历史

作者梳理了Windows UI框架的演进历程：

**Win32 C API → MFC → WinForms → WPF → WinRT XAML → UWP XAML → WinUI 3**

这种频繁的框架更迭导致：
1. 每代框架都存在功能缺口，需要回退到旧框架
2. 开发者需要学习大量即将过时的技术
3. 代码中充满互操作（interop）胶水代码

### 四、核心技术困境

#### 1. 框架选择的痛苦

- **C++选项**：性能好但内存不安全，2026年在新项目中采用C++是"犯罪"
- **框架依赖部署**：理想情况下应分发C#字节码，但Windows 11仅预装.NET 4.8.1（当前版本为10），会导致用户需要下载安装.NET运行库
- **.NET AOT编译**：将整个.NET运行时编译进二进制，结果是简单的工具应用也要9 MiB

#### 2. 分发方式的困境

- **MSIX格式**：依赖代码签名证书，非美国居民每年需花费200-300美元
- **侧载体验**：需要管理员权限的PowerShell命令，用户体验极差
- **微软商店**：因缺乏"独特持久价值"被拒绝

#### 3. API功能缺失

即使使用最新的Windows App SDK，大多数功能仍需P/Invoke调用Win32 API：

| 功能 | 状态 |
|------|------|
| 显示器枚举 | 可用，但监视变化需P/Invoke |
| 无边框窗口 | 大部分可用，非激活需P/Invoke |
| 全局快捷键 | 需P/Invoke |
| 开机自启动 | 可用 |
| 持久化设置 | 可用 |
| 托盘图标 | 完全不可用，需P/Invoke |

#### 4. 互操作技术的缺陷

- **CsWin32**：微软的P/Invoke封装工具，仍处于"pre-1.0"状态，无法正确处理结构体中的字符串
- **C#语言本身的不足**：C#无法优雅地表达Win32 API中的可选参数类型

#### 5. 二十年的停滞

- WPF早在2006年引入双向数据绑定，但创建可绑定类所需的样板代码（INotifyPropertyChanged）在20年后几乎未变
- C#团队在20年间未能将"原生可观察类"纳入语言特性

### 五、行业现状

- **微软自身**：许多第一方应用（VS Code、Outlook、开始菜单）使用Web技术构建
- **社区选择**：更多人转向第三方框架如Avalonia、Uno Platform，或直接使用Electron/Tauri
- **问题追踪器**：充满开发者遇到的痛苦bug和功能缺口，微软工程师响应甚少
- **Windows App SDK更新**：主要添加机器学习API，而非解决核心问题

### 六、结论

Windows原生应用开发已陷入恶性循环：
1. 框架频繁更迭但每代都有缺口
2. 微软自己都不优先投资
3. 开发者转向跨平台/Web技术
4. 进一步削弱Windows原生生态

作者认为，除非微软在以下方面做出改进，否则情况不会好转：
- 通过Windows Update分发.NET更新
- 提供可作为依赖的MSIX格式.NET包
- 降低代码签名证书成本
- 真正投入资源改善WinUI 3

但作者表示不会抱太大期望，并明确表态：考虑到Windows应用平台的混乱程度，他宁愿选择Web技术栈（配合Electron或Tauri来访问必要的Win32 API）。
