# Go 命名约定：实用指南

## 概述

选择正确的名称是 Go 编程中重要（有时也很困难）的部分。好的命名使代码更清晰、更易预测、更易导航；反之则会产生负面影响。Go 对命名有相当强的约定和一些硬性规则。

## 标识符规则

### 硬性规则

- 标识符只能包含 Unicode 字母、数字和下划线
- 标识符不能以数字开头
- 不能使用 Go 关键字作为标识符

### 命名风格

- 未导出标识符使用 camelCase，导出标识符使用 PascalCase
- 避免使用 snake_case、Pascal_Snake_Case、SCREAMING_SNAKE_CASE
- 缩写词（如 API、URL、HTTP）在标识符内保持一致的大小写：`apiKey` 或 `APIKey` 都可以，但 `ApiKey` 不可以
- ID 作为"identity"或"identifier"的缩写时，应全大写：`userID` 而非 `userId`
- 尽量使用 ASCII 字母，避免使用非 ASCII 字符
- 避免与 Go 内置类型冲突（如 `int`、`bool`、`any`）
- 避免在标识符中包含类型名（如 `fullNameString`、`scoreInt`），除非需要进行类型转换区分
- 避免与标准库包名冲突（如 `json`、`url`、`log`）

## 导出与未导出标识符

标识符首字母大写表示导出，对包外可见。应该默认使用未导出标识符，只在真正需要时才导出。这使得包内重构更容易。main 包中的标识符通常应全部未导出。

## 标识符长度与描述性

作用域越小（如循环内），可以使用越短的名字；作用域越大或使用距离声明越远，应使用更具描述性的名称。没有精确的标准，需要根据具体情况选择"合适的长度"。

## 包命名

- 只使用小写 ASCII 字母和数字
- 名称应简短、易输入，能反映包的内容
- 多单词用小写连接，无分隔符（如 `ordermanager`）
- 可使用缩写（如 `expvar`、`strconv`）
- 避免与标准库包名冲突
- 避免以 `.` 或 `_` 开头
- 避免使用 `vendor`、`testdata`、`internal` 作为包名
- 避免使用泛泛的包名如 `common`、`util`、`helpers`、`types`

## 文件命名

- 理想的文件名应总结文件内容，一个词，全小写
- 多单词文件名可使用下划线连接或直接拼接，需保持一致
- 避免使用特殊前缀/后缀除非需要触发 Go 的特殊行为：
  - `_test.go` 仅由 `go test` 运行
  - `_linux.go`、`_windows.go` 等为平台特定文件
  - `_amd64.go`、`_arm64.go` 等为架构特定文件

## 避免冗余（Chatter）

导出函数时，避免重复包名。例如 `customer` 包中的 `NewCustomer()` 是冗余的，直接用 `New()` 即可。类型命名同样如此，`Address` 比 `CustomerAddress` 更简洁。但包名与类型名相同的情况（如 `time.Time`）是常见例外。

## 方法接收者

接收者名称应简短（1-3 个字符），通常是类型名的缩写。避免使用 `this`、`self`、`me` 等通用名称。同一类型的所有方法应使用相同的接收者名称。

## Getter 和 Setter

Go 中通常不需要为结构体字段创建 getter/setter，直接访问字段即可。例外情况是当字段未导出但需要从包外访问时。约定是：setter 以 `Set` 开头，getter 不以 `Get` 开头。

## 接口命名

单方法接口通常以 `-er` 后缀命名，如 `Reader`、`Writer`、`Stringer`。避免在接口名中包含类型名（如 `UserInterface`）。

## 打破约定

在极少数情况下，打破约定可以使代码更清晰是可以接受的，特别是私人代码库。但绝大多数情况下，应遵循本文讨论的命名规则，因为它们使代码更可预测、更一致，更容易被其他 Gopher 快速理解。
