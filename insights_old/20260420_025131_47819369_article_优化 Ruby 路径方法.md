# 洞察报告：优化 Ruby 路径方法

## 基本信息

- **洞察链接**: https://byroot.github.io/ruby/performance/2026/04/18/faster-paths.html
- **原文标题**: Optimizing Ruby Path Methods
- **作者**: byroot
- **发布日期**: 2026年4月18日
- **主题**: Ruby 路径方法性能优化

## 摘要

本文介绍了作者在 Intercom 工作期间，针对 Ruby 应用启动时间进行的多项性能优化工作。主要涉及 Bootsnap 的加载路径缓存优化和 Ruby 核心路径方法（File.join 等）的改进。

## 核心内容

### 1. 背景：CI 构建中的启动时间问题

作者在 Intercom 处理一个拥有 1350 个并行工作进程的 CI 系统。即使是削减启动时间中的每一秒，都能产生巨大影响：
- 削减 1 秒启动时间 = 节省 1350 秒计算时间
- 每次构建可节省超过 20 分钟的计算成本

### 2. Bootsnap 加载路径缓存优化

#### 问题分析
- Ruby 的 `require` 方法需要在 `$LOAD_PATH` 中进行线性搜索
- 每次 require 可能触发最多 `2*N` 次文件系统查询（`N` 为加载路径数量）
- 随着 gem 数量增加，成本呈 `O(N*M)` 增长（N=加载路径数，M=已加载特性数）

#### N+1 系统调用问题
Bootsnap 的路径扫描器对每个目录条目调用 `File.directory?`，导致 N+1 次 `stat(2)` 系统调用。在 Linux/BSD 系统上，`readdir(3)` 的 `d_type` 成员可以无需额外系统调用判断文件类型。

#### 解决方案：Dir.scan
- 作者向 Ruby 提交了 `Dir.scan` 方法的功能请求
- 该方法在遍历目录时同时返回文件类型信息
- 基准测试显示：扫描速度提升 **2 倍**
- 扫描约 32k 文件和 10k 目录只需 230ms（之前需要 500ms）
- 该功能将随 Ruby 4.1.0 正式发布

### 3. File.join 优化（提升 7 倍以上）

作者发现 `File.join` 比字符串插值慢 4 倍，通过分析发现多个优化点：

#### 3.1 多字节编码处理优化
- 原因：Ruby 需要处理日文 Shift_JIS 编码中 `0x5c` 既可能是反斜杠也可能是多字节字符的延续字节
- 解决：添加快速路径，对于 UTF-8、ASCII 等 ASCII 兼容编码使用单字节快速算法

#### 3.2 反向搜索优化
- 原实现从字符串开头扫描查找尾部路径分隔符
- 优化为从字符串末尾开始搜索，减少不必要的遍历

#### 3.3 C 字符串处理优化
- 将 `rb_string_value_cstr` 替换为 `rb_str_null_check`
- 避免不必要的 NULL 终止检查

#### 3.4 可变参数优化
- 原来使用数组接收参数，需要额外分配和复制
- 改为直接从栈读取参数，减少内存分配

#### 优化结果
- 两个字符串连接：**7.8 倍提升**
- 多个字符串连接：**18.8 倍提升**
- 在 Ruby 4.1.0dev 上，`File.join` 现在比字符串插值更快（1.14x）

### 4. 其他路径方法优化

作者还将类似优化应用到：
- `File.basename`
- `File.dirname`
- `File.extname`
- `File.expand_path`

## 关键收获

1. **性能优化要找热点**：N+1 系统调用是 Bootscan 的主要瓶颈
2. **不假设任何代码是完美的**：即使是 Ruby 核心方法也有优化空间
3. **关注边界情况**：理解编码问题（如 Shift_JIS）对性能的影响
4. **Ruby 社区协作**：通过向 Ruby 提交补丁推动语言层面的改进
