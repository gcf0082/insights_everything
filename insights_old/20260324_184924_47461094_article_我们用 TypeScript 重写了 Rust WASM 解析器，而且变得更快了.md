# 洞察报告：将 Rust WASM 解析器重写为 TypeScript

**洞察链接**: https://www.openui.com/blog/rust-wasm-parser

**来源**: OpenUI Blog (Thesys Engineering Team)

**发布日期**: 2026年3月13日

**总结日期**: 2024年

---

## 背景

OpenUI 团队将用 Rust 编写的 WASM 解析器重写为 TypeScript，性能提升了 **3倍**。

OpenUI Lang 解析器将 LLM 生成的 DSL 转换为 React 组件树，需要在每个流式块上运行，因此延迟非常关键。

### 解析管道（6个阶段）

```
autocloser → lexer → splitter → parser → resolver → mapper → ParseResult
```

-   **Autocloser**: 通过追加最少的闭合括号/引号，使部分（流中）文本在语法上有效
-   **Lexer**: 单遍字符扫描器，发射类型化标记
-   **Splitter**: 将标记流切割成 `id = expression` 语句
-   **Parser**: 递归下降表达式解析器，构建 AST
-   **Resolver**: 内联所有变量引用（提升支持，循环引用检测）
-   **Mapper**: 将内部 AST 转换为 React 渲染器使用的公共 `OutputNode` 格式

---

## 问题：WASM 边界税

每次调用 WASM 解析器都会产生强制性的开销，无论 Rust 代码本身运行得多快：

```
JS 世界                              WASM 世界
────────────────────────────────────────────────────────
wasmParse(input)
  │
  ├─ 复制字符串: JS 堆 → WASM 线性内存   (分配 + memcpy)
  │
  │                                 Rust 解析   ✓ 快速
  │                                 serde_json::to_string()  ← 序列化结果
  │
  ├─ 复制 JSON 字符串: WASM → JS 堆       (分配 + memcpy)
  │
  JSON.parse(jsonString)                        ← 反序列化结果
  │
  返回 ParseResult
```

Rust 解析本身从来不是慢的部分。开销完全在边界上：将字符串复制进去，将结果序列化为 JSON 字符串，将 JSON 字符串复制出来，然后 V8 将其反序列化为 JS 对象。

---

## 尝试修复：跳过 JSON 往返

使用 `serde-wasm-bindgen` 让 WASM 直接返回 JS 对象，跳过 JSON 序列化步骤。结果是**慢了 30%**。

原因：JS 无法将 Rust 结构体的字节作为原生 JS 对象从 WASM 线性内存读取——两个运行时使用完全不同的内存布局。要从 Rust 数据构造 JS 对象，`serde-wasm-bindgen` 必须递归地将 Rust 数据实现为真实的 JS 数组和对象，这涉及每次 `parse()` 调用时许多细粒度的运行时边界转换。

与 JSON 方法相比：`serde_json::to_string()` 在纯 Rust 中运行，零边界穿越，产生一个字符串，一次 `memcpy` 将其复制到 JS 堆，然后 V8 的原生 C++ `JSON.parse` 以单一优化流程处理它。**更少、更大、更优化的操作胜于许多小的操作**。

### 基准测试：JSON 字符串 vs 直接 JsValue（1000次运行，微秒/调用）

| Fixture    | JSON 往返 | serde-wasm-bindgen | 变化       |
|------------|-----------|---------------------|------------|
| simple-table | 20.5      | 22.5                | -9% 更慢   |
| contact-form | 61.4      | 79.4                | -29% 更慢  |
| dashboard   | 57.9      | 74.0                | -28% 更慢  |

团队立即回滚了这个更改。

---

## 真正的修复：完全消除边界

团队将完整的解析管道移植到 TypeScript。相同的六阶段架构，相同的 `ParseResult` 输出形状——无需 WASM，无需边界，完全在 V8 堆中运行。

### 基准测试方法：一次性解析

**测量内容**：在完整输出字符串上调用单次 `parse(completeString)`。这隔离了每次调用的解析成本。

**运行方式**：30次预热迭代以稳定 JIT，然后使用 `performance.now()`（微秒精度）进行1000次计时迭代。报告中间值。Fixtures 是每个格式真实流式语法中序列化的真实 LLM 生成的组件树。

**Fixtures**：
-   `simple-table` — 根 + 一个包含3列5行的表（~180字符）
-   `contact-form` — 根 + 表单布局，包含6个输入字段 + 提交按钮（~400字符）
-   `dashboard` — 根 + 侧边栏导航 + 3个指标卡片 + 图表 + 数据表（~950字符）

### 结果：一次性解析（中间值微秒，1000次运行）

| Fixture    | TypeScript | WASM   | 加速比 |
|------------|-----------|--------|--------|
| simple-table | 9.3      | 20.5   | **2.2x** |
| contact-form | 13.4     | 61.4   | **4.6x** |
| dashboard   | 19.4      | 57.9   | **3.0x** |

## 算法问题：O(N²) 流式处理

消除 WASM 修复了每次调用的成本，但流式架构仍然存在更深层的低效问题。

解析器在每个 LLM 块上调用。朴素方法累积块并每次从头重新解析整个字符串：

```
Chunk 1:  parse("root = Root([t")              → 14 字符
Chunk 2:  parse("root = Root([tbl])\ntbl = T") → 27 字符
Chunk 3:  parse(full_accumulated_string)        → ...
```

对于以20字符块传输的1000字符输出：50次解析调用处理累计约25,000字符。**O(N²) 复杂度**。

---

## 修复：语句级增量缓存

深度为0的新行终止的语句是**不可变的**——LLM 永远不会回来修改它们。团队添加了一个流式解析器来缓存完成的语句 AST：

```
状态: { buf, completedEnd, completedSyms, firstId }

每次 push(chunk):
  1. 从 completedEnd 扫描 buf 寻找深度为0的新行
  2. 对于每个完整语句：解析 + 缓存 AST → 推进 completedEnd
  3. 待处理（最后一个，不完整）语句：自动关闭 + 重新解析
  4. 合并缓存 + 待处理 → 解析 + 映射 → 返回 ParseResult
```

完成的语句永远不会被重新解析。只有尾部进行中的语句每次块被重新解析。**O(总长度) 而不是 O(N²)**。

### 基准测试方法：全流式总解析成本

**测量内容**：**一个完整文档的每个块调用累积的总解析开销**。这与一次性基准测试不同——它测量的是真实流期间所有解析调用的总和，而不是单次调用。这才是影响用户实际感知响应性的数字。

**运行方式**：文档以20字符块重放。每个块触发 `parse()`（朴素）或 `push()`（增量）调用。记录所有调用的总时间。100次完整流式重放，取中间值。

### 结果：全流式总解析成本（所有块的中位微秒）

| Fixture      | 朴素 TS（每次块重新解析） | 增量 TS（缓存完成） | 加速比 |
|--------------|--------------------------|---------------------|--------|
| simple-table | 69                       | 77                  | 无（单语句，无缓存收益） |
| contact-form | 316                      | 122                 | **2.6x** |
| dashboard    | 840                      | 255                 | **3.3x** |

`simple-table` fixture 是单个语句——没有东西可以缓存，所以两种方法是等价的。收益随着语句数量增加而增加，因为每个块会缓存和跳过更多文档。

## 总结

| 方案                    | 每次调用成本 | 全流式总成本 | 备注                   |
|-------------------------|-------------|--------------|------------------------|
| WASM + JSON 往返        | 20-61µs     | 基线         | 每次调用复制开销       |
| WASM + serde-wasm-bindgen | 22-79µs   | +9-29% 更慢  | 数百次内部边界穿越     |
| TypeScript（朴素重新解析）| 9-19µs    | 69-840µs     | 无边界，但 O(N²) 流式  |
| TypeScript（增量）       | 9-19µs    | 69-255µs     | 无边界 + O(N) 流式     |

**最终结果：每次调用快 2.2-4.6 倍，总流式成本降低 2.6-3.3 倍。**

## WASM 真正有帮助的场景

这次经历优化了我们对 WASM 正确用例的思考：

✅ **计算密集型且最小互操作**：图像/视频处理、加密、物理模拟、音频编解码器。大输入 → 标量输出或原地变更。边界很少穿越。

✅ **可移植原生库**：将 C/C++ 库（SQLite、OpenCV、libpng）移植到浏览器，无需完整的 JS 重写。

❌ **将结构化文本解析为 JS 对象**：无论哪种方式都要支付序列化成本。解析计算速度足够快，V8 的 JIT 消除了任何 Rust 优势。边界开销占主导地位。

❌ **频繁调用小输入函数**：如果函数在流中调用50次，计算需要5µs，则无法摊销边界成本。

---

## 关键要点

1.  **在选择实现语言之前，先分析时间实际花在哪里。** 对我们来说，成本从来不在计算中——总是在 WASM-JS 边界上的数据传输中。

2.  **通过 `serde-wasm-bindgen` 的"直接对象传递"并不便宜。** 从 Rust 逐字段构造 JS 对象比单一 JSON 字符串传输涉及更多的边界穿越，而不是更少。边界穿越发生在单个 FFI 调用内部，难以察觉。

3.  **算法复杂度改进主导语言级优化。** 从 O(N²) 到 O(N) 的流式情况变化比从 WASM 切换到 TypeScript 有更大的实际影响。

4.  **WASM 和 JS 不共享堆。** WASM 有一个平面线性内存（`WebAssembly.Memory`），JS 可以将其作为原始字节读取，但这些字节是 Rust 的内部布局——指针、枚举判别符、对齐填充——对 JS 运行时完全不透明。转换总是必需的，总是要付出代价。
