## 文章概述

作者 lisyarus 在 2025 年 12 月开始设计自己的编程语言 pslang（psemek language），目前已有约 10k 行 C++ 代码实现。本文详细介绍了他创建这门语言的原因、设计目标、语言特性、编译器架构及未来计划。

## 一、为什么创建自己的编程语言？

作者编程约 17 年，三个因素促使他开始动手：

1. **实现长久以来的愿望**：作者过去做过一些玩具级解释器（如 FALSE 语言），但一直想做一个真正能用于生产、不是玩具的语言。
2. **游戏模组（Modding）需求**：作者正在开发一个高度模拟型的游戏（Village Builder），需要一种既能满足 ECS 实体迭代、又易于沙箱化、还简单易用的模组方案。
3. **重新对汇编产生兴趣**：Matt Godbolt 的 Advent of Compiler Optimisations 系列重新点燃了作者对底层编程和编译器优化的热情。

## 二、模组方案分析

### 2.1 Lua 方案的问题

- 很难彻底沙箱化，需要手动删除大量标准库函数。
- Lua 是高层动态类型语言，不了解 C 指针，无法高效桥接 ECS 实体迭代。
- 标准 Lua 和 LuaJIT 已经分叉，维护成本高。

### 2.2 C++ 原生方案的问题

- 二进制分发需要为所有平台提供开发环境和构建存储；源码分发需要捆绑笨重的编译器（LLVM 占用磁盘空间是游戏的 10-20 倍）。
- 沙箱化几乎不可能，加载原生 DLL 后无法阻止其访问文件系统、网络等。

## 三、语言设计目标

- **无缝 C 互操作**：从原生游戏代码到模组代码的桥接应像函数调用一样简单。
- **底层**：需要处理原始实体数组。
- **实用、符合人体工程学**：让模组开发者能较容易地编写代码。
- **易于沙箱化**：能一键禁用所有 IO 等危险操作。
- **编译器体积小**：不想在 50MB 的游戏中嵌入 1GB 的编译器。
- **编译速度快**：不想让玩家等待数小时进行模组编译。
- **真正跨平台**：支持主流桌面平台，假设 64 位和 IEEE754 浮点。
- **性能合理**：比大多数动态语言快即可。
- **避免重造 C++**：虽然最终可能仍未完全成功。

## 四、语言特性

### 4.1 概述

pslang 是一门**命令式**、**及早求值**、**传值调用**的**底层**编程语言，拥有**静态**、**严格**、**标称**类型系统。语法融合了 C++、Rust、Python 和 Zig 的特点。

### 4.2 作用域

- 使用基于缩进的作用域（Tab 缩进）。
- 每个函数、循环体、`if` 体等都会创建新作用域。
- 顶级作用域被包装成匿名函数，充当入口点。
- 局部函数可以定义在任意作用域中，但局部函数**不是闭包**，无法访问定义处作用域的变量。

### 4.3 原始类型

共 13 种原始类型：

| 有符号整数 | 无符号整数 | 浮点数 |
|---------|---------|-------|
| i8 i16 i32 i64 | u8 u16 u32 u64 | f16 f32 f64 |

- 所有整数算术采用**二进制补码**，溢出**无未定义行为**。
- `bool` 占 1 字节。
- `unit` 类型是特殊类型，只有一个值 `unit()`，用于无返回值函数的返回类型。

### 4.4 数值字面量

- 默认 `10` 为 `i32`；`10b` 为 byte；`10s` 为 short；`10l` 为 long。
- 无符号加 `u` 后缀：`10u` 为 u32，`10ul` 为 u64。
- 浮点默认 `f32`；`10.0h` 为 f16；`10.0d` 为 f64。
- 所有数值字面量都有**明确无歧义的类型**。

### 4.5 数组

- 数组是内置的**一等公民**类型，可整体传递给函数或从函数返回。
- 声明：`i32[5]`
- 字面量：`[1, 2, 3, 4, 5]`
- 数组大小在**编译时已知**。
- 内存对齐与元素类型一致。

### 4.6 函数类型

- 语法：`(a, b, c) -> d`
- 单参数可省略括号：`a -> b`
- 内部实现为普通函数指针，不是闭包。

### 4.7 指针

- 声明：`i32*`（默认**不可变**），`i32 mut*`（可变）。
- 支持取地址 `&x`、`&mut x`，解引用 `*p`，以及指针算术。

### 4.8 结构体

- 语法：
  ```rust
  struct string_view:
      size: u64
      data: u8*
  ```
- 通过构造器函数创建：`string_view(10, data)`
- 通过点语法访问字段：`v.x`
- 结构体字段**没有可变性修饰符**，字段的可变性与对象本身的可变性一致。
- **没有访问控制符**，字段始终是公开的。

### 4.9 空类型

- `unit` 和空结构体（无字段的结构体）称为"空类型"。
- **不占用任何内存**（大小为 0 字节）。
- 这意味着 pslang 不遵循 C++ 中每个对象都有唯一内存地址的规则。

### 4.10 变量

- 不可变：`let x = 10`
- 可变：`mut x = 20`
- 可显式声明类型：`let x: i32 = 10`
- 类型推导是**无歧义的**，因为存在类型严格的规则。

### 4.11 函数

- 语法：`func foo(x: A, y: B) -> C:`
- 所有函数遵循平台原生 C ABI，便于 C 互操作。
- 声明顺序无限制（前向声明通过声明顺序解决）。

### 4.12 控制流

- `if-else` 语句和表达式：`if A then B else C`
- `while` 循环
- **没有 for 循环**（计划将来作为基于集合的循环）
- 短路求值：`&&` 和 `||` 不计算第二个操作数；`&` 和 `|` 始终计算两个操作数（按位运算）。

### 4.13 外部函数（Foreign Functions）

```rust
foreign func sin(x: f64) -> f64
```
- 通过 `dlsym` 从解释器可执行文件本身链接。
- 这是与 C 库及其他第三方库对接的主要机制。

### 4.14 类型转换

- **没有隐式类型转换**。
- 手动使用 `as` 操作符：`x as f32`
- 所有数值类型可互相转换；指针类型可互相转换（可变指针不能隐式转为不可变指针）；指针可与 `u64` 互相转换。

## 五、编译器架构

项目分为多个库：

- `types` – 类型系统定义
- `ast` – 抽象语法树定义和工具
- `parser` – 解析器
- `ir` – 中间表示
- `interpreter` – 解释器
- `jit` – JIT 编译器

解释器和编译器都是简单的 CLI 应用，如需嵌入语言，只需使用 `parser` + `jit` 库。

### 5.1 解析器

- 使用 **Bison** 解析器生成器（`lalr1.cc` 骨架），生成 C++ 解析器类。
- 词法分析器（lexer）和语法分析器（grammar）规则分别定义。
- 作者多次修复了 shift/reduce 冲突，通过 `-Wcounterexamples` 诊断。
- Bison 执行已集成到 CMake 构建步骤中。

### 5.2 缩进处理

- 由于语言使用缩进语法，语法**不是上下文无关的**。
- 解决方案：每行单独解析为一条语句，并附带缩进级别编号，然后通过**简单的线性遍历**根据缩进级别解析作用域。
- 这个阶段还检查 `break`、`continue`、`return` 等语句是否在正确的作用域中。

### 5.3 类型检查

- **标识符解析**：将标识符节点链接到其引用的变量/函数/结构体定义节点。
- **类型推导和检查**：大多数情况下是简单的条件判断。例如 `if` 或 `while` 的条件必须是 `bool` 类型，加法的两个操作数类型必须相同等。

### 5.4 解释器

- "树遍历解释器"（tree-walking interpreter）。
- 主要函数 `exec()`（执行语句）和 `eval()`（求值表达式）。
- 使用 C++ `variant` 存储语言中所有可能的值类型。
- 无法执行 `foreign` 函数（参数数量和类型在编译时未知）。

### 5.5 JIT 编译器 V1

- **仅支持 Apple M1 (Aarch64)**。
- JIT 编译器将代码生成为内存块（映射为可执行）。
- 编译策略：栈式编译器，表达式的结果放置位置与 AAPCS64（C 调用约定）返回值的放置方式相同（整数/指针在 `x0`，浮点在 `v0`）。
- 跳转指令通过输出偏移为零的指令，然后在知道目标偏移后再注入实际偏移值。
- 作者**手动编写了所有 Aarch64 指令**，参考 ARMv8 指令手册。

### 5.6 Aarch64 特殊性

- 所有指令都是 32 位，无法在单条指令中加载 32 位常量，需要分成 16 位补丁或从常量内存加载。
- **没有 push/pop 指令**，栈操作基于 SP（栈指针）寄存器。
- SP 必须 16 字节对齐，偏移量受 9 位或 12 位限制。
- 结构体传递/返回规则复杂（寄存器 vs 内存）。

### 5.7 IR（中间表示）

- 类似于 **SSA**（但不是真正的 SSA，因为允许重新赋值，不使用 phi 节点）。
- IR 是节点序列（`std::list`），每节点可以是字面量、操作、跳转、函数调用等。
- 支持 `assign` 指令重新赋值节点。
- 条件跳转分为 `jump_if_zero` 和 `jump_if_nonzero` 两个独立指令。
- 结构体支持：`alloc` 节点分配栈上空间，`copy` 节点提取嵌套字段，`assign` 支持对嵌套字段赋值。

### 5.8 JIT 编译器 V2

- 分为两部分：AST → IR 编译器 和 IR → Aarch64 编译器。
- 当前实现质量较差，因为尚未实现寄存器分配器，每个函数的栈帧分配了所有 IR 节点所需的内存。
- 未来需要实现**线性扫描寄存器分配器**。

## 六、未来计划

### 6.1 编译器/解释器

- **寄存器分配器**：使用标准线性扫描分配器。
- **IR 优化**：常量传播、算术简化、死代码消除、内联、循环展开。
- **IR 解释器**：直接求值 IR，而非遍历 AST。
- **生成可执行文件**：支持 ELF、Mach-O、PE 格式。
- **调试支持**：支持 DWARF 调试信息格式。

### 6.2 语言特性

- **结构体构造器**：支持同名函数作为构造函数。
- **全局变量**：通过 `global` 关键字声明。
- **方法调用语法**：`x.f(y)` 语义化为 `f(&x, y)` 或 `f(&mut x, y)`。
- **多态**：
  - C++ 风格：函数重载 + 无限制模板。
  - Haskell/Rust 风格：显式 trait + 约束泛型。
- **操作符重载**：需要先有多态支持。
- **For 循环**：计划作为基于集合的循环，需要范围/迭代器接口。
- **自动资源管理**：RAII + move、`defer` 关键字 或 线性类型。
- **多态字面量**：处理空数组 `[]` 和 `null` 的类型推导问题。
- **编译时求值**：`const` 变量 + 编译时函数调用。
- **类型计算**：编译时类型操作，支持类似 C++ concepts 的概念。
- **协程**：`async/await`。

### 6.3 标准库

- **模块系统**：`import lib.sublib`，支持作用域规则。
- **Prelude**：隐式包含的基础工具模块。
- **字符串字面量**：计划将字符串内容放在可执行内存中，字面量转为 `string_view`。
- **标准库**：数学库、内存管理、动态数组、动态字符串、哈希表、文件 I/O、文件系统助手、时间/时钟助手、网络。

## 七、关键要点

1. **语言定位**：pslang 是介于底层和高级之间的实用语言，专为游戏模组场景设计，核心优势是无缝 C 互操作和易于沙箱化。
2. **编译器体积**：目前 10k 行 C++ 代码，远小于 LLVM 等主流编译器，适合嵌入到小型应用中。
3. **设计哲学**：作者有意避免"重造 C++"，但在实践中仍受到 C++ 思维的强烈影响。
4. **类型系统**：静态、严格、标称类型系统，所有类型推导都是无歧义的，这简化了编译器的实现。
5. **JIT 策略**：作者选择手动编写目标 CPU 指令，而非依赖第三方库，以保持编译器的最小化。
6. **空类型的创新**：不占用内存、不遵循"每个对象有唯一地址"的规则，这是一个大胆的设计选择。
7. **项目进度**：语言核心功能已基本可用（能用 pslang 写出一个完整的 Monte-Carlo 路径追踪器），但距离生产级质量还很远。
8. **现实态度**：作者坦言项目可能永远无法真正被广泛使用，主要目的是"玩得开心"，且目前优先级仍是游戏开发本身。
