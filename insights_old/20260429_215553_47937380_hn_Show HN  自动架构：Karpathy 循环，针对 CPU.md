## 项目概述

该项目将 Karpathy 提出的"LLM 循环"方法应用于 CPU/FPGA 架构优化。该方法的核心思想是：
1. 让 LLM 随机扰动系统
2. 测量系统性能
3. 如果改进则保留，否则丢弃
4. 重复循环

项目针对 Gowin GW2A-LV18 FPGA（Tang Nano 20K）进行优化，通过 3-seed nextpnr P&R 获得了显著的性能提升。

## 核心洞察

### 1. 验证器是核心
- **关键观点**: "If you can write the rules down, an agent will satisfy them faster than your team will."
- 验证器（verifier）的质量决定了整个系统的上限
- 有好的测试套件就是一个有效的验证器

### 2. 实用案例反馈
- 有用户使用 Codex + GPT-5.4xhigh 优化 CUDA 内核，获得 **20x 吞吐量提升**
- 10x 来自架构变化，2x 来自微优化
- 另一位用户将提示词 token 减少了 25%，并迁移到更小/更便宜的模型

### 3. 面临的挑战
- 遗传算法可能产生"vibecoded mess"的短期决策
- 很难产生需要跨越多个步骤的重大架构变革
- 代理会尽可能"欺骗"验证器的任何漏洞
- 时区处理等问题几乎无法发现，需要专门的验证代理

## 相关讨论

### 创新性争议
- 许多评论者指出这种方法**并非新idea**
- **AlphaEvolve** (Google) 使用类似的循环
- **OpenEvolve** 有开源实现
- Schmidhuber 的 Gödel Machine (2006-2007)、Optimal Ordered Problem Solver (2002) 都相关
- 总结：**Stigler 定律** —— 没有想法是真正原创的

### 技术可行性
- 传统遗传算法有"交叉"操作，而 Karpathy 循环只有"选择最佳"
- 这是否算真正的遗传算法存在争议
- 现在由于新模型变得**更加实用**

### 扩展应用
- 数据库优化
- 整个公司运营
- 逆向工程闭源FPGA架构和比特流

## 总结

Karpathy 循环是一种将 LLM 作为变异operator应用到优化问题的方法。虽然不是全新概念，但现在由于 LLM 能力的提升变得切实可行。核心启示是：**验证器质量决定系统上限**——一个好的测试套件可以充当有效的验证器。该方法已在硬件优化、提示词优化、CUDA内核优化等领域展现出显著效果。
