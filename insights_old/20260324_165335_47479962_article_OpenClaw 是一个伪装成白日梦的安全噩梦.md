## 执行摘要

OpenClaw 是一个基于 Anthropic Claude Opus 4.5 模型的 AI 助手，能够与本地系统文件、终端、浏览器、Gmail、Slack 以及家庭自动化系统进行交互。然而，这一强大功能背后存在严重的安全隐患。本文深入分析了 OpenClaw 的安全漏洞，包括 ClawdHub 技能市场的恶意软件、提示词注入攻击、认证滥用、内存污染等问题，并提供了详细的安全加固建议。

---

## 一、OpenClaw 概述

### 1.1 产品愿景

OpenClaw 旨在实现高度自动化的个人 AI 助手。用户只需通过 bot 或语音发出指令，即可自动处理以下任务：

- 清理收件箱
- 安排会议并附加预 notes
- 预订航班
- 控制智能家居设备

这种自动化体验被比作"生活在未来"，仿佛拥有了一个数字超级助手。

### 1.2 积极用例

支持者认为 OpenClaw 具有以下优势：

- **学习能力**：随着使用时间增长，bot 能够从模式中学习，创建工具、工作流和技能
- **工作流存储**：可将工作流和技能存储在数据库或文件夹中以供将来参考
- **自动化细节**：例如处理餐厅预订取消费用时，会主动告知费用、确认是否不可退款，并将最后期限包含在日历事件中

## 二、安全威胁分析

### 2.1 ClawdHub 技能市场的安全隐患

OpenClaw 严重依赖 Skills（技能），这些技能从 SkillHub 拉取，由用户自行上传。**关键问题在于：没有任何人对任何事负责。** 没有安全检查，没有防护措施。

#### 2.1.1 恶意软件投放向量

1Password 的 Jason Melier 发现，**当时下载量最高的技能是一个"Twitter"技能**，看似正常，但实际上：

1. 该技能首先要求安装一个名为"openclaw-core"的"必需依赖"
2. 安装步骤包含指向"文档"的链接
3. 这些链接指向恶意基础设施
4. 攻击流程：
   - 技能概述要求安装前置程序
   - 链接指向一个设计用于诱导 agent 执行命令的分页页面
   - 命令解码混淆的有效载荷并执行
   - 有效载荷获取第二阶段脚本
   - 脚本下载并运行二进制文件，包括移除 macOS 隔离属性以绕过 Gatekeeper

最终确认该恶意软件为**信息窃取木马**，可通过 ClawdBot 访问 Cookies、保存的凭据、SSH 密钥等可用于账户接管的任何数据。

#### 2.1.2 供应链后门攻击演示

安全研究员 Jamieson O'Reilly 模拟了一次真实的供应链后门攻击：

- 创建了一个名为"What would Elon Do"的恶意技能
- 使用简单漏洞将下载量虚增至 4000+，使其成为下载量第一的技能
- 观察到来自 7 个不同国家的真实开发者在他们的机器上执行任意命令
- 有效载荷仅向攻击者服务器发送 ping 以证明执行发生，但故意排除了主机名、文件内容、凭据等数据

#### 2.1.3 大规模技能安全审计

Snyk 对 ClawdHub 市场的 **3,984 个技能**进行了安全分析，发现：

- **283 个技能（约 7.1%）** 存在关键安全漏洞
- 这些漏洞会在 LLM 上下文窗口和输出日志中以明文形式暴露敏感凭据

**后续措施**：OpenClaw 已与 VirusTotal 合作，扫描 SkillHub 上的技能以检测潜在风险。

### 2.2 永久性提示词注入威胁

提示词注入是 LLM 的固有特性，无法逃避。但在 OpenClaw 的背景下，问题被放大因为：

- 攻击面太大
- 任何人都可以发送消息或嵌入指令来攻击 agent

OpenClaw 是 Simon Willison 所提出的**"致命三要素"的完美候选**：

| 三要素 | OpenClaw 实际情况 |
|--------|-------------------|
| **访问您的私人数据** | 工具的核心目的之一 |
| **暴露于不可信内容** | 通过 WhatsApp、Telegram、邮件接收消息 |
| **外部通信能力** | 可用于数据泄露 |

由于 agent 在 WhatsApp、Telegram 上运行并读取电子邮件，**任何随机消息都是对拥有系统访问权限的 agent 的输入**。有动机的黑客可以轻易绕过 LLM 的原生防护栏。

### 2.3 被入侵的集成应用

OpenClaw 目前有 **50+ 个集成**，包括 Slack、Gmail、Teams、Trello、Perplexity 网络搜索等。

**问题**：每增加一个集成，就会增加潜在攻击面。

如果攻击者获得 OpenClaw 实例的访问权限，他们可以：

- 访问私人聊天
- 访问电子邮件
- 获取 API 密钥
-访问密码管理器
- 控制家庭自动化系统

**任何您授予 OpenClaw 访问权限的服务，如果 OpenClaw 被入侵，都会被攻陷。**

### 2.4 认证滥用与过度授权令牌

许多与集成相关的风险源于**认证处理不当和过度范围的令牌**。

OpenClaw 必须存储凭据，包括 API 密钥和 OAuth 访问/刷新令牌。根据 OpenClaw 文档，刷新令牌存储在本地 auth profile 文件中。

**风险**：由于许多部署优先考虑便利性（弱认证、暴露的网关、反向代理配置错误），从"互联网暴露"到"令牌盗窃"的路径可能非常短。

一旦令牌被窃取，攻击者无需欺骗模型。他们可以直接在 Slack 和 Gmail 中冒充您，拉取数据，发送消息，并在您的组织内横向移动。

### 2.5 内存污染问题

OpenClaw 的内存完全是 **Markdown 文件的集合**，没有任何机制阻止被入侵的 agent 重写自己的内存文件。

**后果**：

- 攻击者可以入侵 agent
- 用户永远不会察觉到任何异常
- agent 可以静默执行内存文件中指定的任务
- 攻击者的服务器可以静默泄露个人数据和凭据

**技能感染是急性的，而内存污染可以在您不知情的情况下毒化整个实例。**

### 2.6 大规模实例暴露

在热度高峰期，人们纷纷部署 OpenClaw 实例而未考虑安全后果：

- **Censys**：2025年1月27日至31日，发现约 **21,000 个**暴露的实例
- **BitSight**：2025年1月27日至2月8日，扫描发现 **30,000+ 个**易受攻击的 OpenClaw/Clawdbot/Moltbot 实例

#### 关键漏洞

初始版本的 ClawedBot 存在一个关键漏洞：来自 localhost 的任何流量都被视为合法流量（因为它可能是 bot 的所有者）。

**问题**：在大多数实际部署中，OpenClaw 位于同一台机器上的 nginx 或 Caddy 反向代理后面。每个连接都来自 127.0.0.1/localhost。因此，每个连接都被视为本地连接，**无需认证即可自动批准**。

此漏洞在发现后已快速修补。

## 三、OWASP Top 10 代理安全风险映射

Palo Alto Networks 将 OWASP Top 10 代理安全风险与 OpenClaw 进行了映射：

| OWASP 风险 | OpenClaw 实现 |
|-----------|---------------|
| **A01: 提示词注入（直接与间接）** | 网络搜索结果、消息、第三方技能注入 agent 执行的指令 |
| **A02: 不安全的代理工具调用** | 工具（bash、文件 I/O、邮件、消息）基于包含不可信内存源的推理进行调用 |
| **A03: 过度代理自主性** | 单个 agent 具有文件系统 root 访问权限、凭据访问权限和网络通信能力，没有权限边界或审批关卡 |
| **A04: 缺少人在环控制** | 无需审批即可执行破坏性操作（rm -rf、凭据使用、外部数据传输），即使受到旧的、不可信的内存影响 |
| **A05: 代理内存污染** | 所有内存未按来源区分。网络抓取、用户命令和第三方技能输出以相同方式存储，没有信任级别或过期机制 |
| **A06: 不安全的第三方集成** | 第三方"技能"以完全代理权限运行，可直接写入持久内存，没有沙箱隔离 |
| **A07: 不足的权限分离** | 单个 agent 处理不可信输入接收和高权限操作执行，共享内存访问 |
| **08: 供应链模型风险** | agent 使用上游 LLM，未验证微调数据或安全对齐 |
| **A09: 无界限的代理间操作** | OpenClaw 作为单一整体 agent 运行，但未来多代理版本可能启用不受约束的代理通信 |
| **A10: 缺乏运行时监控和防护** | 没有策略执行层在内存检索 → 推理 → 工具调用之间。没有内存访问模式或时间因果关系的异常检测 |

## 四、安全加固建议

### 4.1 容器化隔离环境

**绝对不要**在您的主要计算机上运行 OpenClaw，更不要使用 root 权限。

#### 本地部署（Docker 加固）

1. **使用旧游戏笔记本电脑**，在 Docker 容器中安装
2. **不要挂载整个主目录**。只给**一个工作目录**（例如 `/srv/openclaw/work`）
3. **使用操作系统权限**：以**单独用户**运行（例 `openclaw`），最小化文件访问，**默认没有 admin/sudo 权限**
4. **丢弃 Docker 权限**：在容器内以非 root 用户运行（`USER`），尽可能使用 `read_only: true` 文件系统，只挂载工作目录为可写
5. **永远不要 Docker socket**：不要将 `/var/run/docker.sock` 挂载到容器中
6. **丢弃 Linux capabilities**：将容器能力降至最低要求
7. **使用 Docker 默认的 seccomp 配置**

#### 云 VPS 部署

1. **网络方面：不暴露到公共网络**。将网关绑定到 `127.0.0.1`，仅通过 VPN 或私有隧道访问（WireGuard、Tailscale）
2. **防火墙保护**：只允许从您的 IP 或 VPN 范围 SSH，不向 `0.0.0.0` 开放 OpenClaw 端口
3. **窄化配置 trusted-proxy**：只信任来自实际代理 IP 的身份 header
4. **优先使用无 root 的 Docker**
5. **保持 seccomp 开启**
6. **制定令牌轮换计划**

### 4.2 独立的 OpenClaw 账户

将 OpenClaw 视为一个独立的实体。为它提供：

- 独立的 Gmail 账户
- 独立的日历
- 独立的每个集成服务
- 独立的 1Password 账户来存储凭据

这类似于拥有一个具有独立身份的个人助理，而不是一个自动化工具。

### 4.3 安全的集成

考虑使用 **Composio** 来添加安全的应用集成，而不是让 OpenClaw 直接存储 OAuth 令牌。Composio 提供：

- **托管 OAuth**：不在 agent 的工作目录中存储原始 OAuth 密钥
- **集中化 scope 和认证配置**：可以严格限制权限（例如，先只读 Gmail，然后添加发送权限）
- **更好的运营控制**：将凭据生命周期管理作为一等公民

### 4.4 最小权限访问

使用 Composio 的精细化 scope 控制：

- **按风险级别分离 agent**：一个用于"收件箱分类和起草"（只读 scope），另一个用于"执行操作"（写 scope）
- **时间限制 elevated 访问**：只在需要时临时授予写权限，然后撤销
- **按资源而非仅按操作进行 scope**：优先"此日历"而非"所有日历"
- **破坏性操作需要选择加入并审批**
- **像审计依赖一样审计 scope**

### 4.5 工具执行可观测性

使用 Composio 查看 agent 在应用集成方面的执行时间和内容。这使得追踪整个工具执行历史变得更加容易，能够理解问题发生的位置和方式。

## 五、TrustClaw：安全替代方案

Composio 构建了 **TrustClaw** 作为 OpenClaw 的安全替代方案，主要区别：

| 特性 | OpenClaw | TrustClaw |
|------|----------|-----------|
| **OAuth 管理** | 本地磁盘存储 | Composio 托管整个令牌生命周期 |
| **访问 scope** | 全局访问 | 可定义 agent 只能访问您允许的内容 |
| **代码执行** | 本地执行，风险高 | 远程沙箱执行 |
| **设置** | 复杂，需要手动配置 | 一键设置 |
| **Agent 运行时间** | 需要手动启动 | 24/7 运行，可安排任务 |
| **可观测性** | 有限 | 完整的执行视图 |

## 六、结论

尽管 AI 取得了巨大进步，但它仍然处于青春期。我们必须：

- 谨慎处理并用合理的防护栏约束它
- 假设它会被操纵
- 设计能够contain并从中恢复的系统

OpenClaw 是一个强大的产品，但它存在严重的安全漏洞。虽然一些问题（如社会工程和幻觉）是 LLM 固有的，需要整个行业来解决，但其他问题（如部署问题、应用集成和访问 scope 挑战）在很大程度上是可以解决的。
