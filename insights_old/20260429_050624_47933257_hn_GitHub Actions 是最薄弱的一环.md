## 核心洞察

### 1. 安全建议：使用 Commit SHA 而非标签

讨论的核心建议是：在 GitHub Actions 的 `uses:` 语句中使用完整的提交 SHA 而非版本标签。

- **原因**：标签可被恶意篡改，攻击者可推送恶意版本并使用相同标签名（如 `v5.2.3`）
- **例外**：对于知名官方 action（如 `actions/checkout`），标签相对安全
- **风险**：虽然 SHA 碰撞攻击在实践中几乎不可能，但通过 action 的 NPM 依赖进行供应链攻击的可能性更大

### 2. 传递依赖风险

即使使用 SHA 固定，你的 action 仍可能受到传递依赖攻击：
- 你的 action 引用的 composite action 可能使用可变标签
- GitHub 没有锁文件来保护传递依赖
- 解决方案：GitHub 即将推出 "Immutable Releases" 功能

### 3. 安全工具推荐

| 工具 | 用途 |
|------|------|
| **Renovate** | 自动将标签更新为 SHA，并保留版本注释 |
| **zizmor** | 安全审计工具，可检测 SHA pinned actions 的已知漏洞 |
| **hasp** | 运行时沙箱，强制 SHA 固定并验证注释与 SHA 一致性 |
| **ratchet** | 外部 Actions/Workflows 的 SHA 固定工具 |

### 4. 组织级别防护

- 可在组织层面强制只允许 SHA 固定的 actions
- 可设置白名单，但白名单并非完全可靠（如 Checkmarx KICS 曾被攻击）

### 5. 其他观点

- **YAML 编程问题**：有人认为 YAML 不适合编程，缺少调试工具
- **替代方案**：Dagger、WarpBuild、Buildkite 等
- **pull_request_target 安全风险**：被批评为"极其危险"，应被禁用
- **GitHub Actions 性能问题**：近期出现服务缓慢问题

---

## 总结

这次讨论反映了 GitHub Actions 用户对供应链安全的担忧。建议开发者在 `uses:` 语句中使用完整 commit SHA 以提高安全性，同时关注 GitHub 即将推出的 Immutable Releases 功能。社区也开发了多种工具来帮助缓解这些安全风险。
