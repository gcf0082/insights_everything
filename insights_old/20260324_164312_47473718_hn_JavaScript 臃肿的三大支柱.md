## 摘要

本文作者 James Garbutt 讨论了现代 JavaScript 生态系统中依赖树日益臃肿的问题，并将其根源归纳为三大支柱。作者是 e18e 社区的活跃成员，该社区致力于优化 JavaScript 依赖、清理冗余包。

---

## 三大支柱详解

### 支柱一：过时的运行时支持（Old runtime support）

**问题描述**：npm 生态中存在大量为极旧浏览器（如 IE6/7）或早期 Node.js 设计的"安全"工具包。

**原因**：
1. **支持 ES3 引擎**：部分开发者需要支持约15年前的旧环境，这些环境缺少 ES5 特性（如 `Array.prototype.forEach`、`Object.keys` 等）
2. **防止全局命名空间被修改**：部分维护者效仿 Node.js 的 "primordials" 模式，通过重新导出原生对象来避免全局被污染
3. **跨域值（Cross-realm values）**：例如在 iframe 和父窗口之间传递值时，`instanceof` 检查会失效，需要特殊处理

**问题**：绝大多数开发者使用的是近10年的运行时环境，根本不需要这些兼容性代码。这些小众兼容包却成了日常包的"必经之路"，所有人都为此付出了不必要的成本。

**典型示例**：
- `is-string`（可以用 `typeof` 替代）
- `hasown`（可以用 `Object.hasOwn` 替代）
- `math-intrinsics`（重新导出 `Math.*` 函数）

---

### 支柱二：原子化架构（Atomic architecture）

**问题描述**：社区曾流行将代码拆分成极细粒度的"原子"包，理论上是创建可复用的构建块。

**典型示例**：
- `arrify`：将值转换为数组（`Array.isArray(val) ? val : [val]`）
- `slash`：将文件路径中的反斜杠替换为正斜杠
- `path-key`：获取当前平台的 PATH 环境变量名
- `shebang-regex`：匹配 shebang 的正则（仅一行代码）

**问题**：
1. **单用途包**：这些原子包绝大多数只被一个包使用（如 `shebang-regex` 几乎只被 `shebang-command` 使用）
2. **重复**：同一包可能有多个版本共存于依赖树中（如 nuxt 项目中 `is-docker`、`is-wsl` 各有2个版本）
3. **供应链风险**：一个维护者的账户被攻击可能导致数百个小包被入侵

**结论**：这些代码本应内联，而非独立成包。内联使重复几乎免费，而独立成包则代价高昂。

---

### 支柱三：过时的 Ponyfill

**背景**：
- **Polyfill**：在全局环境注入缺失功能（会污染环境）
- **Ponyfill**：通过导入方式使用替代实现，不污染全局环境

**问题**：这些 ponyfill 曾经有用，但当特性已被所有主流引擎支持后，它们往往没有被移除。

**典型示例**：
- `globalthis`：提供 `globalThis`（2019年起广泛支持，每周下载4900万次）
- `indexof`：`Array.prototype.indexOf` 的 ponyfill（2010年起支持，每周下载230万次）
- `object.entries`：`Object.entries` 的 ponyfill（2017年起支持，每周下载3500万次）

---

## 解决方案

1. **使用 knip**：检测和移除未使用的依赖
2. **使用 e18e CLI**：分析可替换的依赖
   - 示例：`npx @e18e/cli analyze` 可检测如 `chalk` 等可用原生功能替代的包
3. **使用 npmgraph**：可视化依赖树，识别冗余分支
4. **参考 module-replacements 项目**：获取哪些包可用原生功能或更优替代品替换的信息

---

## 结论

整个 JavaScript 社区为一个非常小众的群体支付着不必要的代价——这些人需要极端的向后兼容性或特殊架构。作者认为应该反过来：这个小众群体应该使用他们专属的特殊技术栈，而其他人应该使用现代、轻量且被广泛支持的代码。
