# 洞察报告：Cloudflare 如何应对 "Copy Fail" Linux 漏洞

**洞察链接**：https://blog.cloudflare.com/copy-fail-linux-vulnerability-mitigation/

**发布日期**：2026-05-07

**报告生成时间**：2026-05-09

---

## 基本信息

- **漏洞名称**：Copy Fail（CVE-2026-31431）
- **漏洞类型**：Linux 内核本地权限提升漏洞
- **披露时间**：2026年4月29日
- **影响范围**：使用 algif_aead 内核模块的系统
- **客户影响**：无，Cloudflare 确认零影响

---

## 漏洞概述

### 技术背景

Copy Fail 漏洞存在于 Linux 内核的 `algif_aead` 模块中，该模块提供用户态程序访问内核加密 API 的能力。漏洞源于 2017 年的一项"原地操作"优化，该优化允许加密操作在相同内存区域进行，但缺乏边界检查。

### 漏洞原理

1. 攻击者利用 `AF_ALG` socket 家族与内核加密 API 交互
2. 通过 `splice()` 系统调用将目标文件的页缓存页面链接到 scatterlist
3. 在执行 `recvmsg()` 时，`authencesn` 包装器执行越界写入，写入 4 字节数据到目标偏移位置
4. 由于页缓存是全局共享的，修改 `/usr/bin/su` 等 setuid-root 二进制文件的页缓存即可实现权限提升

### 上游修复

上游修复（commit a664bf3d603d）回滚了 2017 年的原地操作优化，消除了该漏洞的利用条件。

### 检测覆盖验证

Cloudflare 的服务器运行行为检测，持续监控进程执行模式。该检测不依赖已知漏洞知识，而是监控跨集群的异常行为模式。在内部验证期间，检测平台在几分钟内标记了整个攻击链：脚本解释器 → 内核加密子系统 → 权限提升二进制文件。

### 威胁狩猎

安全团队假设在漏洞公开前可能已被利用，因此在公开前 48 小时内进行全集群日志搜索，寻找漏洞利用痕迹。分析结果：所有系统二进制文件未被篡改，加密哈希与已知良好包清单一致，未发现持久化机制，网络连接无异常。确认无恶意活动。

### 缓解措施

**方案一：移除模块**
简单删除 `algif_aead` 内核模块并禁止重新加载会影响使用内核加密 API 的软件，因此未采用。

**方案二：bpf-lsm**
使用 BPF Linux Security Module 程序在 `socket_bind` LSM hook 处进行控制：
1. 非 `AF_ALG` socket 允许通过
2. `AF_ALG` socket 检查调用二进制的白名单
3. 非白名单二进制拒绝绑定

通过 prometheus-ebpf-exporter 钩子 `socket()` syscall，跟踪整个集群中每个二进制文件的 `AF_ALG` 使用情况。结果确认只有已知内部服务是合法用户。

**方案三：内核补丁**
上游 backport 适用于主 LTS 分支后，内部自动化构建了带补丁的内核并在暂存数据中心测试，然后通过重新启动自动化全面部署。

## 后续改进

1. **增强内核 API 依赖的可见性**：审查生产服务中的内核子系统使用情况，以便快速缓解漏洞利用而不中断服务
2. **改进运行时缓解**：增强 bpf-lsm 工具，包括更快的部署、更好的操作手册、更好的日志和可见性
3. **减少 Linux 内核攻击面**：审查和审计内核配置，主动识别未使用的模块或功能以便从构建中完全移除

---

## 结论

尽管 Copy Fail 漏洞披露时 Cloudflare 的主内核线尚未包含上游补丁（该补丁已在上游主分支存在一个月），但团队在几小时内完成了带补丁内核的部署。在此期间，bpf-lsm 提供了无需重启的精确缓解。每个集群机器均受到补丁内核或 bpf-lsm 程序的保护。整个事件期间无客户影响。

## 关键要点

1. **响应速度**：从漏洞披露到缓解完成仅用约 24 小时

3. **行为检测有效性**：现有检测在无需签名更新或规则更改的情况下分钟内捕获漏洞利用
