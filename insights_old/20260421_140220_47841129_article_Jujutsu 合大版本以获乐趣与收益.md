## 文章概述

本文是写给中级Jujutsu用户以及对Jujutsu感兴趣的Git用户。作者分享了他在日常开发中使用的"megamerge"（巨型合并）工作流程，这是一种强大且高效的多分支管理方式。

## 核心概念

### 1. 合并提交的本质

- 合并提交本质上就是一个普通的commit，只是它有多个父提交
- 合并提交可以有超过两个父提交，这种超过两个父提交的合并被称为"章鱼合并"(octopus merge)
- 章鱼合并是megamerge工作流程的核心

### 2. 什么是Megamerge

Megamerge是一个 octopus merge commit，它作为你关心所有工作分支的子提交。包含：
- bugfix分支
- 功能分支
- 等待PR的分支
- 需要与之协同工作的他人分支
- 本地环境设置分支
- 私有提交

**关键原则：不推送megamerge，只推送它包含的分支。**

## Megamerge的优势

1. **始终在组合工作上工作**：如果工作副本能编译运行，就知道所有工作都能正常交互
2. **减少合并冲突担忧**：由于Jujutsu原生支持冲突处理，且随时都在合并，不会再有意外的合并冲突
3. **任务切换摩擦减少**：无需在VCS中切换任务，直接编辑即可
4. **保持分支更新更容易**：一条rebase命令即可更新整个megamerge

## 如何创建Megamerge

```bash
# 创建一个新的空提交，包含所有想要的分支作为父提交
jj new x y z
jj commit --message "megamerge"
```

## 如何提交更改

### 更改归属现有提交

- 使用 `squash --to` 将更改移入下游提交
- 使用 `squash --interactive` 交互式选择要移动的特定部分
- 使用 `jj absorb` 自动识别并自动向下游 mutable commit 压缩

```bash
# 压缩整个WIP commit（默认 --from @）
jj squash --to x --from y

# 交互式压缩部分内容
jj squash --to x --from y --interactive

# 自动吸收
jj absorb --from x
```

### 更改为新提交

```bash
jj commit
jj rebase --revision x --after y --before megamerge
jj bookmark move --from y --to x
```

## 推荐的配置别名

```toml
[revset-aliases]
# 返回离"to"最近的合并提交
"closest_merge(to)" = "heads(::to & merges())"

[aliases]
# 将指定revset插入megamerge下方
stack = ["rebase", "--after", "trunk()", "--before", "closest_merge(@)", "--revision"]

# 将整个栈（megamerge之后的所有提交）包含进来
stage = ["stack", "closest_merge(@).. ~ empty()"]

# 将更改rebase到trunk()
restack = ["rebase", "--onto", "trunk()", "--source", "roots(trunk()..) & mutable()"]
```

## 如何保持更新

使用 `restack` 别名可以只 rebase 你实际控制的提交，保留你不控制的分支和基于他人分支的工作：

```bash
jj restack
```

这让你可以轻松处理包含多个贡献者的巨型九重混合megamerges。

## 总结

Megamerge工作流程让你能够同时工作在多条不同的工作流上，几乎不需要 effort 就能在任务之间切换。虽然初次看起来可能有些吓人，但一旦尝试，很可能会爱上这种方式。

需要记住：megamerge 本身不需要推送到远程，它是让自己看到全貌的便捷方式，仍然需要像往常一样单独发布各个分支。
