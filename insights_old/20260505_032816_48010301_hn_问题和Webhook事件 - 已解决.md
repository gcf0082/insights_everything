## 摘要

本报告讨论了 GitHub 近期出现的 Issues 和 Webhooks 服务问题及其解决方案。事件的根源在于 AI 代理编程（agentic coding）的爆发式增长导致 GitHub 基础设施面临前所未有的压力。GitHub 首席运营官（COO）在 2026年4月3日公布了令人震惊的使用数据，显示平台活动正在急速增长。

## 关键数据

### GitHub 提交量增长
- **2025年全年**: 10 亿次提交
- **现在**: 每周 2.75 亿次提交
- **预计2026年**: 140 亿次（如果保持线性增长，实际可能更快）

### GitHub Actions 使用量增长
- **2023年**: 每周 5 亿分钟
- **2025年**: 每周 10 亿分钟
- **2026年本周**: 每周 21 亿分钟

## 主要讨论观点

### 1. AI 代理编程是主要驱动力

评论者指出，2025年12月被认为是代理编程的重大转折点，得益于 AI 工具链和 LLM 的改进。用户现在可以运行多个代理进行实验，只需偶尔检查进度。

### 2. 基础设施扩展挑战

- GitHub 的技术栈主要基于 Ruby on Rails 单体架构
- 需要将性能敏感代码从 Ruby 迁移到 Go
- Webhooks 服务正在从 MySQL 迁移到其他后端
- 向 Azure 的迁移增加了复杂性

### 3. 免费午餐时代结束

评论者认为，GitHub 曾经通过补贴学生和新用户来获取市场份额，但现在这种模式无法持续。随着 AI 产品的激增，平台面临成本压力，可能需要：

- 限制免费层使用
- 提高付费层价格
- 实施更严格的速率限制

### 4. 对手选择的机会

一些评论者指出，GitHub 的可靠性下降可能为竞争对手（如 GitLab）带来机会。数据显示 GitLab 没有出现类似的扩展问题。

### 5. 企业客户影响有限

有评论指出，大多数企业客户使用 GitHub Enterprise Server (GHES)，这是本地部署版本，不受 云服务 问题影响。这可能是 GitHub 收入的主要来源。

## 技术分析

### 效率问题

- 主要仓库页面和 PR 列表使用搜索查询实现，而不是专门的 API
- 搜索后端失败时，PR 无法加载
- GitHub Actions 缓存性能不佳，经常需要重新下载数百 MB

### 架构限制

- Ruby 不是大规模应用的理想语言选择
- 需要将关键服务迁移到更高效的语言（如 Go 或 Rust）
- MySQL 成为 webhooks 服务的瓶颈

## 社区反应

1. **担忧与质疑**: 用户对 GitHub 可靠性下降表示担忧
2. **市场竞争**: 认为 GitLab 等竞争对手可能从中受益
3. **批评与建议**: 对 GitHub 的定价策略和技术决策提出批评
4. **理解与支持**: 部分用户理解 GitHub 面临的挑战

## 结论

GitHub 面临的挑战是多方面的：
- AI 代理编程的爆发式增长带来了前所未有的基础设施压力
- 从 Ruby 到其他技术的迁移正在进行但需要时间
- 免费层的成本压力可能导致未来服务调整
- 微软收购后的 Azure 迁移增加了复杂性

这些问题需要 GitHub 在技术创新和成本控制之间找到平衡。
