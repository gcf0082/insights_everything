# Docker Engine containerd 镜像存储洞察报告

**洞察链接**: https://docs.docker.com/engine/storage/containerd

**来源**: Docker 官方文档

**日期**: 2026-05-06

---

## 概述

containerd 镜像存储是 Docker Engine 29.0 及更高版本在新安装中的默认存储后端。如果从早期版本升级，守护进程将继续使用传统的 graph drivers (overlay2)，直到您手动启用 containerd 镜像存储。

containerd 作为行业标准的容器运行时，使用 snapshotters（快照器）而非传统存储驱动来存储镜像和容器数据。

> **注意**: containerd 镜像存储在使用用户命名空间重映射 (userns-remap) 时不可用。

---

## 为什么使用 containerd 镜像存储

containerd 镜像存储使用 snapshotters 管理镜像层在文件系统上的存储和访问方式，这与传统的 overlay2 等 graph drivers 不同。

containerd 镜像存储支持以下功能：

1. **构建和存储多平台镜像**: 传统存储驱动需要外部构建器才能构建多平台镜像，containerd 支持本地构建
2. **支持证明 (attestations) 的镜像**: 包括 provenance 和 SBOM，使用传统存储不支持的镜像索引
3. **运行 Wasm 容器**: 支持 WebAssembly 工作负载
4. **高级快照器**: containerd 支持可插拔的快照器，提供以下功能：
   - 懒拉取镜像 (stargz)
   - 点对点镜像分发 (nydus, dragonfly)

对于大多数用户来说，切换到 containerd 镜像存储是透明的，存储后端改变但工作流程保持不变。

---

## 磁盘空间使用

containerd 镜像存储比传统存储驱动使用更多磁盘空间来存储相同的镜像。这是因为 containerd 同时存储压缩和非压缩格式的镜像，而传统驱动只存储非压缩层。

拉取镜像时，containerd 保留压缩层（从 registry 接收的）并将其解压到磁盘。这种双重存储意味着每个层占用更多空间。压缩格式可以加快拉取和推送速度，但需要额外的磁盘容量。

当多个镜像共享相同基础层时，这种差异尤其明显。使用传统存储驱动时，共享基础层在本地只存储一次，支持镜像复用。而 containerd 中，每个镜像都存储共享层的压缩版本，尽管通过快照器仍然可以对非压缩层进行去重。压缩存储的开销与使用这些层的镜像数量成正比。

**如果磁盘空间有限，建议**：
- 定期使用 `docker image prune` 清理未使用的镜像
- 使用 `docker system df` 监控磁盘使用情况
- 配置数据目录到空间充足的分区

## 在 Docker Engine 上启用 containerd 镜像存储

如果从早期 Docker Engine 版本升级，需要手动启用 containerd 镜像存储。

> **重要**: 切换存储后端会暂时隐藏使用其他后端创建的镜像和容器。数据保留在磁盘上。如果要再次访问旧镜像，请切换回之前的存储配置。

将以下配置添加到 `/etc/docker/daemon.json` 文件：

```json
{
  "features": {
    "containerd-snapshotter": true
  }
}
```

保存文件并重启守护进程：

```console
$ sudo systemctl restart docker
```

重启后，验证是否正在使用 containerd 镜像存储：

```console
$ docker info -f '{{ .DriverStatus }}'
[[driver-type io.containerd.snapshotter.v1]]
```

Docker Engine 默认使用 `overlayfs` containerd 快照器。

> **注意**: 启用 containerd 镜像存储后，现有的 overlay2 镜像和容器保留在磁盘上但会被隐藏。如果切换回 overlay2，它们会重新出现。要在 containerd 镜像存储中使用现有镜像，请先将它们推送到 registry 或使用 `docker save` 导出。

## 实验性自动迁移

Docker Engine 包含一个实验性功能，可以在某些条件下自动切换到 containerd 镜像存储。**此功能是实验性的**。提供该功能是为了让想要测试的用户使用，但建议采用[从头开始启用](#在-docker-Engine-上启用-containerd-镜像存储)的方式。

> **警告**: 自动迁移功能是实验性的，可能无法在所有场景中可靠工作。使用前请创建备份。

要启用自动迁移，请在 `/etc/docker/daemon.json` 中添加 `containerd-migration` 功能：

```json
{
  "features": {
    "containerd-migration": true
  }
}
```

您还可以设置 `DOCKER_MIGRATE_SNAPSHOTTER_THRESHOLD` 环境变量，使守护进程在满足以下条件时自动切换：
- 没有运行或停止的容器
- 镜像数量低于或等于阈值

对于 systemd：

```console
$ sudo systemctl edit docker.service
```

添加：

```ini
[Service]
Environment="DOCKER_MIGRATE_SNAPSHOTTER_THRESHOLD=5"
```

如果您没有运行或停止的容器且镜像数量不超过 5 个，守护进程会在重启时切换到 containerd 镜像存储。overlay2 数据保留在磁盘上但会被隐藏。

---

## 总结

containerd 镜像存储是 Docker Engine 的现代存储后端，提供多平台镜像支持、Wasm 容器运行、高级快照器等功能。尽管会占用更多磁盘空间，但其优势在于更灵活的功能和更好的兼容性。Docker 官方建议新安装直接使用 containerd 镜像存储，现有用户可通过手动配置或实验性自动迁移功能进行切换。
