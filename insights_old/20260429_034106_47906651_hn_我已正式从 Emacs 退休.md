## 主题概述

这是一篇关于 Emacs 长期用户 Chris Wellons（别名 skeeto）正式宣布从 Emacs 退休的讨论帖。Chris 是知名的 Emacs 开发者，创建了 Elfeed（RSS 阅读器）等热门插件。帖子引发了对 Emacs 社区、LLM 对编程的影响、以及软件工程未来的广泛讨论。

## 主要讨论内容

### 1. Emacs 社区的损失

多位用户表达了对 Chris 离开 Emacs 的惋惜：

- **farfatched**: "Emacs 社区的巨大损失！emacs-aio 很棒。"
- 提到 Chris 在 GitHub 上清理了自己的 dotfiles，不再使用 Mutt、自建邮件服务器等工具
- 转而使用 KDE on Wayland，放弃了之前的 Openbox、Tridactyl、Xorg、xterm 等配置

### 2. LLM 如何改变工作方式

这是讨论最热烈的方向。部分用户分享了 LLM 时代的新体验：

- **iLemming**: LLM 让他与 Emacs 的关系更紧密了，因为 Emacs 本质上是一个 Lisp REPL，内置编辑器。当给 LLM 一个闭环系统，它可以在 live REPL 中评估代码并观察结果，从而停止猜测，开始根据实证推理。

- **mark_l_watson**: 他的 .emacs 配置有所改进，并编写了自己的 Emacs 编程助手。

- **minikomi**: 使用 gptel 和 org mode 进行深入的开发工作，这是一种非常不同的交互模式。

- **devin**: Clojure 开发中，将应用连接到 live REPL 为 LLM 提供了极好的反馈循环。

- **iLemming 的具体设置**: 使用 gptel 和 ECA 作为主要工具。MCP 都是用 Clojure 编写，因为给 AI 一个 Lisp REPL 更有意义。

### 3. 关于"摩擦"和"维护"的讨论

- **squigz**: 质疑所谓"强大用户工具固有的摩擦"，他的配置多年基本不变。

- **pavon**: 当经常使用其他电脑时，高度定制的设置会带来更多摩擦。分享了自己从 Dvorak + Emacs 转换的经历。

- **viksit**: "大多数人是无法很好设置这种重度定制终端环境的。"

### 4. Emacs vs Vim 的讨论

- **wwarner**: 他的 Emacs 使用方式非常 Vim 化，尝试过几次切换。Vim 确实更快，光标定位更简单直观，但仍有一些功能差距。

- **jb1991**: 使用 Emacs 多年，后来转向 Vim 或其他带 Vim 模式的编辑器。个人认为实时编辑和编写代码的总体过程在 Emacs 中更高效。

### 5. 对 AI 编程的态度分歧

#### 支持方：

- **bitwize**: "AI 是软件开发的必要条件，这是不可谈判的。生产力提升太大了。100% 人工编写的代码时代已经结束。"

- **rirze**: AI 相当于一个没有很好持久记忆的初级开发者。使用 AI 的开发者是在审查、指导并验证 AI 的工作，就像对待初级开发者一样。

- **jesse_dot_id**: "AI 泡沫破裂"是不可能的。模型已经训练完成，工具已经构建。没有一个行业会在效率面前顾全艺术性。

#### 反对方：

- **lowsong**: 对人们"陷入 AI 精神病"感到非常不安。认为"气泡破裂后"人们将不得不面对他们被骗了。

- **anthk**: "他们需要更好的代码分析器、词法分析器等。而 LLM 恰恰相反，因为它们永远无法给出简洁的答案。"

- **otabdeveloper4**: "我是不会用任何 vibe-coded 的东西，所有这些应用都是垃圾。"

### 6. 关于生活与工具的平衡

- **bitwize**: "我们的生活远不止计算环境。通过放弃一点对计算环境的控制，我们解放我们的大脑去关注生活中其他事情：亲人、宠物、园艺、家庭维护、其他爱好和运动..."

- **skydhash**: "对我的工具保持稳定和最小摩擦能带来内心平静。但如果你的电脑使用方式像烤面包机，那你就不会那么在意强大功能。"

### 7. 替代工具的推荐

- **tptacek**: Lazygit 是最接近 Magit 的替代品。

- **cmrdporcupine**: 推荐 gitu (GitHub)。

- **sirn**: 分享了如何将 Magit 作为独立程序运行的配置方法。

## 洞察总结

1. **Emacs 社区的重要人物离去**: Chris Wellons (skeeto) 这样的核心开发者转向新工具，反映了技术生态的变迁。

2. **LLM 与 Emacs 的协同效应**: 许多用户认为 Emacs 的 Lisp REPL 特性使其成为 LLM 辅助编程的理想平台，两者的结合提供了强大的即时反馈循环。

3. **"摩擦"的相对性**: 对于高度定制的工具，摩擦因使用场景而异。在多机器环境下，标准化可能更重要。

4. **软件工程未来的分歧**: 人们对 AI 编程的态度截然不同——有人视为革命性提升，有人担忧其局限性和长期影响。

5. **工具与生活平衡**: 有用户主张放弃部分控制以换取生活其他方面的投入，也有人坚持将计算环境视为"创意实验室"。
