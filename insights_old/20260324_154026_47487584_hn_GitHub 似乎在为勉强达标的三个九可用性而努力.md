# GitHub 正常运行时间洞察报告

## 基本信息

- **洞察链接**: https://news.ycombinator.com/item?id=47487584
- **标题**: GitHub appears to be struggling with measly three nines availability
- **来源**: Hacker News
- **发布时间**: 21小时前
- **得分**: 450 points
- **评论数**: 232 comments
- **原始链接**: https://www.theregister.com/2026/02/10/github_outages/

## 核心发现

### GitHub 正常运行时间现状

根据第三方状态追踪网站 mrshu.github.io/github-statuses 的数据：

- **平台整体可用性**: 90.21%（仅一个"9"）
- **API服务**: 99.69%（两个"9"）
- **Git服务**: 98.98%（两个"9"）
- **Copilot**: 约一个"9"
- **Actions**: 约一个"9"

GitHub的企业SLA承诺为99.9%（三个"9"），但在过去90天内，没有任何单一服务达到此标准。

### 社区反应分析

1. **对正常运行时间计算方式的争议**
   - 有评论者认为不应将任意功能故障都算作"GitHub宕机"
   - 另一方指出，即使是核心服务（API、Git、Actions）也未能达到三个"9"

2. **对企业级客户的影响**
   - 大型企业客户通常会就有SLA违约进行谈判
   - 赔偿通常是服务积分，用于未来消费，而非直接退款
   - GitHub Enterprise基础套餐中型公司需支付每年100万美元以上

3. **关于"三个9"的理解**
   - 99.9%正常运行时间意味着每年最多8.76小时停机时间
   - 90.21%意味着每年约35.73小时停机
   - 但实际影响更大，因为故障分散在多个短暂时段，导致持续的构建和重试问题

### 技术和业务层面的讨论

1. **Azure迁移问题**
   - 2025年GitHub宣布将基础设施迁移至Azure
   - CTO曾承诺"可用性是首要任务"
   - 迁移似乎导致了可靠性问题

2. **AI编码工具带来的压力**
   - AI代理可以瞬间生成大量提交和PR
   - GitHub架构未针对此场景设计
   - 流量急剧增长

3. **功能退化问题**
   - PR页面需要多次点击才能展开所有上下文
   - 浏览器内搜索功能损坏
   - 用户体验下降

4. **安全问题**
   - GitHub Actions存在安全漏洞
   - 可变引用被滥用导致CI攻击
   - 社区多年呼吁但未获修复

### 不同观点

- **乐观派**: 对于免费服务来说，87小时/年的停机可以接受
- **悲观派**: 企业客户支付高额费用，理应获得更高可用性
- **中间派**: 应该区分核心功能（Git、API、Actions）和附加功能（Copilot）的可用性

## 结论

GitHub目前面临严重的可用性问题，不仅未能达到其承诺的99.9% SLA标准，实际上多个核心服务的可用性仅维持在90-99%之间。这一问题可能与Azure迁移、AI工具带来的流量激增以及持续的功能开发和安全技术债务有关。

对于企业用户而言，建议：
1. 明确区分核心功能与附加功能的依赖
2. 建立内部应急预案
3. 考虑自托管方案作为备选
