## 文章摘要

作者分享了其为家庭构建共享汽车应用程序 OurCar 的经历和学到的教训。

### 背景问题

- 家庭成员共用一辆车，需要跟踪：谁正在用车、车辆位置、加油记录、里程表读数
- 手动记录太不可靠且耗时

### 技术方案

- 使用 Flutter 开发跨平台应用
- 使用 LLM (ChatGPT) 辅助命名和生成管理后台流程
- 团队信任为基础，不强制自动化追踪

### 学到的重要教训

1. **小规模项目的黄金法则**：信任是核心——如果家庭成员之间不信任，应用的帮助有限
2. **技术选型**：Flutter 适合跨平台，但从 UI 角度 HTML/PWA 可能更简单
3. **LLM 的作用**：LLM 可以弥补技能缺口，但不适合所有场景
4. **持续迭代**：实际项目的摩擦往往来自"谁最后更新了共享状态"

## 评论热点

### 1. 关于技术实现细节

- **OBD-II 读取器**：有评论建议使用蓝牙 OBD  dongle 读取车辆数据（油量、里程等）
- **PDF vs CSV**：数据分享格式的讨论，PDF 因手机原生支持更方便
- **AirTag 追踪**：有人建议用 AirTag 追踪车辆位置，但作者因隐私要求未采用

### 2. 关于解决方案的简化

- **"过度工程化"争议**：有评论认为可以用纸笔解决，但作者认为 app 更适合实际使用场景
- **LLM 生成应用**：很多人分享用 LLM 构建小型家庭应用的经验，认为这是 LLM 的最佳用例

### 3. 关于用户体验

- **共享状态问题**：核心摩擦不在于" gauge 显示什么"，而在于"谁最后更新了"
- **部分参与者**：如果有人不遵守规则，非正式的记录会悄悄产生分歧

### 4. 其他有价值的建议

- 用 LLM 逆向工程汽车 APP 的 API
- SMS/Telegram 机器人作为低成本方案
- NFC 贴纸配合 PWA

---

## 核心要点

1. **项目源于真实需求**：解决家庭用车共享的实际痛点
2. **技术是手段非目的**：选择 Flutter 是为了更好的体验，但也面临跨平台挑战
3. **信任是基础**：应用帮助的是愿意协作的人，无法强制不负责任的用户
4. **小规模项目的价值**：低风险、高反馈循环，适合学习和实验新技术
5. **LLM 是很好的辅助工具**：但不能替代所有开发工作
