## 内容摘要

这篇文章探讨了 C/C++ 中一个经典的未定义行为（Undefined Behavior, UB）问题：**表达式 `a = a++ + ++a` 的结果是什么？**

### 核心发现

由于两处未定义行为，该表达式可能产生三个不同的正确结果：**11、12 或 13**。

### 第一个未定义行为：操作数求值顺序

不确定哪个 `a` 会先被读取（从内存读取到寄存器）：

- **情况1**：先读取第一个 `a`，再执行前置递增，然后读取第二个 `a`
  - `a = 5 + 6 = 11`

- **情况2**：先执行前置递增，存储结果后再读取 `a`
  - `a = 6 + 6 = 12`

### 第二个未定义行为：后置递增的时机

`a++` 的递增操作何时生效：

- **情况1**：后置递增"消失"——递增写入内存后被覆盖
  - 结果：11 或 12

- **情况2**：后置递增延迟到最后执行
  - 结果：12 或 13

### 组合结果汇总

| 组合 | 结果 |
|------|------|
| 情况1 + 情况1 | 11 |
| 情况2 + 情况1 | 12 |
| 情况1 + 情况2 | 12 |
| 情况2 + 情况2 | 13 |

## 实验结果

文章收集了多种编译器下的测试结果，验证了三个结果（11、12、13）都确实存在：

| 编译器 | 版本 | a++ + ++a 结果 |
|--------|------|---------------|
| gcc | 2.95 - 4.6.0 | 12-13 |
| clang | 2.8, 1.6 | 11-12 |
| PHP | 5.2.10 | 11-12 |
| Java | 1.6.0 | 11-12 |
| C# | 2.0, 4.0 | 11-12 |
| Microsoft C/C++ | 16.00.30319.01 | 12-13 |
| Intel C++ | 12.0.1.127 | 12-13 |
| Keil C | 9.02 | 11-12 |
| SDCC | 3.0.1 | 11-12 |
| Rpi5 gcc | 14.2.0 | 11 |

## 关键结论

1. **C/C++ 中此类表达式结果是未定义的**，不同编译器可能产生不同结果
2. **Java 和 C# 有明确定义**：它们保证从左到右求值，结果分别为 11
3. **实际开发中应避免**在表达式中对同一变量同时进行自增和自减操作
4. 结果可能受**编译器版本和优化级别**影响

## 测试代码（附录）

```c
#include <stdio.h>

int main(void) {
    int a = 5, b = 5, c = 5, d = 5, e = 5, f = 5;

    a = a++ + a++;
    printf("%i\n", a);  // Code 1

    b = b++ + ++b;
    printf("%i\n", b);  // Code 2

    c = ++c + c++;
    printf("%i\n", c);  // Code 3

    d = ++d + ++d;
    printf("%i\n", d);  // Code 4

    e = e++;
    printf("%i\n", e);  // Code 5

    f = f + ++f;
    printf("%i\n", f);  // Code 6

    return 0;
}
```
