## 核心观点

Windows API（Win32）虽然设计上并非为了跨平台，但已成为事实上的跨平台桌面 API。通过 WINE、Crossover 和 Proton 等兼容层，Windows 可执行程序可以在 Linux 和 macOS 上运行，这使得 Windows API 实际上拥有了接近 100% 的桌面操作系统市场份额覆盖率。

---

## 背景

### Windows API 的历史演进

- **早期 Windows**：从零开始编写，未考虑 POSIX 兼容性，作为 GUI 层构建在 MS-DOS 内核之上
- **第一个 SDK**：仅包含两个头文件（STYLE.H 和 WINDOWS.H），WINDOWS.H 仅 2000 多行代码
- **Win32 时代**：微软重写 API 以支持 32 位计算，WINDOWS.H 不再是单一头文件
- **Windows NT**：微软重写 Windows，API 保持不变，扩展支持宽字符和 Unicode

### 市场地位

- Windows 桌面操作系统市场份额超过 80%
- 微软构建了 Visual Studio（强大的 IDE）和 MSDN（全面的 API 文档）
- 为独立软件供应商（ISV）提供了良好的开发环境

---

## 跨平台实现方式

### WINE 项目

WINE（Wine Is Not an Emulator）是一个兼容层，可以加载 Windows 可执行文件并将 Windows API 调用重定向到 Linux 或 macOS 系统调用。大多数 Windows 程序可以通过 WINE 在非 Windows 系统上运行。

### 商业实现

- **Codeweavers**：将 WINE 包装成商业产品 Crossover
- **Proton**：Valve 基于 WINE 为 Linux 游戏开发的解决方案，被 Steam 采用

### 游戏兼容性

Proton 实现了极高的游戏兼容性，许多原本仅支持 Windows 的游戏可以在 Linux 上运行。

---

## 关键洞察

### API 成功的新定义

传统观点认为跨平台 API 应该能够编写一次代码在多处运行。但 Windows API 走出了一条不同的成功之路：

> "在90年代和2000年代初期，Java 被营销为一种可以在多个平台运行 bytecode 的平台。但实际上，对于非 x86 平台才是如此。然而，在大多数情况下，Portable Executable（PE 可执行格式）是'可移植的'，因为它可以无需修改和重新编译即可执行。现在，您可以在 Linux 和 macOS 上运行最初仅针对 Windows 编写的桌面应用程序。即使 Windows 的市场份额只有 80%，Windows API 却拥有 100% 的市场覆盖率，因为它可以在所有三个主流桌面操作系统上运行。"

### 无心插柳的标准化

- Windows API 和可执行文件格式并非通过任何标准组织强制推行
- 微软甚至曾阻止将 Windows API 标准化为 ISO
- 这种采用完全是出于必要性和实用性

## Hacker News 社区讨论摘要

### 赞同观点

1. **反向采用**：Linux 实际上采用了 Win32，因为它们确实有效。Linux 用户空间设计一团糟，以至于在 Linux 上运行游戏的最佳 API 是 Win32。

2. **用户体验优先**：微软优化的是终端用户想要的东西——稳定性、向后兼容性。代码可以跨越数十年无需重新编译即可运行。

3. **市场驱动成功**：成功的路径是通过市场营销、市场份额、应用 dominant（吸引开发者）和极致的向后兼容性。用户不关心代码的优雅性，他们关心程序能持续运行。

4. **向后兼容性典范**：一位用户写道："我在 1995 年写的 Windows 程序至今仍能运行。它们在所有版本的 Windows 上运行，无需重新编译。"

### 质疑观点

1. **成本问题**：使 Windows API ��其他平台上运行花费了数百名开发者、多年的努力和大型公司的支持，这是巨大的工程代价。

2. **如何定义成功**：一个需要巨大努力才能实现跨平台的技术，能说是成功的跨平台 API 吗？

3. **技术 elegance**：技术社区追求的是技术卓越和优雅，而市场需要的是不同的东西。

## 结论

Windows API 的成功案例提供了一个重要的商业启示：

- **技术成功 vs 市场成功**：一个技术上可能不优雅的 API，可以通过市场策略和极致的向后兼容性实现事实上的跨平台
- **用户需求导向**：用户真正想要的是稳定性和持续可用性，而非技术上的 elegant
- **生态系统力量**：通过吸引开发者建立应用生态，即使不原生支持其他平台也能通过第三方兼容层实现跨平台

Windows API 的成功不是来自于代码设计，而是来自于微软对 ISV 的长期照顾、对向后兼容性的坚持，以及通过市场力量建立的生态系统。
