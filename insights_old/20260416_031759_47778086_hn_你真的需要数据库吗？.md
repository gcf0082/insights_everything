## 核心观点

这篇文章探讨了是否可以用平面文件（flat files）替代传统数据库来存储应用数据。作者认为对于许多应用来说，直接使用文件系统可能比使用数据库更简单、更高效。

## 主要讨论要点

### 1. 数据库的必要性

- **原子性需求**：如果需要任何形式的原子性操作，就需要数据库。直接在文件系统上实现原子写入非常脆弱。
- **数据损坏风险**：断电后文件可能变成空文件。建议：写入临时文件→fsync文件→fsync目录→重命名覆盖原文件。
- **并发写入**：多进程同时写入时，文件系统方案需要处理锁和并发问题，容易出现问题。

### 2. 使用文件系统的风险

- 文件系统操作看似简单，但实际上需要处理：
  - 原子性保证
  - 崩溃恢复
  - 并发控制
  - 数据一致性
- 许多人最终会重新实现数据库的功能，只是以更低质量、测试不足的方式

### 3. SQLite 的优势

- 高度可靠、经过充分测试
- 性能实际上优于直接的文件读写
- 避免重复造轮子
- 数据存储不是应用的独特价值所在，应该外包给专业解决方案

### 4. NoSQL vs SQL 的讨论

- **NoSQL 的适用场景**：大规模场景下 ACID 成为瓶颈时；数据没有固定模式时
- **常见误区**：许多团队从 DynamoDB 开始，最终在应用层重新实现关系型数据库
- **建议**：大多数应用场景应优先使用 PostgreSQL，它支持 JSONB，可以处理非结构化数据

### 5. YAGNI 原则

- "你不会需要它"（YAGNI）是重要的软件工程原则
- 等到真正需要数据库时再迁移到 SQLite 通常不晚
- 简单方案失败时再根据实际需求重新设计

## 社区共识

1. 对于简单场景，平面文件可以工作，但规模扩大后会遇到问题
2. 使用 SQLite 通常是更好的权衡，避免了自行处理复杂的数据存储问题
3. 大多数应用的性能瓶颈不在数据库类型，而在于代码质量和系统设计
4. PostgreSQL 已成为一个非常灵活的选择，可垂直扩展到极高水平

## 结论

虽然这篇文章提出了一个有趣的观点——对于某些简单的应用，文件系统可能就足够了——但社区的反馈表明，SQLite 提供了更好的性价比。除非有充分的性能理由，否则使用成熟的数据库解决方案是更稳妥的选择。
