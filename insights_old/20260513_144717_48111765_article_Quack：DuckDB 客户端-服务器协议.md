## 摘要

DuckDB 实例现在可以使用 Quack 远程协议相互通信。这使得 DuckDB 可以运行在客户端-服务器架构中，支持多个并发写入器。Quack 易于设置，构建在 HTTP 等成熟技术之上，性能优异，能够支持从批量操作到小规模事务的各种工作负载。

---

## 背景：数据库架构

数据库早期没有"客户端"和"服务器"的区别，整个数据库运行在单台计算机上。1980年代，Sybase 首次引入了数据库"服务器"和"客户端"的概念。SQLite（2000年）和 DuckDB（2019年）代表了不同的声音，采用进程内架构，无需客户端/服务器协议，只有低级 API 调用。这种架构非常适合交互式使用场景，如数据科学中的 Jupyter 笔记本。但对于多个进程同时修改同一个数据库文件的场景，效果不佳。

存在一些变通方案：自定义 RPC 解决方案、使用 Arrow Flight SQL 协议的第三方项目（如 GizmoSQL）、MotherDuck 的自定义协议、或切换到 PostgreSQL 等传统数据库。

## Quack 协议介绍

### 基本使用

两个 DuckDB 实例可以通过 Quack 协议相互通信。一个实例充当服务器，另一个充当客户端。

**DuckDB #1 (服务器)**:
```sql
INSTALL quack FROM core_nightly;
LOAD quack;
CALL quack_serve('quack:localhost', token = 'super_secret');
CREATE TABLE hello AS FROM VALUES ('world') v(s);
```

**DuckDB #2 (客户端)**:
```sql
INSTALL quack FROM core_nightly;
LOAD quack;
CREATE SECRET (TYPE quack, TOKEN 'super_secret');
ATTACH 'quack:localhost' AS remote;
FROM remote.hello;
```

还可以使用 `remote.query()` 函数将整个查询发送到远程执行。

## 协议设计

### 基于 HTTP
Quack 构建在 HTTP 之上，这是 2026 年的最佳选择。HTTP 在负载均衡、认证、防火墙等方面都有成熟支持。HTTP 还允许 DuckDB-Wasm 原生支持 Quack，使浏览器中的 DuckDB 可以直接连接到云端服务器。

### 请求-响应模式
Quack 交互始终由客户端驱动，采用请求-响应模式。消息包括连接请求（使用令牌认证）、查询执行请求、结果获取请求等。

### 序列化
请求和响应使用新的 MIME 类型 `application/duckdb` 编码，利用 DuckDB 内部的高效序列化原语。这些原语已在 WAL 文件中使用多年，经过充分优化和实战测试。

### 加密
Quack 服务器默认生成随机认证令牌，默认只绑定 localhost。建议通过 nginx 等反向代理暴露到互联网，并让代理终止 SSL。客户端会对非本地连接默认启用 SSL。

### 往返优化
Quack 经过优化，查询只需一次往返即可完成。同时针对高效批量响应传输进行了优化，可能是目前通过 socket 传输表数据最快的方式。

### 认证与授权
Quack 带有默认认证方法，默认授权无限制，两者都可以通过用户提供的代码覆盖。默认情况下，服务器生成随机令牌并与客户端提供的令牌比较，但可以自定义认证函数（如查询 LDAP 目录、读取文件等）。授权函数可以检查每个查询并与之前的认证字符串关联。

### 默认端口
默认端口为 `9494`，94 是 Netscape Navigator 发布的年份。

## 性能测试

测试在 AWS m8g.2xlarge 实例（8 vCPU，32 GB RAM）上进行，客户端和服务器在同一可用区，ping 时间约 0.280ms。

### 批量传输

| 行数 | DuckDB Quack | Arrow Flight | PostgreSQL |
|------|-------------|--------------|------------|
| 100k | **0.07s**   | 0.07s        | 0.20s      |
| 1M   | **0.24s**   | 0.38s        | 2.20s      |
| 10M  | **0.89s**   | 2.90s        | 25.64s     |
| 60M  | **4.94s**   | 17.40s       | 158.37s    |

Quack 在 5 秒内传输了 6000 万行数据，远超 Arrow Flight SQL 和 PostgreSQL。

### 小规模写入

| 线程数 | DuckDB Quack   | Arrow Flight | PostgreSQL |
|--------|---------------|--------------|------------|
| 1      | **1,038 tx/s** | 469 tx/s     | 839 tx/s   |
| 2      | **1,956 tx/s** | 799 tx/s     | 1,094 tx/s |
| 4      | **3,504 tx/s** | 1,224 tx/s   | 2,180 tx/s |
| 8      | **5,434 tx/s** | 1,358 tx/s   | 4,320 tx/s |

Quack 在高并发场景下表现优异，在 8 个并行线程时达到约 5,500 tx/s，超过了 PostgreSQL。

## 应用场景

- **多进程数据收集**：从多个进程收集遥测数据到中央 DuckDB 实例，同时驱动仪表板查询
- **DuckLake 集成**：Quack 将集成到 DuckLake，使 DuckDB 可以作为远程访问的目录服务器
- **数据架构扩展**：DuckDB 从进程内分析数据库向现代数据架构的核心构建块演进

## 总结

Quack 协议的发布解锁了完整的多人体验，多个独立进程（本地或远程）现在可以并行修改表内容，无需相互锁定。Quack 使这种场景比使用 DuckLake 更简单，性能更高。DuckDB 正在从最初的进程内分析数据库角色扩展为现代数据架构的核心组件。

## 附录：为什么不用 Arrow Flight SQL？

Arrow Flight SQL 是数据交换 API（类似 ODBC 和 JDBC），旨在减少系统间数据交换的摩擦。但 DuckDB 担心使用外部控制的格式会限制创新。此外，Arrow Flight SQL 每个查询需要至少两次协议往返（CommandStatementQuery 和 DoGet），这对于小规模更新和高延迟环境不理想。Quack 设计为能够对小型查询执行单次往返查询和获取。
