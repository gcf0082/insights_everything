# GitHub Actions 是最弱的一环

**洞察链接：** https://nesbitt.io/2026/04/28/github-actions-is-the-weakest-link.html

**发布日期：** 2026年4月28日

**作者：** Andrew Nesbitt

**标签：** GitHub, 安全, 包管理器, 供应链

---

## 执行摘要

过去十八个月中，几乎所有开源供应链事件都可以追溯到 `.github/workflows` YAML 文件。GitHub Actions 作为一个最初为私有仓库企业 CI 工具设计的平台，其默认配置对于匿名 fork 和临时 pull request 场景存在严重安全隐患。该平台的多项"便利"功能组合起来极易造成危险，而全球大多数开源项目的构建和发布都运行在这个平台之上。

---

## 主要安全事件

### 1. spotbugs 事件 (2024年11月)

- **攻击方式：** 利用 `pull_request_target` 触发器，从不受信任的 fork 中检出代码并在具有完整 secret 访问权限的上下文中执行
- **后果：** 攻击者通过恶意 PR 窃取了维护者的 PAT，进而影响了 reviewdog 和 tj-actions/changed-files

### 2. Ultralytics 事件 (2024年12月)

- **攻击方式：** 通过 `pull_request_target` 触发器向 GitHub Actions 缓存中注入恶意代码
- **后果：** 当合法发布流程恢复缓存时执行了加密货币挖矿程序，两个版本的 ultralytics 被推送到 PyPI

### 3. tj-actions/changed-files 事件 (2025年3月)

- **攻击方式：** 利用标签劫持，将 `v1` 标签重新指向恶意提交
- **后果：** 23,000 个下游仓库受到影响，运行了内存抓取程序，将 runner secret 转储到公开构建日志中
- **影响范围：** CISA 为此发布了官方警报

### 4. nx/s1ngularity 事件 (2025年8月)

- **攻击方式：** 在 `pull_request_target` 工作流中，将 pull request 标题直接插入 shell 步骤，利用 `$` 模板语法注入代码
- **后果：** 恶意 nx 版本试图查找 AI 编码助手的凭据，导致超过 5,000 个私有仓库短暂公开

### 5. Trivy 事件 (2026年2-3月)

- **攻击方式：** 通过 `pull_request_target` 工作流攻破 Trivy 的 action 仓库
- **后果：** 攻击者篡改了 76/77 个历史版本标签，即使用户已固定到旧版本仍会运行凭据窃取程序

### 6. prt-scan 活动 (2026年3-4月)

- **攻击方式：** 攻击者运行大规模活动，在数百个存在 `pull_request_target` 错误配置的仓库中批量创建 pull request
- **特征：** 使用生成的语言相关 diff 看起来像合理的贡献

### 7. elementary-data 事件 (2026年4月)

- **攻击方式：** 通过 issue_comment 触发器，在工作流中将 `$` 注入 bash
- **后果：** 攻击者推送了伪造的 github-actions[bot] 作者的提交，在十分钟内将凭据窃取 wheel 发布到 PyPI 和 GHCR
- **讽刺：** 该漏洞在作者自己运行 zizmor 扫描 PyPI 包工作流时已被标记为 High/High，三周后才被实际利用

## 共同特征分析

以下 GitHub Actions 特性在每个事件中反复出现：

| 特性 | 风险描述 |
|------|----------|
| `pull_request_target` 触发器 | 让不受信任的 fork 在具有完整 secret 访问权限的上下文中运行代码 |
| `issue_comment` 触发器 | 类似地允许未授权用户触发工作流 |
| `$` 模板展开 | 在 shell 脚本中进行文本替换，无引号保护 |
| `GITHUB_TOKEN` 默认值 | 2023年2月之前创建的仓库默认获得写权限 |
| Action 版本 | 可变 git ref，可被劫持 |
| 缓存跨信任边界 | 缓存键基于分支，恶意数据可被下游使用 |

## 信任发布的隐忧

包注册中心（PyPI、npm、RubyGems、crates.io）都已采用基于 OIDC 的信任发布机制，用 CI 中的短期令牌替代长期 API 令牌。然而，这意味着这些注册表的完整性保证现在取决于 GitHub Actions 工作流的安全性。

过去十年，包管理器通过 lockfile、双因素认证、签名、审计日志和 provenance 证明来加强安全，但通过 OIDC 将所有信任连接到 CI 平台，实际上是将原本分散在数千个维护者凭据上的信任集中到一个不具备这些安全属性的 CI 平台上。

---

## GitHub 的响应

GitHub 于2026年3月发布了安全路线图，包含以下改进：
- 工作流 lockfile，直接和传递 action 依赖固定到 SHA
- 策略控制，可完全禁止 `pull_request_target`
- 秘密作用域限制到特定工作流
- 托管 runner 的出口防火墙

然而，所有功能都是可选的，且"公开预览"需要三到六个月。路线图中的 lockfile 需求比提出该功能的Issue被关闭为"不计划"晚了约三年。

## 建议措施

### 短期措施

1. **使用 zizmor：** 在仓库中添加 `zizmorcore/zizmor-action`，约四行 YAML
2. **固定 SHA：** 所有 action 引用使用具体 SHA 而非标签
3. **设置权限：** 在每个工作流文件顶部设置 `permissions: {}`
4. **假设最坏情况：** 假设任何未经身份验证的用户可以在 PR 标题、分支名称或 issue 评论中输入的内容最终都会成为 shell 脚本

### 长期建议

- 将 token 默认值改为只读（针对所有公共仓库）
- 拒绝在 `run:` 步骤内展开 `github.event.*`
- 拒绝在 `pull_request_target` 作业中恢复缓存
- 在请求 `id-token: write` 的任何工作流中要求不可变引用

---

## 统计数据

- 91% 使用第三方 action 的 PyPI 包至少通过可变标签引用一个
- 三分之二的仓库至少有一个工作流没有 `permissions:` 块
- tj-actions 事件一年后，仍有数百个包通过标签指向该 action

---

## 结论

GitHub Actions 的安全问题并非"bug"，而是平台设计选择的结果。这些功能不会消失，最佳的防御是在使用该平台时保持警惕。随着包注册表越来越依赖 CI 平台的安全性，GitHub 有责任采取更果断的行动，即使是会破坏现有工作流的改变。
