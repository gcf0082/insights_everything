## 摘要

Ultrathink 团队在生产环境中使用 SQLite 运行真实的电商商店，处理真实的 Stripe 支付和真实客户订单。本文总结了他们在 SQLite 生产实践中获得的经验教训，特别是关于并发部署导致订单丢失的教训。

---

## 一、系统架构：四个数据库，一个存储卷

### 1.1 数据库配置

Rails 8 环境下，生产环境配置了四个 SQLite 数据库：

- **primary**: 处理订单、产品、用户数据
- **cache**: 作为 Rails 缓存存储
- **queue**: 运行 Solid Queue 后台任务
- **cable**: 处理 Action Cable 连接

所有数据库文件都存放在映射到 Docker 命名卷的 `storage/` 目录中。

### 1.2 WAL 模式

SQLite 默认的日志模式会在写操作时锁定整个数据库，这对于处理并发请求的 Web 应用来说是不可行的。

WAL（Write-Ahead Logging）模式改变了这一机制：
- 写操作追加到单独的 `-wal` 文件
- 读操作继续从主文件读取
- 多读者和单写者可以并发操作
- Rails 8 默认启用 WAL 模式

作者表示，对于他们的流量级别，WAL 模式可以轻松处理所有操作，包括产品页面浏览、购物车操作和结账流程。`timeout: 5000` 配置给予写操作最多 5 秒获取锁的时间，在正常操作中从未达到这个限制。

## 二、重大事故：丢失两个订单

### 2.1 事故经过

2月4日，团队在两小时内向 main 分支推送了 11 次提交。每次推送都会通过 GitHub Actions 触发部署。Kamal 运行蓝绿部署——启动新容器，健康检查后停止旧容器。在切换期间，两个容器同时运行，都挂载了 `ultrathink_storage` 卷。

问题是：部署重叠窗口开始重叠。容器 A 仍在排空请求，容器 B 启动，然后容器 C 的部署在 B 完全健康之前就开始了。三个进程同时打开同一个 WAL 文件，都尝试写入。

### 2.2 后果

订单 16 和 17 在 Stripe 中成功完成，客户已被扣款，支付意向显示 `succeeded`。但订单记录从未写入数据库。在 WAL 文件争用过程中，写入丢失了。

诊断方法是通过 `sqlite_sequence` 表：

```
SELECT * FROM sqlite_sequence WHERE name='orders';
-- seq: 17

SELECT MAX(id) FROM orders;
-- 15
```

自增计数器显示已分配 17 个 ID，但只存在 15 行。两个订单已创建然后消失——Stripe 有钱，但数据库没有记录。

### 2.3 解决方案

解决方法是程序性的，而非技术性的：停止如此频繁地推送到 main。

团队添加了一条规则——批量相关更改，避免快速连续的推送。这条规则被记录在他们的 CLAUDE.md（AI 代理治理文件）中：

> "避免快速连续的推送到 main —— 2 小时内 11 次推送导致重叠的 Kamal 部署和并发 SQLite 访问。尽管 Stripe 成功扣款，订单 16/17 丢失。"

这不是 SQLite 问题，而是部署管道问题，SQLite 让这个问题变得可见。Postgres 配合蓝绿部署可以很好地处理这个问题，因为连接通过 TCP 套接字——新容器连接到同一个 Postgres 服务器，数据库引擎管理写操作顺序。SQLite 的写操作顺序依赖于文件系统级别的共享 Docker 卷锁，当容器重叠时就会出现问题。

## 三、sqlite_sequence：法医工具

`sqlite_sequence` 表是 SQLite 中最被低估的调试工具。它跟踪每个表曾经分配的最高自增值——即使该行后来丢失了。

团队使用它来统计历史任务队列工作量。AI 代理完成并清除数千个任务，但 `sqlite_sequence` 记得：

```ruby
def count_historical_tasks
  result = ActiveRecord::Base.connection.execute(
    "SELECT seq FROM sqlite_sequence WHERE name='work_queue_tasks'"
  )
  result.first&.fetch("seq", 0) || 0
end
```

`WorkQueueTask.count` 返回约 300（当前行数）。序列显示 3,700+（每个曾经创建的任务）。如果这些数字意外偏离，说明某些行被不该删除的代码删除了。

## 四、没人提到的坑

### 4.1 没有 ILIKE

PostgreSQL 开发者直觉地使用 `WHERE name ILIKE '%term%'`。SQLite 会抛出语法错误。使用 `WHERE LOWER(name) LIKE '%term%'` 代替。

### 4.2 json_extract 返回原生类型

`json_extract(data, '$.id')` 如果值存储为数字，则返回整数。与字符串比较会静默失败。当需要字符串比较时，始终使用 `CAST(json_extract(...) AS TEXT)`。

### 4.3 每个 kamal app exec spawn 一个新容器

在 t3.small（2GB RAM）上，Web 容器使用约 780MB。Spawn 一个 exec 容器进行快速数据库查询会增加约 500MB。部署期间两个并发的 exec 调用？那就是 2.5GB 在 2GB 机器上。OOM killer 会杀死 Web 进程。

### 4.4 timeout: 5000 是安全网，不是解决方案

如果经常遇到 `SQLITE_BUSY`，说明有配置无法修复的并发问题。在应用层面减少写操作争用。

## 五、结论：还会选择 SQLite 吗？

**是的**。对于单服务器部署和中等写音量，SQLite 消除了一整类基础设施复杂性：
- 无需连接池调优
- 无需数据库服务器升级
- 无需复制延迟
- 备份简单：`sqlite3 production.sqlite3 '.backup backup.sqlite3'` —— sqlite3 `.backup` API 可以安全处理 WAL 模式和并发写入（不同于原始 `cp` 可能在文件半写时捕获）

约束是真实的：单服务器，仔细规划部署节奏。当需要水平扩展或真正的多写者并发时，将迁移到 Postgres。Rails 使这个切换变得简单——更改适配器，运行迁移，更新使用 SQLite 特定语法的查询。

在此之前，一个单文件处理一切。你只需尊重它的本质。
