## 核心观点

Anthropic 工程师 Thariq Shihipar 分享了他放弃 Markdown，转而使用 Claude Code 生成 HTML 作为主要输出格式的实践经验。他认为 Markdown 虽然曾是 AI 与人类沟通的主要文件格式，但随着 AI 能力越来越强大，Markdown 已经变成一种束缚。他开始偏好 HTML 作为输出格式，并发现 Claude Code 团队的其他成员也越来越多地使用 HTML。

---

## 选择 HTML 的六大理由

### 1. 信息密度更高

HTML 能够传达比 Markdown 丰富得多的信息。它不仅能处理标题和简单排版等基础文档结构，还能完美呈现：
- 表格展示数据列
- CSS 展现设计细节
- SVG 绘制精美插图
- 嵌入代码片段
- 结合 JavaScript 实现动态交互
- 绘制清晰的工作流图表
- 展示空间分布数据
- 直接插入图片

作者认为，只要 Claude 能读懂的信息，几乎没有不能用 HTML 高效展现的。

### 2. 视觉清晰度与易读性更佳

随着 Claude 能够处理越来越复杂的工作，它写出的需求规格说明和实施计划也变得越来越庞大。在实际工作中，作者发现自己基本不会去读超过 100 行的 Markdown 文件。但 HTML 文档就好读多了，Claude 可以通过选项卡、插图、链接等视觉元素，把文档结构整理得井井有条，还能做到移动端自适应。

### 3. 易于分享

分享 Markdown 文件很让人头疼，因为大多数浏览器不能很好地渲染它们，通常只能作为附件发送。但 HTML 文件只需上传到云端（如 S3），就能轻松分享链接，任何人可以在任何设备上打开阅读。

### 4. 支持双向交互

HTML 允许与文档进行真实互动。例如，可以添加滑块或旋钮来直观调整设计效果，或提供选项微调算法参数，甚至可以加按钮一键复制微调后的参数为提示词。

### 5. 数据摄取与理解能力强

Claude Code 能够摄取极其庞大的上下文信息。作者让 Claude Code 自动遍历电脑里的代码文件夹，找出所有由它生成的 HTML 文件，进行分组归类，生成全新的 HTML 页面展示每类文件的特征。Claude Code 还能通过 MCP 接入 Slack、Linear 等多种数据源。

### 6. 充满乐趣

用 Claude 制作 HTML 文档本身就是一件极其好玩的事，让用户更深度地参与创造过程。

## 五大核心应用场景

### 1. 需求、计划与探索

对 Claude 来说，HTML 是一块可以深入探讨问题的广阔画布。作者不再期望 Claude 给出单薄的 Markdown 计划，而是期望它生成由多个 HTML 文件交织而成的思考网络。

**提示词示例：**
- "我还没想好新手引导页面要走什么风格。请生成 6 种截然不同的方案——在布局、语气和信息密度上做出差异——并把它们放在同一个 HTML 文件的网格布局里，方便我并排对比。"
- "请在一个 HTML 文件里创建一份详尽的实施计划。记得画一些视觉草图，展示数据流向，并补充上我可能需要重点审查的代码片段。"

### 2. 代码审查与理解

在 Markdown 文件里生啃代码绝对是一件痛苦的事。但有了 HTML，可以优雅地渲染出代码差异对比、详细的页边注释、流程图以及模块结构图。作者发现这种方式比 GitHub 自带的差异对比视图好用一万倍，现在每次提交 PR 都会附带一个 HTML 格式的代码解读页面。

**提示词示例：**
- "帮我审查这个 PR，生成一个 HTML 制品来向我解释它的逻辑。我对数据流和背压逻辑不太熟悉，所以请重点剖析这部分。请渲染出真实的代码差异，并在旁边加上行内注释。"

### 3. 设计与原型制作

Claude Design 的底层逻辑就是 HTML，因为 HTML 在表达设计理念方面具有无可匹敌的优势。可以制作丝滑的交互原型，比如动画效果或用户操作链路。

**提示词示例：**
- "我想为一个新的结账按钮做个交互原型：点击它时，它会播放一段动画，然后迅速变成紫色。请生成一个带有几个滑块和选项的 HTML 文件，让我能反复测试这套动画的不同参数配置。"

### 4. 报告、研究与学习

Claude Code 极其擅长整合海量的多源数据，并提炼成可读性极强的报告。可以让 Claude 搜索 Slack 聊天记录、代码库、Git 提交历史甚至整个互联网，生成一目了然的精美报告。

**提示词示例：**
- "我一直搞不懂我们的限流器到底是怎么工作的。请阅读相关代码，并为我生成一个单页的 HTML 讲解文档：包含一个令牌桶机制的数据流向图、3 到 4 段带有详细注释的核心代码片段。"

### 5. 自定义编辑界面

有时候单纯靠文字输入框很难准确描述复杂需求。作者会让 Claude 专门为手头工作快速搭建一个"用完即走"的临时可视化编辑器，一个专为特定数据量身定制的单一 HTML 文件。关键是要设计导出功能，如"复制为 JSON"或"复制为提示词"的按钮。

**提示词示例：**
- "我需要重新梳理这 30 个 Linear 任务单的优先级。请给我做一个 HTML 文件，把每个任务做成一张可拖拽的卡片，横跨分为"现在/接下来/以后再说/砍掉"四个栏目。"

## 常见问题解答

**Q: 这样不会很浪费 Token 效率吗？**

A: 确实，Markdown 通常消耗的 Token 更少。但作者发现 HTML 极强的表现力以及它极高的人工阅读率，让他整体上获得了好得多的输出结果。在 Opus 4.7 模型高达 100 万的庞大上下文窗口里，多花的这点 Token 几乎可以忽略不计。

**Q: 那你现在什么时候还会用 Markdown？**

A: 说实话，现在几乎干什么都不用 Markdown 了，不过作者承认可能已经在"HTML 极端主义者"的道路上走得太远了。

**Q: 这生成起来不比 Markdown 慢吗？**

A: 确实更慢！生成 HTML 的时间可能是生成 Markdown 的 2 到 4 倍，但作者亲身测试下来，生成的结果绝对物超所值。

**Q: 那版本控制怎么办？**

A: 老實說，這確實是 HTML 最大的痛點之一。相比起清爽的 Markdown，HTML 文件在版本控制工具里的差异对比非常杂乱。

**Q: 怎么让 Claude 生成的页面符合我的审美？**

A: Claude 内置的前端设计插件已经能帮它生成相当不错的 HTML 页面了。但如果想让页面完全契合公司品牌风格，可以让 Claude 扫描代码库，生成一个专属的"设计系统 HTML 文件"，之后让它"照猫画虎"保持风格一致。

---

## 总结

作者认为，自己钟爱 HTML 的根本原因在于：它让他真切地感觉到，自己依然在这个循环之中，依然在与 Claude 并肩作战。之前一度很恐惧，既然几百行的 Markdown 计划书都懒得仔细看了，那以后是不是只能任凭 Claude 自己盲目做决定？但现在很高兴地说，因为有了 HTML，感觉比以往任何时候都更紧密地参与到了人机协同的创作旅程中。
