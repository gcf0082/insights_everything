# 洞察报告：我对 Bun 感到担忧

**链接**: https://wwj.dev/posts/i-am-worried-about-bun/

**创建日期**: 2026-05-02

**更新日期**: 2026-05-02

**作者**: wwj

**摘要**: Bun 是一款优秀的软件，但作者担心其被 Anthropic 收购后可能会走上相同的"垃圾化"（enshittification）道路。

---

## 核心观点

### Bun 的优点
- Bun 是一款出色的 JavaScript 运行时
- 运行快速、实用，团队持续发布更新
- 让 TypeScript 在小型脚本、应用、测试和工具中成为一种乐趣
- 提供内置的 TypeScript 支持（无需构建步骤）、打包器、测试功能
- 包管理速度快，支持 monorepo，磁盘占用合理

### 担忧的根源

#### 1. Anthropic 收购了 Bun
- 2025年12月，Anthropic 收购了 Bun
- 官方承诺 Bun 保持开源 MIT 许可证，同一团队继续开发，路线图继续专注于高性能 JavaScript 工具和 Node.js 兼容性
- 问题：Anthropic 是一家似乎完全不关心其软件产品的公司

#### 2. Claude Code 正在"垃圾化"
- **Claude Code 曾经很棒**：一年前，它是首批让人相信开发者工作流会从主要是自动补全转变为 Agent 工具的 AI 编程工具之一
- **现在 Claude Code 变差了**（2026年4月）：
  - 质量下降
  - 限制行为
  - 第三方工具限制
  - 混乱的计费
  - 沟通缓慢
- Anthropic 发布的工程事后分析报告将问题归因于：默认推理能力降低、过期会话 bug、提示词变更影响编码质量
- **OpenClaw 事件**：仅仅在 git 历史中包含 "OpenClaw" 就可能导致 Claude Code 拒绝请求或额外收费

#### 3. 集成风险
- Bun 被嵌入到 Claude Code 中
- 随着 Bun 和其团队进一步融入 Anthropic，他们的政策也会随之而来
- 导致 Claude Code 崩溃的相同政策可能会影响到 Bun
- 作者担心 Bun 也会开始出现类似团队甚至不自己测试产品的问题

---

## 作者的决定

### 改用 pnpm
- 对于目前使用 Bun 的项目，作者正在迁移到 pnpm
- 对于新的 JavaScript 或 TypeScript 项目，推荐使用 pnpm
- pnpm 虽然只是包管理器，但足以满足日常工作需要的高速安装、monorepo 支持和合理的磁盘占用

### 建议
- 不要把作者的建议当作金科玉律，应该自己决定什么最适合
- 对于新项目，pnpm 是合理的
- 对于现有项目，除非有充分理由，否则可以继续使用 Bun

---

## 展望

作者希望他是错的：
- 希望 Bun 保持卓越
- 希望 Bun 团队继续提供出色的工作
- 希望 Anthropic 给他们机会为 JavaScript 生态系统做出正确的选择
- Bun 仍然可以变得更强大——Anthropic 有资金、分销渠道，以及真正关心 Bun 性能和稳定性的理由

但作者不再像12月那样信任这个局面。Claude Code 曾经感觉像 Anthropic 理解开发工具的证明，现在感觉像 Anthropic 不知道如何随着时间维护和改进产品的警告。

---

*Bun 仍然很棒。我只是不知道它接下来会走向何方。一年后情况可能会完全不同，所以我将跟进并看看我的预测是否正确。*
