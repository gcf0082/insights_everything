## 内容摘要

本文是一篇幽默的技术博客，作者模仿了 "Dear Sir, You Have Built a Compiler" 的风格，调侃那些为了避免使用 Kubernetes 而最终构建了类似系统的人。

### 核心观点

作者指出：许多开发者为了"选择无聊的技术"而避免使用 Kubernetes，认为它"过度复杂"。然而，随着业务需求增长，他们最终通过 shell 脚本、Docker Compose、Tailscale、Ansible 等工具的组合，实际上构建了一个与 Kubernetes 功能相似的系统。

### 典型的演进路径

1. **起点**：使用 shell 脚本部署容器
2. **第一阶段**：转向 Docker Compose
3. **第二阶段**：添加部署、滚动更新、回滚和扩展功能
4. **第三阶段**：扩展到多台服务器，配置防火墙规则
5. **第四阶段**：引入 Tailscale 实现覆盖网络和服务发现
6. **第五阶段**：使用 Ansible 实现基础设施即代码
7. **终点**：创建一个 API 服务来安全访问 Docker API

### 结论

当以上所有功能组合在一起时，实际上就是 Kubernetes 的核心功能。作者提醒读者：在 dismissing Kubernetes 之前，应该先理解它解决什么问题。

## 关键引述

> "Do I really need a separate solution for deployment, rolling updates, rollbacks, and scaling?" you ask yourself?

> "Surely not. Your app is so simple; a backend, a reverse proxy, postgres, and a job runner."

## 总结

这篇文章以幽默的方式提醒开发者：有时候我们为了避免"复杂性"而选择自己构建解决方案，但最终可能发现我们只是在重新发明轮子。理解工具解决的问题，比盲目拒绝"复杂"的工具更重要。
