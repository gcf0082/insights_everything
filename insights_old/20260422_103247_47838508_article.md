## 核心内容摘要

### 一、变更背景

代理工作流（Agentic workflows）从根本上改变了 Copilot 的计算需求。长时间运行的并行会话现在消耗的资源远超原有套餐结构的设计预期。随着 Copilot 代理能力的快速扩展，代理工具执行的工作量越来越大，越来越多的用户达到了原本为维护服务可靠性而设计的用量限制。如果没有进一步措施，服务质量将会下降，影响所有用户。

### 二、主要变更措施

#### 1. 新用户注册暂停
- **暂停对象**：GitHub Copilot Pro、Pro+ 和学生套餐的新注册
- **目的**：使现有客户获得更好的服务

#### 2. 用量限制收紧
- **Pro+ 套餐**：提供超过 Pro 套餐 5 倍以上的限制
- **Pro 套餐用户**：需要更高限制的用户可以升级到 Pro+
- **可视化**：用量限制现在显示在 VS Code 和 Copilot CLI 中，方便用户避免达到限制

#### 3. 模型可用性调整
- **Pro 套餐**：Opus 模型不再可用
- **Pro+ 套餐**：Opus 4.7 仍然可用
- **Pro+ 将移除**：Opus 4.5 和 Opus 4.6（已于2026年4月16日公告）

### 三、用量限制机制

GitHub Copilot 目前有两种用量限制：

| 限制类型 | 说明 |
|----------|------|
| **会话限制** | 主要确保服务在高峰期不会过载。大多数用户不应受到影响。达到限制后需等待使用窗口重置 |
| **周限制（7天）** | 限制用户一周内可以消耗的 token 总数。用于控制并行化、长轨迹请求带来的高成本 |

- **Premium Request**（高级请求）：决定可以访问哪些模型以及可以发出多少请求
- **Usage Limit**（用量限制）：基于 token 的护栏，限制在给定时间窗口内可以消耗的 token 数量
- 用户可能仍有高级请求额度，但已达到用量限制

### 四、如何避免意外限制

1. **使用乘数较小的模型**：简单任务使用较小乘数的模型，限制消耗速度更慢
2. **升级到 Pro+**：Pro 用户升级到 Pro+ 可将限制提升 5 倍以上
3. **使用计划模式**：在 VS Code 和 Copilot CLI 中使用计划模式（Plan mode）提高任务效率
4. **减少并行工作流**：如 `/fleet` 等工具会产生更高的 token 消耗，应谨慎使用

### 五、变更原因

代理和子代理在处理复杂编码问题方面展现了巨大价值，用户已意识到这些工具的重要性。这些长时间运行的并行工作流虽然可以产生巨大价值，但也挑战了 GitHub 的基础设施和定价结构——少数请求产生的成本现在已经经常超过套餐价格。这些问题需要 GitHub 解决，此次采取措施旨在一段时间内为现有用户提供最佳体验，同时开发更可持续的解决方案。

### 六、退款政策

- 如果用户遇到意外限制或变更不适合，可在 **2026年5月20日** 前访问账单设置取消 Pro 或 Pro+ 订阅，并获得剩余订阅时间的退款
