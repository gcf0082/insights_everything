## 内容摘要

本文作者解释了为何要创建 httpx 的分支版本 **httpxyz**。

### 背景

HTTPX 是一个非常流行的 Python HTTP 客户端库，有大量代码依赖于它。

### 分支原因

1. **发布问题**：作者在 2024 年贡献了 zstd 内容解码功能，该功能被合并并发布。但随后发现功能存在问题，作者提交了修复补丁，但修复被忽视，自 2024 年 11 月以来从未有任何版本发布。作者多次请求发布包含修复的版本，甚至单独给作者发邮件，得到的回复是"1.0 开发正在进行中"。然而 1.0 开发已经计划了两年多。

2. **隐藏问题**：近期 GitHub 仓库中的 issues 被隐藏，讨论区被关闭。这使得人们更难以参与贡献或使用该包，因为代码和文档中到处都有问题链接。

3. **1.0 版本前景不明**：关于即将发布的 1.0 版本已经讨论了多年（可追溯到 2020 年左右）。2024 年 10 月的讨论中，有人指出 1.0 可能与当前版本的 httpx 有很大不同。

4. **依赖方采取保护措施**：由于上述原因，依赖 httpx 的 openai 和 anthropic Python 包都在 pyproject.toml 中设置了防护措施，防止安装 1.0 版本。

5. **破坏性变更**：作者之前在次版本更新中添加过破坏性变更，造成了很大的麻烦。

6. **类似模式**：作者还创建了 Django REST Framework 和 MkDocs，同样关闭了 issues 和讨论区，这使得使用这些产品的人们生活更加困难。

### 计划

- 维护一个稳定的 httpx 分支
- 不进行重大重写
- 不做破坏性变更
- 不大幅改变项目设置
- 座右铭："移动得更快，但不破坏东西"

### FAQ

**Q: 为什么不在 GitHub 上托管代码？**

作者认为技术社区过度依赖单一平台。GitHub 多年来为开发者提供了很好的服务，但作者担心这种单一文化。Codeberg 是一个非营利组织运营的平台，虽然还有一些"粗糙"，但大多数情况下都能正常工作。

**Q: 现在应该迁移所有代码吗？**

如果一切正常运作，不一定需要迁移。但如果需要，httpxyz 提供了一条稳定的路径。也可以考虑使用其他包，如 niquests（requests 的一个分支，支持 HTTP/2、HTTP/3 和异步支持）。

## 结论

作者理解维护者的倦怠问题，也理解倾向于开发"下一代"以及工作之外的生活，但作者认为对于这样一个高知名度的模块，既不进行维护也不让其他人参与维护是有问题的。

作者希望原开发者能继续开发更优秀的软件，期待 HTTPX2 的出色表现！
