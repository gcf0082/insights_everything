## 内容摘要

### 项目背景

TJ Green 在 Tiger Data（一家 Postgres 云服务商，主要业务是时序数据）工作。去年夏天，公司希望拓展到新兴的 AI 工作负载，需要在 Postgres 中提供最先进的混合搜索栈。他们已经内部构建了 pgvectorscale 来扩展语义搜索（超越 pgvector 的内存限制），但还需要一个可扩展的关键词搜索解决方案。

### 面临的挑战

- Postgres 内核不提供此功能
- 领先的 Postgres BM25 扩展 ParadeDB 使用 AGPL 许可证
- 开发自己的扩展看起来很艰巨：需要一个小团队，6-12 个月，可能仍无法达到 Parade/Tantivy 的性能水平

### 解决方案

TJ 凭借 25 年数据库系统内部工作经验，结合最新的 AI 工具（Claude Code + Opus），决定独自在一个季度内完成项目。

### 成果

- 实际花费时间略超预期（两个季度）
- 社区也提供了一些帮助
- **pg_textsearch v1.0** 发布，采用 PostgreSQL 许可证开源
- 在 MS-MARCO 基准测试中，查询吞吐量比 Parade/Tantivy 快 **4.7 倍**
- 索引构建速度：138M 文档（约 50GB 数据）约 18 分钟；200 万行数据约 1 分钟

## 热门评论摘要

### 技术讨论

- **索引构建时间**：作者确认 138M 文档约 18 分钟，2M 行数据在低端硬件上约 1 分钟
- **过滤功能**：目前主要处理 BM25 排名，SQL 上下文中可以通过后过滤实现 term 过滤
- **分区表支持**：已有稳健的分区表支持，表达式索引有社区 PR 在进行
- **混合搜索**：支持与向量搜索结合，用于 RAG 应用
- **更新频率**：针对批量索引和查询性能优化，高频更新功能尚未完成

### 云平台支持

- 已在 Tiger Data Cloud 可用
- GCP Alloy 正在支持中
- Supabase 和 AWS Aurora 悬而未决

### 用户反馈

- 已有生产环境使用案例，反馈比 ParadeDB 更稳定
- 开发者响应迅速
- 期待在 Neon 等平台使用

## 结论

pg_textsearch 为 Postgres 生态系统带来了重要的 BM25 全文搜索能力，采用更宽松的 PostgreSQL 许可证，且在性能上超越了现有的开源替代方案。对于构建 RAG 和混合搜索应用的开发者来说，这是一个值得关注的选择。
