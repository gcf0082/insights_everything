## 内容摘要

本帖是关于 **jj (Jujutsu)** 一个分布式版本控制系统(与Git兼容)的讨论。帖子链接了一篇教程文章，介绍jj的概念和使用方式。

### jj的核心概念

1. **无索引设计** - jj没有类似Git的暂存区(working directory即change)
2. **先描述后变更** - 使用`jj new`创建新提交，再描述，区别于Git的先变更后提交
3. **提交是计算值** - jj将提交视为基于目录内容等计算的稳定值，而非变更单位
4. **自动重基** - jj会自动重基，在push之前标记为不可变，防止意外重写

### 用户反馈与讨论焦点

#### 正面观点
- `jj split`可以很好地拆分多个功能/抽象的变更
- `jj commit [files]`可以只提交文件的子集
- Git兼容后端，无缝切换
- `jj undo`恢复能力强，AI代理更难搞砸

#### 争议观点
- **工作流反向**: 用户习惯先变更后提交，jj要求反向思考
- **自动提交**: 编辑文件会自动提交，需使用`jj new`创建空提交来避免污染历史
- **GitHub PR审查**: 每次force push后，审查者丢失上次审查的delta，需要手动追踪

#### 主要批评
- 学习成本高，与Git心智模型差异大
- 社区反馈：试用几个月后，没有一次"用它会更轻松"的感觉
- 互操作性并非无缝，需要不断导入操作

### 代码审查工作流问题

用户提到使用jj时GitHub的堆叠PR问题：force push后审查者丢失上次审查的commit。GitHub在最近6小时内已添加对堆叠PR的支持。

## 关键要点

| 要点 | 描述 |
|------|------|
| 核心优势 | Git兼容、可堆叠提交、自动重基、undo能力强 |
| 主要障碍 | 学习曲线、与Git心智模型相反、GitHub审查集成不完善 |
| 适用场景 | 个人项目、多分支开发、需要频繁重基的工作流 |
| 不适用场景 | 团队协作(需审查流程)、已习惯Git工作流的用户 |
