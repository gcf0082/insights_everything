## 内容摘要

本文是C3语言博客作者对五年前做出的一个语言设计决策的回顾与反思——关于无符号整数（unsigned）是否应该用于表示大小和索引。

## 核心讨论

### 背景

系统程序员对无符号整数又爱又恨。Java世界的整数模型是以零为中心的有符号数，但这仍然存在边界条件需要处理。在低级编程中，有时确实需要使用完整的字长范围来表示非负值。

### 主要观点

**支持无符号整数的理由**：
- 需要使用完整的字长范围（如8位、16位、32位数据类型）
- 低级编程中无符号类型 essential
- 位运算和模运算需要无符号类型
- 在某些场景下（如HPC图形和模拟），为了内存/缓存性能，需要使用uint32_t而不是int64_t
- 对于TCP序列号等环形缓冲区的模运算

**反对无符号整数的理由**：
- C中的无符号整数是模运算，而有符号整数更符合数学直觉
- 无符号整数更难调试，溢出是隐式定义的
- 将负值赋给无符号类型会静默失败
- 数组索引操作用有符号类型更直观
- 学生从小学习有符号整数，模运算是大学才接触的概念

### 各语言的处理方式

| 语言 | 大小/索引类型 | 特点 |
|------|---------------|------|
| Java | 有符号 | 没有原始无符号类型，通过工具方法支持无符号运算 |
| Rust | 无符号 | len()返回usize，但设计有多种运算变体 |
| C/C++ | 无符号 | 需要注意隐式转换警告 |
| Go | 有符号 | int类型为有符号，有边界检查 |

## 关键讨论点

1. **无符号类型名称的误导性**：无符号并不等于非负数，将负值赋给无符号类型会静默转换而不是报错。

2. **C23的变化**：C23更新了数组下标操作符的定义，现在不允许对数组类型使用负下标。

3. **工具的重要性**：有符号溢时有 sanitizer 和编译器警告帮助，而无符号回绕是明确定义的，难以自动检测是故意还是错误。

4. **代数结构的差异**：有符号整数和无符号整数在加法、减法、乘法上都是模2^n的同构环，区别主要在比较运算和除法/取模运算。

5. **语言设计建议**：使用有符号整数作为默认值，仅在需要位运算和模运算时使用无符号类型；建议启用 -Werror -Wsign-conversion 编译器选项。

## 结论

五年的实践让作者认识到，对于大小和索引，有符号整数是更好的默认选择。虽然无符号类型在某些低级场景下有用武之地，但作为通用默认值，有符号类型更安全、更直观。
