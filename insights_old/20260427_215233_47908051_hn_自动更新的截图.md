## 文章摘要

这是一篇关于如何实现文档截图自动更新的技术文章。作者 James Adam 分享了他为 Jelly 应用构建的帮助中心系统，该系统能够自动从运行中的应用中捕获截图，并在每次重建时自动更新所有截图。

### 核心问题

维护文档截图的痛苦：手动截取、裁剪、上传后，一旦UI发生变化（调整颜色、移动按钮、更新文案），所有涉及的截图都会过时。开发者知道截图过时了，用户可能注意不到，但这让开发者很困扰。

### 解决方案

通过在Markdown中嵌入特殊的HTML注释来标记需要截图的位置，然后使用Rake任务启动无头Chrome浏览器（Capybara + Cuprite）自动访问对应页面并捕获截图。

#### 语法示例

```markdown
<!-- SCREENSHOT: acme-tools/inbox | element | selector=#inbox-brand-new-section -->
![The "Brand New" section](images/basics-brand-new-section.png ':screenshot')
```

#### 截图模式

- **element**：按CSS选择器截图特定DOM元素
- **full_page**：捕获整个页面，可选裁剪高度
- **viewport**：仅捕获浏览器视口可见区域

#### 高级选项

- `click`：点击按钮后截图（用于捕获弹窗等状态）
- `wait`：等待毫秒数（等待动画完成）
- `crop`：裁剪区域坐标
- `torn`：应用撕纸边缘效果
- `hide`：临时隐藏不需要的元素（如开发者工具栏、cookie横幅）

### 使用方式

只需运行一行命令即可完成所有截图捕获和页面构建：

```bash
rails manual:build
```

## 社区讨论要点

1. **Nix方案**：有开发者提到使用Nix的`.#screenshots`派生（derivation）来实现类似功能，前期工作量大但几乎零维护。还可以生成浅色/深色主题配对，使用`<picture>`元素根据`prefers-color-scheme`自动切换。

2. **移动端截图**：有开发者分享使用Nix生成临时Android模拟器来捕获移动应用截图，无需预先设置。

3. **文档与UI不同步问题**：有评论指出自动化截图可能导致文档文字与实际UI不一致（如"Options > Customize"变成"Preferences > Advanced"），造成混淆。

4. **移动端代码块样式问题**：多位开发者指出文章中的代码块在移动端无法横向滚动，文字溢出到白色背景上。

5. **网站技术栈**：有人提到网站使用Django，间歇性出现"NoMethodError"错误，建议关闭Django调试模式。

6. **替代方案**：
   - Fastlane：用于自动生成App Store截图
   - shot-scraper：另一个类似工具
   - 静态HTML嵌入：对于Web项目，可考虑使用iframe代替截图
   - Textual使用SVG而非截图

7. **与CI/CD集成**：建议将截图作为E2E测试的一部分，在测试中捕获截图并与文档一起版本化管理。

8. **视觉差异检测**：可以在CI pipeline中加入视觉差异检测，防止错误截图流出。

## 总结

这篇文章展示了一个优雅的解决方案，通过将截图捕获指令直接嵌入Markdown文档，实现文档截图的自动化更新。虽然前期需要处理各种边界情况（滚动到视口、点击弹窗、图像裁剪等），但一旦建成，更新文档的摩擦几乎消失，开发者更愿意频繁更新帮助中心。该方法值得在需要维护大量文档截图的项目中推广。
