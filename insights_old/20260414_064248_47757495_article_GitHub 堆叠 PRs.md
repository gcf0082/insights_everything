## 一、产品概述

GitHub Stacked PRs（堆叠式 Pull Request）是 GitHub 推出的一项创新功能，旨在帮助开发者将大型代码变更拆分为小型、可独立审查的 Pull Request 序列。这些 PR 相互依赖、层层堆叠，最终一起合并到主分支。该功能结合了 `gh stack` CLI 工具和原生 GitHub UI 支持，为开发者和审查者提供了无缝的体验。

## 二、核心功能特性

### 2.1 PR 堆叠结构

堆叠式 PR 形成一个依赖链：
- 最底层的 PR 指向主分支（如 main）
- 每个上层 PR 指向其下方 PR 的分支
- 示例结构：`main ← auth-layer (#1) ← api-endpoints (#2) ← frontend (#3)`

这种结构确保每个 PR 代表一个原子性的、可审查的变更单元，可以独立审查和迭代。

### 2.2 原生 GitHub UI 集成

**堆叠导航器（Stack Map）**
当 PR 属于一个堆叠时，PR 页面顶部会显示堆叠地图，展示堆叠中每个 PR 的状态，并支持一键导航到任何层级。这为审查者提供了清晰的上下文，帮助理解当前 PR 在整体变更中的位置。

**规则和 CI 强制执行**
- 分支保护规则（如 CODEOWNER 审批）在堆叠中的每个 PR 上强制执行，即使中间层的 PR 并未直接指向 main
- CI 检查针对最终目标分支（main）运行，为整个堆叠中的所有 PR 触发

**合并功能**
- 支持从下到上逐步合并整个或部分堆叠
- 支持直接合并和合并队列两种方式
- 合并后，剩余的 PR 自动 rebase 到主分支，准备好进行审查和合并

### 2.3 合并方法支持

堆叠 PR 支持所有三种合并方式：
- **Merge commit**：为整个合并组创建一个 merge commit，保留每个 PR 的完整提交历史
- **Squash merge**：为每个 PR 创建一个干净的压缩提交，将该 PR 的所有提交合并为一个
- **Rebase merge**：将每个 PR 的提交重新播放到目标分支，创建线性历史

### 2.4 简化的 Rebasing

Rebasing 是堆叠式 PR 操作中最复杂的部分，GitHub 提供了自动化支持：
- PR UI 中提供 rebase 按钮，可触发整个堆叠的级联 rebase
- CLI 命令 `gh stack rebase` 在本地执行相同的级联 rebase
- 部分合并后，剩余分支自动 rebase，确保下一个 PR 指向 main

## 三、gh stack CLI 工具

`gh stack` CLI 是管理本地开发工作流程的核心工具，主要命令包括：

| 命令 | 功能描述 |
|------|----------|
| `gh stack init` | 初始化堆叠，创建第一个分支 |
| `gh stack add` | 在堆叠顶部添加新分支 |
| `gh stack rebase` | 执行级联 rebase |
| `gh stack push` | 推送所有分支到远程 |
| `gh stack submit` | 推送分支并创建/更新 PR |
| `gh stack up/down/top/bottom` | 在堆叠层级间导航 |
| `gh stack sync` | 一次性执行 fetch、rebase、push 和更新 PR 状态 |
| `gh stack unstack` | 拆除堆叠，移除 GitHub 和本地的跟踪 |
| `gh stack checkout <pr-number>` | 从 GitHub 拉取整个堆叠到本地 |

CLI 不是使用堆叠 PR 的必要条件，底层 git 操作是标准的，但 CLI 大大简化了工作流程。

## 四、为什么使用堆叠式 PR

### 4.1 解决的问题

大型 Pull Request 面临诸多挑战：
- **审查困难**：大的变更难以全面审查，容易遗漏问题
- **合并缓慢**：大型 PR 需要更多时间通过所有检查
- **冲突频繁**：大型变更更容易与其他分支产生冲突
- **上下文丢失**：审查者难以理解单个变更与整体的关系
- **反馈质量下降**：审查者面对大量变更时难以给出高质量反馈

### 4.2 优势

- 每个 PR 代表一个聚焦的变更层，可以独立审查
- 审查者获得清晰的上下文，了解变更在整体中的位置
- 更快的审查周期和更少的合并冲突
- 更好的变更追踪和历史记录

## 五、堆叠结构设计原则

### 5.1 依赖链

堆叠分支形成依赖链：基础变更（模型、共享类型、数据库schema）放在下层，依赖它们的代码（API 路由、UI 组件）放在上层。

原则：如果一层中的代码依赖于另一层的代码，依赖必须在同一分支或更低的分支中。

### 5.2 何时创建新分支

在以下情况下创建新分支：
- 从后端工作切换到前端工作时
- 从核心逻辑转到测试或文档时
- 下一组变更有不同的审查受众时
- 当前分支的 PR 已经足够大可以审查时

## 六、快速开始指南

### 6.1 安装 CLI 扩展

```bash
gh extension install github/gh-stack
```

### 6.2 设置 AI Agent 集成（可选）

```bash
npx skills add github/gh-stack
```

### 6.3 创建第一个堆叠

```bash
cd my-project
gh stack init  # 初始化堆叠，创建第一个分支
# 编写代码并提交
git add .
git commit -m "Add auth middleware"
# 添加更多分支
gh stack add api-routes
# 编写代码并提交
git add .
git commit -m "Add API routes"
# 推送所有分支
gh stack push
# 创建 PR
gh stack submit
# 查看堆叠状态
gh stack view
```

## 七、总结

GitHub Stacked PRs 为现代软件开发工作流程提供了一种优雅的解决方案，通过将大型变更拆分为可管理的堆叠结构，实现了更高效的代码审查和更顺畅的合并流程。结合强大的 CLI 工具和原生 UI 集成，这一功能特别适合追求高质量代码审查的开发团队和 AI Agent 工作流程。目前该功能处于私人预览阶段，开发者可以通过 waitlist 注册体验。
