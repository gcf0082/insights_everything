## 事件概述

TanStack 的多个 NPM 包（包括 @tanstack/router 等）被一个名为"Mini Shai-Hulud worm"的攻击者入侵。攻击者通过劫持 CI/CD 流水线，窃取开发者密钥，并将恶意代码注入到包的 postinstall 脚本中。

## 关键发现

### 1. 恶意负载分析

攻击 payload 会：
- 在 `~/.local/bin/gh-token-monitor.sh` 安装一个"死手开关"(dead-man's switch)
- 在 Linux 上创建 systemd 用户服务，在 macOS 上创建 LaunchAgent
- 每 60 秒轮询 `api.github.com/user`
- 如果 token 被撤销（HTTP 40x），执行 `rm -rf ~/`

### 2. Trusted Publishing 的局限性

多名评论者指出，Trusted Publishing 模式本身不足以防止此类攻击：
- 攻击者可以通过入侵 CI/CD 流水线获取 OIDC token
- 如果 repo admin token 被窃取，可以绕过所有 GitHub 内部的安全检查
- 没有 registry 层面的第二因素认证

### 3. 社区建议

- **使用最小发布年龄设置**：npm 支持 `min-release-age`，推荐至少 7 天
- **锁定所有依赖**：使用 package-lock.json 固定精确版本
- **考虑使用 pnpm**：因为 postinstall 脚本的危险性
- **在 VM/独立环境中运行项目**：容器不适合作为安全边界
- **手动审批发布流程**：Astral 等项目使用手动审批 gated releases

### 4. 生态系统对比

| 生态 | 安全性 |
|------|--------|
| npm | 每周都有供应链攻击报道 |
| Go modules | 默认锁定依赖，攻击较少 |
| Cargo | 攻击面与 npm 类似，但规模较小 |
| Maven (Java) | 文化上倾向于谨慎升级 |

### 5. GitHub 机器人问题

评论者指出，GitHub 上的机器人情况比想象中糟糕，有人用"GitHub bot law"来形容这个问题的严重性。

## 主要观点

1. **NPM 是最大的攻击面**：因为其庞大的用户群和无限制的 postinstall 脚本执行
2. **Trusted Publishing 不够安全**：需要 registry 层面的第二因素
3. **社区需要更安全的默认设置**：而不只是依赖开发者手动配置
4. **锁定依赖是必要的**：package-lock.json 能防止恶意版本替换
