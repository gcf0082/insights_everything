## 文章摘要

本文探讨了异步编程范式的演进历程，从回调函数（Callbacks）到 Promise/Future 再到 async/await，每一代解决方案都解决了前一代的核心问题，但也引入了新的问题。作者认为这种"波浪式"的演进体现了计算机科学中一种深层的模式：每次修复都只是针对症状，而没有改变根本结构。

## 核心内容总结

### 问题起源：C10K 问题

1999年，Dan Kegel 提出了"C10K问题"——如何处理 thousands of concurrent connections。传统的线程模型为每个连接分配一个线程，但：
- OS 线程需要约 1MB 的栈空间
- 创建线程需要约 1 毫秒
- 上下文切换在内核空间发生，消耗 CPU 周期

这导致了资源耗尽问题，催生了异步编程的三波演进。

### 第一波：回调函数（Callbacks）

**解决的问题**：线程资源耗尽问题
- 使用事件循环（select, poll, epoll, kqueue）将数千个连接复用到少数线程上
- 让出线程而不是阻塞等待 I/O 操作完成
- Node.js 和 Nginx 建立在这种模型之上

**引入的新问题**：
1. **控制流反转**：代码从"做A，然后B，然后C"变成嵌套的回调函数
2. **回调地狱**：JavaScript 开发者甚至为这个问题建立了一个网站（callbackhell.com）
3. **错误处理碎片化**：每个回调都需要自己的错误路径，错误无法沿着调用栈自然传播
4. **无取消机制**：一旦启动异步操作，无法通用地停止它

### 第二波：Promise 和 Future

**核心理念**：异步操作立即返回一个代表最终结果的对象，而不是传入回调函数

**解决的问题**：
- 可组合性：`promise.then(f).then(g)` 形成了管道式的线性结构
- 错误处理统一：`.catch()` 可以处理链条中任何步骤的失败
- Promise 是头等值，可以存储、传递、从函数返回

**引入的新问题**：
1. **一次性限制**：Promise 只能resolve一次，不适合 modeling 流、事件、重复消息
2. **组合笨拙**：`Promise.all` 处理两个独立异步操作容易，但条件分支、循环等模式变得复杂
3. **错误静默消失**：早期 JavaScript 中，没有 `.catch()` 处理的 Promise 会静默吞掉错误，Node.js 后来不得不将未处理的拒绝改为进程崩溃
4. **类型分裂**：每个函数要么返回值要么返回 Promise 的值，调用者需要知道是哪一个

### 第三波：async/await

**解决的问题**：Promise 链仍然不像顺序代码
- C# 2012年首创，JavaScript (ES2017)、Python 3.5、Rust 1.39 等语言相继采用
- 让异步代码看起来像顺序代码，可以直接使用 `try/catch` 和循环

**引入的新问题**——"函数着色"问题：

Bob Nystrom 2015年发表的"What Color is Your Function?"用红蓝函数类比：
- async 函数是"红色"，sync 函数是"蓝色"
- 红色可以调用蓝色，但蓝色调用红色需要特殊处理（阻塞或重构）
- 这个选择像病毒一样传播，影响整个代码库

具体表现：
1. **函数级别的着色税**：添加一个 I/O 调用会改变函数的签名、返回类型和调用约定，变化波及整个调用图
2. **库级别的着色税**：作者需要决定写 sync 版本还是 async 版本，或者维护两个版本（Python 的 requests 和 aiohttp 就是栗子）
3. **生态系统级别的着色税**：Rust 的 Tokio、async-std、smol 等运行时互不兼容，`reqwest` 只支持 Tokio

**新 bug 类别**：
- "futurelocks"：async Rust 中特有的死锁类别，一个 future 获取锁后停止轮询，另一个 future 等待获取同一把锁

### 顺序陷阱（Sequential Trap）

async/await 最大的优势（让异步代码看起来像顺序代码）也是一个认知陷阱：

```javascript
async function loadDashboard(userId) {
    const user = await getUser(userId);
    const orders = await getOrders(user.id);
    const recommendations = await getRecommendations(user.id);
    return render(user, orders, recommendations);
}
```

这段代码顺序执行：`getRecommendations` 要等 `getOrders` 完成后才开始。但这两个操作实际上是独立的，可以并行运行。代码看起来正确，但实际上浪费了性能。

要实现并行，程序员必须显式重构：

```javascript
async function loadDashboard(userId) {
    const user = await getUser(userId);
    const [orders, recommendations] = await Promise.all([
        getOrders(user.id),
        getRecommendations(user.id)
    ]);
    return render(user, orders, recommendations);
}
```

顺序语法掩盖了依赖关系，而依赖关系恰恰是确定什么可以并行运行的关键信息。

### 什么做对了

尽管有这些问题，一些语言从异步编程的教训中学到了正确的经验：

- **Go**：选择 goroutines 而非 async/await，承担更重的运行时但完全避免了函数着色
- **Java 21 的 Project Loom（虚拟线程）**：轻量级线程看起来和行为像普通线程，无需代码变色，明确引用了函数着色作为要避免的问题
- **Zig**：完全移除了编译器级别的 async/await，围绕 I/O 接口参数重建，函数签名不会因为调度方式而改变

## 核心洞见

### 累积的成本模式

| 波浪 | 解决 | 引入 |
|------|------|------|
| 回调 | 线程资源耗尽 | 控制流反转、错误处理碎片化、回调地狱 |
| Promise | 嵌套、错误合并 | 一次性限制、错误静默消失、类型分裂 |
| async/await | 线性异步序列的 ergonomics | 函数着色、生态系统分裂、新死锁类别、顺序陷阱 |

**关键结论**：每一波都让编写单个异步函数的局部体验更愉快，但维护大型代码库、管理运行时兼容性、寻找并行机会的全局体验却更复杂了。

### 深层模式

这可能是"如何管理并发执行？"这个问题的最清晰例证：
- 每次询问这个问题都会在每个抽象层级生成新问题
- 每个修复都是对前一个症状的合理回应
- 但十五年和多次迭代后，累积的代价是显著的
- 模式清晰可见：每个修复都治疗症状，而让结构保持不变

## 结语

回调、Promise 和 async/await 的设计者都是在解决真正的问题，每一步都是对前一步失败的合理回应。但十五年过去了，累积的税收是相当大的，一个模式清晰可见：这不仅仅是糟糕的工程，而是一种深层的系统性问题——通过询问"如何管理并发执行？"我们不断在每个抽象层级生成新问题。

这系列文章的主题再次得到验证：那些从"如何管理并发执行？"开始的方案keep generating new problems at every level of abstraction。
