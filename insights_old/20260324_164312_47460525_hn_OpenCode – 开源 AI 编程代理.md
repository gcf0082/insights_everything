## 核心洞察

### 1. 开发实践问题

用户反馈 OpenCode 开发团队采用极高的发布频率，几乎没有时间进行测试或修复问题。在这种快速迭代的节奏下，功能被频繁添加、移除、修改和破坏，缺乏完善的变更日志记录。

### 2. 资源消耗问题

OpenCode 是一个大型且复杂的 TypeScript 代码库，性能表现不佳。作为一个 TUI（终端用户界面）应用，它经常占用超过 1GB 的内存，这与同类工具形成鲜明对比。用户提到 Claude Code（Electron 应用）在相同机器上会使用多个 GB 的 RAM 和 100% CPU，而 Codex（Rust 实现）仅使用 80MB 内存和 6% CPU。

### 3. 安全隐患（关键问题）

#### 3.1 默认配置不安全
- OpenCode 默认允许执行所有操作
- 默认从网络（provider-based URL）拉取配置
- 登录时，provider 网站可以提供任意 shell 命令供 OpenCode 执行，即使命令是 `rm -rf /home` 也会被直接执行

#### 3.2 隐私数据泄露
- 默认将用户 prompts 发送至 Grok 免费版
- Grok 免费版会使用用户提交的信息进行训练，X AI 可用于构建广告画像等用途
- 即使只配置了本地模型，用户的 prompts 仍会被发送到云端用于生成"会话标题"
- 使用 mitmproxy 实测：即便配置了本地 chat completions，OpenCode 仍会静默调用 opencode zen 并使用 gpt-5-nano，该模型有 30 天保留期且明文接受 OpenAI 人工审查

#### 3.3 潜在 RCE 漏洞
- GitHub 上存在相关安全 issue，可能导致远程代码执行

### 4. 用户界面问题

- TUI 界面过于复杂且存在 bug
- 功能过多，很多功能用户并不需要，反而增加了使用难度
- 记住所有功能的交互方式困难
- 退出时会清空屏幕，导致终端历史记录丢失（使用 alternate screen）

### 5. Windows 平台问题

在 Windows 上被杀毒软件标记为病毒，因为该工具能执行 shell 脚本、打开 URL 等操作，这些行为容易被杀毒软件检测为木马行为。

## 社区观点

### 替代方案比较

| 工具 | 编程语言 | 资源占用 | 特点 |
|------|----------|----------|------|
| OpenCode | TypeScript | >1GB | 功能丰富但资源消耗大 |
| Claude Code | Electron/TypeScript | 多GB | 使用 alternate screen |
| Codex | Rust | ~80MB | 可远程连接，支持客户端/服务器分离 |

### 语言选择讨论

社区讨论了 AI 编程代理应该使用什么语言：
- **Rust**: 性能更好，有更强的编译时验证，适合边缘设备
- **Go**: 更容易学习，可交付单一二进制文件，跨平台兼容性好
- **TypeScript**: 当前主流，但资源消耗大

## 总结

OpenCode 作为一个开源 AI 编程代理，虽然功能丰富但存在以下主要问题：

1. **安全性不足**：默认配置允许危险操作，会向第三方云服务发送用户数据
2. **资源消耗过高**：作为 TUI 应用，内存占用不合理
3. **开发节奏过快**：频繁发布导致质量不稳定
4. **隐私风险**：即使配置本地模型，仍可能向云端发送数据

用户建议的使用方式：
- 为 OpenCode 创建专用用户账户，限制其文件访问权限
- 使用 Docker 容器运行，并将网络访问限制在本地
- 明确配置 small_model 以禁用默认的云端模型
