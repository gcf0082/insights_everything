# 洞察报告：Building a CLI for all of Cloudflare

**洞察链接**：https://blog.cloudflare.com/cf-cli-local-explorer/

**发布日期**：2026-04-13

**作者**：Matt "TK" Taylor, Dimitri Mitropoulos, Dan Carter

---

## 摘要

Cloudflare 发布了全新统一 CLI 工具 `cf`（下一代 Wrangler）以及 Local Explorer 功能，旨在为开发者和 AI Agent 提供一致的命令行体验。`cf` CLI 将覆盖 Cloudflare 全部近 3,000 个 API 操作，而 Local Explorer 则允许用户在本地调试 KV、R2、D1、Durable Objects 等资源。

---

## 核心洞察

### 1. 新一代 CLI（cf）的诞生

- **背景**：Cloudflare 拥有超过 100 个产品、近 3,000 个 HTTP API 操作。许多产品没有 CLI 命令，而 AI Agent 非常喜欢使用 CLI。
- **解决方案**：重新构建 Wrangler CLI，发布技术预览版 `cf`，可通过 `npx cf` 或 `npm install -g cf` 安装试用。
- **特点**：
  - 支持所有 Cloudflare 产品的命令
  - 可通过基础设施即代码（IaC）进行配置
  - 命令输出针对 Agent 和人类用户均进行了优化

### 2. 代码生成管道的革新

- **现有问题**：Cloudflare 已使用 OpenAPI schema 生成 SDK、Terraform provider 和 MCP server，但 CLI、Workers Bindings、wrangler.jsonc、Agent Skills 等仍需手动维护。
- **新技术**：引入 TypeScript schema 系统，能够定义：
  - 完整 API 范围
  - CLI 命令和参数
  - 上下文信息
  - 生成的任意接口（包括 OpenAPI schema）
- **优势**：TypeScript 是 Cloudflare 的通用语言，可更好地表达 API、CLI 命令和绑定。

### 3. Agent 与 CLI 的一致性

- **核心挑战**：Agent 期望 CLI 命令命名一致（如 `get` 而非 `info`），手动审查无法保证大规模组织的一致性。
- **规则制定**：
  - 统一使用 `get`（不用 `info`）
  - 统一使用 `--force`（不用 `--skip-confirmations`）
  - 统一使用 `--json`（不用 `--format`）
- **本地与远程一致性**：Wrangler 支持本地和远程资源（如 D1、R2、KV），Agent 可能会混淆两者，导致数据操作错误。统一的默认值和输出提示至关重要。

### 4. Local Explorer — 本地调试功能

- **功能**：允许开发者在本地环境中检查和操作模拟资源，包括 KV、R2、D1、Durable Objects 和 Workflows。
- **使用方式**：运行 Wrangler CLI 或 Cloudflare Vite 插件时，按 `e` 键打开 Local Explorer。
- **API 端点**：本地应用运行在 `/cdn-cgi/explorer/api`，提供 OpenAPI 规范，Agent 可直接管理本地资源。
- **目标**：提供 Cloudflare API 的本地镜像，仅修改本地数据，使本地和远程使用相同的 API 形状。

### 5. 未来计划

- 将 `cf` 与现有 Wrangler 功能整合
- 扩展支持整个 Cloudflare API 表面
- 根据反馈优化命令输出
- 持续添加新命令（如 DNS 记录、缓存规则等）

## 总结

Cloudflare 通过发布 `cf` CLI 和 Local Explorer，大幅简化了开发者和 AI Agent 与其平台的交互方式。通过引入 TypeScript schema 系统，实现了跨工具的一致性自动生成，解决了手动维护的难题。Local Explorer 为本地开发提供了类似生产环境的调试能力，尤其对 AI Agent 工作流具有重要意义。
