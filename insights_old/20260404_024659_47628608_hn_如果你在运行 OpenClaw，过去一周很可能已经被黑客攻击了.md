## 内容摘要

这是一个关于 OpenClaw 安全漏洞（CVE-2026-33579）的讨论帖子。OpenClaw 创作者 steipete 在评论中解释了漏洞的实际影响。

### 漏洞技术细节

**根本原因**: 之前的修复不完整。早期公告加固了网关 RPC 路径（设备审批），但 `/pair approve` 插件命令路径仍然调用相同的批准函数而没有传递 `callerScopes`，核心逻辑在该参数缺失时默认失败开放。

**最强利用路径**: 已经拥有网关访问权限的客户端，可以使用 `chat.send` 配合 `/pair approve latest` 来批准请求更广泛权限（包括 `operator.admin`）的待处理设备请求。这是一个权限天花板绕过——从配对/写入级别访问提升到管理员权限。

**影响范围**: 这不是 Telegram 或 Discord 特有的漏洞，而是存在于共享插件命令处理程序中。任何已授权的命令发送者只要能访问 `/pair approve` 就可以利用。

### 社区讨论要点

1. **代码库规模**: OpenClaw GitHub 仓库包含超过 200 万行代码（cloc 统计），其中 160 万行是 TypeScript，有近 26000 次提交。

2. **暴露风险**: 135,000+ OpenClaw 实例暴露在公网，其中 63% 运行零身份验证。

3. **安全历史**: OpenClaw 有 400+ 安全问题和漏洞。

4. **实际风险评估**: OpenClaw 创作者表示，对于作为单用户个人助手使用的情况，实际风险非常低。团队正在与来自 Nvidia、ByteDance、Tencent 和 OpenAI 的人员一起努力加固代码库。

5. **用途讨论**: 用户讨论 OpenClaw 的实际用途，包括自动化任务（如健身房预约、代码编辑测试、市政会议追踪等）。
