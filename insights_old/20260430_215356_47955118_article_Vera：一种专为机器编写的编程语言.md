## 项目概述

Vera 是一个开创性的编程语言项目，其核心理念是「代码不需要正确，只需要可检查」。与传统编程语言不同，Vera 从设计之初就考虑到了 AI 编程代理的需求，试图解决大语言模型在代码生成时面临的核心问题。

根据项目的实证研究，模型在编程时最大的问题不是语法，而是跨规模的一致性保持。模型在维护代码库不变量、理解变更的连锁反应、以及随时间推理状态方面存在困难。实证文献表明，模型特别容易受到命名相关错误的影响，如选择误导性名称、错误重用名称、以及丢失对值的追踪。

Vera 通过使一切变得显式和可验证来解决这些问题。名称被结构化引用取代，合同是强制性的，效应是类型化的。每个函数都是一个规范，编译器可以针对其实现进行验证。

## 核心技术特性

### 1. 类型化 De Bruijn 索引

Vera 最有特色的设计是没有变量名。`@Int.0` 表示最近绑定的 `Int`，`@Int.1` 表示之前的那个。这种类型化 De Bruijn 索引方案完全消除了命名歧义，模型只能依赖结构化信息。

例如，一个安全的除法函数：

```
public fn safe_divide(@Int, @Int -> @Int)
  requires(@Int.1 != 0)
  ensures(@Int.result == @Int.0 / @Int.1)
  effects(pure)
{
  @Int.0 / @Int.1
}
```

这里没有变量名，只有位置索引和类型注解。

### 2. 强制性合同

Vera 要求每个函数都必须声明：

- `requires()`：前置条件
- `ensures()`：后置条件
- `effects()`：效应声明

这些合同由 Z3 SMT 求解器静态证明。与一些将合同作为可选注解的语言不同，Vera 的合同是强制性的，确保每个函数的行为都是机械可检查的。

### 3. 代数效应系统

Vera 默认是纯函数式的。调用 LLM 的函数必须在签名中声明 `effects(<Inference>)`，进行 HTTP 请求的函数必须声明 `effects(<Http>)`。调用者如果不许可相应的效应，就无法调用该函数。

内置效应包括：

- `IO`：控制台和文件 I/O
- `Http`：网络请求
- `State<T>`：类型化可变状态
- `Exn<T>`：类型化异常
- `Async`：异步操作
- `Inference`：LLM 调用
- `Random`：随机数生成

### 4. 三层验证体系

Vera 的合同检查分为三层：

**第一层：Z3 静态验证（可判定片段）**

编译器生成验证条件并发送给 Z3。如果 Z3 返回 `unsat`，则合同对所有输入都被证明。

**第二层：Z3 引导验证（扩展片段）**

通过 `assert` 语句和引理函数提供提示，覆盖函数调用、量词和数组属性。这一层已规范但尚未实现。

**第三层：运行时回退**

如果 Z3 返回 `unknown` 或超时，合同编译为运行时检查。

### 5. WebAssembly 编译目标

程序编译为 WebAssembly，可在命令行通过 wasmtime 运行，或在浏览器中通过自包含的 JavaScript 运行时运行。`vera compile --target browser` 生成一个包含 `.wasm` 二进制文件、JS 运行时和 HTML 的自包含包，无需构建步骤或打包器。

## 设计原则

1. **可检查性优先于正确性**：代码必须能机械检查，错误提供自然语言解释和具体修复方案。

2. **显式性优先于便利性**：所有状态变更声明，所有效应类型化，所有函数合同强制性。

3. **单一规范形式**：每个构造只有一个规范的文本表示，没有风格选择。

4. **结构化引用优于名称**：绑定通过类型和位置索引引用，消除命名一致性错误。

5. **合同作为真相来源**：每个函数声明其需求和保证，编译器尽可能静态验证。

6. **受限的表达性**：更少的有效程序意味着模型犯错的机会更少。

## 洞察结论

Vera 是一个高度创新和精心设计的编程语言项目，专门针对 AI 编程代理的需求。其核心理念——「可检查性优先于正确性」——非常有见地，因为它认识到 AI 模型在代码生成时的局限性，并提供了一种机制来确保生成的代码能被验证而不是盲目相信其正确性。

**主要优势**：

1. 类型化 De Bruijn 索引完全消除了命名歧义问题
2. 强制性合同确保每个函数的行为都是可验证的
3. 代数效应系统使副作用显式和类型化
4. Z3 SMT 验证提供了强大的静态保证
5. WebAssembly 目标确保了可移植性和沙箱安全
6. 浏览器运行时支持开箱即用

**潜在挑战**：

1. 新语言意味着训练数据稀缺
2. 强制性合同可能增加开发摩擦
3. 开发者需要适应无变量名的编程风格
4. 基准测试仍处于早期阶段，单次运行结果方差较大

**项目成熟度**：项目处于活跃开发状态，核心语言已完成，基准测试显示有前景的结果（Kimi K2.5 在 Vera 上达到 100% 正确率）。项目拥有完善的文档、规范和测试基础设施，四个战略里程碑规划清晰。
