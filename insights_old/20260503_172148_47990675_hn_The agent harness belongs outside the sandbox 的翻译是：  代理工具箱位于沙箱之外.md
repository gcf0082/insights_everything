## 核心话题

本文讨论了 AI Agent 的 harness（工具框架）是否应该位于沙箱内部的问题。作者认为 harness 应该放在沙箱外部，理由是：

### 主要观点

1. **信任问题**：作者认为目前对 harness 的信任不应该高于 LLM，因为两者都在快速演进。如果 harness 具有 LLM 没有的能力，并且 LLM 能触发这些能力，那么必须假设 LLM 最终会找出触发条件并执行它们。

2. **长期架构**：一些组件需要放在沙箱之外（否则谁创建沙箱？）。长期来看，应该有一个独立的安全层，类似于 hypervisor，位于所有事物之外，基于上下文、用户等进行授权。

3. **沙箱外方案**：不使用沙箱，而是给 agent 一台计算机，但将其与敏感资源隔离。使用 tokenization 或代理来处理凭证和密钥。

4. **架构选择**：许多解决这些问题的方式都有各自的优势，目前人们过于依赖特定路径，可能错失很多有价值的选择。

### 作者的反驳

文章作者 aluzzardi 解释了他们的架构：

- 他们构建的是用于构建 agent 的 harness，不是运行第三方 agent
- 既然是他们自己构建的，他们对 harness 的信任与对 API 服务器的信任一样
- agent loop 本身不需要沙箱化，大多数工具调用也不需要
- 需要沙箱的工具可以按需路由到专门的计算机，而不是整个软件都在沙箱中运行

## 关键洞察

1. **"lethal trifecta"概念**：如果 harness 有 LLM 没有的能力，且 LLM 能触发这些能力，就构成了一个危险的组合。这不是通过更好的编码就能解决的问题（如 SQL 注入），只能管理风险。

2. **沙箱 vs 机器**：两种主要方案——在沙箱中运行 harness + agent，或在独立机器上运行 harness 并将 agent 操作路由到沙箱。各有优缺点。

3. **权限管理**：路径路由（如 `/workspace` 到沙箱，`/memory` 到数据库）可能需要配合权限层来控制用户、会话或 agent 的访问范围。

4. **harness 定义**：去掉模型后剩下的部分就是 harness——包括工具、记忆、沙箱化、引导等。

5. **领域演进**：从 prompt engineering (2023-24) → context engineering (2025) → harness engineering (2026)。

## 总结

这场讨论揭示了 AI Agent 安全架构的核心挑战：如何在赋予 agent 能力和限制其风险之间取得平衡。关键共识包括需要将凭证与 agent 隔离开，并仔细考虑哪些组件应该驻留在沙箱内外。不同的架构选择反映了新兴领域的探索性质，目前尚无标准答案。
