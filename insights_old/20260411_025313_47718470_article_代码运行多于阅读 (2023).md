# 洞察报告：Code is run more than read

**链接**: https://olano.dev/blog/code-is-run-more-than-read/  
**发布时间**: 2023-11-30  
**作者**: Facundo Olano  
**标签**: software

---

## 核心洞察

作者提出了一个层层递进的优先级模型，用于指导软件开发和决策：

```
biz > user > ops > dev
```

即：**业务 > 用户 > 运维 > 开发**

---

## 主要观点

### 1. 代码写得多，读得少 → 维护者 > 作者
这是程序员的基本常识：代码被维护者阅读的时间远多于作者编写的时间，因此要注重代码可维护性。

### 2. 代码用得多，读得少 → 用户 > 维护者
软件的价值在于服务用户，即使代码写得再好，如果用户体验不好也毫无意义。应尽早将程序展示给用户并收集反馈。

### 3. 代码运行得多，读得少 → 运维 > 开发
"运行"不仅指执行程序，还包括部署、升级、监控、审计、维修、退役等全部运维工作。长期维护系统可靠性的成本远超构建时的便利。

### 4. 还有商业 → 业务 > 用户
软件有用且运行良好就会对组织有价值的假设有时过于简化。需要考虑预算、营销、截止日期、投资者利益等商业因素。

---

## 反模式（Smells）

| 反模式 | 表现 | 原因 |
|--------|------|------|
| 无法维护的代码 | maintainer > author | 过度优化、只有特定人能改的代码 |
| 不可用的软件 | dev > user | 不倾听用户、技术的先 |
| Works on my machine | dev > ops | 未考虑运维，过度设计 |
| 做正确的事 | dev > biz | 代码被视为终点 |
| 简历驱动开发 | dev > * | 开发者在做想做的事 |
| 幻想软件 | biz > user > ops > dev | 软件建了但很少上线 |

---

## 结论

作者强调，尽管商业因素有时会优先于用户，但软件工程师应该坚守职业道德，不伤害用户。用户与业务之间的关系是复杂的平衡：

```
user > ops > dev
biz > ops > dev
biz ≹ user
```

软件不仅仅是一门技术，更是一种责任。
