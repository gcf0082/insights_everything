## 摘要

本文详细记录了作者将Mac OS X 10.0 Cheetah移植到任天堂Wii的过程。Wii自2007年推出以来，已有多款操作系统成功移植（如Linux、NetBSD、Windows NT），而Mac OS X成为最新加入的成员。作者从头开发引导程序、内核补丁编写、驱动开发，最终实现了在Wii上运行完整Mac OS X桌面的目标。

---

## 1. 可行性调研

### 1.1 硬件兼容性

- **处理器**: Wii使用PowerPC 750CL处理器，与G3 iBooks和部分G3 iMac使用的PowerPC 750CXe有密切的血统关系，CPU不是阻碍因素。
- **内存**: Wii配备88MB内存（24MB 1T-SRAM + 64MB GDDR3 SDRAM），Mac OS X Cheetah官方要求128MB，但非官方可在64MB下启动。作者使用QEMU在64MB内存下验证了可行性。
- **其他硬件支持**: 串口调试输出（通过USB Gecko）、SD卡、帧缓冲视频输出、USB端口（键鼠）等。

### 1.2 软件兼容性

Mac OS X基于开源的Darwin（包含XNU内核和IOKit驱动模型），辅以闭源组件。移植的核心思路是修改开源部分以在Wii上运行Darwin，闭源部分可直接运行。

**Mac启动流程**:
1. **Open Firmware**: 最低层软件环境，负责硬件检测、设备树构建、加载引导程序
2. **BootX**: Mac OS X的引导程序，读取设备树、加载XNU内核
3. **XNU内核**: 初始化处理器、虚拟内存、IOKit、BSD等

---

## 2. 移植方案

评估了三种方案：
1. 移植Open Firmware，运行原版BootX
2. 移植BootX并修改，绕过Open Firmware
3. 从零编写自定义引导程序

最终选择方案3（参考Wii Linux项目的做法），因为Mac OS X一旦运行就不再依赖Open Firmware或BootX，且自定义引导程序更简洁，只需：
- 初始化Wii硬件
- 从SD卡加载内核
- 构建设备树和启动参数
- 将控制权交给内核

---

## 3. 引导程序开发

### 3.1 基础框架

基于Wii的低级示例代码ppcskel构建，该项目提供系统初始化、SD卡读取、帧缓冲绘图、调试日志输出等功能。

### 3.2 内核加载

XNU内核以Mach-O格式存储，需要解析load commands将不同节加载到内存指定位置。加载后的内存布局包括：异常向量、__TEXT、__DATA、__KLD、__LINKEDIT、__SYMTAB等段。

### 3.3 调用内核

通过将内核入口地址转换为函数指针并调用：
```c
(*(void (*)())kernel_entry_point)(boot_args_address, MAC_OS_X_SIGNATURE);
```

### 3.4 调试与追踪

内核启动后，作者通过二进制补丁方式让内核闪烁Wii前面板LED来追踪进度。通过分析XNU源码并用Hopper Disassembler交叉参考，成功定位并修改编译后的内核二进制。

追踪到的执行路径：
1. start.s: start → allStart → nextPVR → donePVR → doOurInit → noFloat → noVector → noSMP → noThermometer
2. ppc_init.c: ppcInit
3. pe_init.c: PE_INIT_PLATFORM
4. device_tree.c: find_entry（崩溃于300异常）

### 3.5 设备树创建

设备树是表示系统硬件的数据结构。由于Wii硬件固定，作者参考Wii Linux项目的设备树，在引导程序中硬编码了最小设备树（根节点包含cpus和memory子节点），后续逐步扩展。

---

## 4. 内核补丁

内核在设置视频和I/O内存时遇到问题。XNU这个时代假设了视频和I/O内存的位置，并重新配置BAT（块地址转换），与Wii的内存布局不兼容。

**开发环境**：
- Mac OS X Cheetah虚拟机（QEMU运行，无头模式）
- 主机通过NFS共享XNU源码，虚拟机通过NFS挂载
- 主机通过SSH控制虚拟机

构建好环境后，修复了BAT设置问题，并添加小补丁将控制台输出重定向到USB Gecko，使视频输出和串口调试日志正常工作。虚拟内存、IOKit、BSD子系统均成功初始化。

---

## 5. 驱动开发

### 5.1 IOKit驱动模型

Mac OS X驱动基于IOKit构建，采用面向对象编程。关键概念：
- **Driver**: 管理特定硬件的类
- **Nub**: 作为挂载点的类，用于驱动之间的通信

### 5.2 Wii硬件表示

Wii不使用PCI总线，而是使用名为Hollywood的系统芯片（SoC），包含GPU、SD卡、WiFi、蓝牙、中断控制器、USB等。Hollywood还包含ARM协处理器Starlet，通过IPC与主PowerPC处理器通信。

由于无法使用现有IOKit驱动家族（如IOPCIFamily），作者实现了等效的Hollywood驱动，创建nubs来表示其所有子设备。

### 5.3 Hollywood驱动

创建`NintendoWiiHollywood`驱动，通过设备树节点匹配。驱动遍历Hollywood节点的子设备，创建并发布`NintendoWiiHollywoodDevice` nubs供其他驱动挂载。

### 5.4 SD卡驱动

继承IOBlockStorageDevice，实现读取/写入SD卡。通过IPC调用Starlet上的MINI来实现SD卡通信（使用IPC_SDMMC_SIZE、IPC_SDMMC_READ、IPC_SDMMC_WRITE命令）。

**关键调试发现**: 缓存内存问题——SD卡数据写入缓存内存时，PowerPC CPU可能读到过期数据。解决方法是使用非缓存内存作为缓冲区。

成功后，系统找到根文件系统，启动日志显示：
```
Waiting on <dict>...<key>IOProviderClass</key><string>IOService</string>...
NintendoWiiSDCard: started
Got boot device = ...
BSD root: disk0s4
devfs on /dev
```

### 5.5 帧缓冲驱动

继承IOFramebuffer，实现显示相关方法。`getApertureRange`返回帧缓冲在内存中的位置（0x01700000，640x480，16位色）。

**颜色问题**: Wii视频编码器期望YUV像素数据，而Mac OS X使用RGB。解决方案是使用双帧缓冲——Mac OS X使用RGB帧缓冲，每秒60次转换为YUV帧缓冲供Wii硬件显示。

**控制台问题**: 早期启动的文本帧缓冲仍在使用。解决方法是从`isConsoleDevice`返回true，让系统认为新的IOKit帧缓冲与之前使用的是同一个。

## 6. USB支持

### 6.1 挑战

作者希望重用IOUSBFamily和AppleUSBOHCI驱动，但遇到三个主要障碍：

**障碍1**: Mac OS X Cheetah和Puma的IOUSBFamily源代码未开源，调试能力严重受限。

**障碍2**: AppleUSBOHCI要求其provider是IOPCIDevice，但Wii不使用IOPCFamily。解决方法是创建继承IOPCIDevice的假nub，使AppleUSBOHCI能匹配。

**障碍3**: AppleUSBOHCI假设小端字节序，而Wii使用reversed-little-endian（硬件层面字节交换），导致"双重交换"问题。

### 6.2 解决

通过在IRC上寻求帮助，获得了IOUSBFamily的CVS仓库。从源码打补丁重新构建后，USB键盘和鼠标成功工作。

## 7. 改进与完善

### 7.1 引导程序改进

- **多分区启动**: 解析Apple Partition Map（APM），列出所有可启动分区，让用户选择
- **驱动程序注入**: 不修改内核，而是让引导程序通过设备树的`/chosen/memory-map`节点加载Wii专用驱动，使能从未修改的Mac OS X安装分区启动

### 7.2 内核简化

将驱动移出内核后，内核修改简化为：
- BAT设置补丁（适配Wii的I/O地址和帧缓冲内存）
- 从设备树的"hollywood"节点获取I/O基地址
- 帧缓冲缓存一致性修复
