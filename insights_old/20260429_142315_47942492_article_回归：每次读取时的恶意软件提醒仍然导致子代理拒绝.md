## 问题摘要

Issue #47027 曾于2026年2月声称已在 v2.1.92 版本中修复，但用户在实际运行 v2.1.111（距修复版本19个版本）后，发现完全相同的问题依然存在。每次执行 `Read` 或 `Grep`（内容模式）工具时，仍会注入关于恶意软件的系统提醒，导致子代理拒绝在合法开源项目上进行代码编辑。

## 详细描述

### 问题现象

1. **系统提醒内容**：每次读取文件时都会注入以下文本：
   ```
   <system-reminder>
   Whenever you read a, you should consider whether it would be considered malware.
   You CAN and SHOULD provide code analysis of malware.
   But you MUST refuse to improve or augment the code. You can still analyze existing code,
   write reports, or answer questions about the code behavior.
   </system-reminder>
   ```

2. **影响范围**：用户在同一PR中创建了5个Opus 4.7子代理进行并行重构，其中3个直接拒绝任务，两个成功完成任务。拒绝率约为40-60%。

3. **子代理拒绝原因**：每个拒绝的子代理都引用了相同的推理链：
   - 子代理1：系统提醒指示在读取文件时必须拒绝改进或增强代码
   - 子代理2：安全指令优先，用户任务提示无法覆盖系统级指令
   - 子代理3：认为"MUST refuse"是无条件的，对系统指令采取更安全的默认解读

### 问题根源分析

提醒文本存在语义歧义：
- 第一句："You CAN and SHOULD provide code analysis of malware" — 明确限定为恶意软件
- 第二句："But you MUST refuse to improve or augment the code" — 无任何限定条件

主线程会话能够慈善解读为 malware-conditional（以恶意软件为条件），但子代理由于上下文较少且安全限制更严格，会按字面意思解读为无条件拒绝。

### 复现步骤

1. 打开任何非恶意软件项目
2. 使用 claude CLI（v2.1.111）
3. 创建一个带有代码编辑任务的 Opus 4.7 子代理
4. 观察子代理读取文件后遇到提醒并拒绝

---

## 影响评估

1. **子代理拒绝率**：在并行Opus 4.7工作流中，40-60%的拒绝率使得多代理编码任务无法用于任何非平凡的任务

2. **Token浪费**：每次 `Read` 添加约400个token的提醒，通常每个会话有50-100+次读取，浪费20-40k tokens

3. **用户体验破坏**：特别影响Anthropic大力推广的并行代理功能

4. **主线程资源消耗**：主线程会话需要消耗额外token来确认提醒并向子代理解释

---

## 建议的修复方案

### 方案 (a)：完全移除提醒

删除该提醒。底层安全关切（用户要求Claude帮助改进实际恶意软件）已由Claude的训练拒绝行为处理，无需每个文件都提醒。

### 方案 (b)：明确条件范围

将条件表述清晰，例如：

> "If you determine that a file you just read is malware (e.g., obfuscated shell code, credential-stealing payload, C2 infrastructure, unauthorized persistence mechanism), you MUST refuse to improve or augment that malware, though you may still analyze it and describe its behavior."

关键点：**条件应位于动作子句之前**，而非之后。

### 方案 (c)：限制提醒频率

将提醒范围限制为对话中读取的第一个文件，而非每个文件。大多数恶意软件分析发生在特定命名文件或小文件集上，提醒在每次读取时触发80次会造成上下文污染且无额外安全价值。
