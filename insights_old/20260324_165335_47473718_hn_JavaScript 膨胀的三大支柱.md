## 三大膨胀根源

### 1. 旧运行时支持（包含安全性和跨域问题）

**表现**：npm 依赖树中充斥着大量看似简单却包含许多深层依赖的工具函数。

**产生原因**：
- **支持极旧的引擎**：部分开发者需要支持 ES3 环境（如 IE6/7 或早期 Node.js），这些环境缺乏现代 JavaScript 特性
- **防止全局命名空间被修改**：Node.js 使用 "primordials" 模式保护自身，部分开发者将此模式应用于包开发
- **跨域值处理**：从 iframe 传递的值与父页面的同名类型不是同一引用（如 `window.RegExp !== iframeWindow.RegExp`）

**问题**：这些兼容性需求只服务于极少数人，但它们却成为了"热门路径"中的默认依赖。

---

### 2. 原子架构

**表现**：包被拆分到几乎原子化的粒度，每个小功能都成为一个独立的 npm 包。

**典型示例**：
- `shebang-regex` - 仅一行正则表达式：`/^#!(.*)/`
- `arrify` - 将值转换为数组
- `cli-boxes` - 包含盒子边框数据的 JSON 文件
- `path-key` - 获取当前平台的 PATH 环境变量名

**问题**：
- **单用途包**：许多原子包只有一个消费者，本质上等于内联代码但却增加了获取成本
- **重复**：同一包的不同版本在依赖树中重复出现（如 nuxt 的依赖树中 `is-docker`、`is-stream` 各有2个版本）
- **供应链风险**：维护者被攻击会导致大量小包被攻破

---

### 3. 过时的"Ponyfill"（马填充）

**背景**：
- **Polyfill**：在目标环境不支持时提供替代实现，会修改全局环境
- **Ponyfill**：导入式的 polyfill，不修改全局环境，库作者可以安全使用

**问题**：当特性已被所有主流引擎支持后，这些 ponyfill 仍然被保留。

**具体示例**：
- `globalthis` - `globalThis` 的 ponyfill（2019年已广泛支持，周下载量4900万次）
- `indexof` - `Array.prototype.indexOf` 的 ponyfill（2010年已广泛支持，周下载量230万次）
- `object.entries` - `Object.entries` 的 ponyfill（2017年已广泛支持，周下载量3500万次）

## 解决方案

### 使用 knip 移除未使用的依赖
knip 可以帮助发现和移除未使用的依赖、死代码等。

### 使用 e18e CLI 分析依赖
```bash
npx @e18e/cli analyze
npx @e18e/cli migrate --all
```

### 使用 npmgraph 可视化依赖树
npmgraph.js.org 可视化依赖图，帮助发现冗余包。

### 模块替换项目
module-replacements 项目记录了可用原生功能或更优替代品替换的包。

## 核心观点

> 我们都在为一个非常小众的群体付出代价——他们需要特殊的架构或超前的向后兼容性。

作者认为，这些兼容性需求在当初可能是正确的选择，因为当时的平台功能确实有限。但如今我们已经拥有这些特性多年，却仍然在下载所有的膨胀代码。

**解决思路**：反转局面。那一小部分特殊需求的用户应该使用他们自己的特殊栈，而不是让所有人共同承担成本。
