## 摘要

本文讲述了作者在开发一个实验性的Type-2 hypervisor时，发现并修复了一个Linux内核中的sign-extension bug。该bug导致在多核环境下使用hypervisor时系统崩溃，最终作者将此补丁提交并被合并到Linux内核主线。

---

## 背景

作者通过"重新造轮子"的方式学习虚拟化技术，开发了一个名为evmm的实验性hypervisor。虚拟化目前由硬件辅助实现，Type-2 hypervisor本质上是一个内核模块，向用户空间暴露API，由VMM（如QEMU或Firecracker）来协调虚拟机。

---

## 核心概念

### x86任务状态段（TSS）

在保护模式下，x86架构定义了TSS来保存任务信息（通用寄存器、段寄存器等）。现代64位系统采用per-core TSS，主要用于：
- 保存当前线程的内核栈指针
- 为NMI（不可屏蔽中断）和Double Fault提供已知的好栈

TR寄存器包含可见部分（16位偏移）和隐藏部分（基地址、限长、访问权限），避免CPU每次都从GDT中查找TSS。

### Hypervisor与TSS

Hypervisor本质上是操作系统之间的任务切换器。VMCS结构包含：
- Host-state区域：`HOST_TR_SELECTOR`和`HOST_TR_BASE`
- Guest-state区域：`GUEST_TR_SELECTOR`、`GUEST_TR_BASE`、`GUEST_TR_LIMIT`、`GUEST_TR_ACCESS_RIGHTS`

---

## 问题症状

作者在虚拟化Fedora开发环境（3个vCPU）上开发hypervisor，一切正常。但在真实物理机器（Intel Core i7-12700H，14核心）上运行几秒后系统崩溃，表现为：
1. NMI触发VM-Exit，CPU尝试从TSS读取内核栈
2. CPU遭遇致命页面故障，导致Kernel Oops
3. CPU中断被禁用，变成"僵尸"状态
4. 其他CPU等待IPI响应，陷入无限等待
5. RCU子系统饥饿，系统进入完全死锁

---

## 调试过程

1. 将hypervisor固定到单一核心后问题消失，说明问题出在跨核心迁移
2. 作者排除了自己编写代码的问题，开始怀疑"借用"的Linux内核代码
3. 在QEMU上增加vCPU数量后，发现虚拟机会立即重启（三重故障）
4. 最终将注意力转向TR和TSS

---

## 根本原因

问题代码使用了`get_desc64_base`函数从GDT中提取TSS地址：

```c
return ((uint64_t)desc->base3 << 32) |
       (desc->base0 | ((desc->base1) << 16) | ((desc->base2) << 24));
```

**问题**：C标准的整数提升规则导致sign-extension。当`base2`（uint8_t）的最高位为1时，左移24位后变成有符号int，转换为uint64_t时会发生符号扩展，导致高32位全部变成1，破坏了实际的base3值。

**示例**：
```
base0 = 0x5000
base1 = 0xd6
base2 = 0xf8  // 最高位为1！
base3 = 0xfffffe7c

预期结果: 0xfffffe7cf8d65000
实际结果: 0xfffffffff8d65000  // base3被破坏
```

---

## 解决方案

修复方法是将所有值在位移操作前强制转换为uint64_t：

```c
return (uint64_t)desc->base3 << 32 |
       (uint64_t)desc->base2 << 24 |
       (uint64_t)desc->base1 << 16 |
       (uint64_t)desc->base0;
```

此补丁已被合并到Linux内核主线：https://lore.kernel.org/kvm/20251222174207.107331-1-mj@pooladkhay.com/

## 经验教训

1. 不要盲目信任任何代码，即使是内核开发者也可能会犯错
2. AI在调试时可以帮助分析日志和提取要点，但不应完全依赖它进行调试
3. 问题的复现条件特定：只有当base2的最高位为1时才会触发
