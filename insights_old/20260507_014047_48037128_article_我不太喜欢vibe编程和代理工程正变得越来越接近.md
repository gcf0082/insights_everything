## 摘要

Simon Willison 在与 Joseph Ruscio 讨论 AI 编程工具时，意识到一个令人不安的现象：**vibe coding（氛围编码）**和**agentic engineering（代理工程）**之间的界限正在模糊。

## 核心观点

### 1. 两种模式的原始定义

- **Vibe Coding**: 不看代码的编程方式，用户可能不会编程，得到能用的东西就太好了。适用于个人工具，不适合为他人构建软件。
- **Agentic Engineering**: 专业软件工程师使用 AI 工具，理解安全、可维护性、运维等，使用工具提升自己的能力。

### 2. 两者开始融合

随着编码代理变得越来越可靠，Simon 不再审查代理编写的每一行代码，即使是其生产级别的代码。他举例：如果让 Claude Code 构建一个 JSON API 端点来运行 SQL 查询并输出结果，代理会正确完成，不需要审查。

这带来了内疚感：如果没有审查代码，在生产环境中使用是否真的负责任？

### 3. 对比信任来源

Simon 通过类比来缓解这种不适：在大型组织中，其他团队构建的软件会被当作半黑盒使用——查看文档，使用服务，只在出现问题时才深入代码。但关键区别是：**人类团队有专业声誉**，而 Claude Code 没有。

### 4. 软件评估的新挑战

以往通过查看 Git 仓库的提交、README、自动化测试来判断项目质量。现在可以在半小时内生成类似外观的仓库。Simon 意识到，他更看重的是**是否有人真正使用过这个工具**——一个被使用了两周的 vibe coding 产品比刚生成但几乎未使用的更有价值。

### 5. 瓶颈已转移

如果日产 200 行代码变成 2000 行，整个软件开发生命周期都需要重新设计。上游的设计流程也需要更灵活，因为构建成本大幅降低。

### 6. 为什么不怕失业

- 与 AI 工具交互对大多数人来说是一门"外星语言"
- 这些工具是现有经验的放大器
- 生产软件是极其困难的事情
- Matthew Yglesias 的观点："我不想 vibe coding——我希望专业管理的软件公司使用 AI 编程辅助来为我制造更好/更多/更便宜的软件"

## 关键引用

> "The goal is to build high quality production systems: if you're building lower quality stuff faster, I think that's bad. I want to build *higher* quality stuff faster."

> "I want somebody to have *used* the thing. If you've got a vibe coded thing which you have used every day for the past two weeks, that's much more valuable to me than something that you've just spat out and hardly even exercised."

## 结论

Simon Willison 认为，虽然 vibe coding 和 agentic engineering 的界限正在模糊，但这未必是坏事。关键在于理解何时使用何种方式，并保持对代码质量的追求。
