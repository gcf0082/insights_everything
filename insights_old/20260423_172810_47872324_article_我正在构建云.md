## 文章核心内容总结

### 作者背景

David Crawshaw 是 Tailscale 的联合创始人之一，正在创建一家新公司 exe.dev。本文是他关于为何要重新构建云计算服务的深度思考。

### 当前云计算的核心问题

作者认为当今云服务存在几个根本性的设计缺陷：

1. **虚拟机抽象错误**：虚拟机被绑定到 CPU/内存资源，而实际上 Linux 虚拟机只是另一个 Linux 进程中的 cgroup，应该可以任意运行多个，而非被云厂商的选择所限制。

2. **存储性能问题**：云厂商推荐使用的远程块设备在机械硬盘时代有意义（寻道时间 10ms，网络 RTT 1ms 可以接受），但进入 SSD 时代后，寻道时间降至 20 微秒，远程存储的 IOPS 开销从机械硬盘的 10% 上升到 SSD 的 10 倍以上。配置 200k IOPS 的 EC2 虚拟机需要每月支付约 1 万美元，而作者的 MacBook 就拥有 500k IOPS。

3. **网络成本高**：主流云厂商的出站流量费用是普通数据中心的 10 倍，这限制了开发者构建经济可行的服务。

4. **痛苦的 API 设计**：Kubernetes 试图掩盖这些痛苦，但它本身就是在尝试解决一个不可能的问题——让云服务既可移植又好用。

### 解决方案与 exe.dev 的诞生

作者认为，随着 AI 代理（agents）的出现，软件编写变得更容易，这将创造更多软件（Jevons 悖论）。人们需要更私密、更易管理的计算环境。

exe.dev 的核心理念是回归计算本质：

- 不再预配置虚拟机，而是提供 CPU 和内存，由用户自行决定运行何种虚拟机
- 自动配置 TLS 代理和认证代理
- 本地 NVMe 磁盘，异步复制到其他机器
- 全球多区域部署，通过 anycast 网络提供低延迟入口

### 对未来的展望

作者表示还有很多功能需要构建（ 如静态 IP、历史磁盘快照等），他们将从零开始，在数据中心托管服务器，重新思考软件栈的每一层。

## 关键洞见

1. **抽象层面的根本问题**：云厂商提供的抽象（VM、远程块存储）与其底层硬件能力之间存在巨大鸿沟，这种抽象反而限制了用户的能力。

2. **技术演进与定价策略的矛盾**：SSD 技术已普及，但云厂商的存储定价和产品仍基于机械硬盘时代的架构，用户在为此付出额外代价。

3. **AI 时代的新需求**：AI 代理使软件开发更普及，预期将产生更多软件，对云服务的需求和质量期望也在提升。

4. **回归基础的可能性**：真正的创新来自于回归基础，重新思考每一个软件层的实现方式，而非在现有抽象之上修修补补。

## 对行业的启示

这篇文章反映了云用户长期以来对主流云服务的不满——被锁定在厂商特定的抽象中、为不必要的性能损失付费、面临复杂的 API 和定价结构。exe.dev 的尝试代表了 一种「回归本质」的设计思路：让用户更直接地控制底层资源，而不是被层层抽象所束缚。
