# 洞察报告：GitHub可用性更新

**链接：** https://news.ycombinator.com/item?id=47932422

**来源：** Hacker News

**发布日期：** 2026年4月29日

**摘要：**

GitHub发布了关于其可用性问题的更新声明，承认过去一年面临严重的服务中断问题，并承诺将"可用性置于首位，其次是容量，然后是新功能"。然而，用户对这份声明反应冷淡，认为缺乏具体解决方案和实质性承诺。评论者指出，GitHub图表缺乏明确数据，令人质疑其透明度。

---

## 核心要点

### 1. 爆炸性增长压力

GitHub声称经历了惊人的增长：
- **提交（commits）**：同比增长13倍
- **合并请求（PRs）**：从数百万增长到9000万
- **新仓库**：2026年初的增长超过了此前三年的总和
- 这种指数级增长主要来自AI代码生成工具和自动化代理

### 2. 服务中断现状

- GitHub Actions已近一年时间处于"边缘可用"状态
- 过去14天内发生两次服务中断，其中一次导致**数据丢失**
- 用户报告PR列表显示不完整数据
- 网站UI存在可靠性问题

### 3. Azure迁移问题

- GitHub原本计划完全迁移至Azure
- 迁移过程中发现问题，现已转向**多云架构**
- 评论者普遍认为Azure是三大云服务商中**最不可靠**的选择
- 有用户提到微软内部也在避免使用Azure（如LinkedIn）

### 4. 用户反馈

社区反应强烈批评：

| 批评点 | 具体内容 |
|--------|----------|
| 缺乏透明度 | 图表未标注Y轴，数据无法验证 |
| 行动滞后 | 问题持续一年多才发布声明 |
| 优先级矛盾 | 一边说暂停新功能，一边每周都在发布新功能 |
| 数据问题 | 付费用户数据展示不完整 |

### 5. 商业影响

- 企业用户每年为每位开发者支付$250，但服务可靠性持续下降
- 有用户怀念Jira——这在以前是不可想象的
- 有人开始考虑自建Git服务或迁移到Azure DevOps

## 技术分析

### 增长来源
AI代理和自动化工具产生的代码提交是主要增长驱动力。这些代理产生的代码价值存疑——有用户指出现在平均提交的价值可能不到2018年的十分之一。

### 云架构问题
- Azure缺乏其他云厂商的可靠性
- 从自有数据中心迁移到云服务商本身就是挑战
- 多云策略本应是为了供应商独立性，但对于微软内部公司来说理由不充分

### 前端工程的局限
有评论指出，前端工程师无法解决底层架构可靠性问题，这需要专门的多云扩展经验。

---

## 结论

这份声明被广泛认为是**公关性质的空话**。用户的主要诉求是：
1. 具体的改进时间表和里程碑
2. 真实的数据透明化
3. 真正的资源调配到可靠性工作

目前的承诺——"可用性第一，容量第二，功能第三"——与用户实际经历不符。社区普遍持怀疑态度。
