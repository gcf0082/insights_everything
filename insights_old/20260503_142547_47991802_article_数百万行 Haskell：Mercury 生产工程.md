## 核心洞察

### 1. 可靠性哲学：从"防止失败"到"吸收变化"
传统可靠性思维聚焦于防止失败（列出所有可能出错的地方，添加检查、写测试），但这不够。

**更有效的思维**：系统能够**吸收变化**——优雅降级、运营者能理解和调整架构、让正确的事情变得容易，错误的事情变得困难。

在快速成长的公司（每年2倍增长），一半同事永远只有不到一年的经验。可靠性不仅是 absence of failure，更是 presence of adaptive capacity（自适应能力）。

### 2. 类型系统作为运营工具
类型系统的价值不仅是排除某些错误，更是**编码机构知识**——以 surviving the departure of the person who wrote it 的形式存在。

通过类型编码关键操作流程，确保新工程师无法绕过正确做法。例如：将数据库写入和事件发布封装为 `Transact` 类型，使得 commit 是唯一执行路径。

### 3. 纯性是边界，不是属性
Haskell的纯性不是语言属性，而是**接口强制执行**的边界。

- `bytestring`、`text`、`vector` 内部使用 mutable allocation、unsafe coercions
- `runST` 的 rank-2 类型确保 mutable 引用无法逃逸
- 内部可以是 imperative，外部表现是 pure

**生产经验**：目标不是完全避免 mutation，而是 containment mutation（contain mutation, make the containment legible, verify that it stays contained）。

### 4. 持久执行框架：Temporal
Mercury采用 Temporal 作为持久执行框架，处理跨多服务、多步骤、多失败模式的分布式工作流。

关键优势：
- 工作流作为普通顺序代码编写
- 平台记录每一步到 event history
- worker崩溃时另一个 worker 重放 deterministic prefix
- 平台处理 retries、timeouts、cancellation、error handling

Mercury开源了 `hs-temporal-sdk`（Haskell SDK for Temporal）。

### 5. 领域模型与传输层分离
常见错误：让 invoking system leak into domain model。

例如：HTTP status code exceptions 被错误地用于 cron jobs、background workers、Temporal workflows。

**解决方案**：将领域错误建模为领域类型（`InsufficientFunds`、`DuplicateRequest`），在每个边界编写薄翻译层。

### 6. 类型编码权衡
类型编码 powerful 但 expensive。

**最佳实践**：
- 编码防止 silent corruption 的不变量（违规会产生产生错误数据但无立即错误）
- 对立即失败的场景使用 runtime checks
- 不要试图用类型建模整个领域
- 类型是为团队服务的，不是为编译器

关键：encapsulate complexity（将复杂性封装在模块内，对外暴露简单的 API surface）。

### 7. 可观测性设计
**核心原则**：You cannot operate what you cannot see。

Haskell 没有 monkey patching，需要通过 records of functions 提供可注入性：

```haskell
data HttpClient = HttpClient
  { sendRequest :: Request -> IO Response
  , getManager  :: IO Manager
  }
```

这种模式允许在运行时包装、instrument、mock、替换任何函数。

**Monoid 模式**：组合多个 interceptor 为 trivial task：
```haskell
appTemporalInterceptors = mconcat [retargetingInterceptor, otelInterceptor, sentryInterceptor, ...]
```

### 8. 库作者的建议
- 考虑依赖 `hs-opentelemetry-api` 添加 instrumentation
- **不要直接从库代码 logging**（提供 callback 或 logger 参数）
- 考虑暴露 `.Internal` modules（帮助用户避免 fork）

### 9. Haskell招聘的现实
- **优势**：对Haskell岗位的需求超过市场供给（反转通常的招聘动态）；Haskell 是 baseline developer quality 的良好代理
- **挑战**：Haskell 吸引理想主义者，需要培养实用主义文化

### 10. 投资回报
- **早期更慢**：生态系统不如 Next.js/Rails 完善
- **后期回报**：refactor 时间从天级缩短到小时级；新工程师通过类型签名理解契约；在生产事故中 "impossible state was actually impossible"

## 关键要点总结

| 主题 | 核心观点 |
|------|----------|
| 可靠性 | 关注 adaptive capacity，不仅仅是防止失败 |
| 类型系统 | 作为运营工具，编码机构知识 |
| 纯性 | 边界导向，不是语言属性 |
| 正确路径 | 通过类型让正确的事情变得 easy |
| 分布式工作流 | 使用 Temporal 持久执行框架 |
| 可观测性 | 设计时考虑，提供 escape hatches |
| 库设计 | records of functions、避免直接 logging |
| 实用主义 | 类型是 power tool，不是 religion |
