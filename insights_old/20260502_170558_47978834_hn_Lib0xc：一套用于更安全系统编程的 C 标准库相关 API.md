## 项目概述

Lib0xc 是微软发布的一套 C 标准库相关的 API，旨在提高系统编程的安全性。该项目利用 GNUC 扩展和 C11 特性，将更安全的 C 编程实践和模式固化为实际的 API，包含完整的文档和测试。

### 主要特性

- **context_t**: 带标签的指针类型，减少与 `void *` 之间的转换
- **call_t**: 类型检查的延迟函数调用
- **struct_field_t**: 结构体字段描述符
- **__cast_signed_unsigned**: 有符号到无符号整数的安全转换，避免 `-Wint-conversion` 警告
- **bounds-safety**: 支持 clang 的 bounds-safety 扩展
- **系统工具**: 日志、单元测试、缓冲区对象、统一 Mach-O 和 ELF 链接器集

### 许可

MIT 许可证，由微软持有版权并允许发布。

## 讨论要点

### 1. 与现代语言的对比

有用户询问 lib0xc + clang bounds-safety 与 Zig 或 Odin 等现代语言的区别。作者回复称 Zig 等语言仍有许多优势，例如在编译时评估任意表达式的能力。但作者也指出，lib0xc 提供的是**边界安全**（bounds-safety），而 Zig 和 Rust 提供的是**类型安全**（type-safety），后者是超集。

### 2. C 语言缺少的特性

作者认为如果 C 语言要达到 Zig 的用户体验，需要以下特性：
- 协议/ trait
- 从调用者栈帧分配内存
- stdint.h 类型的 printf 格式说明符
- 将函数参数列表表达为结构体
- 新的 sprintf 系列，返回值始终小于等于传入的大小

### 3. defer 机制

关于 defer 块的支持，作者表示 C2y 提案中已包含 defer，且 GCC 9 和 clang 22 已支持。lib0xc 提供了变量注解形式的清理属性，可自动释放堆分配或关闭文件描述符。

### 4. 增量兼容性

作者强调 lib0xc 设计为增量兼容。例如可以将 `sprintf` 替换为 `ssprintf`，函数签名相同。任何打印到字符数组的代码都可以直接工作。整数转换也是如此，可以逐步替换。

### 5. Microsoft 的立场

作者澄清：Microsoft 支持内存安全，Rust 是新项目的首选方向。但现有的 C 代码库不太可能完全重写，此类项目可以从内存安全的增量改进中受益。

### 6. 标准委员会的问题

评论指出 C 标准委员会数十年来拒绝承认问题的存在。C 语言的章程有"不发明"规则——任何特性需要先在实践中证明，然后再纳入标准。这导致了先有鸡还是先有蛋的问题：编译器没有动力实现更安全的 API，因为非标准 API 无法保证可移植性。

### 7. 嵌入式和底层开发

关于 `void *` 是否会伤害嵌入式开发或复古游戏开发，作者表示不会——仍然可以按字节寻址内存。问题在于传递 `void *` 时没有关联长度信息，这表明语言层面的失败。
