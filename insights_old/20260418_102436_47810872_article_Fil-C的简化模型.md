## 摘要

本文介绍了一个关于 Fil-C（一个 C/C++ 内存安全实现）的简化模型。Fil-C 通过引入自动内存管理机制，将不安全的 C/C++ 代码转换为安全代码。该模型核心包括：指针变量的伴随分配记录、边界检查、垃圾回收器等。

##Fil-C 核心机制

### 1. AllocationRecord 结构体

Fil-C 为每个指针变量引入一个伴随的 `AllocationRecord*` 类型的局部变量：

```c
struct AllocationRecord {
  char* visible_bytes;    // 可见内存区域
  char* invisible_bytes; // 不可见内存区域（存储指针的伴随记录）
  size_t length;        // 长度
};
```

### 2. 指针操作的转换规则

| 原始代码 | Fil-C 转换后 |
|---------|-------------|
| p1 = p2; | p1 = p2, p1ar = p2ar; |
| p1 = p2 + 10; | p1 = p2 + 10, p1ar = p2ar; |
| p1 = (T1*)x; | p1 = (T1*)x, p1ar = NULL; |
| x = (uintptr_t)p1; | x = (uintptr_t)p1; |

### 3. 内存分配与释放

Fil-C 使用 `filc_malloc` 和 `filc_free` 替代标准的 `malloc` 和 `free`：

```c
void* filc_malloc(size_t length) {
  AllocationRecord* ar = malloc(sizeof(AllocationRecord));
  ar->visible_bytes = malloc(length);
  ar->invisible_bytes = calloc(length, 1);
  ar->length = length;
  return {ar->visible_bytes, ar};
}
```

### 4. 边界检查

当解引用指针时，会执行边界检查：

```c
assert(p1ar != NULL);
uint64_t i = (char*)p1 - p1ar->visible_bytes;
assert(i < p1ar->length);
assert((p1ar->length - i) >= sizeof(*p1));
x = *p1;
```

### 5. 垃圾回收器

Fil-C 集成了一个垃圾回收器（GC），主要功能：

- 自动释放不可达的 `AllocationRecord` 对象
- 对已释放的 `AllocationRecord`，其指针最终会变为空
- 忘记调用 `free` 不再导致内存泄漏

### 6. 局部变量地址提升

如果编译器检测到局部变量的地址可能超出其生命周期，会将其提升为堆分配（通过 `malloc`），而不是栈分配。

### 7. memmove 的特殊处理

对于跨越指针边界的 `memmove` 操作，需要同时移动对应的 `invisible_bytes` 区域。

## 生产级版本的额外复杂性

- **线程安全**：并发使 GC 复杂化，`filc_free` 无法立即释放内存
- **函数指针**：需要额外的元数据来标记可执行代码
- **内存使用优化**：延迟分配 `invisible_bytes` 等优化策略
- **性能优化**：通过各种技巧弥补内存安全的性能开销

## 使用场景

1. 大型 C/C++ 代码需要内存安全保证，愿意接受 GC 和性能开销
2. 作为 ASan 的替代方案，运行代码以发现内存错误
3. 编译时安全评估（如 Zig 语言的编译时求值）
4. 指针来源（pointer provenance）的具体实现示例
