## 核心洞察

### 1. 词法分析/语法解析比你想象的简单

- **手写递归下降解析器（Recursive Descent Parser）完全够用**，不需要 lex/yacc/bison/ANTLR 等工具
- 编译器课程教这些工具是因为《龙书》等经典教材的影响，但实践中这些工具往往过度设计
- lex/parser 解析从来不是性能瓶颈，实际上一个普通语言的手写 lexer + parser 一个周末就能完成
- 配合 Pratt 解析技术，可以处理大多数语言的语法
- 使用正则表达式做词法分析也被认为过于繁琐，手写 lexer 更直接高效

### 2. 最大的失败原因：无法限制项目范围

- 这是几乎所有语言项目失败的根本原因
- 开发者从有限的功能集开始，完成整个编译器/虚拟机工具链后，就会开始"贪婪"，对类型系统修修补补，添加从未使用过的功能
- 一旦扩展范围，就需要修改从开始到结束的每一个阶段
- 建议：严格控制功能范围，避免过度设计

### 3. LLM 正在改变游戏规则

- 在 LLM 的帮助下，写一个编程语言只需 15 分钟（如果你有明确的语法规范）
- 有开发者用 LLM 来迭代语言规范和虚拟机规范，消除了歧义和不必要的功能
- 有开发者利用 LLM 作为"配对设计师"，让 LLM 总结各种设计决策的影响
- 有人用 LLM 写了超过 5000 个测试用例
- 有人用 "vibe engineering" 方式，几乎不亲自审查代码，交给 LLM 迭代

### 4. 生产级语言 vs 爱好者语言

- **爱好者语言**: 手写递归下降解析器 + 编译到 C 或 LLVM 就足够了
- **生产级语言**: 需要考虑错误报告与恢复、跨平台支持、大量工作投入
- 真正有创新的成熟语言需要多年开发
- Python 主导的世界里，生产级编译器的需求可能并不迫切
- 真正困难的不是写解析器，而是中间端（middle end）——元编程、类型系统、代码优化等

### 5. 社区推荐的学习路径

- 推荐从 Nystrom 的 **Crafting Interpreters** (https://www.craftinginterpreters.com/) 入手，而不是直接看龙书
- Crenshaw 的 "Let's Build a Compiler" 也很有用
- 实际编写 DSL 和配置语言的 parser 比看书更有价值

### 6. 语言设计建议

- **语法只是表面问题**：很多人对语法非常固执己见，但语法只是很小的一部分
- 简单命令式语言（类似 BASIC）很容易理解，适合作为入门
- 有开发者共享了自己用 LLM 开发语言的项目（Sapphire、Loonlang 等）
- 游戏开发专用语言可以有：向量数学、异步支持、特殊的 render state 管理、帧级内存管理等特性
- FFI/C interop 是语言设计中需要权衡的重要因素

### 7. 语言成功真正的难点

- **不是写编译器，而是生态**：库、工具链、文档、诊断信息、优化
- Specification（规范）、Implementation（实现）、Documentation（文档）、Diagnostics（诊断信息）、Optimization（优化）、Tooling support（工具支持）、Standard library（标准库）这些都是需要投入大量工作的方面
- 一门语言能否成功，很大程度上取决于已有的库生态

## 总结

构建自己的编程语言入门门槛比传统认知低得多——手写递归下降解析器配合 LLM 能在极短时间内完成基础框架。但真正将语言变成可用、成熟产品的挑战是：严格控制项目范围（避免功能蔓延）、投入大量时间构建生态和工具链。LLM 的介入大幅降低了从规范到代码的转化成本，使得"语言极客"可以更专注于语言设计和用户价值，而不是被实现细节困住。
