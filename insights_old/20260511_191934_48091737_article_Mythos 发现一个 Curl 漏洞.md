**标题**：Mythos finds a curl vulnerability

**发布日期**：2026年5月11日

**作者**：Daniel Stenberg（curl 项目负责人）

**来源**：daniel.haxx.se

## 摘要

2026年5月11日，curl 项目负责人 Daniel Stenberg 发布博客文章，介绍了 Anthropic 公司开发的 AI 模型 Mythos 对 curl 代码库进行安全扫描的结果。Mythos 报告发现了5个"确认的安全漏洞"，但经 curl 安全团队深入分析后，仅确认了1个低严重级别的漏洞，其余4个为误报或普通缺陷。这一结果表明，虽然 AI 驱动的代码分析工具在发现安全缺陷方面表现出色，但关于 Mythos 的营销宣传可能过于夸大。

## 背景

### Anthropic Mythos 模型

2026年4月，Anthropic 宣布其新 AI 模型 Mythos"非常擅长"发现源代码中的安全漏洞，引发广泛关注。Anthropic 决定暂不公开发布该模型，而是通过 Alpha Omega 项目与选定的公司合作。

### curl 项目的 AI 扫描合作

通过 Linux Foundation 的 Alpha Omega 项目，curl 获得了 Mythos 的扫描机会。扫描于2026年5月6日完成，针对 curl git 仓库的 master 分支（特定提交），分析了 src/ 和 lib/ 目录共178,000行代码。

### curl 的现有 AI 扫描历史

在 Mythos 扫描之前，curl 已使用多种 AI 工具进行过扫描：
- **AISLE**：已触发约200-300个 bug 修复
- **Zeropath**：发现多个确认的漏洞并发布 CVE
- **OpenAI Codex Security**：同样发现多个漏洞

这些工具在近8-10个月内共发现并修复了约200-300个 bug，其中十几个被确认为漏洞并以 CVE 形式发布。

## 扫描结果

### Mythos 报告的发现

Mythos 报告共发现：
- **5个"确认的安全漏洞"**
- **约20个非安全类 bug**

### curl 安全团队验证

经过数小时深入分析后，curl 安全团队将结果缩减为：
- **1个确认的漏洞**：严重级别为"低"，计划与2026年6月底发布的 curl 8.21.0 同步发布 CVE
- **3个误报**：被标记的缺点在 API 文档中已有说明
- **1个普通 bug**：非安全相关

### 关键发现

- **零内存安全漏洞**：未发现任何内存安全问题
- **热门路径无问题**：HTTP/1、TLS、URL 解析核心等热点区域未发现问题
- **误报率极低**：报告的 bug 描述清晰准确，几乎没有误报

### curl 代码库规模

- **代码行数**：176,000行 C 代码（不含空行）
- **代码量**：660,000个单词，比《战争与和平》英文版多12%
- **代码重写率**：平均每行生产代码被重写4.14次
- **贡献者**：573人当前活跃，历史上共1,465人贡献过代码
- **已发布 CVE**：188个
- **安装量**：超过200亿个实例
- **支持平台**：110+操作系统，28种 CPU 架构

## 技术细节

### Mythos 分析方法

- 采用手动驱动分析，使用 LLM 子代理进行并行文件读取
- 每个候选发现都通过直接源代码检查进行重新验证
- 使用 curl 自己的 vuln.json 构建 CVE 到变体映射
- 未使用自动化 SAST 工具

### curl 的防御基础设施

报告中提到 curl 的系统性防御措施：
- 处处使用受限的动态缓冲区（capped dynbufs）
- `curlx_str_number` 对每次数值解析设置明确上限
- `curlx_memdup0` 溢出保护
- CURL_PRINTF 格式字符串强制执行
- 每个协议响应大小限制
- pingpong 64KB 行限制

### 扫描覆盖范围

覆盖范围包括：所有次要协议、所有文件解析器、所有 TLS 后端验证路径、HTTP/1/2/3、FTP 完整深度、mprintf、x509asn1、DoH、所有认证机制、内容编码、连接重用、会话缓存、CLI 工具、平台特定代码、CI/构建供应链。

## 分析结论

### 关于 Mythos 的评价

Daniel Stenberg 认为：
- Mythos 的营销宣传可能过于夸大
- 未发现 Mythos 在发现漏洞方面比其他工具显著更优秀
- 即便有所改进，也未达到在代码分析领域产生实质性影响的程度
- 这仅代表一个代码仓库的情况，可能在其他场景中表现更好

### AI 代码分析工具的整体评价

尽管对 Mythos 持保留态度，Daniel 强调：
- **AI 驱动的代码分析工具显著优于传统静态分析工具**
- 所有现代 AI 模型在发现安全缺陷方面都表现出色
- 尚未使用 AI 工具扫描的项目应该会从这类工具中发现大量缺陷
- 不使用 AI 代码分析工具意味着给对手留下发现和利用漏洞的时间和机会

### AI 工具与传统工具的区别

- 可以发现代码注释与实际行为不一致的情况
- 可以检查无法运行的平台和配置
- 了解第三方库及其 API 的细节
- 了解 curl 实现的协议细节，能够发现与协议规范相矛盾的问题
- 能够总结和解释缺陷，通常比传统分析器更清晰
- 常常能够生成和提供修复补丁（尽管通常不是100%完美的修复）

### 未来展望

- AI 工具会发现更多问题，但都是已知的漏洞类型
- 尚未发现 AI 报告过全新类型的漏洞
- AI 工具和人类安全研究人员都在不断改进
- 仍会继续使用 Mythos 和其他 AI 工具进行更多扫描

## 总结

本次 Mythos 对 curl 的扫描验证了 AI 工具在代码安全分析方面的能力，但并未展现出比现有工具更显著的优势。curl 作为最经过 fuzz 测试和审计的 C 代码库之一，其系统性防御措施有效减少了漏洞发现。对于 AI 代码分析工具的价值，应该保持理性预期——它们是强大的辅助工具，但并非万能的安全解决方案。
