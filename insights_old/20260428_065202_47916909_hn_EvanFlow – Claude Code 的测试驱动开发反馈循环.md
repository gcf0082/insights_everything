## 项目概述

EvanFlow 是一个为 Claude Code 设计的 TDD（测试驱动开发）驱动反馈循环框架。它通过一个结构化的工作流程引导功能开发，在每个关键决策点都需要人工批准，确保开发过程的质量和可控性。

## 核心功能

### 1. 单一 TDD 驱动循环
用户只需说 "let's evanflow this"，EvanFlow 就会引导完成整个开发流程：
- **Brainstorm**（头脑风暴）
- **Plan**（计划）
- **Execute**（执行）
- **TDD**（测试驱动开发）
- **Iterate**（迭代）
- **STOP**（停止）

每个阶段都设有真实的检查点，包括设计审批和计划审批。

### 2. 垂直切片 TDD
- 一个失败的测试 → 最小实现 → 下一个测试
- 在编写通过测试的实现之前，先观察每个测试失败
- 这看起来很明显，但几乎没有任何 AI Agent 默认这样做
- 根据 HumanEval 研究，约 62% 的 LLM 生成的测试断言是错误的

### 3. 嵌入式 grill（质询）
在决策点进行深度质询：
- 如果用户做 X 会发生什么？
- 回滚方案是什么？
- 什么明确不在范围内？
- 这能够在设计阶段早期发现缺陷，此时修复成本最低

### 4. 迭代直到清洁（Iterate-until-clean）
- 硬性上限 5 轮
- 重新阅读差异，对比死代码、命名、删除测试、断言正确性
- 五种失败模式检查：幻觉行为、范围蔓延、级联错误、上下文丢失、工具误用
- 对于 UI：通过无头 Chromium 进行截图验证

### 5. 开发者全程控制
- 从不自动提交
- 从不自动暂存
- 从不主动提出集成
- 每一个 Git 操作都由开发者决定

## 技术特点

- **vendor-agnostic（供应商无关）**：不依赖特定测试运行器
- **多 AI 助手兼容**：支持 Claude Code、Codex 和 GitHub Copilot
- **验证器上下文**：能看到最近的会话历史，帮助更好地处理重构等场景
- **大型计划处理**：对于涉及 3 个或更多独立单元共享类型的计划，会分叉为并行的 coder/overseer（编码器/监督者）编排模式
- **集成测试**：触点处的集成测试即作为内聚契约

## 相关讨论

评论区中有用户质疑 TDD 在 2026 年的必要性，认为 TDD 的主要好处是为系统提出合理的架构，而 LLMs 如果被正确指导已经可以做到这一点，因此看不到 TDD 的意义。
