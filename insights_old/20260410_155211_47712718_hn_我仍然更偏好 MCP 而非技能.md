## 洞察摘要

这是一场关于 **MCP (Model Context Protocol)** 与 **Skills** 哪个更好的讨论。原文作者倾向于支持 MCP，但评论中出现了大量不同的声音。

---

## 主要观点

### 1. 支持 CLI + Skills 的理由

- **更喜欢已有的 CLI 工具**：许多开发者更倾向于使用已有的 CLI 工具，而不是安装新的 MCP 服务器
- **更低的 token 开销**：CLI + Skills 组合在 token 消耗上比 MCP 更高效
- **更好的可组合性**：CLI 工具可以轻松组合使用，而 MCP 的工具调用无法直接组合
- **本地执行更安全**：可以更好地控制密钥和秘密，不会暴露在上下文窗口中

> "I don't want APIs, I want agents to use the same CLI tooling I already use that is locally available."

### 2. 支持 MCP 的理由

- **企业环境友好**：MCP 在企业环境中更有意义，特别是对于无法访问 CLI 的聊天机器人
- **更易用的认证**：MCP 支持 OAuth，可以更方便地处理认证
- **对非技术用户更友好**：用户只需安装 MCP 即可使用，无需配置 CLI 环境
- **远程 MCP 解决了分发和更新问题**

> "MCP is great for chatbots that don't have access to a full CLI."

### 3. 互补性观点

大多数评论者认为 **MCP 和 Skills 是互补关系**，而非非此即彼的选择：

- **MCP** 适用于：外部数据/工具的标准化接口（基础设施层）
- **Skills** 适用于：项目特定的行为上下文（编排层）

> "This isn't a zero-sum game or a choice of one over the other. They solve different layers of the developer experience."

### 4. 批评 MCP 的观点

- **上下文膨胀**：MCP 会向 agent 上下文添加大量信息
- **安全问题**：认证令牌随每次调用发送到上下文，存在泄露风险
- **复杂性**：相对于简单的 CLI 来说， MCP 增加了不必要的复杂性
- **不稳定**：部分 MCP 实现质量较差，如 Jira MCP

### 5. 批评 Skills 的观点

- **指令容易被忽略**：即使明确且简洁的指令也经常被模型忽视
- **需要执行环境**：如果 agent 无法执行 CLI，Skills 就无用武之地

---

## 总结

这场讨论反映了 AI agent 开发领域的两个主要方向：

1. **本地优先 (CLI + Skills)**：适合个人开发者，喜欢掌控和简化
2. **企业优先 (MCP)**：适合需要统一管理和安全控制的组织

两者各有优势，最佳实践可能是根据具体场景选择使用或结合使用。
