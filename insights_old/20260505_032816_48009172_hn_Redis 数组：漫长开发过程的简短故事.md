## 内容摘要

### 背景

这是 Redis创始人 antirez 发表的一篇关于 Redis 新增 Array（数组）数据结构的开发博客。他详细描述了如何使用 LLM（大型语言模型）辅助完成这个功能开发的全过程。

### 开发规模

- **开发时间**：4个月
- **代码行数**：约5000行（包含注释）
  - sparsearray.c: ~2000行（稀疏数组实现）
  - t_array.c: ~2000行（数组命令和上层实现）
  - AOF/RDB 持久化代码：~500行
- **额外依赖**：TRE 正则表达式库（用于 ARGREP 功能）

### 技术特点

1. **Array 数据结构**：支持稀疏表示的数组类型
2. **ARGREP 功能**：集成 TRE 库实现高效的数组正则表达式匹配
3. **Ring Buffer 支持**：高效支持环形缓冲区操作
4. **范围查询优化**：提供高效的范围查询能力

### LLM 使用经验

antirez 分享了他使用 LLM 辅助开发的经验：

1. **提效明显**：即使没有 LLM，4个月完成这个实现也是可能的，但有了 LLM 他能做更多工作
2. **模型迭代**：从 Opus 到 GPT 5.3 是巨大的飞跃，后续版本只是增量改进
3. **提示技巧**：使用细微的提示差异来引导模型聚焦特定细节或工程思维
4. **"Machiavelli" 方法**：遇到复杂问题时，让 AI 扮演名为 Machiavelli 的杰出计算机科学家，有时能得到意想不到的好点子
5. **代码质量**：某些实现由 AI 编写，质量令人印象深刻

### 社区讨论要点

1. **AI 辅助开发 vs "vibe coding"**
   - 有经验的开发者使用 AI 能放大其专业能力
   - 不能简单要求所有开发者转向 AI 编程工具

2. **开发流程**
   - 多人倡导"对抗性 AI 审查"方法：用一个 AI 开发和，另一个 AI 审查
   - 需要人机结合：必须阅读代码，AI 会遗漏细节

3. **效率提升**
   - 有开发者报告约 10x 的产出提升
   - 设计和文档部分提升尤为明显

4. **代码审查挑战**
   - 5000 行代码的 PR 对审查者来说工作量很大
   - Redis 的开发模式一贯是个人主导，社区反馈作为补充

### 未来展望

antirez 认为：
- 某些项目可以做到"不完全看代码"编程，只拥有概念、算法和思路
- 服务器软件的开发方式将会改变
- 程序员的角色将与 Linus 管理内核的方式类似

## 总结

本文记录了 Redis 创始人 antirez 如何在 LLM 辅助下，用4个月时间开发了新的 Array 数据类型。文章展示了当前 AI 辅助编程的最新实践，尽管 antirez 是经验丰富的顶级开发者，AI 仍然显著提升了他的开发效率。社区讨论反映了 AI 编程工具的现状：它们是有力的加速器，但需要熟练的工程师正确使用，人机协作仍是最佳实践。
