## 项目概述

作者 Nick Kossolapov 是一名拥有8年以上工作经验的软件工程师，他表示自己一直以来并不真正理解计算机的工作原理。为了学习计算机硬件基础，他决定通过编写一个 Game Boy 模拟器来深入理解。他首先完成了 "NAND to Tetris" 课程来打牢基础知识，然后编写了一个 CHIP-8 模拟器来练手，最终成功开发了名为 "Fame Boy" 的 Game Boy 模拟器，支持声音功能，可在桌面和 Web 端运行。

## 核心架构

### 前端与核心的接口设计

模拟器采用了清晰的接口设计，前端与核心之间仅通过两个数组和两个函数进行交互：

- **framebuffer**: 160x144 的像素数组，每个像素有四种色度（白、浅、暗、黑）
- **audiobuffer**: 采样率为 32768 Hz 的环形音频缓冲区，包含读写指针
- **stepEmulator()**: 执行一条 CPU 指令并返回消耗的周期数
- **getJoypadState(state)**: 前端向模拟器传递手柄状态的回调函数，通常每帧调用一次

### 模拟器组件结构

1. **CPU (Cpu/)**: 采用与真实硬件 Sharp LR35902 相同的设计，仅知道内存映射图，是代码中最具 F# 风格的模块
2. **内存 (Memory.fs)**: 存储 RAM，作为 CPU、IO 控制器和卡带之间的内存映射和总线，同时与 PPU 共享 VRAM 和 OAM RAM
3. **IO 控制器 (IoController.fs)**: 处理所有硬件寄存器，虽然真实硬件中不存在这个组件，但简化了接口并增加了安全性
4. **主步进器 (Emulator.fs)**: 将所有组件的步进函数组合起来，协调各组件的执行顺序

## 关键技术实现

### F# 在 CPU 建模中的应用

作者选择了 F# 编写模拟器，原因有两个：首先，F# 的类型系统非常适合建模 CPU 指令；其次，作者之前工作中经常使用 F#，很喜欢这门语言。

#### 领域建模

作者参考 Gekkio 的 Game Boy 完整技术参考文档对指令进行分组，使用 F# 的联合类型来建模。例如：

```fsharp
type LoadInstr =  
    | Load8Immediate of uint8
    | Load8Direct of Register
    | Load8Indirect
```

通过提取 "From" 和 "To" 这样的通用概念，将 CPU 指令从 512 个操作码减少到仅 58 条指令。F# 的类型系统确保了非法状态无法被表达，只有能编译的代码才是有效的领域模型。

#### 标志位设置

在测试驱动开发的帮助下，作者实现了一个优雅的 Flags 模块：

这个 16 行的 Flags 模块被作者认为是他写过的最喜欢的 F# 代码之一，函数可组合、可测试，且由于是内联函数不需要堆分配，性能提高了约 10%。

### PPU (图片处理单元) 实现

Game Boy 没有 GPU，只有 PPU（图片处理单元）。作者从读取显存中的瓦片和背景地图开始实现，这帮助他在整个开发过程中都能看到进展。虽然这种实现方式与真实硬件的 FIFO 队列方式有所不同（真实硬件逐个像素输出到屏幕，而模拟器在绘制周期开始时渲染整条扫描线），但对于大多数游戏来说已经足够。

### 手柄处理

手柄的实现遇到了一个问题��游戏通常会在短时间内多次读取手柄寄存器来获取所有按钮的状态，但如果将寄存器缓存起来不更新，就会导致部分按钮失效。最终解决方案是让 IO 控制器仅在 CPU 读取时才更新手柄寄存器。

### 声音实现 (APU)

声音是整个项目中最具挑战性的部分。作者花费了大量时间来理解不同的声音寄存器和通道。APU 的采样率选择为 32768 Hz，因为这个数字恰好是 CPU 频率 1048576 Hz 的约 1/128，方便整数对齐。最终采用音频驱动模式来同步模拟器，因为在所有设备上更可靠。

### Web 端移植 (Fable)

作者使用 Fable 将 F# 代码转译为 JavaScript。在调试过程中发现了一个关键问题：JavaScript 的位操作是 32 位的，导致 8 位无符号整数的值溢出，变成负数。修复方法是在所有需要 8 位值的地方进行截断处理。修复后 Web 打包体积仅约 100 KB。

## 性能优化

### 内存访问优化

作者起初使用领域驱动开发的方式来实现内存访问，每次读写都会创建 MemoryRegion 对象，导致每秒分配数百万个对象到堆中，性能严重下降。移除联合类型和映射函数后，FPS 直接翻倍。基准测试显示，85% 的性能提升来自移除分支和局部调用点的 JIT 优化，其余 15% 来自使用结构体联合类型。

### 基准测试结果

| 平台 | Flag | Roboto | Merken |
|------|------|--------|-------|
| Ryzen 9 7900 (桌面) | 1785 FPS | 1943 FPS | 1422 FPS |

禁用 PPU 可提高约 250 FPS，禁用 APU 可提高约 500 FPS。

## AI 的使用

作者在项目中主要将 AI 作为辅助工具使用，如代码审查、头脑风暴、解释技术文档等。只有两个地方使用了 AI 生成的代码：最后的性能优化（使部分基准测试的性能翻倍）以及解决 "timer winter" 问题（由于定时器只每指令触发一次而非每周期触发，导致模拟器运行慢了 2-3 倍）。

## 关键经验教训

1. **学习路径**: 从 NAND to Tetris 到 CHIP-8 模拟器再到 Game Boy 模拟器，循序渐进
2. **测试驱动开发**: 使用单元测试确保实现的正确性，大幅提高开发效率
3. **适当妥协**: 在性能关键区域可以适当放弃idiomatic F# 风格
4. **调试工具**: 使用 JProfiler 进行性能分析，使用浏览器的开发者工具进行 Web 调试
5. **发布模式**: Release 模式比 Debug 模式快得多（从约 45 FPS 提升到 1000 FPS）
