## 概述

本文讨论了在 Rust 中使用 `Box` 和其他字符串类型来节省内存的各种技术方案。

## 核心洞察

### 1. String 的内存布局

Rust 的 `String` 本质上是 `Vec<u8>` 加上 UTF-8 文本承诺，包含三个词（word）：长度、指针、容量。

### 2. 替代方案

| 类型 | 特点 | 适用场景 |
|------|------|----------|
| `Box<str>` | 无容量，只需 2 个词 | 不需要预分配空间的场景 |
| CompactString | 24 字节内联 SSO | 短字符串为主的场景 |
| ColdString | 8 字节内联 SSO | 极短字符串为主的场景 |
| Atoms | 使用 u32/u32 引用，字符串去重 | 重复字符串较多的场景 |
| Bump allocator | 高性能字符串构建 | 需要高效构建字符串的场景 |

### 3. ColdString 的实现原理

利用指针对齐产生的 spare bits：
- 堆分配使用 4 字节对齐，指针低两位为 0
- 通过旋转这些 bits 到第一个字节
- 实现单指针存储 + 短字符串优化

### 4. Option<T> 的内存开销

`Option<T>` 可能导致显著的内存开销：
- 某些类型的 `None` 使用 T 的无效值表示（如 `Option<OwnedFd>` 的 None 是 -1）
- 但如果 T 的所有位模式都是有效的，需要额外的 tag 字节
- 大 enum 变体可能导致显著的内存浪费
- 可用 `Box` 包裹较大的变体来优化

### 5. SSO（短字符串优化）的开销

虽然 SSO 看起来美好，但有开销：
- 每次访问都需要分支判断
- `String` 转 `&str` 不是零成本操作
- 零长度字符串场景下，普通的 `String` 已经跳过分配

### 6. 实用建议

- 使用 **dhat-rs** 工具分析内存使用情况
- 使用 **Clippy** 的 `large_enum_variant` lint 检测大 enum
- 使用 **heaptrack** 或 **perfetto** 进行堆分析
- 谨慎使用 `Option<T>`，考虑用 `Box` 包裹大型变体

## 技术细节

### 代码示例

```rust
// 使用 Box<str> 节省内存
struct Config {
    name: Box<str>,
    description: Box<str>,
}

// 使用 Box 优化 enum
enum Message {
    Small(String),      // 变体小时使用栈
    Large(Box<Data>),  // 变体大时使用堆
}
```

### 专用类型库

- `string_cache` crate: 提供 Atom 类型
- `compactstring` crate: 提供 CompactString
- `cold_seg` crate: 提供 ColdString

## 总结

内存优化需要权衡：
- 不是所有场景都适合用 `Box`
- 要考虑数据的实际分布（短字符串 vs 长字符串）
- 测量实际的内存使用情况，而不是猜测
- 架构改变（如 Atoms）可能带来显著收益，但也增加了复杂性
