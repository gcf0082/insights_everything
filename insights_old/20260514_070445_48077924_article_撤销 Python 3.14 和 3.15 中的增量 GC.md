# 洞察报告：Python 3.14 和 3.15 中回滚增量垃圾回收器

**洞察链接**：https://discuss.python.org/t/reverting-the-incremental-gc-in-python-3-14-and-3-15/107014

**基本信息**：
- **来源**：Python 官方讨论论坛 (discuss.python.org)
- **分类**：核心开发 (Core Development)
- **发布日期**：2026年4月16日
- **作者**：Hugo van Kemenade
- **标签**：release

---

## 摘要

Python 3.14 发布了新的增量垃圾回收器（Incremental GC），但收到了大量生产环境中内存压力显著增加的报告。经过评估，Python 核心团队决定在 Python 3.14 和 3.15 中回滚该特性，恢复使用 3.13 中的代际垃圾回收器（Generational GC）。

---

## 背景

Python 3.14 引入了增量垃圾回收器，旨在减少 GC 暂停时间的最大值。然而，该特性没有经过 PEP 流程评估，并且在 3.13 发布前曾被回滚过一次。

### 问题表现

- **内存压力显著增加**：生产环境报告指出内存使用大幅上升
- **Sphinx 性能下降**：此前发现的 Sphinx 减速问题与增量 GC 有关
- **性能权衡**：增量 GC 虽然减少了最大暂停时间，但带来了更高的内存占用和更长的运行时间

---

## 决策依据

### 为什么回滚？

1. **生产环境问题**：大量报告表明内存压力不可接受
2. **维护成本**：同时维护两个 GC 版本（3.13 和 3.15 各一个）会增加维护难度和风险
3. **未经 PEP 流程**：增量 GC 未经过标准的 PEP 评估流程
4. **历史经验**：3.13 发布前曾因类似问题回滚过一次

### 时间安排

- **Python 3.15**：处于 alpha 阶段，原计划 2026-05-05 发布第一个 beta，可能在接下来一周内发布额外的 alpha 9
- **Python 3.14**：原计划 2026-06-09 发布 3.14.5，将提前发布以包含回滚后的 GC

---

## 技术细节

### 性能对比（来自 Neil Schemenauer 的测试）

| 指标 | 增量 GC | 代际 GC | 变化 |
|------|---------|---------|------|
| 最大 GC 暂停 | 1.3ms | 26ms | 减少约 95% |
| 峰值内存 (RSS) | 2.7x | 1x | 增加约 170% |
| 运行时 | 更慢 | 基准 | 增加 10-25% |

**最坏情况下**：
- 内存使用可达 **5 倍**增长
- 垃圾回收时间更长

### 代码维护

- 增量 GC 约增加 1600 行代码
- 将 gc_inc.c 和 gc_gen.c 作为独立编译单元不会导致明显的性能回归
- 但长期维护两个版本会增加复杂性

---

## 社区讨论

### 是否应保留两个 GC 选项？

**反对意见**（Hugo van Kemenade）：
- 维护成本过高
- 3.14 中有两个 GC，但 3.13 和 3.15 各只有一个，风险更大

**支持意见**（Serhiy Storchaka、Antoine Pitrou）：
- 可以在 3.14 和 3.15 中都保留两个 GC
- 旧 GC 设为默认，新 GC 作为实验性选项

**结论**：维持单一 GC 是更安全保守的做法。

### 关于补丁发布

Tim Peters 认为，对于补丁发布而言，"首先不做损害"（first do no harm）是核心原则：
- 改动涉及所有程序的基础实现
- 代码微妙且复杂
- 性能影响因应用而异，难以预测

Gregory P. Smith 建议：是否可以为补丁发布提供发布候选版本（RC）以便更广泛测试？

---

## 未来计划

如果未来需要重新引入增量 GC，应通过正式的 PEP 流程进行更彻底的评估。

Java 采用了类似做法：保持旧 GC 为默认，新 GC 作为可选实验特性。
