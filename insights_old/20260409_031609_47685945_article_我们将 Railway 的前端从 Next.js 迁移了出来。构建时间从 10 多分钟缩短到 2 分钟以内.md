# 洞察报告：Railway 将前端从 Next.js 迁移至 TanStack Start + Vite

## 基本信息

- **洞察链接**：https://blog.railway.com/p/moving-railways-frontend-off-nextjs
- **来源**：Railway Blog
- **作者**：Victor Ramirez
- **发布日期**：2026年4月3日
- **主题**：前端框架迁移经验分享

## 摘要

Railway 将整个生产环境前端从 Next.js 迁移到 Vite + TanStack Router，整个迁移通过两个 PR 完成，实现了零停机迁移。本文总结了迁移的原因、技术选型、具体实施过程以及放弃的功能。

## 1. 迁移背景：Next.js 不再适合 Railway

Next.js 曾帮助 Railway 从零开始发展到服务数百万用户的生产级应用，但后来逐渐显现出以下问题：

### 1.1 构建时间过长
- 前端构建时间超过 10 分钟
- 其中 6 分钟是 Next.js 本身耗时
- 约一半时间卡在"finalizing page optimization"阶段
- 对于每天多次发布的团队来说，这严重影响了迭代效率

### 1.2 产品特性不匹配
- Railway 应用主要是客户端应用
- Dashboard 是富状态界面
- Canvas 是实时应用
- 大量使用 Websocket
- Next.js 的服务端优先特性对 Railway 来说没有实际价值

### 1.3 路由和布局问题
- 仍在使用 Pages Router
- 共享布局只能通过变通方案实现
- App Router 虽然能解决部分问题，但它偏向服务端渲染模式，与 Railway 的客户端驱动架构不符

## 2. 技术选型：为什么选择 TanStack Start + Vite

### 2.1 选择理由
- **类型安全的路由**：路由参数和搜索参数自动推断，整个路由树支持自动完成，路由从文件系统自动生成
- **原生布局支持**：无路径布局路由替代了之前的变通方案，更易于组合和预测
- **极快的开发循环**：即时 HMR，启动时间几乎为零，修改代码到看到结果的时间几乎可以忽略
- **按需 SSR**：仅在营销页面、更新日志、招聘页等场景使用 SSR，其他地方全部使用纯客户端渲染
- **明确的模型**：TanStack 依赖较少的框架魔法，对底层工作原理有更多控制权

### 2.2 团队反馈
多个团队成员在假期期间试用 TanStack Start，反馈一致认为使用体验良好。对于 Railway Dashboard 这样的产品，这种开发愉悦感与任何基准测试同样重要。

## 3. 迁移过程：两个 PR，零停机

### 3.1 PR 1：移除 Next.js 依赖
- 替换 `next/image`、`next/head`、`next/router`
- 每个依赖都替换为原生浏览器 API 或框架无关的替代方案
- 此 PR 不改变框架本身，只是移除对 Next.js 的依赖，为 PR 2 做好准备

### 3.2 PR 2：框架替换
- 迁移 200+ 条路由
- 首先将页面文件中所有非路由相关内容提取为独立的 React 组件
- 然后从原始页面树生成所有路由
- 添加 Nitro 作为服务器层
- 用 Nitro 配置替换 `next.config.js`，整合 500+ 重定向、安全头和缓存规则
- 用浏览器原生替代方案替换 Node.js API（如 Buffer、url.parse 等），代码反而更简洁

### 3.3 部署结果
- 在一个周日凌晨合并
- 团队立即进行内测，Discord 上建立实时问题处理房间
- 当天修复了一系列问题
- 零停机完成迁移

## 4. 放弃的功能

迁移并非没有代价：

- **内置图片优化**：用 `<img>` 标签 + Fastly 边缘图片优化替代 `next/image`
- **部分生态系统工具**：用内部小工具替代 `next-seo`、`next-sitemap` 等
- **成熟度**：TanStack Start 是新框架，可能存在更多边缘情况。团队认为可以接受，因为方向正确、维护者响应积极，且他们赞助了 Vite 和 TanStack

## 5. 部署架构

Railway 的生产前端采用与用户相同的方式运行：
- 每个 PR 都有预览部署
- 健康检查
- 零停机滚动更新

流量架构：
- Fastly 直接从边缘服务大部分流量
- 营销页面使用缓存
- 动态页面在需要时使用 ISR
- 前端服务器大部分处于空闲状态

Vite 的资源模型特别适合这种架构：每个模块有独立的内容哈希块，账单模块的变更只会使该块失效，回访用户只需下载几千字节而非几兆字节。

## 6. 为什么现在迁移

前端迭代速度的重要性前所未有：

- 构建时间从 10+ 分钟降至 2 分钟以内
- 开发服务器瞬时启动
- 路由边界处进行类型检查
- 布局无需变通方案即可组合

代码编写到用户面前展示之间的差距是瓶颈，框架切换、边缘缓存、资源模型都是为了缩小这一差距。Vite + TanStack 为前端变更几乎即时发布的世界奠定了基础，而这正是团队努力的方向。

## 总结

Railway 通过本次迁移实现了：
1. 构建时间减少 80% 以上
2. 200+ 路由零停机迁移
3. 更符合产品特性的前端架构
4. 更快的迭代速度

关键经验：选择框架时应优先考虑与自身产品特性的匹配度，而非盲目追求流行或全面的解决方案。
