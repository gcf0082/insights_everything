## 内容摘要

### 问题背景

在将数据输入加密算法（Sign、Encrypt、MAC或Hash）之前，如何对数据进行打包？这个问题数十年来一直没有得到充分的解决方案。需要解决两个重要问题：

1. **规范编码（Canonical Encoding）**：不同的编码不能解码为相同的内存数据（如比特币的交易延展性问题的根源）
2. **域分离（Domain Separation）**：更重要的，编码系统需要解决域分离问题

### 问题示例

假设系统有两种消息类型：
- `TreeRoot`：包含透明树的根
- `KeyRevoke`：表示密钥被吊销

如果这两种数据结构字段完全对齐（timestamp和hash字段顺序相同），攻击者可以将为`TreeRoot`生成的签名应用到`KeyRevoke`消息上，欺骗验证者签署 signer 从未意图签署的声明。

### 历史攻击案例

这种攻击并非理论上的，已在多个系统中成功实施：
- Bitcoin
- Ethereum DEXs
- TLS
- JWTs
- AWS

### 现有方案的局限性

目前的域分离方案都是临时性的：
- Solana 使用周围程序方法名的哈希
- Ethereum 使用 EIP-712 最佳实践
- TLS v1.3 使用"context strings"

### FOKS 的解决方案：Snowpack IDL

FOKS 提出的解决方案是将随机、不可变的域分隔符直接嵌入 IDL 中：

```foks
struct TreeRoot @0x92880d38b74de9fb {
   timestamp @0 : Uint;
   hash @1 : Blob;
}
```

关键特性：
1. **64位随机域分隔符**：每个结构都有唯一的域分隔符
2. **类型系统强制执行**：`Sign`和`Verify`方法要求对象实现`GetUniqueTypeID()`接口
3. **规范编码**：通过 Msgpack 编码，确保规范输出
4. **向前向后兼容**：支持添加和删除字段

### 工作原理

- 签名时：将域分隔符与对象序列化拼接，然后输入签名原语
- 验证时：同样重构拼接内容进行验证
- 域分隔符不出现在最终序列化中，节省字节

### 安全性保证

- 随机生成的域分隔符在全球范围内几乎不会冲突
- 类型系统确保没有域分隔符的结构无法被签名
- 编译器确保同一项目内所有域分隔符唯一

## 结论

域分离错误已经多次困扰真实系统。现有的缓解措施是临时性的：context strings、method-name hashes 和手写的前缀容易忘记且难以审计。

Snowpack 采用了不同的方法：随机、不可变的 64 位域分隔符直接存在于 IDL 中，类型系统确保你无法签署、加密或 MAC 缺乏域分隔符的对象。
