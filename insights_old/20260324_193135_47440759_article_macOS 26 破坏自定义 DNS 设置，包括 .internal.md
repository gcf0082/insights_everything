## 问题概述

macOS 26 中的 `/etc/resolver/` 域名 DNS 解析器机制对于不在 IANA 根区域中的任何 TLD 静默失效。`mDNSResponder` 会拦截自定义/私有 TLD 的查询，并将它们作为 mDNS（多播 DNS）处理，从不咨询解析器文件中指定的单播域名服务器。这破坏了之前在 macOS 25 上正常工作的整类本地开发和私有网络 DNS 工作流程。

---

## 问题背景

macOS 支持通过放置在 `/etc/resolver/` 中的文件进行按域名的 DNS 解析器配置。在 `/etc/resolver/internal` 文件中创建包含 `nameserver 127.0.0.1` 的配置，可以指示 DNS 堆栈将所有 `*.internal` 查询发送到 127.0.0.1 的本地域名服务器。

此机制记录在 `man 5 resolver` 中，自 macOS 10.6 以来一直可靠工作。它被广泛用于运行本地 DNS 服务器（dnsmasq、bind、unbound）的开发人员来解析私有域名后缀。

---

## 复现步骤

1. 安装 dnsmasq 并配置为应答自定义域
2. 启动 dnsmasq: `brew services start dnsmasq`
3. 验证 dnsmasq 直接应答: `dig @127.0.0.1 probe.example-private A +short`
4. 创建解析器文件: `sudo sh -c 'echo "nameserver 127.0.0.1" > /etc/resolver/example-private'`
5. 刷新 DNS 缓存并重启 mDNSResponder
6. 验证 `scutil --dns` 显示解析器已注册
7. 尝试通过系统解析器解析: `ping -c1 probe.example-private` → 失败

---

## 预期行为

`ping`、`curl` 和任何使用 `getaddrinfo()` 的应用程序应将 `probe.example-private` 解析为 `127.0.0.1`，这正是 macOS 25.x 上发生的情况。

---

## 实际行为

所有通过 `getaddrinfo()` 的解析（即每个真实应用程序）失败，显示"Unknown host"。没有 DNS 流量到达 dnsmasq。相反，`mDNSResponder` 拦截查询并立即返回带有异常大 TTL（~108002 秒）的缓存"No Such Record" mDNS 响应。

证据：
- `dns-sd -G v4 probe.example-private` 显示"No Such Record"
- `tcpdump -i lo0 -n port 53` 捕获期间没有数据包到达 dnsmasq

---

## 影响范围

### 受影响的 TLD

| TLD | 状态 | 备注 |
|-----|------|------|
| `.internal` | 失效 | IETF 草案专用 TLD |
| `.test` | 失效 | RFC 6761 §6.2 — 明确保留用于本地测试 |
| `.home.arpa` | 失效 | RFC 8375 — IANA 保留用于住宅私有网络 |
| `.lan` | 失效 | 广泛使用的约定 |
| 任意自定义 TLD | 失效 | 任何不在 IANA 根区域的 TLD |

**`.test` 特别严重**：RFC 6761 第 6.2 节明确保留 `.test` 用于本地/私有 DNS 测试。

### 受影响的用户群体

- 使用 dnsmasq + `/etc/resolver/` 的开发人员
- Docker Desktop 的容器名称解析
- Vagrant、Tailscale 等工具
- Kubernetes 本地开发工具（minikube、kind、k3d）

---

## 变通方案

唯一可靠的变通方案是手动将条目添加到 `/etc/hosts`，这完全绕过 mDNSResponder。但这对于动态用例不切实际。

---

## 技术分析

问题的根本原因是 `mDNSResponder` 拦截了自定义 TLD 的查询，并将其作为 mDNS 处理，而不是按照 `/etc/resolver` 配置将其发送到指定的单播域名服务器。这是一个回归问题：在 macOS 25.x 中，自定义 TLD 的查询会被正确路由到 `/etc/resolver` 中指定的本地 DNS 服务器。
