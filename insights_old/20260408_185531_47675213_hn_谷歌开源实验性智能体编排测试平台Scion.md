## 内容摘要

Google 开源了一个实验性的 Agent 编排测试平台 **Scion**，被称为"Agent 的超级监视器"，具有 Kubernetes 风格的设计理念，为 Agent 执行提供了底层基础架构。

## 主要讨论话题

### 1. 与 Gastown 的比较

- **Gastown** 是一个更成熟的 Agent 编排工具，具有"公式"(formulas)功能，被用户认为是"改变游戏规则"的存在
- Scion 目前更接近 Gastown 计划中的 "gascity"——由用户自己提供编排角色和定义
- 有开发者正在开发将 Gastown 移植到 Scion 上运行的版本
- Scion 主要提供**并发**能力，而 Gastown 也更侧重于并发而非循环；新项目 **Fabro** 试图同时实现循环和并发

### 2. 多模型协作趋势

- 用户倾向于不再依赖单一模型提供商，而是根据任务成本-质量-速度的平衡选择不同模型
- 常用组合：Gemini、K2.5、GLM-5 等
- 工作流应支持确定性路径，可使用 RAW API 调用、CLI 和 Agent

### 3. 架构设计

- Scion 使用 **容器** 作为隔离层，而非微虚拟机
- 提供多层状态和遥测：Hook 系统 → OpenTelemetry 标准化 → 云收集器
- 支持长时间运行的 Agent 和容器间通信
- 类似于 Kubernetes 的控制平面设计

### 4. 安全与隔离

- "约束优于隔离"的哲学
- 容器提供了边界但缺乏内部运行的可见性
- 有用户建议使用微虚拟机（如 kata containers）来获得更好的隔离

### 5. 实际应用案例

- 用户使用 Gastown 在 3-4 个 5 小时会话中完成了一个 Rust/Bevy 经济模拟游戏 85% 的功能代码
- 总成本约 50 美元的 VM 时间
- Agent 编排被比喻为"购买一台巨大的推土机"——熟练操作员可以移山，但仍需大量手动规划和协调工作

### 6. 批评与担忧

- 有人认为 Google 有太多不同的 Agent 工具（gemini-cli, antigravity, AI studio, Scion, Gemini app），缺乏统一性
- ADK（Agent Development Kit）被认为是更好、更持久的选择
- Google 的工具通常是为大型复杂组织设计的，可能不适合小型团队

### 7. 数据合规问题

- 讨论了 Agent 处理欧盟用户数据时的 GDPR 合规问题
- 开源了一个 PII 检测路由层，当发现个人数据时强制使用欧盟推理
