## 内容摘要

这是一个关于为 Claude Code 设计纯文本认知架构的Show HN项目，旨在解决AI编程助手在长时间项目中的记忆问题。

## 核心观点

### 1. 记忆组织的最佳实践
- 记忆应该具有"目的性"（purpose-driven），定义问题和答案的结构
- 记忆系统需要明确"记住什么"和"何时记住"的规则
- 项目文档（包含类、函数映射）比通用记忆更有效

### 2. Google Antigravity 的记忆结构参考
- 聊天完整文本日志 (Conversation)
- 聊天摘要
- 文件存储 (Brain)
- 隐式笔记 (Implicits)
- 注释 (Annotations)
- 知识项 (Knowledge items) - 可相互链接
- 自定义工作流
- 项目规则 (rules.md)
- 保存的重要文件 (Artifacts)
- 自定义技能 (Skills)

### 3. 持久化挑战
- 单会话运行良好，但进程终止或电脑关闭后，内存中的上下文无法恢复
- 需要持久化层来保存上下文

### 4. 置信度与衰减机制
- 存储的信息并非同等可靠，需要引入置信度分数和时间戳
- 对未强化的信息进行衰减
- 矛盾记录日志很有价值，让冲突的观察同时保留

### 5. 个性化记忆
- 记忆结构应根据用户类型（程序员、家庭主妇、游客等）定制
- 大公司只能构建"大桶"模式，而个性化、任务驱动的记忆更有价值
- 应将控制权交给用户

### 6. 与现有方案的对比
- 与 Superpowers' episodic-memory skill 和 Anthropic's Auto Dream 的区别在于 "/foresight"（预见）功能
- 与 CLAUDE.md 或 auto memory 相比，结构化程度不同
- 嵌套记忆系统更接近人类记忆模式：短期记忆、长期记忆、归档记忆

### 7. 替代方案
- 使用 copilot-instructions.md / CLAUDE.md + Readme.md + 聊天历史即可满足多数需求
- 记忆可能更适合个人助手用例，而非编程代理
- 规则 (Rules) 和技能 (Skills) 比记忆更明确、更少噪音、更易维护

### 8. 上下文管理的实际效果
- Codex 在会话持久性方面优于 Claude，不会遗忘 Claude 会丢弃的内容
- 极简上下文能获得更好结果，这可能是"代理群体"有效的原因
