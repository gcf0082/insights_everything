## 概述

本帖介绍了一个名为 "Get Shit Done" (GSD) 的开源项目，这是一个元提示（meta-prompting）、上下文工程和规范驱动的开发系统。该系统在 Hacker News 上获得了广泛关注，引发了关于 AI 辅助开发工作流的激烈讨论。

## 核心内容

### 1. GSD、Superpowers 与 Plan 模式的比较

许多开发者分享了使用 GSD、Superpowers 和 Claude Code 内置 Plan 模式的体验：

- **Plan 模式足够:** 部分用户认为 Claude Code 的 Plan 模式已经足够，更倾向于自行掌控 AI
- **Superpowers 的优势:** Superpowers 在头脑风暴和探索阶段表现出色，具有多层交叉检查和自我审查机制
- **GSD 的特点:** GSD 生成的代码更具前瞻性，考虑了项目整体和未来可能性，但运行时间较长

### 2. Token 消耗问题

讨论中一个重要观点是这些框架的 Token 消耗问题：

- 这些框架比直接使用 AI 多消耗 **10 倍** 的 Token
- 用户经常触及 Max plan 限制，但产出并未明显提升
- 有用户报告：为 500 行代码花费了约 $25

### 3. 规范驱动开发的争议

关于 Spec-Driven Development (SDD) 存在显著分歧：

**支持方观点：**
- 规范有助于提高代码对齐性和清晰度
- 好的规范可以提高测试的成功率
- 通过调整规范而非修改代码来解决问题

**反对方观点：**
- 自然语言规范存在歧义，无法精确描述系统行为
- 规范容易过时，难以与实际代码保持同步
- 自动化测试才是系统行为的真正验证

### 4. 框架与"缰绳"的重要性

一个被广泛认同的观点是：**模型只是引擎，框架（harness）是司机和底盘**

- 即使拥有最强大的模型，如果框架设计不当，也无法获得良好结果
- Claude Code、Copilot CLI、Codex CLI 等工具的框架设计差异显著影响实际表现

### 5. 开发者的反思

- 简单 Ralph 循环（实现者+审查者）重复运行直到通过审查和单元测试，可以完成 90% 的工作
- 最有效的方法是保持简单、清晰，只提供任务的基本信息
- AI 开发者需要注意"过度工程化"的倾向

## 主要结论

1. **没有银弹:** 复杂的 AI 开发框架并非总是优于简单的直接交互方式
2. **Token 成本:** 自动化程度越高，Token 消耗越大，需要权衡效率与成本
3. **框架重要性:** AI 模型的表现很大程度上取决于其运行框架的设计
4. **务实方法:** 许多资深开发者倾向于保持简单的工作流，而非使用复杂的元框架
5. **规范 vs 测试:** 关于规范驱动开发是否能取代测试驱动开发存在根本性分歧
