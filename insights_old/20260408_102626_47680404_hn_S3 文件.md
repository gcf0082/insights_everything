## 概述

AWS 推出了 S3 Files 功能，使 S3 存储桶可以像传统文件系统一样访问。该服务实际上是在 S3 之前放置了一个 EFS（AWS 托管的 NFS 服务）作为缓存层。

## 核心功能

1. **缓存机制**: 使用 EFS 作为缓存层，小文件（<128KB）存储在 EFS 中，大文件直接从 S3 流式传输
2. **同步机制**: 文件变更大约每 60 秒聚合提交回 S3 作为单个 PUT 操作
3. **双向同步**: 当其他应用直接修改 S3 中的对象时，S3 Files 会自动检测并反映到文件系统视图中

## 社区讨论焦点

### 1. 价格担忧

许多用户对定价表示担忧：
- 写入成本较高，对于写密集型应用可能是阻碍
- 128KB 的缓存阈值被认为过低
- 使用 S3 作为文件系统的初衷是降低成本，但这增加了额外费用

### 2. 性能考量

- NVMe 延迟在 10-100µs 范围（128KB 块）
- S3 延迟约为 100ms
- 缓存阈值应设置在数十到数百兆字节，而非千字节级别

### 3. 与现有工具对比

- **s3fs**: 纯 FUSE 实现，机器崩溃可能导致数据损坏
- **Mountpoint S3**: AWS 官方客户端
- **goofys**: Go 实现的类似工具
- **JuiceFS**: 使用共享数据库存储元数据
- **FiberFS**: 新兴的解决方案，内置缓存、CDN 兼容、JSON 元数据

### 4. 适用场景

- 读密集型工作流，小文件适合缓存
- 数据湖场景（如 DuckLake）
- 不适合的场景：
  - SQLite 数据库（SQLite 锁机制不支持 NFS）
  - 大文件的随机访问（成本高昂）

### 5. 技术限制

- S3 本质上是不可变的，修改文件需要重新上传整个文件
- 冲突解决：版本冲突的文件会移动到 `.s3files-lost+found` 目录
- 潜在的数据一致性问题

## 总结

S3 Files 是 AWS 为满足用户将 S3 作为文件系统使用的需求而推出的产品。虽然技术上解决了问题，但高昂的缓存成本和潜在的复杂性使其更适合特定用例，而非通用文件存储解决方案。对于真正需要文件系统语义的场景，传统文件系统（如 EBS、EFS）可能仍然是更好的选择。
