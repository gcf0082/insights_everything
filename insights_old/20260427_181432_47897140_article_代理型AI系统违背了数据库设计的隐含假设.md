# 洞察报告：数据库并非为AI代理系统设计

## 基本信息

- **洞察链接**: https://arpitbhayani.me/blogs/defensive-databases/
- **原文标题**: Databases Were Not Designed For This
- **原文作者**: Arpit Bhayani
- **原文发布时间**: 2025年
- **洞察时间**: 2026-04-27 18:14:32

## 核心洞察

传统数据库架构基于一系列隐式假设建立，这些假设在传统应用系统中成立，但在AI代理（Agentic AI）系统面前逐一失效。文章揭示了五个核心假设的破裂及其防御性应对策略。

### 假设一：确定性调用者

传统假设：查询由人类开发者编写、审查、测试后部署，行为可预测。

AI代理reality：代理通过推理生成查询，不同推理路径产生不同查询，可能生成从未见过的复杂JOIN语句。传统索引和连接池基于历史模式设计，无法应对这种动态性。

防御策略：在角色级别设置语句超时和事务超时。

### 假设二：写入是故意的

传统假设：所有写入操作都经过人类审查，是有意图的行为。

AI代理reality：代理可能基于错误理解写入、在循环中写入非预期数据、重试时重复写入、甚至在毫秒级写入数千行。真实失败案例：代理调用遗留API时连接池耗尽导致静默失败，代理将"无数据"理解为"无问题"，继续处理500条不完整数据。

防御策略：
1. 软删除：永远不允许代理硬删除，使用`deleted_at`和`deleted_by`列
2. 追加日志：金融记录等高风险操作用append-only日志表
3. 幂等键：每个操作携带幂等键，数据库通过唯一约束拒绝重复

### 假设三：连接是短暂的

传统假设：每个请求短暂使用一个连接，连接池基于预期并发峰值配置。

AI代理reality：
1. 代理在推理过程中保持连接开放（查询→LLM推理→再查询）
2. 一个高级任务派生的子代理会扇出，导致并发会话激增
3. 开发环境3个代理，生产环境30个代理，配置未更新

防御策略：
1. 为代理工作负载建立独立的连接池
2. 使用PgBouncer的事务级池化模式，用20个实际连接服务500个代理连接
3. 设置`pool_timeout=3`秒，快速失败而非无限排队

### 假设四：错误查询会大声失败

传统假设：慢查询或错误查询会超时，工程师可通过EXPLAIN ANALYZE发现问题。

AI代理reality：代理无法区分"数据不存在"和"查询错误"，会基于错误结果继续执行，写入基于坏数据的决策。这是一种新型失败——查询返回了行，但不是正确的行。

防御策略：在查询中注入代理上下文（agent_id, task_id, step），通过`pg_stat_statements`和慢查询日志追踪每个查询的责任方。

### 假设五：Schema是为开发者设计的

传统假设：列名、结构、nullable设计考虑开发者便利性。

AI代理reality：当代理通过Text-to-SQL或MCP服务器访问数据库时，schema成为与语言模型的合约。"usr_id"和"stat_cd"这样的列名对代理毫无意义。

防御策略：
1. 为代理设计可读懂的schema：清晰的列名、CHECK约束、NOT NULL
2. 建立代理专用视图层翻译遗留列名
3. 为列编写docstring式的COMMENT

### 核心模式： blast radius（爆炸半径）

代理获得的数据库访问权限决定了故障的影响范围。传统应用中，应用代码是guard rail；但代理是通用推理器，可生成未预期的查询。

防御策略：按代理类型分配最小权限角色，在数据库级别强制执行，而非依赖应用层。

## 关键模式总结

| 模式 | 目的 |
|------|------|
| 语句超时 | 防止推理循环 |
| 软删除 | 防止意外删除，支持审计 |
| 追加日志 | 完整审计轨迹，支持"撤销" |
| 幂等键 | 防止重复写入 |
| 独立连接池 | 隔离代理与人类流量 |
| PgBouncer | 事务级池化放大多连接能力 |
| 查询注释 | 可观测性，定位问题 |
| 代理可读schema | 减少错误查询 |
| 最小权限角色 | 限制爆炸半径 |

## 结论

AI代理系统与传统的根本区别在于：传统系统中人类始终在循环中某处；代理系统移除了这个保证。软删除、append-only日志、幂等键、最小权限角色、查询标记——这些模式从"最佳实践"变成"负载-bearing infrastructure"。数据库并未为这个调用者设计，但防御工具早已存在。
