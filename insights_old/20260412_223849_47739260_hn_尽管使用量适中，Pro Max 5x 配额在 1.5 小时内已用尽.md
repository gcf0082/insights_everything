## 问题概述

用户报告Claude Pro Max 5x订阅配额在仅1.5小时的适度使用后即耗尽，引发广泛关注和讨论。

## 主要问题

### 1. Claude Code 性能下降
- 用户反映Claude明显变差，即使指向具体文件进行inspect，仍会进入5分钟以上的长时间探索循环
- 模型有时会"thinking forever"但不执行任何实际任务
- 有用户怀疑Anthropic在负载高峰时使用量化模型

### 2. 缓存TTL变更
- 3月6日的变更将缓存TTL从1小时改为5分钟
- 缓存过期后，所有token需要重新构建，导致使用量急剧增加
- 这个变更被用户认为导致成本增加而非减少

### 3. 配额的"实验性"调整
- 有用户观察到问题似乎是临时性的，推测Anthropic可能在进行"实验"
- Plan模式引入后，模型收到相互矛盾的指令（plan模式要求探索 vs 用户明确要求聚焦）

### 4. 定价与价值不符
- $200/月的订阅在1小时内耗尽
- 有用户报告单个问题就消耗15%的配额

## 用户建议与解决方案

### 1. 切换到其他工具
- 部分用户转向Codex ($100计划) 或Cursor
- Codex在配额和准确性方面表现更好，但"人格"令人反感
- 建议尝试$20计划对比Codex和Cursor

### 2. 优化使用技巧
- 开启MAX_THINKING_TOKENS可节省token（但可能导致性能下降）
- 保持活动会话活跃，避免缓存过期
- 200k token后尽快压缩上下文
- 激进地修剪上下文和明确任务范围
- 建议使用更便宜的模型如Haiku处理简单任务

### 3. 本地LLM作为替代
- 有用户开源了token高效的编码代理dirac
- 目标是通过更短的上下文实现超过50%的成本节省

## 社区观点

### 担忧
- 担心AI编程工具正在经历"enshittification"（质量恶化）
- 担心配额会继续收紧，价格会上涨
- 认为" subsidized GenAI compute"的黄金时代可能即将结束

### 不同体验
- 欧洲用户报告较少遇到配额问题
- 部分用户（$100计划）每天使用2-4小时，从未超过20%配额
- 存在明显的区域性和时段性差异

### Anthropic的回应
- GitHub issue被"Closed as not planned"
- 用户认为回复看起来像是AI生成的，没有实际帮助
- 有用户将Anthropic的情况比作Twitter早期的fail-whale问题

## 关键结论

1. **核心问题**: Claude Code的配额消耗异常快速，与3月份的缓存TTL变更直接相关
2. **用户情绪**: 普遍不满，认为Anthropic正在重蹈Google Gemini的覆辙
3. **替代方案**: Codex和Cursor成为热门替代选择
4. **长期趋势**: 社区担忧AI编程工具将逐步涨价并收紧配额
