## 文章概述

本文介绍了 Stripe 如何使用 rubyfmt 对其超过 2500 万行 Ruby 代码进行统一的代码格式化。Stripe 选择在一个周末完成整个代码库的格式化，以避免合并冲突。尽管测试套件给了很高的信心，但面对如此大的差异，GitHub 甚至无法渲染。

## 核心洞察

### 1. 代码格式化策略

**一次性格式化 vs 增量格式化**

- Stripe 选择了"大爆炸"方式，在一个周末完成整个代码库的格式化
- 有开发者建议采用增量方式：只格式化不在任何 open PR 中被修改的文件
- 第一次运行格式化 95% 的文件，然后每天运行，约两周后达到 99.5%
- 使用 `git blame.ignoreRevsFile` 可以保持历史记录的有用性

### 2. Dart 格式化器的安全检查机制

Dart 格式化器有一个内部 sanity check：
- 并行遍历格式化和未格式化的字符串，跳过所有空白字符
- 如果任何非空白字符不匹配，立即中止
- 这确保格式化器唯一改变的是空白字符
- 这个检查在奇怪 bug 出现时挽救过作者

### 3. 关于代码规模

- Stripe 目前的代码库已增长到 4200 万行
- 这是一个令人震惊的规模，但大部分代码并不直接处理金融交易
- 使用 Sorbet（Stripe 的 Ruby 类型检查器）会增加代码行数
- 代码库增长是指数级的

### 4. 关于 Ruby 用于金融系统

- Stripe 将信用卡支付迁移到了 Java
- 非卡支付（如 ACH、支票、各种钱包）仍在使用 Ruby 处理
- PCI 相关代码存在于单独锁定保护的 repo 中，使用 Go 和 Ruby 混合
- 一旦建立了账户余额和账本的基础设施，处理支付并不困难

### 5. 关于 AI 生成代码

- LLM 倾向于生成"企业级 fizzbuzz"风格的代码
- 喜欢定义类和导入，重新发明轮子
- 一个有用的常量可能在 15 行后用不同的名字重新定义
- 生成的代码看起来令人印象深刻，但实际上是垃圾代码
- AI 是在糟糕的例子上训练的，因为人类自己也不知道自己在做什么

### 6. 关于格式化争议

- gofmt 的风格是"不是每个人的最爱"（Rob Pike 提到）
- 许多冲突无法解决，因为每个人都同意丑陋/不必要的 diff 的痛苦大于轻微格式化分歧的痛苦
- 这种被动抵抗问题正是技术领域的问题所在

## 技术要点

1. **格式化工具应该只改变空白字符**：这是安全运行的基础
2. **选择合适的时间窗口**：周末或低峰期可以避免合并冲突
3. **使用 ratcheting 机制**：可以在团队不知情的情况下偷偷引入
4. **准备 ignoreRevsFile**：使历史记录保持有用
5. **代码规模管理**：现在是时候考虑代码格式化了，否则随着规模增长只会更难
