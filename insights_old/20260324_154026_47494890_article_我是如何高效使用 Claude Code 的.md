## 文章总结

Neil Kakkar 分享了他在使用 Claude Code (原 Claude Code) 进行高效开发方面的实践经验。以下是核心内容：

### 1. 自动化繁琐工作

- 以前手动创建 PR：暂存更改、编写提交信息、创建 PR 描述、推送、创建 PR
- 编写了第一个 Claude Code skill：`/git-pr`
- PR 描述比手动写的更详细，因为可以读取完整 diff 并总结更改
- 真正的收获是减少了心智负担：不再需要在写代码和描述代码之间切换上下文

### 2. 消除等待时间

- 原来服务器构建需要约1分钟，时间不长不短足以打断专注
- 切换到 SWC 构建后，服务器重启时间降到 1 秒以下
- 这是一个看似很小但意义重大的改变：亚秒级重启意味着永远不会离开心流状态

### 3. 让 Claude 看到实际效果

- 以前每个 UI 更改都需要人工检查：本地预览、肉眼检查、决定是否符合预期
- 使用 Claude Code 的预览功能，让 Agent 能够设置预览、保持会话数据、查看 UI 实际外观
- 工作流程：将"完成"的定义改为 Agent 必须自己验证 UI
- 这样 Agent 可以捕获自己的错误，大大延长了自主运行时间

### 4. 并行处理一切

- 原来只能同时舒适地处理一个任务
- 多个 worktree 会遇到端口冲突问题（前端和后端各需不同端口）
- 建立了系统：创建 worktree 时为每个服务器分配唯一端口范围
- 从同时处理 2 个分支提升到可以同时运行 5 个 worktree
- 工作方式：同时在不同的 worktree 上启动多个 Agent，分别构建不同功能

### 核心观点

> **这是基础设施的问题，而不是 AI 的问题**

- 角色转变：从独自解决复杂问题的开发者，变成构建让 Agent 高效工作的基础设施的管理者
- 最有价值的工作不是写功能，而是构建让提交从涓涓细流变成洪水般的基础设施

### 消除摩擦的循环

1. `/git-pr` 消除了**格式化的摩擦**——将代码更改变成可展示的 PR
2. SWC 消除了**等待的摩擦**——修改和看到结果之间的停滞时间
3. 预览功能消除了**验证更改的摩擦**——快速查看正在发生什么
4. worktree 系统消除了**上下文切换的摩擦**——同时处理多个工作流而不会冲突

每消除一种摩擦，下一种就变得可见。这是经典的约束理论——修复一个，系统立即向你展示下一个。

## 关键洞见

1. **AI 时代的开发者角色**：从"实现者"转变为"管理者"，管理一群 Agent 完成实现工作
2. **基础设施决定效率**： plumber 工作（即基础设施）决定了是在心流中还是在与环境搏斗
3. **反馈循环至关重要**：当循环足够紧密时，工程变成了提高速度的娱乐游戏
4. **并行能力的价值**：消除并行障碍后，可以同时运行多个 Agent，大幅提升产出
5. **自动化 grunt work**：管理者应该自动化团队的所有繁琐工作
