## 项目概述

Zerobox 是一个跨平台、单二进制文件的进程沙箱化 CLI 工具，使用 Rust 编写。它使用了 OpenAI Codex 仓库中的沙箱化 crate，并增加了额外功能，如密钥注入、SDK 等。

### 核心特性

1. **默认拒绝策略**: 与 Deno 相同的沙箱策略，默认只允许读取文件，所有写入和网络 I/O 被阻止
2. **无 VM/Docker**: 不需要虚拟机、Docker 或远程服务器
3. **MITM 代理**: 用于阻止网络调用和在网络层面注入凭证
4. **密钥注入**: 通过占位符模式注入 API 密钥，安全的密钥不会暴露给沙箱内的进程
5. **跨平台**: 在 Linux 上使用 BubbleWrap，macOS 上使用 Seatbelt

### 使用示例

```bash
# 阻止读取 /etc
zerobox --deny-read=/etc -- cat /etc/passwd

# 密钥注入
zerobox --secret OPENAI_API_KEY=$OPENAI_API_KEY --secret-host OPENAI_API_KEY=api.openai.com -- bun agent.ts
```

## 社区反馈分析

### 积极评价

1. **Simonw (高票评论)**: CLI 设计优秀，特别是密钥/网络代理模式。称赞基于 OpenAI Codex 实现，密钥注入功能是"真正很棒的想法"。

2. **volume_tech**: 认为通过 MITM 代理进行凭证注入是最有趣的部分。标准方法是环境变量，但代理层可以在网络调用时替换凭证，这是运行不受信任的代理代码时的有用方案。

3. **gbibas**: 在生产环境运行 AI 编码代理时，最可怕的时刻是需要 API 访问但无法信任代理会如何处理凭证。Zerobox 的方向是正确的。

4. **smallerfish**: 建议参考其 bubblewrap-tui 项目，使用 DNS 代理处理不尊重 HTTPS_PROXY 环境变量的进程。

### 主要关切

1. **文档和可信度 (Simonw)**:
   > "这类项目面临的最大挑战是建立可信度。底层框架很容易被错误配置，需要看到架构设计完善且经过广泛测试的可靠证据。"

2. **macOS 限制**:
   - 作者承认在 macOS 上，代理是"尽力而为"的模式，不尊重 HTTPS_PROXY 的程序可以直接连接
   - 默认行为（无网络）在内核级别完全强制执行
   - 域名过滤依赖于程序是否遵守代理环境变量

3. **安全考虑**:
   - **lights0123**: Linux 默认允许所有用户读取运行进程的 CLI 参数，任何其他进程都可以读取密钥
   - **jbverschoor**: 采用黑名单模式很难正确配置

4. **技术债务**:
   - **EGreg**: 指出替代方案是能力模型系统，而非传统的 OS 级沙箱 + 代理方式

### 功能建议

1. **交互式配置文件构建** (Simonw):
   - 建议首次运行时通过交互式提示让用户批准文件访问
   - 类似于 `--build-profile claude-profile.txt -- claude`

2. **预定义配置文件**: 作者考虑添加预定义配置文件（如 Claude 配置，默认阻止所有 .env 读取）

3. **审计日志**: 多个用户请求更完善的审计日志功能，支持 JSONL/OpenTelemetry 集成

4. **Docker 集成**: 用户希望在 Docker 容器内使用 Zerobox

## 技术细节

### 性能

- Apple M5 平均开销约 10ms
- 已测试用于 Claude 包装，无延迟问题

### 网络拦截

- Linux: 通过网络命名空间强制，所有流量通过代理
- macOS: 依赖代理环境变量，不尊重代理的程序可绕过

### 密钥注入机制

1. 首次运行时在 `~/.zerobox/cert` 创建证书
2. 使用该证书进行 MITM 拦截
3. 注入 HTTP_PROXY/HTTPS_PROXY 等环境变量

## 总结

Zerobox 代表了 AI 代理沙箱化的重要方向，特别是在凭证管理方面。其 MITM 代理模式的密钥注入是核心创新，解决了 AI 代理安全运行的关键痛点。主要挑战在于需要更完善的文档、建立可信度，以及在 macOS 上强制网络过滤的能力限制。社区对配置文件系统和审计日志的强烈需求表明，生产级使用需要更精细的权限管理和监控能力。
