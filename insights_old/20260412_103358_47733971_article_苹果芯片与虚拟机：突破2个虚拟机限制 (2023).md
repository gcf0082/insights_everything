# 洞察报告：Apple Silicon 虚拟机突破 2 VM 限制

## 基本信息

- **洞察链接**: https://khronokernel.com/macos/2023/08/08/AS-VM.html
- **发布日期**: 2023年8月8日
- **作者**: Mykola Grymalyuk (khronokernel)
- **主题**: Apple Silicon 虚拟机限制突破方法

## 摘要

本文介绍了如何在 Apple Silicon Mac 上突破 macOS 虚拟机的 2 个 VM 限制。作者通过深入研究 XNU 内核，找到了 Apple 隐藏的 VM 配额机制，并成功实现了同时运行超过 2 个 macOS 虚拟机。

## 核心内容

### 问题背景

Apple Silicon 主机最多只能同时运行 2 个 macOS 客户虚拟机，超过限制时会收到以下错误：

> "The number of virtual machines exceeds the limit. The maximum supported number of active virtual machines has been reached."

这一限制来源于 macOS 软件许可协议（SLA）第 2.B.iii 条，规定个人非商业用途或软件开发最多只能运行 2 个额外的 macOS 虚拟机副本。

### 技术实现

#### 1. 定位限制机制

作者通过逆向工程发现：

- 限制并非存在于用户空间的 `Virtualization.framework`
- 实际限制嵌入在 XNU 内核的封闭源代码部分
- 关键变量为 `hv_apple_isa_vm_quota`
- 核心函数为 `hv_vm_destroy_0()` 和 `hv_trap_vm_create()`

#### 2. 突破方法

需要使用两个启动参数：

- `hypervisor=0x1`：作为门控检查
- `hv_apple_isa_vm_quota=0xFF`：将 VM 配额从默认的 2 个提升至 255 个

#### 3. 实现步骤

1. **获取 Kernel Debug Kit (KDK)**
   - 从 Apple Developer Portal 下载与主机系统版本匹配的 KDK

2. **构建开发内核集合**
   - 使用 `kmutil create` 命令创建自定义内核集合
   - 示例命令：
   ```bash
   sudo kmutil create \
     --arch arm64e \
     --no-authorization \
     --variant-suffix development \
     --new boot \
     --boot-path VirtualMachine.kc \
     --kernel /Library/Developer/KDKs/.../kernel.development.t6020 \
     ...
   ```

3. **配置启动**
   - 在恢复模式（Recovery）中执行：
   ```bash
   csrutil disable
   bputil --disable-boot-args-restriction
   kmutil configure-boot --volume /Volumes/Macintosh\ HD --custom-boot-object ...
   nvram 40A0DDD2-77F8-4392-B4A3-1E7304206516:boot-args='kcsuffix=development hypervisor=0x1 hv_apple_isa_vm_quota=0xFF'
   ```

4. **验证配置**
   - 使用 `sysctl kern.osbuildconfig` 和 `nvram boot-args` 验证

### 实验结果

作者在 M2 Pro MacBook Pro 上成功同时运行了 9 个 macOS 虚拟机。

### 注意事项

- 使用自定义内核集合后，系统更新会失败，需要在更新前恢复默认内核集合
- 恢复方法：在恢复模式中使用 `bputil` 重置启动策略

## 技术亮点

1. **逆向工程方法**：通过比较 Intel 和 Apple Silicon 内核的函数和字符串差异来定位限制代码
2. **巧妙绕过**：利用开发内核绕过 `AppleInternal` 检查
3. **社区贡献**：作者感谢 CoreOS Virtualization 团队提供了未文档化的选项

## 结论

这篇博客展示了 Apple Silicon macOS 虚拟化内部工作机制，证明了通过技术手段可以突破官方限制。同时也提醒用户，这是不受支持的使用场景，可能会带来兼容性风险。
