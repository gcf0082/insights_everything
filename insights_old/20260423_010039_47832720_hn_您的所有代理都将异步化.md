## 文章摘要

本文作者 Zak Knill（Ably员工）探讨了AI代理从同步交互向异步运行模式的转型问题。他指出，当前主流的聊天式AI应用基于HTTP+ SSE流式响应构建，但当代理需要异步运行时，这种传输模式无法正常工作。文章分析了行业现状和四种HTTP无法处理的场景，并提出需要将"持久状态"与"持久传输"分离的解决方案。

---

## 核心问题��传输不匹配

### 当前架构的局限性

传统AI聊天应用采用以下架构：
```
HTTP请求 → 提示词 → LLM推理 → SSE流式返回token
```

这种模式的核心问题在于：
- **LLM响应在HTTP连接断开时丢失**（如页面刷新）
- 代理的工作生命周期与单个HTTP连接的生命周期耦合
- 无法支持异步、长生命周期、跨设备的代理任务

### HTTP无法处理的四种场景

1. **代理生命周期超过调用者**：定时任务或长时间运行的代理完成后，原调用者可能已断开连接
2. **代理主动推送**：代理需要主动向用户推送结果（如完成夜间任务、需人工审批）
3. **调用者切换设备**：用户可能在不同设备上恢复任务
4. **多用户共享会话**：团队协作时，代理需同时向多人推送更新

## 行业现有解决方案

### Anthropic的回应
- **Channels**：基于MCP，允许从外部聊天系统异步推送消息
- **`/loop`和`/schedule`**：定时和循环任务
- **Routines**：后台运行的代理能力
- **Remote Control**：跨设备恢复会话

### 其他厂商
- **ChatGPT**：定时任务触发异步代理
- **Cursor**：云端后台代理
- **Cloudflare**：Sessions API + Email for Agents

### 现有方案的问题

这些方案主要解决的是**持久状态**问题（会话存储），而非**持久传输**问题。获取响应仍依赖轮询或HTTP请求，无法实现真正的异步推送。

## 评论核心观点

### 1. 持久化上下文管理
**edg5000** 指出问题核心在于行业对"将消息拼接为会话流"的执念。代理应能自主决定将哪些消息拉入上下文或从中移除，实现"永久代理"的上下文管理。

**vanviegen** 开发了 [maca](https://github.com/vanviegen/maca) 尝试这一方向，但发现模型未针对非标准代理方式训练，存在"对抗本能"的问题。

### 2. 上下文容量与缓存的权衡
**alehlopeh** 和 **edg5000** 讨论了动态上下文组成会破坏KV缓存的问题。虽然平均上下文需求可能更小，但缓存优势丧失。

### 3. 消息系统类比
**ljm** 提议借鉴Smalltalk的消息传递和Erlang的Actor模型来设计AI代理架构。

### 4. 推送 vs 拉取架构
**\_pdp\_** 通过实验发现推送模式效果不佳，新插入的消息对代理影响有限。更好的架构是**拉取模式**：代理拥有查询待处理消息的工具，所需通信内容直接出现在上下文底部。

**sudosteph** 采用全局锁文件方案，每个"代理"是独立Linux用户，通过changelog文件交换状态。

### 5. 会话状态标准化
**tuo-lei** 指出真正的问题是各工具的会话状态完全不兼容：
- Claude Code使用JSONL
- Cursor使用SQLite + JSONL
- 无统一schema

应优先制定**共享会话格式**，而非再增加一层pub/sub。

### 6. 跨代理结果共享
**nexustoken** 发现真正的问题是重复工作（不同代理处理相同输入）。通过哈希规范化输入JSON，缓存命中率可达十几个百分点。

### 7. 异步vs同步的权衡
**sonink** 认为随着代理速度提升，同步监督可能更合理：
- 上下文保持连续
- 廉价token减少，用户更有动力"在场"监督
- 除非代理在长周期任务上更可靠，否则异步实用性有限

### 8. 行业洞察
**bozdemir** 认为问题本质是UX设计，而非技术问题。SSE + Last-Event-ID + 事件溯源可解决80%的需求，无需发明新原语。

### 9. WebSocket方案
**sasipi247** 提到OpenAI Responses API的WebSocket模式可实现良好性能，结合NATS JetStream实现命令路由。

## 洞察结论

1. **AI代理正在从"同步聊天"向"异步长生命周期"转变**，这对底层传输架构提出了根本性挑战

2. **问题分解**：需要同时解决"持久状态"（会话存储）和"持久传输"（实时消息）两个问题

3. **核心需求**：
   - 代理能主动推送消息给用户或其他代理
   - 支持跨设备恢复会话
   - 支持多用户共享会话
   - 会话状态标准化

4. **技术方向**：
   - 拉取模式优于推送模式
   - 消息传递类架构（参考Smalltalk/Erlang）
   - WebSocket或pub/sub作为传输层
   - 共享会话格式标准化

5. **商业机会**：作者所在公司Ably正在构建基于"会话"概念的AI代理持久传输方案
