## 洞察摘要

ClojureScript 最新版本正式支持 async/await，这是该语言期盼已久的功能。讨论涵盖了多个热门话题：

### 1. Async/Await 支持的重要意义

- 此前添加 async/await 需要对 CLJS 编译器进行深度修改，shadow-cljs 作者 Theller 曾尝试并得出结论认为不可行
- 此次由 Borkdude 实现突破，他曾在 Clojurians Slack 表示不相信添加支持不切实际，最终付诸实现
- 这对于 JavaScript 互操作至关重要，无需引入额外库即可使用原生 async 函数

### 2. Clojure 在 AI 编程时代的复兴

- **REPL 驱动开发**: 与 AI 代理配合良好，代理可以与 REPL 交互而不必每次都承受启动开销
- **语言简洁性**: 没有类型检查的负担、很少无效语法错误和保留关键字
- **数据导向**: 默认支持不可变和持久化数据结构
- **向后兼容**: 10年前写的 Clojure 代码至今仍可正常运行
- **文档驱动的类型**: 可使用 Malli 进行运行时类型检查，为 LLM 提供清晰的参数结构说明

### 3. Typed 语言 vs Dynamic 语言与 AI

- **强类型系统的优势**: 在重构时能帮助 AI 纠正幻觉错误，特别是大型无类型 Python 代码库
- **Clojure 的应对方案**: 使用 Malli 或 Spec 进行类型检查/契约验证
- **LLM 在 Clojure 中的表现**: 只需解决"括号闭合"问题，配合良好的文档字符串和 Schema 定义，LLM 表现良好

### 4. Clojure 近期受关注的原因

- **Clojure 纪录片**: https://clojure.org/about/documentary 激发了人们对 Clojure 的重新兴趣
- **HTMX 的兴起**: 服务端渲染+HTMX 被认为是比复杂 SPA 更好的前端方案

### 5. core.async 与 async/await 的对比

- **core.async 的问题**:
  - 增大 JS 产物包体积
  - 没有固有的错误处理模型
  - 转换为状态机代码难以阅读/调试
  - go 宏鼓励过大的函数
- **async/await 的优势**: 原生支持，更小的包体积，与 JavaScript 生态更好集成

### 6. 其他前端替代方案讨论

- **Gleam + Lustre**: Elm 风格的 Web 框架
- **HTMX**: 服务端渲染的新范式
- **Mint**: 内置一切的语言
- **Scala.js**: 强大的类型系统

## 关键结论

1. ClojureScript 添加 async/await 是一项重大改进，简化了与 JavaScript 生态的互操作
2. 在 AI 辅助编程时代，Clojure 的 REPL 驱动开发模式和简洁性成为优势
3. 虽然动态语言在 AI 编码方面面临挑战，但通过 Malli 等工具可以在 Clojure 中实现类型安全
4. Clojure 纪录片的上映可能是该语言近期获得关注的原因之一
