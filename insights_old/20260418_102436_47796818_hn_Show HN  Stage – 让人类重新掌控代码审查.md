## 一、项目简介

Stage 是一个代码审查工具，旨在引导审查者逐步阅读 PR，而不是面对一个杂乱的 diff。它将 PR 的变更组织成小的、逻辑清晰的"章节"（chapters），并按最合理的顺序排列。每个章节会告诉审查者具体改了什么，以及需要重点检查什么。

**核心功能：**
- 自动分析 diff，将相关变更分组为"章节"
- 按逻辑顺序呈现章节
- 提供变更摘要和重点检查项
- 支持 GitHub 账号登录，与现有工作流无缝集成

**团队背景：** 两位工程师 Charles 和 Dean 因对 GitHub 代码审查 UI 不满意而构建了 Stage。他们观察到随着 AI 编码代理的兴起，PR 变得越来越大、越来越难理解，审查 backlog 迅速堆积。

## 二、主要讨论议题

### 1. 缺少上下文语境

有评论指出，Stage 的方案似乎只展示了"什么"变更，但审查者需要知道"为什么"变更、最初的需求是什么、以及变更与需求的关联。开发者建议可以整合 Linear、GitHub Issues 等工具获取上下文。

作者回应正在考虑在 PR 摘要中加入更多"为什么"的内容，并考虑引入类似 git-ai 的工具来捕获代理上下文。

### 2. AI 是否能解决问题

部分评论认为仅靠添加更多 AI 无法解决"不理解代码"的问题——如果缺乏理解是问题所在，将人与代码距离拉得更远只会恶化情况。

Stage 团队回应：他们不是在构建一个取代人类审查的 bot，而是用 AI 来引导人类注意力完成审查过程。

### 3. AI 生成的代码审查问题

有评论指出 AI 倾向于误导，例如 Claude 经常说服审查者相信问题并不严重。AI 生成的代码"工程化地隐藏"问题，需要大量精力去检测。担心 Stage 会给 AI 更多工具来操控叙事，使问题更加严重。

作者回应：只在大型功能开发时才会遇到代理误导的问题，但生成 PR 摘要的独立代理没有同样的动机去撒谎。

### 4. 既然 AI 能总结，为什么不能直接审查？

评论质疑：如果 AI 能解释变更并指出重点，为什么不能直接完成审查？担心用户的建议会变成待办清单，开发者只会看摘要而不真正审查代码。

作者认为软件设计始终有主观成分，需要权衡取舍，工程师们仍希望参与决策过程。

### 5. 章节 vs 提交

有评论质疑：按逻辑分组变更难道不是提交（commit）应该做的事吗？作者回应：在实践中很难创建良好的提交，因为编码时新想法不断出现或需要迭代不同设计。他们认为 Git 不是正确的抽象层级，决定在 PR 层面解决。

### 6. 定价与开源

有评论批评没有定价页面。作者回应正在开发中。关于是否开源，团队表示正在考虑。

### 7. 竞品与差异化

被提及的竞品包括：Plannotator（开源本地运行）、CodeRabbit、Greptile、Graphite（stacked PRs 工具）。

## 三、总结与评价

Stage 试图解决 AI 时代代码审查面临的核心挑战：PR 数量激增、代码规模变大、审查者难以建立心智模型。通过将变更组织成逻辑清晰的"章节"，帮助审查者更有序地理解代码变更。

**主要优点：**
- 创新的 UI 设计，将代码审查类比为"阅读书籍章节"
- 与 GitHub 现有工作流集成
- 解决大规模 PR 阅读困难的问题

**潜在问题：**
- 缺乏上下文/需求背景的整合
- 与提交规范的关系不够明确
- 如何处理 AI 可能带来的误导问题
- 商业化模式待验证

社区反馈整体积极，但也提出了关于上下文整合、信任问题、定价透明度等方面的质疑。
