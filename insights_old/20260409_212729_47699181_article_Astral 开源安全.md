# 洞察报告：Astral 开源安全实践

**洞察链接**：https://astral.sh/blog/open-source-security-at-astral

**发布日期**：2026年4月8日

**作者**：William Woodruff

**摘要**：Astral（ruff、uv、ty 的开发团队）分享其在开源安全方面的实践经验，涵盖 CI/CD 安全、仓库与组织安全、自动化、发布安全以及依赖安全管理等方面。

---

## 核心要点

### 1. CI/CD 安全

Astral 在 GitHub Actions 工作流中实施了多项安全措施：

- **禁用危险触发器**：全面禁止 `pull_request_target` 和 `workflow_run` 等高风险触发器，这些几乎无法安全使用。
- **操作 pinning**：要求所有 action 必须固定到具体提交而非标签或分支，使用 zizmor 的 `unpinned-uses` 和 `impostor-commit` 审计进行验证。
- **权限限制**：组织级别默认只读权限，每个工作流从 `permissions: {}` 开始，仅按需扩展。
- **Secret 隔离**：使用部署环境和环境特定的 secret，避免测试或 linting job 访问发布 secrets。

### 2. 仓库与组织安全

- **最小权限原则**：大多数成员只有代码仓库的读写权限，管理员数量最少。
- **强制 2FA**：要求所有成员使用不低于 TOTP 的强 2FA 方法。
- **分支保护**：main 分支禁止强制推送，必须通过 PR 合并。
- **标签保护**：release 标签仅在发布部署成功后创建，且不可修改或删除。
- **管理员限制**：禁止仓库管理员绕过所有保护措施。

### 3. 自动化（Automations）

- 对于无法在 GitHub Actions 中安全执行的操作（如在第三方 issue 上留评论），使用独立的 GitHub App（astral-sh-bot）来处理。
- GitHub App 虽不能消除敏感凭证，但可以避免模板注入等攻击。
- 建议使用 Gidgethub 框架开发 GitHub App。

### 4. 发布安全

- **可信发布（Trusted Publishing）**：消除对长期凭证的需求。
- **Sigstore 认证**：为二进制和 Docker 镜像生成可验证的认证。
- **不可变发布**：使用 GitHub 不可变发布功能防止事后篡改。
- **发布环境隔离**：发布流程在专用部署环境中运行，需要至少一名其他团队成员手动审批。
- **无缓存构建**：发布时不使用缓存以防止缓存投毒攻击。
- **独立安装程序校验**：standalone installer 内置校验和验证。

### 5. 依赖安全

- 使用 Dependabot 和 Renovate 保持依赖更新并提醒漏洞。
- **冷却期（Cooldown）**：新版本发布后不立即更新依赖，避免临时被篡改的包。
- **保守添加依赖**：尽量减少新依赖，审查依赖功能以禁用不需要的功能。
- **财务支持**：通过 OSS Fund 资助依赖项目。

---

## 总结建议

1. **认识 CI/CD 的局限**：某些操作无法在 GitHub Actions 中安全执行，建议使用 GitHub App 或放弃这些功能。
2. **消除长期凭证**：这是最常见的攻击途径，尽量使用可信发布等 OIDC 机制。
3. **强化发布流程**：使用部署环境、审批、标签规则和不可变发布。
4. **关注依赖健康**：保持对依赖树整体健康状况的了解。
