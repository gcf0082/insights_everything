## 核心观点

幂等性不仅仅是"记住key是否被见过"，而是**记住key代表什么意思**。真正的难点在于第二次请求不一定是第一次的简单重放。

---

## 详细洞察

### 1. 幂等性是关于效果的

操作是幂等的，如果执行一次或多次具有相同的预期效果。关键词是"效果"：

- HTTP方法级别语义（如PUT、DELETE）提供基础幂等保证
- 但业务层面仍可能产生重复副作用：审计记录重复、领域事件重复、邮件重复、 provider调用重复
- `POST` 通常不是幂等的，但可以通过服务器存储和强制正确行为来变为幂等
- 唯一性约束只能防止一类重复，但不能让客户端知道操作是否成功

### 2. 需要记住什么

对于 `POST /payments`，持久的幂等性记录需要回答三个问题：
1. 谁拥有这个key？
2. 第一个命令是什么意思？
3. 什么结果可以被重放？

幂等性表的关键字段：

| 字段 | 说明 |
|------|------|
| tenant_id | 作用域隔离，防止跨租户冲突 |
| operation_name | 防止跨操作复用key |
| request_hash | 服务器记忆的第一个命令，用于检测同key不同内容 |
| status | 状态机：IN_PROGRESS, COMPLETED, FAILED_RETRYABLE, UNKNOWN_REQUIRES_RECOVERY 等 |
| locked_until | 处理锁定时间，防止并发问题 |

### 3. 相同key，不同命令

这是幂等性层应该大声捕获的bug：

**场景**：
- 第一次请求：`amount: "10.00"`
- 第二次请求：`amount: "100.00"`（相同Idempotency-Key）

**问题**：返回原响应会隐藏严重客户端bug。客户端请求100欧却得到10欧的响应。

**推荐行为**：对有副作用的API，相同scoped key复用不同命令应返回 `409 Conflict`：

```json
{
  "errorCode": "IDEMPOTENCY_KEY_REUSED_WITH_DIFFERENT_REQUEST",
  "message": "This idempotency key was already used with a different request."
}
```

### 4. 哈希命令而非字节

原始字节比较对JSON API通常太严格：

```json
// 这两个应该等价
{ "amount": "10.00", "currency": "EUR" }
{ "currency": "EUR", "amount": "10.00" }
```

**推荐流程**：
1. 将请求解析为版本化的DTO
2. 规范化等价值：金额、枚举大小写、默认字段、时间戳精度
3. 排除传输层元数据
4. 包含路径参数和操作名
5. 如果header影响操作语义（如API版本），则包含
6. 排除 Authorization 和幂等性key本身
7. 规范序列化后哈希

### 5. 首次插入决定执行所有权

并发场景下，原子插入至关重要：

```sql
INSERT INTO idempotency_requests (...) 
VALUES (...) ON CONFLICT DO NOTHING;
```

**关键逻辑**：
- `rows_inserted == 1` → 本请求拥有执行权
- 否则加载现有记录，比较 request_hash
- 不同hash返回409，相同则根据状态处理

### 6. Provider超时是保证的终点

关键失败路径：
1. API接收请求，插入 IN_PROGRESS 记录
2. 创建本地支付记录
3. 调用下游provider
4. Provider成功，但API在记录结果前崩溃
5. 客户端重试

**状态机建议**：
```
RECEIVED → LOCAL_PAYMENT_CREATED → PROVIDER_REQUEST_SENT → PROVIDER_CONFIRMED → COMPLETED
                                              ↓
                                    UNKNOWN_REQUIRES_RECOVERY
```

**恢复策略**：
- 始终使用稳定的下游操作ID（如 `provider_payment_pay_789`）
- 恢复时查询provider（如果支持）
- 无法确认时标记为 UNKNOWN_REQUIRES_RECOVERY

### 7. 重放是契约而非便利

- 重放的响应应与原始响应完全一致
- 如果存储响应体，注意处理敏感数据（PII、签名URL、一次性令牌等）
- 考虑schema变化：版本2和版本3的响应结构可能不同
- 更好的方式：存储 `resource_type`, `resource_id`, `response_schema_version` + 完整响应体

### 8. 队列消费者面临同样问题

消息队列中的去重：

```sql
-- consumer_inbox
consumer_name, message_id, status, processed_at
UNIQUE(consumer_name, message_id)
```

**陷阱**：
- 在发送邮件前标记已处理然后崩溃 → 邮件永远不会发送
- 发送邮件后标记然后崩溃 → 可能重复发送

**解决方案**：在发送前先持久化副作用（插入唯一约束的行），然后由独立进程发送

### 9. 过期是API契约的一部分

幂等性记录通常不能永久存在：
- 定义明确的过期窗口（如24小时）作为产品决策
- 过期后可能创建新操作，但可能让排队的重试感到意外
- 完成记录可分离保留：元数据保留，响应体删除

**清理 IN_PROGRESS 的注意事项**：
- 不要因为"太旧"就删除，可能意味着卡住的worker或崩溃的进程
- 小批量删除，按 expires_at 分区

### 10. 失败重放是策略决策

| 失败类型 | 行为 |
|----------|------|
| 语法验证失败 | 不需要幂等性存储 |
| 业务拒绝（如余额不足） | 取决于业务规则：是否应该为同一key保留决策 |
| 认证失败 | 不应创建幂等性记录 |
| 授权失败 | 谨慎：重试必须解析到创建原始记录的同一主体 |
| 服务端错误（副作用前） | 通常可重试 |
| 服务端错误（副作用后） | 危险，需要恢复而非乐观重试 |

## 关键建议清单

- [x] 拒绝相同scoped key + 不同规范命令
- [x] 在scoped key上使用唯一约束或原子插入
- [x] 哈希验证后的命令，而非原始JSON
- [x] 将 IN_PROGRESS 视为API可见行为
- [x] 定义完成、可重试失败、重放失败和未知状态
- [x] 存储足够的响应数据以满足重放契约
- [x] 下游调用也要幂等，或有协调机制
- [x] 使用outbox/inbox模式处理事件和队列
- [x] 在持久副作用存在前不标记消息已处理
- [x] 将幂等性窗口定义为API契约的一部分
- [x] 敏感响应体单独保留策略
- [x] 测试并发重复、provider成功后超时、部分失败、过期、schema变更重放
- [x] 监控：不同body复用、stale IN_PROGRESS、过期重试、未知状态、重放率

## 结论

> "The second request is not a repeat until proven."

幂等性不是记住key，而是**精确记住第一个操作**：scoped操作、规范命令、执行状态、结果资源或响应、过期窗口，以及足够的失败状态来避免将不确定性转化为重复副作用。

服务器必须证明第二次请求属于哪种情况：重试、穿同一key的不同操作、竞速第一个请求、provider成功后进程失败、还是清理任务删除唯一记录后重试。

key不是保证。保证是服务器**精确记住第一个操作**以能够重放、拒绝不匹配、或恢复而非猜测。

## 适用场景

| 场景 | 推荐做法 |
|------|----------|
| 支付/金融交易 | 必须使用完整幂等性层 |
| 发送邮件/通知 | 需要持久化副作用 + 唯一约束 |
| 管理操作（重复无害） | 可简化处理 |
| 只读操作 | 通常不需要幂等key |
| 业务key唯一性可替代 | 如 `unique(account_id, merchant_reference)` |
