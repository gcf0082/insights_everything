# Google API Key 安全事件洞察报告

## 基本信息

| 项目 | 内容 |
|------|------|
| 洞察链接 | https://discuss.ai.google.dev/t/unexpected-54k-billing-spike-in-13-hours-firebase-browser-key-without-api-restrictions-used-for-gemini-requests/140262 |
| 事件类型 | API Key 安全漏洞 / 账单异常 |
| 影响范围 | Google Cloud API Keys (Gemini API) |
| 严重程度 | 高 |

## 事件概述

2026年2月，一个墨西哥的三人开发团队遭遇了严重的 API Key 安全事件，其 Google Cloud API Key 被盗用后在短短48小时内累积了高达 **$82,314.44** 的Gemini使用费用。该公司正常的月度支出约为 $180，这意味着遭受了约 **46,000%** 的费用激增。

攻击者主要使用了以下服务：
- Gemini 3 Pro Image
- Gemini 3 Pro Text

## 根本原因分析

### 1. API Key 自动继承新API权限

当 Google Cloud 项目启用了 Generative Language API（即 Gemini 的后端 API）后，该项目中的所有现有 API Key 会**自动获得**访问 Gemini 端点的权限，而无需Key持有者知情或同意。

### 2. 历史遗留的安全假设

Google 长期以来对 Maps 和 Firebase 的文档明确表示：API Key 不是"密钥"，而是用于识别开发者应用项目的标识符。基于这一指导方针，开发者被指示将 Key 直接嵌入前端代码中（如 HTML）。然而，Google 引入 Gemini 后，这些原本用于计费的标识符现在可以访问强大的 AI 能力。

### 3. Key 格式易于识别

Google Cloud API Key 以 "AIza" 开头,这使得攻击者可以轻松地从公开网站源代码、GitHub 仓库或日志中扫描和窃取这些 Key。

## 安全研究机构发现

安全研究人员 **Truffle Security** 扫描了数百万个网站，发现了 **2,863 个** 有效的 Google API Key——这些 Key 原本仅用于 Maps 等无害服务，现在同时可以认证到 Gemini API。

这意味着：
- 攻击者可以访问受害者上传到 Gemini 的文件
- 攻击者可以访问缓存数据
- 攻击者可以将 LLM 使用费用记入受害者账户

## 影响与风险

| 风险类型 | 描述 |
|----------|------|
| 财务风险 | 未授权的 API 调用可导致数千至数万美元的账单 |
| 数据风险 | 攻击者可以访问上传到 Gemini 的文件和缓存数据 |
| 隐私风险 | 敏感数据可能被未授权方访问 |

## 安全建议

### 个人开发者应采取的措施

1. **限制 API Key 权限**
   - 在 Google Cloud Console 中为每个 Key 添加 API 限制
   - 禁止旧 Key 访问 Gemini API
   - 启用应用程序限制（HTTP referrer、IP 地址等）

2. **设置预算警报**
   - 配置 Cloud Billing 预算警报（50%、90%、100%）
   - 注意：预算警报**不会**自动停止服务，仅会发送通知

3. **定期审计 Key**
   - 检查所有活跃的 API Key
   - 删除不再使用的 Key
   - 轮换所有可疑的 Key

4. **使用强认证**
   - 生产环境优先使用 OAuth 2.0 或 Workload Identity
   - 避免在客户端代码中存储长期有效的凭据

### Google 应采取的措施

1. **修改默认行为**
   - 新启用 API 时不应自动向现有 Key 授予权限
   - 要求用户明确确认后才授予新 API 访问权限

2. **加强 Key 保护**
   - 自动检测并禁用泄露到公开源代码中的 Key
   - 实施更严格的默认限制

3. **改进用户通知**
   - 当 Key 的权限范围发生变化时及时通知用户
   - 在用户启用新 API 时提供清晰的权限影响说明

## 结论

这一事件揭示了现代云服务中一个关键的安全隐患：当云平台添加新功能时，原有的凭据可能会意外获得更广泛的权限，而用户往往不知情。对于 AI 服务来说，这个问题尤其严重，因为这些服务的使用成本可以迅速累积。

开发者必须认识到：**API Key 不再是简单的"计费标识符"，而是需要像密码一样保护的敏感凭据**。云服务提供商也有责任确保凭据权限的扩展不会在用户不知情的情况下发生。
