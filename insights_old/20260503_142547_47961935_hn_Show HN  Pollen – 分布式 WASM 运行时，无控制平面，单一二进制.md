## 项目概述

Pollen 是一个用纯 Go 编写的自组织 Mesh 和 WASM 运行时。工作负载可以"播种"到集群中，并有机地扩展和跟随负载运行。没有中央协调器；决策是通过使用 gossiped CRDT 运行时状态作为真相来源进行确定性本地决策。所有节点拥有相同的世界视图，拥有相同的工作负载放置和路由逻辑。

### 核心特性

- **单二进制文件**: 安装在集群中每台机器上，它们会自动组织
- **拓扑派生**: 从 gossiped 状态确定性派生
- **工作负载管理**: 工作负载放置在有容量的节点上，副本向需求迁移，幸存节点从失败节点重新托管
- **安全**: 基于 ed25519 身份和签名属性，任何 TCP 或 UDP 服务都会获得 mTLS
- **连接**: 节点之间直接连接，如果不行则通过相互可访问的节点中继
- **极简运维**: 两三个命令即可建立集群

### 技术实现

- 使用 gRPC 控制 API 暴露在主机上的 socket
- 通过 `pln call seed_name function_name payload` 或传统 gRPC 客户端调用工作负载
- 使用 Extism PDK 编译为 WASM 的任何内容作为工作负载
- 当前仅支持无状态工作负载，计划引入共享状态和持久化

### 性能数据

- 在演示中，约 2500 rps 到达笔记本电脑，4k-5k 在 4-5 个全球节点生成
- 单次轻量级 WASM 函数调用可达到更高的吞吐量
- 内存是主要瓶颈（多跳场景），CPU 是单跳场景的瓶颈

## 社区讨论要点

### 1. 文档和清晰度问题

多个评论者指出项目难以理解：
- 技术术语过多（CRDT、gossip protocol）
- 缺乏实际用例说明
- "seed" 比喻不够直观

作者回应：会添加更多具体示例。

### 2. 潜在应用场景

**开发者设想的使用案例：**
- 替代 Tailscale 用于某些 homelab 应用
- 公司内利用闲置机器作为去中心化沙盒微服务集群
- P2P 计算按需平台
- 分布式 ML 任务分解
- 客户可以访问受控子集的基础设施（可撤销访问）

### 3. 与类似项目对比

- **Wetware**: 相似的 P2P WASM 理念但设计目标不同（多工具代理产品）
- **Lunatic**: 未使用
- **wasmCloud**: 未使用

### 4. 技术挑战

- 状态管理：正在研究如何引入收敛状态
- 网络分割：分区发生时节点在其结果分区内查看剩余对等点，没有分裂脑概念
- 性能优化：内存和 CPU 约束

有人提出愿意招聘 Pollen 类似的开发者全职工作，也有人希望交流技术细节。

## 总结

Pollen 是一个有潜力的早期项目，尝试解决现代分布式系统的运维复杂性。它采用去中心化、无控制平面的设计理念，使用 CRDT 和 gossip 协议实现一致性。主要挑战在于缺乏清晰的应用场景说明和状态管理能力。社区对其技术创新表示认可，但需要更好的文档和实际用例来吸引更广泛的用户。
